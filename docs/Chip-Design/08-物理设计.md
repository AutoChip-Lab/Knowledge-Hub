# 芯片设计完整流程 · 第 ⑧ 步:物理设计

> 「芯片设计完整流程」学习系列第八篇。
> 全流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产
```

**物理设计(俗称"后端")夹在「逻辑综合(⑦)」和「签核/流片(⑨⑩)」之间**:综合给出的门级网表只是"哪些门怎么连"的**逻辑连接关系**,还没有任何位置信息。物理设计就是把这些门**真正摆到硅片平面上**——决定每个单元的坐标、每条线的走向、电源怎么供、时钟怎么分发,最终产出可以送去制造的**版图(layout)**。这是设计**第一次变成"有物理形状的东西"**。

> 一句话定位:**综合给的是"逻辑接线图",物理设计把它变成"有坐标的真实版图"——每个门有位置,每条线有走向,延迟从此变成真账。**

---

## 1. 物理设计到底是什么

一句话:**把门级网表,在目标工艺的物理规则下,布局规划 → 摆放单元 → 建时钟树 → 布线 → 收敛时序/功耗/物理规则,最终产出可制造的版图(GDSII)。**

### 为什么这一步质变:延迟终于"真"了
综合时算的时序是**估计**——线还没布,线延迟靠模型猜。物理设计后,单元有了坐标、线有了实际长度和层,**寄生电阻电容(RC)是真的了**,延迟才接近真账。很多综合阶段"看起来收敛"的设计,布线后原形毕露——后端的核心工作就是**让真实物理下的时序/功耗/可布性同时成立**。

> 灵魂句:**前端的时序是纸上谈兵,后端才见真章——因为只有摆好、布通,线延迟才是真的。**

### 后端的多目标"同时收敛"
物理设计不是单目标,而是要**同时**满足:**时序**(跑得到频率)、**面积**(die 够小、省成本)、**可布性**(线布得通、不拥塞)、**功耗与电源完整性**(IR-drop、EM 不超标)、**物理规则**(DRC/LVS 必须干净)。任何一项崩了都不算收敛。

*在 TPU 上*:⑦ 给出 256×256 阵列等的门级网表;⑧ 要把巨大的阵列、24 MiB Unified Buffer 的 SRAM 宏、累加器等摆到 28nm、≤331mm² 的 die 上,规划电源、把 700MHz 时钟低偏差地分发到整块阵列、把密密麻麻的连线布通,最后产出版图。

---

## 2. 谁来做 + 需要什么输入

**谁来做**:**物理设计 / 后端工程师(PD / backend engineer)**,和综合、时序签核团队紧密协作。

**需要什么输入**:
- **门级网表(⑦)** + **约束 SDC**。
- **物理库与工艺文件**:标准单元的物理抽象(**LEF**)、时序库(.lib)、**工艺设计规则(tech LEF / DRC 规则)**、SRAM/IP 宏的 LEF/时序、**金属层栈信息**。
- **布局规划信息**:die/核尺寸、IO/pad 位置、宏摆放约束、电源方案。
- **功耗意图(UPF/CPF)**、**DFT/scan 信息**。

> 关键心态:后端是"在物理现实里还债"——前端任何乐观假设(时序、面积、布线、功耗),到这里都要用真实坐标和金属层兑现;兑不出来就得返工(改综合甚至改 RTL)。

---

## 3. 物理设计的核心内容(尽量穷尽地列)

每项给:**是什么 + 怎么做(经验法则) + 在 TPU 上长什么样**。

### 3.1 物理设计流程与数据准备
- **是什么**:整条后端流程——数据准备 → 布局规划 → 电源网络 → 布局 → 时钟树 → 布线 → 时序/物理收敛 → 出版图;以及导入网表/LEF/tech/约束等数据。
- **怎么做**:先把库、工艺、约束、宏都准备齐、对得上;后端是**高度迭代**的——每步都回看时序/拥塞/功耗,不行就退回上一步调。
- *在 TPU 上*:导入阵列网表 + 28nm 工艺规则 + SRAM 宏 LEF + 700MHz 约束,准备开摆。

### 3.2 布局规划(floorplanning)
- **是什么**:定 **die/核的尺寸与形状、利用率(utilization)、IO/pad 环、大宏(SRAM/PLL/IP)的摆放、预留布线/电源通道**。
- **怎么做**:布局规划**定全局成败**——宏摆得不好,后面布线拥塞、时序崩、电源走不通。规整大块(如阵列、SRAM)要摆得利于数据流和供电;利用率留余量给布线。
- *在 TPU 上*:把 256×256 阵列摆成规整大块、把 24 MiB Unified Buffer 的 SRAM 宏(占了很大面积)沿阵列边摆好,让"喂数据"路径短。

### 3.3 电源网络规划(power planning / PDN)
- **是什么**:设计**供电网络**——电源环(ring)、电源条/网格(straps/mesh)、把电源送到每个单元的电源轨,并保证 **IR-drop、电迁移(EM)** 不超标。
- **怎么做**:高翻转、大电流的块(如算力阵列)要**足够粗的电源网格**;PDN 太弱 → 压降导致时序失效甚至功能错。电源规划和布线抢金属资源,要平衡。
- *在 TPU 上*:256×256 阵列满负荷时电流巨大,必须铺足够强的电源网格压住 IR-drop,否则阵列会"供不上电"。

### 3.4 布局(placement)
- **是什么**:把**标准单元**摆到核内的合法位置,**时序驱动 + 拥塞驱动**,再合法化(legalization)。
- **怎么做**:让时序关键的单元靠近、减少长线;盯**拥塞(congestion)**——摆太挤会布线布不通。摆放质量直接决定后面时钟树和布线好不好做。
- *在 TPU 上*:阵列的规整性是后端的福音——成排 PE 规则摆放,数据流方向清晰,拥塞可控。

### 3.5 时钟树综合(CTS)
- **是什么**:在此之前时钟被当作"理想"(零 skew、分发网络未实现,插入延迟按 0 或估算值算);CTS **真正搭建时钟分发网络**(缓冲器树 / H-tree / mesh),目标是**低偏差(skew)+ 合理插入延迟**,并整合时钟门控。
- **怎么做**:**不受控/随机的** skew 会被 STA 悲观地同时吃掉 setup 和 hold 两侧裕量——既压低可达频率,又制造 hold 问题;CTS 要把它压低、并把时钟均匀送到成千上万个触发器。(注:skew 是**带符号**的——刻意按路径定向引入的"有用偏差 useful skew"反而能借还裕量帮助收敛,见 §3.7,与此处要消除的不受控偏差是两回事。)CTS 后时序才"接近真实"。
- *在 TPU 上*:把 700MHz 时钟低偏差地分发到整块巨大阵列的所有触发器——阵列越大,时钟分发越是挑战。

### 3.6 布线(routing)
- **是什么**:把所有连接用**真实金属线**实现——**全局布线**(分配大致走廊)+ **详细布线**(具体到每条线在哪层、走哪),并满足 DRC。
- **怎么做**:多金属层分配;盯 DRC(间距/宽度规则)、拥塞、串扰;布不通就回去调布局/floorplan。布线完成,RC 才完全真实。
- *在 TPU 上*:阵列内部规则短连线 + 全局的控制/数据总线;规整结构让布线相对可控,但海量连线仍是重活。

### 3.7 时序收敛(timing closure / ECO)
- **是什么**:在布局后/CTS 后/布线后反复跑 STA,**修 setup(提速)和 hold(加延迟)违例**,直到时序干净;用 **useful skew、ECO(工程变更)** 做增量修复。
- **怎么做**:setup 靠换快单元/缩线/重摆,hold 靠插延迟单元;**这是后端最耗时的拉锯**,贯穿布局到布线全程。综合的乐观时序在这里被真实物理"修正"。
- *在 TPU 上*:阵列关键路径短(一个 MAC),setup 宽裕;重点修长全局路径与 hold——大阵列里 hold 修复尤其琐碎。

### 3.8 寄生提取与信号完整性(extraction / SI)
- **是什么**:**寄生提取(RC extraction)**——从版图算出每条线真实的电阻电容(SPEF),供精确时序;**信号完整性(SI)**——处理**串扰(crosstalk)、噪声**等真实物理效应。
- **怎么做**:提取后用真实 RC 再做时序分析;先进节点的**串扰(SI 电气噪声)**与**双重/多重曝光**(光刻分解/版图着色技术,会让 RC 提取必须区分同色/异色耦合电容,即 mask-aware extraction)都越来越重,需**分别**处理——前者是电气噪声、后者是提取复杂度,别混为一谈。
- *在 TPU 上*:密集阵列的相邻线串扰、长总线的 RC 都要提取并纳入时序,确保 700MHz 在真实寄生下仍成立。

### 3.9 物理验证(DRC / LVS / antenna)
- **是什么**:版图的硬性体检——**DRC**(设计规则检查:线宽/间距等是否合规)、**LVS**(版图 vs 网表:版图连接是否和网表一致)、**antenna** 等。
- **怎么做**:**DRC/LVS 必须 100% 干净**——这是 foundry 收不收你版图的硬门槛,不过就不能流片。后端反复跑、反复修到干净。
- *在 TPU 上*:整块 die 的 DRC/LVS 全清,才能把 GDSII 交给 28nm 产线。

### 3.10 后端的低功耗与 DFT 落地
- **是什么**:把前面的低功耗/DFT 意图在版图上**物理落地**——电源开关、隔离单元、电平移位器的摆放,always-on 区域;scan 链的**物理重排序**以利布线。
- **怎么做**:按 UPF 实现电源域边界与隔离;scan 链按物理就近重连,减少绕线。
- *在 TPU 上*:空闲 PE 列的门控/电源管理在版图上落地;海量 scan 链按物理位置重排,布得更顺。

### 3.11 产出与交接(GDSII 准备)
- **是什么**:后端的最终产物——**版图(GDSII/OASIS)**,以及提取的寄生、最终网表、各类报告,交给 **⑨ 签核**做全面 STA/功耗/EM-IR/物理验证,通过后 **⑩ 流片**。
- **怎么做**:把干净、收敛的版图与数据打包;后端和签核常来回迭代到全绿。
- *在 TPU 上*:28nm 收敛版图 → 交签核 → 通过即可 tape-out。

### 3.12 贯穿视角:低功耗/UPF 这条线怎么贯穿全流程(全景)
低功耗(power intent)和 DFT 一样,是**贯穿各章的横切线**,而且它有一份**贯穿全程的"单一事实来源":UPF/CPF 功耗意图文件**。把散落各处串起来:

```
③ 架构:划电压域/功耗域、定 TDP 预算与低功耗状态(P/C-state)
  → ④ 微架构:门控粒度、操作数隔离、retention 策略(power hooks)
  → ⑤ RTL:写成"带 enable 的更新"让工具识别门控、配合 UPF(别手写门控时钟)
  → ⑥ 验证:power-aware 仿真,验下电/上电、保持(retention)、隔离(isolation)对不对
  → ⑦ 综合:按 UPF 插 ICG 时钟门控、多 Vt、实现电源域边界
  → ⑧ 物理(本章):电源开关、隔离单元、电平移位器的物理摆放,always-on 区域,PDN
  → ⑨ 签核:动态+漏电功耗签核、IR-drop/EM,核对"网表与 UPF 意图一致"
```

- **一句话**:**UPF/CPF 是那份从 ③ 一直传到 ⑨ 的功耗意图契约**——电压域、隔离、电平移位、保持、always-on 都写在里面,各阶段工具照它实现并互相校验。任何一处和 UPF 对不上(漏个隔离单元、电平移位放错),就是"仿真过、硅上崩"或漏电失控。
- *在 TPU 上*:空闲 PE 列的细粒度时钟门控,从 ⑤ 的 RTL enable → ⑦ 插成 ICG → ⑧ 在版图上落电源域边界与隔离 → ⑨ 签核确认漏电与意图一致,是 perf/W 实打实的来源。

## 4. 贯穿始终、必须反复权衡的东西

- **多目标同时收敛**:时序、面积、可布性、功耗/电源完整性、DRC——**任何一项崩了都不算完**,后端就是在它们之间反复拉锯。
- **面积 vs 可布性**:利用率拉太高省面积,但布线拥塞、时序崩;留太松浪费 die。
- **时钟树 skew vs 功耗**:时钟树越平衡 skew 越小,但缓冲器多、功耗大。
- **电源网格 vs 布线资源**:PDN 要粗(压 IR-drop),又和信号布线抢金属层。
- **前端乐观要后端还**:综合的时序/面积假设,后端用真实物理兑现;兑不出就返工到 ⑦ 甚至 ⑤。
- **先进节点的新麻烦**:串扰、双重曝光、EM、IR-drop 在 7nm/5nm 越来越主导。

> 灵魂句:**后端是"还债现场"——前端所有乐观的假设,都要在真实的坐标、金属层、寄生和电源网格上兑现;多目标里只要一项不收敛,版图就出不去。**

---

## 5. 物理设计"怎么做"出来的(方法与工具)

- **后端实现工具**:Innovus(Cadence)、IC Compiler II / Fusion Compiler(Synopsys)、Aprisa——做 floorplan/placement/CTS/routing 的主力。
- **寄生提取**:StarRC、Quantus(出 SPEF)。
- **签核 STA**:PrimeTime(⑨ 用真实 RC 做最终时序)。
- **物理验证**:Calibre、IC Validator、Pegasus——跑 DRC/LVS/antenna。
- **数据格式**:LEF/DEF(物理)、GDSII/OASIS(版图)、SPEF(寄生)、UPF(功耗)。
- **流程**:高度迭代——每步回看时序/拥塞/功耗/DRC,不行就退回调,直到多目标全收敛。

---

## 6. 物理设计阶段的交付物(交给下游什么)

- **版图(GDSII / OASIS)**:最终物理产物,**⑩ 流片的核心数据**(经 ⑨ 签核后)。
- **最终门级网表 + 寄生(SPEF)**:供 ⑨ 做真实时序/功耗签核。
- **时序/功耗/拥塞/DRC-LVS 报告**:收敛证据。
- **物理验证(DRC/LVS)干净的凭证**:流片的硬门槛。

> 后端的产物,是把抽象设计第一次变成"硅片上能造出来的几何图形"——它让前面所有逻辑努力,终于有了可制造的物理形态。

---

## 7. 容易被忽视、却致命的坑

1. **布局规划没做好**:宏摆放/利用率不当 → 后面布线拥塞、时序崩、电源走不通,推倒重来——后端头号坑,且越早错越贵。
2. **电源网络太弱**:IR-drop 过大 → 单元供电不足,时序失效甚至功能错;EM 超标影响寿命。
3. **拥塞(congestion)失控**:摆太挤布不通,再好的时序也白搭。
4. **时钟树 skew 没控住**:跑不到目标频率,还诱发大量 hold 违例。
5. **把综合时序当真账**:不重视布线后真实 RC,布完才发现一堆违例。
6. **DRC/LVS 不干净**:foundry 直接拒收,流片硬门槛过不了。
7. **串扰 / SI 忽视**(尤其先进节点):真实噪声让时序在硅上失效。
8. **hold 修复遗漏**:hold 违例不像 setup 能靠降频救,漏了就是硅上随机错。
9. **低功耗/DFT 物理落地出错**:电源域边界、隔离单元、scan 重排没处理好,功耗或可测性崩。

---

## 8. 小结 + 下一步

**物理设计(后端)= 把门级网表在真实工艺规则下布局规划、摆单元、建时钟树、布线,并让时序/面积/可布性/功耗/物理规则同时收敛,最终产出可制造的版图(GDSII)。** 它的灵魂是 **"延迟到这里才变真账,前端的乐观都要在真实坐标和金属层上还债"**;最该敬畏的硬门槛是 **"DRC/LVS 不干净就流不了片"**;最耗时的拉锯是 **"多目标同时收敛"**。

下一步(第 ⑨ 步)是 **签核(signoff)**:对这份版图做**最终、最严格的全面检查**——用真实寄生做时序签核(STA)、功耗与 EM/IR-drop 分析、物理验证、与 RTL 的等价性——所有项全绿,才敢按下流片(⑩)的按钮。
