# 芯片设计完整流程 · 第 ① 步:产品定义 / 市场需求

> 「芯片设计完整流程」学习系列第一篇(全流程的真正起点)。
> 全流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产
```

**你现在在第 ① 步——整条链的最左端、最上游**(① 决定"要不要做、做什么",② 翻成技术指标,③ 翻成硬件骨架)。它**不画电路、不写代码、甚至不太谈技术实现**,它回答的是商业与产品问题:**这颗芯片为什么存在?给谁用?要达到什么指标才卖得动?值不值得投这笔钱?**

为什么把它当成"第一步"而且要认真对待:**一颗芯片是一笔巨大的、不可逆的赌注**——先进节点下,光掩模/流片(NRE)就上千万美元,从立项到量产 2–4 年,投入动辄几千万到上亿。**这一步定错了方向,后面所有人的努力都是在精确地造一个没人要的东西。** 它的影响力甚至比架构还大。

> **📐 本系列的范围边界(先说清楚,免得"完整流程"被误读)**:这十三步讲的是**数字 SoC / ASIC 的实现主线**。为聚焦,几条相关但不同的线**默认按"外购"或"并行另线"处理、本系列不展开**:① **模拟 / 混合信号(AMS)** 设计流(schematic→SPICE→手工版图)——SerDes、PLL、HBM/DDR PHY 等模拟 IP 一律按**外购成熟硬 IP**当黑盒(它们内部的模拟设计是另一套流程);② **FPGA** 流程——本系列只把 FPGA 当**验证原型**(见 ⑥),不作为最终实现目标(FPGA 无流片/掩模/封测);③ **软件 / 编译器 / 驱动 / 固件栈**开发——是与硬件**并行推进的另一条主线**,这里只在接口处提及;④ **车规功能安全(ISO 26262)**、安全/合规认证——作为**并行合规流程**,只以规格约束的形式出现。一句话:本系列是"**纯数字实现流 + 把模拟/第三方 IP 当黑盒外购**"的视角。

---

## 1. 这一步到底是什么

一句话:**在投入研发之前,搞清楚"做哪颗芯片、为谁做、要多强、卖多少钱、能不能赚钱、什么时候必须上市",并产出一份让全公司对齐的目标。**

核心特征:
- **业务驱动,不是技术驱动**。工程可行性只是约束条件,真正的问题是"**该不该做 + 做了卖不卖得动**"。
- 它的产物是后面**一切技术工作的"北极星"**:性能、功耗、面积/成本、接口、排期这些目标,全都从这一步流下去(正好就是架构设计的输入)。
- 本质是**对 2–4 年后的市场下注**——你在预测未来。

> 一个贯穿全篇的例子:假设你要立项一款**数据中心 LLM 推理加速卡**。下面每一节我都用它举例。

---

## 2. 谁来做这一步

不是芯片工程师主导,而是:
- **产品经理 / 产品市场(Product / Product Marketing)** 牵头;
- **市场、商务(BD)、销售**提供市场和客户信息;
- **架构 / 工程**提供"能不能做、要多少成本/时间"的可行性反馈;
- **关键客户(lead customer)**深度参与,告诉你他们真正要什么。

它是一个**多方拉锯、反复对齐**的过程,不是某个人拍脑袋。

---

## 3. 核心内容(尽量穷尽地列)

### 3.1 市场分析(Market analysis)
- **市场规模与增长**:TAM / SAM / SOM(总体 / 可服务 / 可获得市场)。这个赛道够不够大、在不在涨。
- **目标细分市场**:数据中心?手机?汽车?边缘/IoT?(LLM 推理卡 → 数据中心 / 云。)
- **竞争格局**:现有玩家是谁、他们的 PPA / 价格 / 路线图(对 LLM 卡:NVIDIA、AMD、各家自研 ASIC……)。
- **差异化**:**凭什么客户买你而不是在位者?** 更高能效?更低 TCO?更适配某类模型?没有差异化 = 没有理由存在。
- **时机窗口**:市场窗口会关。早了没生态,晚了被吃掉。

### 3.2 客户与用例(Customers & use cases)
- **谁是客户**(买芯片的人)vs **谁是最终用户**——常常不是同一方。
- **锚定客户 / lead customer**:有没有大客户愿意早期承诺采购?(没有 anchor 客户的芯片,是在豪赌。)
- **具体用例与负载**:要跑哪些模型、什么 batch/序列长度、prefill 还是 decode 为主、精度(FP16/FP8/INT)……**这些直接决定后面架构怎么定。**
- **需求采集**:客户访谈、RFQ、把"我想要更快"翻译成可量化指标。
- **出货量预测**:按客户 / 细分市场估量,这决定了商业模型成不成立。

### 3.3 产品定义(Product definition)
- **产品形态与集成度**:是一颗芯片?一个 chiplet?一块板卡?一个模组?(LLM 卡:die + HBM + 封装 + 板卡 + 散热,整体交付。)
- **功能集**:必须有 vs 最好有(MoSCoW:Must / Should / Could / Won't)。
- **产品家族 / SKU / 衍生品**:一套架构能不能裁/扩出高中低多个型号?(这决定了研发投入的"杠杆率"。)
- **定位**:高端 / 主流 / 走量。

### 3.4 需求规格(Requirements——会往下游流的硬指标)
这一节最关键,因为**它就是架构设计那篇里"输入"的来源**。要把市场语言翻成可量化目标:
- **性能目标**:吞吐(tokens/s、TOPS)、延迟(TTFT / TPOT)、有效利用率——由用例 + 竞品倒推。
- **功耗 / 散热包络**:形态决定一切。手机几瓦、数据中心卡几百瓦。这是算力的天花板。
- **成本 / 目标售价 / 毛利**:→ 反推可接受的 die 面积、封装、用得起哪个工艺节点。
- **必须支持的接口/标准**:PCIe/CXL 第几代、内存用 HBM 还是别的、UCIe(若 chiplet)……市场说了算。
- **软件 / 生态需求**:要不要兼容某框架、某编译栈、某套算子库——**对 AI 芯片,生态常常是生死线**。
- **可靠性 / 质量 / 认证**:数据中心要长时间高可靠;汽车要 AEC-Q100 + 功能安全;医疗/军工各有标准。
- **物理约束**:封装、引脚数、板卡尺寸、散热方案。

### 3.5 商业论证 / 商业案例(Business case)
- **成本结构**:
  - **NRE(一次性工程费)**:设计人力 + IP 授权 + **掩模/流片**(先进节点单次掩模可达千万美元级)。
  - **单位成本**:die(≈ 晶圆成本 /(每片 die 数 × 良率)——面积↑→每片 die 数↓→成本↑,良率↑→成本↓)+ 封装 + 测试。
- **量 × 价 × 毛利**:能不能覆盖 NRE 并盈利?**盈亏平衡点的出货量**是多少?
- **ROI / 回本周期**。
- **自研 vs 外购(make vs buy)**:自己做芯片,还是买现成的?自研只有在"量足够大 + 差异化足够强"时才划算。
- **投资审批**:这一步通常以一个**立项评审(gate review)**收尾——批准了才进下一步。

### 3.6 时机与路线图(Timing & roadmap)
- **上市窗口**:什么时候必须出货才还有意义。
- **工艺节点可用性**:目标时间点上,你要的 node 成熟吗、产能够吗、IP 全吗。
- **产品路线图**:这一代、下一代、衍生品怎么排——芯片很少是"一锤子",要按代际规划。
- **排期可行性 vs 窗口**:做得完吗?做完时窗口还在吗?

### 3.7 风险评估(Risk)
- **市场风险**:2–4 年后这个市场还在/还这么大吗?
- **技术风险**:这套指标到底做不做得出来?
- **竞争风险**:对手会不会中途跳代把你超了?
- **供应链 / 代工风险**:产能、节点准入、关键 IP/内存(如 HBM)拿不拿得到。
- **客户承诺风险**:lead 客户到时候真会买吗,还是只是口头?

### 3.8 约束(Constraints)
- 预算、团队规模与能力、可复用 IP。
- 代工厂关系 / 节点准入。
- 法规 / 出口管制 / IP 授权。
- 标准合规与认证。

## 4. 关键权衡与心态

- **野心 vs 现实**:指标要**够激进才能赢**,又要**够现实才做得出来、按时上市**。这是这一步的核心张力。
- **市场拉动 > 技术推动**:最危险的失败是"工程师觉得很酷,但没人要"。永远从"谁会买、为什么买"出发。
- **目标的杠杆极大**:晚 6 个月、或功耗差 20%,就可能整代错过市场。这一步的几个数字,后面用几千人月去兑现。
- **总成本思维**:别只看 die 成本——NRE、生态/软件、支持、认证都是真金白银。

---

## 5. 交付物(这一步产出什么)

- **MRD(市场需求文档)**:市场需要什么(市场团队主笔)。
- **PRD(产品需求文档)**:产品必须做到什么(产品团队主笔)。
- **商业案例 / 投资建议书**:量、价、成本、ROI、盈亏平衡。
- **目标规格(Target Spec)**:PPA + 成本 + 功能 + 接口 + 排期 的一组"北极星数字"——**这就是交给下游(系统规格 / 架构)的东西**。
- **Go / No-Go 决策**:一个正式的立项闸门。

---

## 6. 容易被忽视、却致命的点

1. **技术驱动而非市场驱动**:为了炫技做了没市场的东西——头号死法。
2. **低估软件/生态**:尤其 AI 芯片,硬件再强,算子库/编译器/框架支持跟不上 = 卖不动。
3. **错过市场窗口**:晚了就死,再好也没用。
4. **过度规格(gold-plating)**:堆了一堆"最好有",把成本和排期拖爆。
5. **只算 die 成本**:忽略 NRE + 生态 + 支持 + 认证的总账。
6. **没有锚定客户**:纯靠预测立项,风险极高。
7. **不做产品家族规划**:一次性单品,研发分摊不下来,单位价值的成本太高。
8. **可行性与目标脱节**:市场拍了个指标,工程根本做不到——这一步必须有工程可行性反馈在场。

---

## 7. 小结 + 下一步

**产品/市场需求 = 在烧钱之前,用商业与市场的视角定清楚"做哪颗芯片、为谁、要多强、卖多少、何时上市、能否赚钱",并产出一份让全公司对齐的目标规格 + 一个 Go/No-Go 决策。** 它是业务驱动的,是整条流程的真正起点,**它的产物(Target Spec)正是架构设计的输入**。

下一步(第 ② 步)是 **系统规格(System Spec)**:把这份偏商业的"目标"翻译成一份**工程可执行的技术规格**——功能清单、接口协议、性能/功耗/面积的量化指标与预算、验收标准——它是产品需求与架构设计之间的桥。
