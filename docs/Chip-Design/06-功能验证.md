# 芯片设计完整流程 · 第 ⑥ 步:功能验证

> 「芯片设计完整流程」学习系列第六篇。
> 编号主流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。
> DFT设计涉及:③ 测试架构与覆盖率目标 → ④/⑤ 测试控制和可测逻辑 → ⑦ Scan/压缩/MBIST 插入与 ATPG → ⑧ Scan 链重排及测试时钟、功耗、拥塞收敛 → ⑨ 测试模式与覆盖率签核 → ⑫/⑬ ATE、诊断与良率学习。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产

DFT设计:③ 测试架构 → ④/⑤ 测试控制和可测逻辑 → ⑦ Scan/压缩/MBIST 与 ATPG → ⑧ Scan 链重排/测试时序功耗 → ⑨ 测试模式与覆盖率签核 → ⑫/⑬ ATE/诊断/良率学习
```

**功能验证和 RTL(⑤)并行、并在它之后收口**:RTL 写出了"硬件代码",但代码对不对、符不符合规格,谁说了算?功能验证就是用海量激励、参考模型、覆盖率,**在流片之前证明这份 RTL 的行为真的符合规格**。它是整条流程里**工作量通常最大的一环**——很多项目里验证工程师比设计工程师还多,验证投入占到一半以上。

> 一句话定位:**RTL 回答"硬件怎么实现",功能验证回答"这实现到底对不对"——它是流片前唯一一道全面的功能正确性防线。**

---

## 1. 功能验证到底是什么

一句话:**在芯片造出来之前,用仿真/形式/加速等手段,系统性地证明 RTL 的行为符合规格——并尽最大努力找出它哪里不符合(找 bug)。**

### 一个关键心态:你的目标是"找 bug",不是"让它过"
新手最大的误区,是把验证当成"跑通几个用例、绿了就行"。真正的验证思维恰恰相反:**你是在拼命攻击这个设计,想方设法让它出错。** 一个抓不到 bug 的测试,几乎是浪费。

> 灵魂句:**验证不是去证明"它能工作",而是去证明"它在哪不工作"。绿灯只意味着"这次没抓到",不等于"没有 bug"。**

### 验证(verification) ≠ 测试(test)
这两个词中文都叫"测",但完全不同:
- **功能验证(⑥,流片前)**:证明**设计本身对不对**(RTL 匹配规格)。问的是"我们有没有把设计做对"。
- **量产测试(⑫,流片后)**:芯片造出来后,筛**制造缺陷**(这颗具体的芯片有没有坏)。问的是"这颗芯片造好了没"。

### 和 RTL(⑤)的关系
RTL 工程师写设计;验证工程师(通常是**独立的人/团队**)对照规格攻击它。**独立很关键**——让设计者自己验自己,会带着同样的理解盲点,看不见自己的错。

*在 TPU 上*:⑤ 写出了 256×256 阵列的 RTL;⑥ 要回答——喂进各种矩阵(含全零、溢出、饱和、随机),它算出来的结果,跟一个独立的"黄金"矩阵乘模型逐一比对,**对得上吗?握手会不会死锁?跨时钟域会不会出错?**

---

## 2. 谁来做 + 需要什么输入

**谁来做**:**验证工程师 / DV(Design Verification)团队**——常是独立于设计的一支大队伍。架构/设计配合答疑,但验证**独立判断对错**(以规格为准,不以 RTL 为准)。

**需要什么输入**:
- **规格(②)——黄金标准**:验证一切以规格为"对错的唯一裁判";**规格里的歧义会直接变成验证的扯皮**(所以②那篇强调"可验证")。
- **RTL(⑤)**:被验证对象(DUT,Design Under Test)。
- **微架构(④)**:理解内部结构,设计有针对性的激励与覆盖点。
- **验证计划**:要验什么、怎么验、覆盖率目标、何谓"验完了"。

> 关键心态:验证团队**早期就要介入**(从②规格阶段就参与评审),否则会混进"测不了"的需求、或等 RTL 写完才开始搭环境而拖垮进度。

---

## 3. 功能验证的核心内容(尽量穷尽地列)

按"做一次完整验证要搭哪些东西、用哪些手段"组织,每项给:**是什么 + 怎么做(经验法则) + 在 TPU 上长什么样**。

### 3.1 验证计划(verification plan)
- **是什么**:一份文档,列清**要验证哪些功能/场景、用什么方法、覆盖率目标、以及"何谓验完"**。
- **怎么做**:从规格逐条拆出可验证的特性(feature),为每条定激励策略与覆盖点;**先有计划再搭环境**,否则验得零散、说不清覆盖了没。
- *在 TPU 上*:列出要验的——矩阵乘正确性、各精度/尺寸、溢出/饱和、握手不死锁、指令序列、CDC……每条配覆盖目标。

### 3.2 测试平台(testbench)架构
- **是什么**:包住 DUT 的整套验证环境——**激励驱动(driver)、监控(monitor)、记分板(scoreboard)、参考模型(reference model)**。
- **怎么做**:driver 把激励喂进 DUT 接口,monitor 抓 DUT 的输入输出,scoreboard 把 DUT 输出和参考模型输出比对报错。**结构清晰可复用**是关键(UVM 就是为此而生)。
- *在 TPU 上*:driver 灌矩阵和指令,monitor 抓阵列输出,scoreboard 对照黄金矩阵乘模型判对错。

### 3.3 激励生成:定向 vs 约束随机
- **是什么**:**定向测试(directed)**——手写具体用例测特定场景;**约束随机(constrained-random)**——在合法约束内随机生成海量激励,自动覆盖你没想到的角落。
- **怎么做**:定向用于关键/边界场景和首次点亮;**约束随机是现代验证主力**——用随机性逼出人想不到的 corner case,再用覆盖率确认测到了哪。
- *在 TPU 上*:定向测"全零矩阵、最大值溢出";约束随机生成各种尺寸/数值的矩阵海量灌,逼出意外组合。

### 3.4 自检与参考模型(scoreboard / golden model)
- **是什么**:一个独立的**参考模型(黄金模型)**算出"应该的结果",scoreboard 自动和 DUT 比对——即**自检(self-checking)**。
- **怎么做**:参考模型要**独立实现**(常用 C/C++/Python/SystemVerilog),**绝不能和 RTL 抄同一份理解**,否则共同犯错就抓不到。无自检的测试(靠人眼看波形)不可规模化。
- *在 TPU 上*:参考模型 = 一个朴素的软件矩阵乘(独立写),每个结果和阵列输出逐元素比对。

### 3.5 覆盖率(coverage):你怎么知道"测够了"
- **是什么**:量化"验证做到什么程度"的指标。**代码覆盖**(行/分支/翻转/状态机/条件,工具自动收)+ **功能覆盖**(你手写的 covergroup,描述"重要场景测到没")。
- **怎么做**:**覆盖驱动验证(CDV)**——盯着覆盖率找空洞,针对性补激励,直到收敛到目标。**注意:覆盖率高 ≠ 验证充分**(覆盖到了不代表检查对了,见 §7)。
- *在 TPU 上*:功能覆盖点——各矩阵尺寸、是否触发溢出、各指令、FIFO 满/空、各 CDC 路径……都被覆盖过没。

### 3.6 UVM 方法学
- **是什么**:**UVM(Universal Verification Methodology)**——基于 SystemVerilog 的业界标准验证方法学,提供可复用的环境框架(agent/sequence/driver/monitor/scoreboard 等组件)。
- **怎么做**:用 UVM 把测试平台搭成标准、模块化、可复用的结构;约束随机 + 覆盖 + 自检都在这套框架里组织。是中大型芯片功能验证的事实标准。
- *在 TPU 上*:为阵列、内存接口、指令接口各搭 UVM agent,组合成 chip-level 环境。

### 3.7 断言(assertions / SVA)
- **是什么**:**SVA(SystemVerilog Assertion)**——把"必须永远成立的性质"写成断言,嵌在设计/环境里,一旦违反立即报错并定位。
- **怎么做**:用断言表达接口协议、握手规则、不该发生的状态("FIFO 不该在空时被读")。断言**在 bug 发生的那一刻、那个位置**就抓住,极大缩短调试;还能喂给形式工具。
- *在 TPU 上*:断言"valid 拉高后 ready 前数据不变""FIFO 不溢出""阵列不在装填未完时输出"。

### 3.8 形式验证(formal verification)
- **是什么**:用数学**穷举证明**某性质对所有输入都成立(或给出反例),不靠跑激励——和仿真互补。
- **怎么做**:适合状态机、仲裁器、协议、CDC、寄存器这类**状态空间可控**的块;能证出仿真永远跑不到的角落,也能证"某 bug 不存在"。代价是大数据通路(如整块阵列)易状态爆炸,要会划范围。
- *在 TPU 上*:用形式证控制 FSM 无死锁、握手协议恒成立、CDC 同步结构正确——这些仿真难穷尽。

### 3.9 仿真加速:emulation / FPGA 原型
- **是什么**:软件仿真太慢(跑完整芯片、启动软件要天文时间),用**硬件仿真器(emulation:Palladium/ZeBu/Veloce)**或 **FPGA 原型**把 RTL 跑在硬件上,快几个数量级。
- **怎么做**:emulation 调试可见性好、用于大规模/长测试(如跑真实负载、软硬件协同);FPGA 原型更快但可见性差,常给软件团队提前用。
- *在 TPU 上*:在 emulator 上跑真实模型推理的完整数据流,验证整芯片级行为与性能,这是纯软件仿真扛不动的。

### 3.10 专项验证(CDC / 低功耗 / X / 寄存器 / 连通性)
- **是什么**:几类**普通功能仿真默认看不到、却最致命**的专门验证。
- **怎么做**:**CDC 验证**(跨时钟域同步是否正确,需专用工具+仿真注入亚稳态)、**低功耗验证**(UPF 下电/上电、保持、隔离对不对)、**X 传播**(复位/未知态是否被乐观掩盖)、**寄存器(CSR)验证**(读写复位值,常自动生成)、**连通性**(顶层连线对不对)。
- *在 TPU 上*:验核心域↔PCIe↔DDR3 的 CDC、空闲 PE 列门控下电恢复正确、CSR 读写、顶层互连连通。

### 3.11 回归、调试与 bug 管理
- **是什么**:把所有测试**自动化批量跑(回归 regression)**,持续保证"改一处没弄坏别处";配套 **debug**(波形定位)和 **bug 跟踪数据库**。
- **怎么做**:夜间/CI 跑全回归;失败自动分诊(triage)→ 定位是设计 bug 还是环境 bug → 进 bug 库跟踪到关闭。**bug 收敛曲线**是项目健康度的关键指标。
- *在 TPU 上*:每晚回归海量随机矩阵 + 定向用例,失败入库,盯 bug 发现/关闭曲线判断能不能流片。

### 3.12 覆盖收敛与验证签核(closure)
- **是什么**:回答那个最难的问题——**"验完了吗?能流片了吗?"** 用覆盖率收敛 + bug 收敛 + 验证计划全勾选来定。
- **怎么做**:签核判据通常是:**功能/代码覆盖率达标、验证计划逐条闭环、bug 收敛(新 bug 趋零、无未决严重 bug)**。这是一个正式闸门——过了才放行去综合/流片。
- *在 TPU 上*:覆盖率全绿、计划全闭环、严重 bug 清零 → 验证签核 → 放行。

## 4. 贯穿始终、必须反复权衡的东西

- **永远验不完 vs 何时算够**:输入空间无穷,你**永远无法证明"没有 bug"**,只能用覆盖率证明"我测够了"。收敛判据就是这条线。
- **随机 vs 定向**:随机逼角落、定向打要害,两者配合。
- **仿真 vs 形式 vs 加速**:精度/速度/穷尽性各有取舍,大项目三者并用。
- **覆盖到 ≠ 检查对**:覆盖率只说"激励走到了这",不说"结果查对了"——必须配强 checker。
- **独立性**:验证要独立于设计思维,否则共享盲点。
- **越早越便宜**:bug 越晚发现越贵(流片后一个 bug 可能=几百万美元 + 几个月)。验证要 shift-left、和设计并行。

> 灵魂句:**验证是个无底洞——你证不完"没有 bug",只能用覆盖率和收敛判据,证明"我已经测到了该测的"。**

---

## 5. 功能验证"怎么做"出来的(方法与工具)

- **仿真器**:VCS / Questa / Xcelium(商业)、Verilator(开源)——跑 testbench 的主力。
- **方法学/语言**:**UVM + SystemVerilog**、**SVA 断言**。
- **形式工具**:JasperGold / VC Formal —— 穷举证明性质、查 CDC/连通性。
- **硬件加速**:Palladium / ZeBu / Veloce(emulation)、FPGA 原型。
- **覆盖率工具**:收集与合并代码/功能覆盖,看收敛。
- **CDC / 低功耗工具**:专门查跨时钟域与 UPF。
- **调试**:Verdi(波形/反向追踪)。
- **回归与 CI**:批量跑、自动分诊、bug 数据库。

---

## 6. 功能验证阶段的交付物(给项目什么)

- **验证计划**(可追溯到规格的每条特性)。
- **测试平台 / UVM 环境**(可复用资产,衍生型号还能用)。
- **覆盖率报告**(代码 + 功能,已收敛)。
- **回归结果 + bug 数据库**(收敛曲线)。
- **验证签核报告**——一个正式闸门:**"RTL 功能正确,可以放行去综合/流片"** 的信心凭证。

> 验证的真正"交付物"不是某个文件,而是**让全项目敢于流片的信心**——它是流片前唯一一道全面的功能正确性闸门。

---

## 7. 容易被忽视、却致命的坑

1. **验证滞后于设计**:不和 RTL 并行,等写完才搭环境 → 项目尾期爆雷、被迫带 bug 流片。
2. **只做定向、没随机/覆盖**:只测想得到的,漏掉想不到的 corner case——硅上才暴露。
3. **覆盖率高 ≠ 验证充分**:覆盖只说"激励走到了",不说"结果查对了";没有强 checker,覆盖率全绿也可能满是漏网 bug。
4. **checker 太弱 / 没自检**:靠人眼看波形,不可规模化,bug 大量溜过。
5. **参考模型和 RTL 共享理解**:抄同一份(可能错的)理解 → 一起错、抓不到。
6. **不验 CDC / 低功耗 / X / 复位**:这些普通仿真默认看不到,却正是硅上随机崩的根源。
7. **寄存器 / 连通性没验**:低级却致命——CSR 复位值错、顶层连线接错,影响全局。
8. **没有清晰的"验完判据"**:不知道何时该停 → 要么过早流片(带 bug),要么无限拖期。
9. **设计 bug 与环境 bug 混淆**:大量时间耗在分诊上;环境本身也要被信任(环境也有 bug)。

---

## 8. 小结 + 下一步

**功能验证 = 在流片前,用仿真/形式/加速 + 参考模型 + 覆盖率,系统性地证明 RTL 行为符合规格,并拼命找出它哪里不符合;它通常是整条流程工作量最大的一环,也是流片前唯一一道全面的功能正确性闸门。** 它的灵魂是 **"目标是找 bug,不是让它过"**;最难的判断是 **"何时算验完"(覆盖 + bug 收敛)**;最该敬畏的真相是 **"覆盖到 ≠ 检查对"**。

下一步(第 ⑦ 步)是 **逻辑综合**:把这份**已被验证为功能正确**的 RTL,映射成目标工艺库的门级网表,并在时序/面积/功耗约束下优化——设计从此第一次变成"真实的门和连线"。
