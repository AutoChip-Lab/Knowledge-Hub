# 芯片设计完整流程 · 第 ⑦ 步:逻辑综合

> 「芯片设计完整流程」学习系列第七篇。
> 全流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产
```

**逻辑综合夹在「RTL(⑤)」和「物理设计(⑧)」之间**:RTL 是用 HDL 写的、抽象的"行为描述";综合就是把它**翻译成由具体工艺库里的标准单元(门)和连线组成的门级网表(netlist)**,并在时序/面积/功耗约束下优化。这是整条流程里**第一次让设计变成"真实的门和连线"**——从此它不再是抽象代码,而是某个工艺、某个库里实实在在的电路。

> 一句话定位:**RTL 是"做什么"的抽象代码,综合把它编译成"用哪些门怎么连"的具体电路——综合器就是硬件世界的"编译器 + 优化器"。**

---

## 1. 逻辑综合到底是什么

一句话:**把可综合的 RTL,在给定的工艺库和约束(时钟、面积、功耗目标)下,自动映射并优化成门级网表——一份用标准单元和连线精确描述、可以拿去布局布线的电路。**

### 综合大致分三步(像编译器)
1. **翻译(elaborate / 通用逻辑)**:把 RTL 解析成与工艺无关的通用逻辑(加法器、mux、寄存器……)。
2. **映射(technology mapping)**:把通用逻辑映射成**目标工艺库里真实的标准单元**(具体的 AND/OR/MUX/触发器,且选驱动强度、阈值电压版本)。
3. **优化(optimize)**:在**时序/面积/功耗**约束下反复优化——这是综合的核心价值。

### 一个必须建立的认知:综合器是"照约束优化的机器人"
综合器不会读心。**你给的约束(SDC)说要多快,它就拼命压时序;你不说的,它就不管。** 所以综合结果的好坏,**一半取决于约束对不对**。

> 灵魂句:**Garbage in, garbage out——约束给错了,综合器会非常"努力地"优化出一个错误的目标。**

### 和 RTL(⑤)、签核(⑨)的边界
- 综合**不该改变功能**:网表和 RTL 必须**逻辑等价**(所以一定要做形式等价检查,见 §3.9)。
- 综合内部的时序分析是**估计**(还没真正布线,线延迟靠模型猜)——**真正的时序签核在后端之后的 ⑨**。综合说"收敛"只是初步,别当真账。

*在 TPU 上*:⑤ 写出了 256×256 阵列的 RTL;⑦ 把它映射成 28nm 库里的真实门,在"700MHz 时钟"约束下优化时序、插入时钟门控、插入 scan 链,产出可以拿去布局布线的门级网表。

---

## 2. 谁来做 + 需要什么输入

**谁来做**:**综合 / 实现工程师(synthesis / implementation engineer)**,常和后端(物理设计)是同一拨或紧邻团队。设计、DFT、低功耗、时序团队配合。

**需要什么输入**:
- **可综合的 RTL(⑤)** + **功能已验证(⑥)** 的信心(别拿没验过的 RTL 去综合)。
- **工艺库(.lib / Liberty)**:目标 foundry/node 的标准单元库,含每个单元的时序/功耗/面积特征。
- **约束(SDC)**:时钟定义、IO 延迟、时序例外(false/multicycle path)、面积/功耗目标——**这是综合的"指挥棒"**。
- **功耗意图(UPF/CPF)**(若多电压/电源域)、**DFT 要求**(scan、压缩、覆盖率目标)。

> 关键心态:综合的产出质量(QoR)= RTL 质量 × 约束质量 × 库与策略。**约束写不对,再好的 RTL 也综合不出好电路。**

---

## 3. 逻辑综合的核心内容(尽量穷尽地列)

每项给:**是什么 + 怎么做(经验法则) + 在 TPU 上长什么样**。

### 3.1 综合流程与产物(全景)
- **是什么**:read RTL → elaborate(通用逻辑)→ technology mapping(映射到标准单元)→ 时序/面积/功耗优化 → 写出门级网表 + 报告。
- **怎么做**:理解每步在干什么;迭代式跑——看报告、调约束/策略、再跑,直到 QoR 达标。
- *在 TPU 上*:把阵列、Unified Buffer 控制、指令译码等 RTL 一路映射成 28nm 标准单元网表。

### 3.2 工艺库与标准单元
- **是什么**:目标工艺的**标准单元库**——一堆预先设计、预先表征好的小电路(门、触发器、mux、加法器……),每个有多种**驱动强度**和**阈值电压(Vt)版本**(LVT 快但漏电大、HVT 省漏电但慢)。
- **怎么做**:综合从库里挑单元来实现你的逻辑;**多 Vt 混用**是省漏电的关键手段(关键路径用快单元、非关键路径用省电单元)。
- *在 TPU 上*:阵列里海量 MAC 用库里的乘加/触发器单元实现;非关键路径多用 HVT 压漏电。

### 3.3 约束(constraints / SDC)
- **是什么**:用 **SDC(Synopsys Design Constraints)** 告诉综合器目标——**时钟频率(create_clock)、输入输出延迟、时序例外(false path / multicycle path)、面积/功耗目标**。
- **怎么做**:**约束必须准确且完整**——少设一条时钟、错设一条 false path,综合就会优化错方向(虚假违例或漏掉真违例)。约束是综合里最容易出错、又最关键的一环。
- *在 TPU 上*:`create_clock 700MHz` 给核心域、给各时钟域定约束、给跨域路径正确标 false/multicycle,给 IO 定延迟。

### 3.4 时序驱动综合与 slack
- **是什么**:综合**以满足时序为第一目标**做优化——内部跑简化的 STA,算每条路径的 **slack(裕量)= 要求时间 − 到达时间**;slack 为负就是**违例(violation)**。
- **怎么做**:盯**关键路径(最差 slack)**优化;不够就换更快单元、重组逻辑、插流水(若允许)、retiming。注意综合的线延迟是**估计**,签核要等后端。
- *在 TPU 上*:阵列关键路径就是一个 MAC(很短),slack 充裕,轻松满足 700MHz;真正吃紧的是长的全局控制/IO 路径。

### 3.5 面积与时序/面积/功耗权衡
- **是什么**:综合的本质是在 **PPA 铁三角**里找平衡——压时序常常涨面积/功耗,缩面积可能伤时序。
- **怎么做**:用约束和综合选项调平衡(面积优先?时序优先?);时序满足后回收非关键路径的面积/功耗(downsizing、换 HVT)。
- *在 TPU 上*:阵列时序宽裕 → 把非关键单元降规格、换省电版本,在保 700MHz 的前提下省面积和漏电。

### 3.6 功耗优化(综合阶段)
- **是什么**:综合阶段就插入省电结构——**自动时钟门控插入(clock gating)**、**多 Vt 优化**、操作数隔离;按 **UPF/CPF** 实现多电压/电源域意图。
- **怎么做**:让综合器把 ⑤ 写成"带 enable 的更新"识别成门控、自动插 ICG 单元;多 Vt 在不破坏时序的前提下尽量用省漏电单元。
- *在 TPU 上*:空闲 PE 列的时钟门控在这里被真正插成 ICG 单元;非关键路径换 HVT,把整芯片漏电压下来。

### 3.7 DFT 综合(scan 插入)
- **是什么**:在综合中(或紧接其后)把普通触发器换成**扫描触发器(scan FF)**并**串成扫描链(scan chain)**,加 **test compression**,使芯片可被量产测试。
- **怎么做**:按 DFT 规划插 scan、定链数与压缩比、跑可测性以达**覆盖率目标**;DFT 友好的 RTL(⑤ §3.10)在这里才好插得进。
- *在 TPU 上*:阵列里成千上万触发器被串进 scan 链;海量 SRAM 接 MBIST 接口,保证量产测得了、算得清良率。

### 3.8 综合策略(层次 vs 展平、retiming)
- **是什么**:怎么组织综合——**层次化(hierarchical)**(按模块分别综合,runtime/内存友好、可复用)vs **展平(flatten)**(跨模块边界一起优化,QoR 可能更好但更重);以及 **retiming**(挪动寄存器平衡流水级)、增量综合等。
- **怎么做**:大芯片常层次化 + 对关键块局部展平;规整重复结构(如阵列)用层次化复用一个综合好的块。
- *在 TPU 上*:256×256 阵列是同一个 PE 的海量复制 → **层次化综合一个 PE,再复用铺满阵列**,省 runtime 又一致。

### 3.9 形式等价性检查(LEC)
- **是什么**:**逻辑等价检查(Logic Equivalence Check)**——用形式方法**数学证明"综合出的网表和 RTL 逻辑完全等价"**,确保综合(及后续优化)没改变功能。
- **怎么做**:每次综合/ECO 后都跑 LEC(Formality / Conformal);**这是不可省的签核项**——综合器也可能因约束误导或工具 bug 改错功能,只有等价检查能兜住。
- *在 TPU 上*:证明阵列、控制逻辑的网表 ≡ 已验证的 RTL —— 功能正确性从 ⑥ 一路守到门级。

### 3.10 综合产出与交接(给后端)
- **是什么**:综合交付——**门级网表(gate netlist)**、综合后 **SDC**、**时序/面积/功耗报告**、scan/DFT 信息。
- **怎么做**:把这些干净地交给 **⑧ 物理设计**;网表 + SDC 是后端的输入,报告供评估 QoR 是否可进后端。
- *在 TPU 上*:28nm 门级网表 + SDC + 报告 → 交布局布线,开始变成有坐标的真实版图。

### 3.11 贯穿视角:DFT 这条线从架构一直走到 ATE(全景)
DFT(可测性设计)在本系列里是**有意"贯穿各章"而非独立成章**的——但它其实是一条有自己 owner、自己工具链、自己覆盖率闸门的独立线。把散落各处的点串起来看:

```
③ 架构:预留 scan/MBIST/JTAG、定覆盖率目标
  → ④ 微架构:留可扫描结构、MBIST 接口、观测点(debug 钩子)
  → ⑤ RTL:按 DFT 友好规则写(不写裸门控时钟/组合环/latch,让所有 FF 可串扫描)
  → ⑦ 综合(本章):FF 换成 scan FF、串 scan 链、插 test compression、
        跑 ATPG 生成图样、做 fault-coverage 收敛(stuck-at / transition)
  → ⑧ 物理:scan 链按物理就近重排序、MBIST/BISR 落地
  → ⑨ 签核:DFT/at-speed 测试时钟的时序一起签
  → ⑫ 封测:ATE 用这些 scan/MBIST 图样逐颗筛制造缺陷
```

- **一句话**:DFT 的覆盖率(能筛多干净)在 ⑦ 这里靠 **ATPG + fault model** 定下来,但它的**地基早在 ③/④/⑤ 就埋好了**——前面没留好可测性结构,这里 scan 插不进、覆盖率上不去,到 ⑫ 就**筛不干净、坏片漏给客户**。
- *在 TPU 上*:海量 PE 的触发器在本章被串进 scan 链 + 压缩,SRAM 接 MBIST;ATPG 把覆盖率跑到目标,这份图样一路用到 ⑫ 的 ATE,保证量产测得了、良率算得清。

## 4. 贯穿始终、必须反复权衡的东西

- **PPA 铁三角(综合阶段具体化)**:压时序↑面积/功耗,缩面积↓时序——用约束和选项找平衡。
- **约束松紧**:欠约束(under-constrain)→ 漏优化、后端打回;过约束(over-constrain)→ 面积/功耗暴涨、综合跑不动。
- **QoR vs runtime**:展平 QoR 好但慢/重,层次化快但可能略逊——按规模取舍。
- **综合 ≠ 签核**:综合内 STA 是乐观估计(线延迟靠猜),**别把综合"时序收敛"当成真收敛**——真账在 ⑨。
- **功能不可变**:综合是优化不是改写,**必须 LEC 兜底**。

> 灵魂句:**综合器是照约束优化的机器人——它只会把你"说出来的目标"做到极致,你没说的、说错的,它一概不负责。约束,就是你对它唯一的指挥棒。**

---

## 5. 逻辑综合"怎么做"出来的(方法与工具)

- **综合工具**:Design Compiler / Fusion Compiler(Synopsys)、Genus(Cadence)、**Yosys**(开源)。
- **库与约束**:Liberty(.lib)标准单元库、**SDC** 约束、UPF/CPF 功耗意图。
- **等价检查**:Formality / Conformal(LEC)。
- **时序/DFT**:PrimeTime(签核 STA,⑨ 用)、DFT Compiler / 测试压缩工具。
- **流程**:迭代——跑综合 → 看时序/面积/功耗报告 → 调约束/策略 → 再跑,收敛 QoR;每轮配 LEC。

---

## 6. 逻辑综合阶段的交付物(交给下游什么)

- **门级网表(gate-level netlist)**:综合的核心产物,**⑧ 物理设计的直接输入**。
- **综合后 SDC**(时序约束,后端继续用)。
- **时序 / 面积 / 功耗报告**(QoR 评估)。
- **DFT/scan 信息**(scan 链、压缩结构)。
- **等价检查通过的凭证**(网表 ≡ RTL)。

> 综合的产物是设计的"第一次落地为门"——它把抽象 RTL 变成了后端能摆放、能布线的真实电路,并用等价检查保证这一步没把功能弄丢。

---

## 7. 容易被忽视、却致命的坑

1. **约束错 / 不全(garbage in, garbage out)**:少设时钟、错设 false/multicycle path → 综合优化错方向,虚假违例或漏真违例——综合头号坑。
2. **不做等价检查(LEC)**:综合/ECO 悄悄改了功能没发现 → 网表和 RTL 不一致,带 bug 往下走。
3. **把综合时序当真账**:综合内 STA 乐观(没真实线延迟)→ 以为收敛,后端布线后原形毕露。
4. **过约束 / 欠约束**:过约束面积功耗暴涨、跑不动;欠约束漏优化、后端救不回。
5. **DFT 没规划好**:scan 插不进、链没串好、覆盖率不达标 → 量产测不了、良率算不清。
6. **多 Vt / 功耗意图没用对**:漏电失控,或电源域意图(UPF)和网表对不上。
7. **层次 / 展平选择不当**:展平把大芯片跑爆(runtime/内存),或层次化边界堵死优化、QoR 差。
8. **RTL 里残留 latch / 不可综合结构**:⑤ 没抓住的,综合这里才暴露,返工。

---

## 8. 小结 + 下一步

**逻辑综合 = 把可综合的 RTL,在工艺库和约束下,自动映射并优化成门级网表——设计第一次落地为真实的门和连线;再用形式等价检查证明它和 RTL 功能完全一致。** 它的灵魂是 **"综合器是照约束优化的机器人,约束是唯一的指挥棒(garbage in, garbage out)"**;最该敬畏的真相是 **"综合时序只是估计,真账在签核(⑨)"**;最不可省的一步是 **"LEC 等价检查——保证优化没改功能"**。

下一步(第 ⑧ 步)是 **物理设计**:把这份门级网表**真正摆到硅片平面上**——布局规划、单元摆放、时钟树综合、布线,让每个门有坐标、每条线有走向,设计从"逻辑电路"变成"有物理形状的版图"。
