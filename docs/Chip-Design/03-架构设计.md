# 芯片设计完整流程 · 第 ③ 步:架构设计

> 「芯片设计完整流程」学习系列第三篇。
> 全流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产
```

**架构设计处在「需求」和「具体实现」之间**,是把"这颗芯片要干什么"翻译成"它大体长什么样、由哪些模块组成、能不能达标"的那一层。它的特点是:**还没写一行 RTL,主要靠文档、模型、仿真和估算来做决策**——而这一阶段的决策,**对最终 PPA(性能/功耗/面积)和成本的影响最大、却又最便宜改**。越往后越贵改,架构阶段拍错一个方向,后面几百人月都救不回来。

---

## 1. 架构设计到底是什么

一句话:**在写代码之前,定义芯片的"骨架"——有哪些大模块、它们怎么连、数据怎么流、并用模型证明这套骨架能在给定的功耗/面积/成本预算内达到性能目标。**

它回答三个根本问题:
1. **What**——这颗芯片由哪些功能块组成?(功能划分)
2. **How**——这些块怎么互连、数据/控制怎么流动?(互连与数据通路)
3. **Will it work**——这套结构能不能达标?(PPA/成本建模与验证)

### 一个常被混淆的区分:架构 vs 微架构
- **架构(Architecture)** = 对外可见的"是什么":指令集(ISA)、寄存器/内存映射、对软件可见的行为、块级结构与接口。换 RTL 实现但不改这一层,软件不用动。
- **微架构(Microarchitecture)** = 内部"怎么实现":流水线多少级、cache 多少路组相联、分支预测器怎么做、缓冲多深。对软件透明。

实务里两者边界是模糊的,"架构设计"通常**横跨到块级微架构**。你现在只要记住:架构是"外部契约 + 宏观结构",微架构是"每个块内部的实现选择",这一步重点在前者 + 宏观层面的后者。

## 2. 开始架构设计前,你手里应该有什么(输入)

架构师不是凭空画框图,而是从一份**产品/系统规格**出发。这份输入通常包含:

- **目标场景与工作负载**:这颗芯片跑什么?(手机 SoC?AI 推理卡?网络交换芯片?)有没有代表性 benchmark / trace。(怎么刻画到合适颗粒度,见 §3.1。)
- **性能目标**:吞吐(TOPS、Gbps、帧率)、延迟、主频区间。
- **功耗预算**:TDP、电池续航、散热上限(决定了你能堆多少算力)。
- **面积/成本预算**:die 大小、封装、BOM 成本、目标售价。
- **工艺节点**:哪家 foundry、哪个 node(7nm/5nm/...),这直接决定密度、速度、漏电、成本和 IP 可用性。
- **必须支持的标准/协议**:PCIe/CXL/UCIe、DDR/LPDDR/HBM、以太网、USB、显示接口……
- **软件/生态约束**:要不要兼容某个 ISA、某套驱动、某个编译器栈。
- **项目约束**:排期(time-to-market)、团队规模与能力、可复用 IP、预算。

### 2.1 这些目标到底怎么定出来?(写给没有经验的人)

先卸下一个心理包袱:**这些数字大多不是架构师"拍"出来的,而是从 ①产品需求 / ②系统规格 流下来的**——到了架构这一步,它们通常已经作为约束摆在你面前了。但你一定会追问:"那它们最初是怎么定的?我接到一张白纸该怎么办?"答案是——**不是靠灵感发明,而是靠"三角定位 + 反推",从三个方向把可行区夹出来:**

1. **对标竞品(从上方锚定门槛)**:把目标负载拿到现有最强的芯片上实测(比如 LLM 卡就测 NVIDIA 当代卡),它的 PPA 就是你的**及格线**;你的差异化主张决定"在哪根轴上、超它多少"。这是最靠谱的起点。
2. **客户拉动(从需求侧锚定)**:lead 客户会给你具体的"我要 X 吞吐 @ Y 延迟 @ Z 成本/TCO"。把模糊的"要更快"逼成可量化的 SLA。
3. **从物理与成本反推(从下方夹逼)**:很多目标其实是**不可动的边界倒推出来的**——形态定功耗、功耗定算力、售价定面积。你从这两头(市场拉动 vs 物理/成本)夹逼,落在中间的可行区。

逐根轴,给小白一个"这个数怎么取"的思路:

- **性能**:别写"要快",写成可量化的工作点——"在 batch=B、seqlen=S、FP8 下吞吐 ≥ N tokens/s、TTFT < T"。数从"竞品实测 + 客户 SLA"来。
- **功耗**:几乎总是被**形态(form factor)硬性决定**的天花板,你是"收到"而不是"选":数据中心 PCIe 卡 ~300–700W、OAM 模组更高;边缘盒子几瓦~几十瓦;电池设备 mW~几百 mW。**先认这个天花板,再看天花板下能堆多少算力。**
- **面积/成本**:从**目标售价倒推**——售价 × 目标毛利 → 可接受的芯片成本 → 晶圆价 ÷ 每片可切 die 数(由面积 + 良率决定)→ 反推出 die 面积上限 → 在该节点的晶体管/算力预算。
- **标准/协议**:由你**必须接入的生态**决定,不是自由选——主机接口(PCIe/CXL 第几代)、内存(HBM/DDR 第几代)、die 间(UCIe)。原则:**选在你量产窗口已经成熟、IP 拿得到、带宽够用的那一代,别盲目追最新**(最新的往往 IP 不全、贵、有坑)。
- **软件/生态**:选客户**已经在用**的栈(对 AI 芯片基本就是 PyTorch + 主流推理 runtime)。AI 芯片里生态兼容常常是生死线;除非你有锁定客户愿意陪你自建栈,否则**默认偏向兼容**。

> **给小白最重要的一条:不要从白纸开始,要从一个"参考设计"开始。** 找一颗公开的竞品 / 上一代芯片当锚,在它的参数上做"扰动"(我要更省电 20%?那砍频率、上低精度、加复用——各动哪个旋钮),用**类比 + 锚定 + 反推**收敛。架构不是凭空发明,是**带着约束在已知设计附近做搜索**。

> 关键心态:架构设计本质是**在一堆相互冲突的约束里找一个可行且最优的折中**,不是追求某一项极致。

## 3. 架构设计的核心内容(尽量穷尽地列)

按"子系统"组织,每一项都给:**要决策什么 + 关键考量 + 怎么入手(经验法则) + 放到 LLM 推理卡上长什么样**。一颗复杂 SoC 几乎每一项都要碰;简单芯片可以裁剪。

### 3.1 工作负载分析(一切的起点)

这是整个架构的地基:**架构是为负载服务的,不是为漂亮服务的。** 你必须先搞清楚负载长什么样,才知道资源该往哪儿堆。

**问题一:工作负载要刻画到什么程度?要把网络固定死吗?**

既不能粗到"我要跑 LLM / 跑 DNN 就行",也**不必、且不应该**把某一个确切网络钉死。正确的颗粒度是三件套:

- **工作负载类 + 几个锚点(anchor)模型**:比如"LLM 推理",锚 2–3 个代表性结构(一个 dense 7B、一个 MoE 大模型、一个 70B-class),覆盖你要打的形态即可。
- **工作点(operating point)**:精度(FP16/FP8/INT8/INT4)、batch 大小、序列长度(prompt / 生成各多长)、prefill / decode 占比、延迟目标(TTFT / TPOT)。
- **压硬件的"不变量"(这才是真正定硬件的东西)**:算术强度(FLOP/Byte)、工作集大小(权重多大、KV cache 多大)、带宽需求、通信模式、可并行度。**你不是为"某个网络"设计,而是为这些不变量设计。**

一句话给颗粒度定标准:

> **凡是会改变你某项资源 sizing 的参数,就必须钉住(至少给个范围);不影响 sizing 的,放开。** 太钉死 → 架构很脆,模型一换就崩;太笼统 → 任何资源都算不出该多大。

正确姿势是定**一个主设计点 + 一个覆盖包络**:主设计点 = 你"必须赢"的那个负载(must-win,资源主要为它配);覆盖包络 = 你"不能在上面翻车"的范围(前后向兼容)。例:主打 70B-class、FP8、中等 batch 的 decode 延迟,同时不能在 prefill-heavy、长上下文场景掉链子。

为什么不钉死确切网络:① 你的芯片 2–4 年后才出,那时的"主力模型"现在可能还没诞生;② 但同一类负载的不变量(访存/算力比、KV cache 随 batch×seqlen 线性增长……)是**稳定**的,硬件吃的正是这些不变量。

**问题二:工作负载分析怎么做?理论分析还是跑仿真?**

两者都要,但有顺序——这是一条**由便宜粗略 → 昂贵精确的"保真度阶梯",只爬到当前决策需要的那一级就停**:

1. **解析 / 第一性原理(roofline)**:手算 FLOPs、字节数、算术强度,画 roofline 看"这个负载在我的算力/带宽配比下,到底被算力卡还是被带宽卡"。**早期 80% 的决策在一张电子表格里就定了。** 工具:Excel / Python、Roofline(Intel Advisor)。
2. **在现有硬件上 profile 真实负载**:拿现成 GPU/CPU 把目标 workload 跑起来,量算子占比、访存量、各 kernel 耗时、数据复用率——给你"真实负载长什么样"的地面真值。工具:NVIDIA Nsight Systems / Nsight Compute、PyTorch Profiler、Linux perf、Intel VTune。
3. **解析性能模型 / 加速器 mapping 工具**:为你设想的架构建 cost model,扫不同 dataflow / tiling 的能效与利用率。工具:Timeloop + Accelergy、MAESTRO、SCALE-Sim(脉动阵列)、CACTI(SRAM/cache 建模)、McPAT(CPU 功耗/面积);LLM 专用的有 Calculon、Vidur、LLMServingSim、GenZ。
4. **cycle 级架构模拟器(要看时序、排队、争用、尾延迟时才上)**:gem5(CPU/SoC)、Accel-Sim / GPGPU-Sim(GPU)、Ramulator / DRAMSim3(DRAM)、BookSim / Garnet(NoC)、Astra-Sim(分布式 ML 通信)、SST。慢,但能抓到解析模型抓不到的拥塞和长尾。
5. **FPGA 原型 / 硬件 emulator**:最高保真、最慢,通常已进入微架构/RTL 期。工具:Synopsys ZeBu、Cadence Palladium、Siemens Veloce,以及 FPGA 原型板。

> **先用 roofline 把设计空间砍掉 90%,再用模拟器在剩下的 10% 里分胜负。** 每往上爬一级,成本约 ×10、洞察 +1——能用电子表格回答的,绝不开模拟器。

*在 LLM 卡上*:**decode 阶段是访存受限**——每生成一个 token 都要把全部权重从 HBM 读一遍,算术强度低 → **HBM 带宽是天花板**;**prefill 阶段是算力受限**——大矩阵乘、算术强度高 → **阵列算力是天花板**。KV cache 大小 ≈ 2 × 层数 × 隐藏维 × seqlen × batch × 精度字节(此式按 **MHA** 计;现代模型多用 **GQA/MQA**,应把"隐藏维"换成"KV头数×head_dim",会小数倍到一个数量级,**MLA**(如 DeepSeek)压成低维潜向量更小),它决定了 HBM 容量,也决定了"能同时服务多少请求"。这几个数一算,HBM 选几代几栈、阵列开多大、片上 SRAM 给多少,就都有了下手点。

### 3.2 功能划分 / 模块分解
- **要决策**:有哪些大块(CPU、GPU、NPU/专用加速器、内存控制器、IO、互连、外设、安全模块……);**软硬件划分**(哪些用硬件、哪些交固件/软件);**可编程 vs 固定功能**。
- **关键考量 & 怎么定**:用 §3.1 的热点画像驱动——只把同时满足"(a) 占运行时间大头、(b) 规格稳定不易变、(c) 专用化能带来数量级能效/面积收益"的核心算子**硬化成专用引擎**,其余留给可编程单元 / 软件。即使硬化,也要留"逃生通道"(能被软件 bypass),防止流片后算子演进把你架空。
- *LLM 卡*:吃掉绝大部分 FLOPs 的矩阵乘 / attention → 专用张量引擎硬化;采样、调度、KV 管理、稀有算子 → 可编程核 + 软件。

### 3.3 计算架构
- **要决策**:ISA(若含通用核:RISC-V / ARM / x86 / 自研);核数与核型(同构 / 异构如大小核);数据通路宽度、SIMD/向量、**脉动阵列**(AI 芯片)、专用引擎;宏观流水线与并行策略(指令级/数据级/线程级)。
- **关键考量 & 怎么定**:先定**并行维度**——负载主要是数据并行(大矩阵)、线程并行还是指令并行?这决定你"堆阵列"还是"堆核"。算力 sizing 从 roofline 反推:目标吞吐 ÷ 单 PE 算力 ÷ 频率 = 需要多少 MAC,再看面积/功耗预算容不容得下。精度决定 PE 设计(支持哪些数、要不要混合精度/稀疏)。**ISA 默认复用成熟的(ARM/RISC-V),别轻易自研**——自研 ISA 等于自建一整套编译器与软件栈。
- *LLM 卡*:核心是大 tensor/systolic 阵列做 GEMM + 向量单元做 norm/softmax/激活;主打 FP8/INT;阵列规模由 prefill 算力目标定,频率被功耗天花板压着。

### 3.4 存储层次(memory hierarchy)——常常是性能的真正决定者
- **要决策**:Cache 层次(L1/L2/L3 大小、组相联、包含/排他、替换);片上 SRAM/scratchpad(大小、bank、带宽);片外内存(DDR/LPDDR/HBM/GDDR 选型、几 channel、带宽);内存控制器;一致性模型(cache coherence,MESI 等);内存一致性模型(对软件可见的读写顺序);地址映射。
- **关键考量 & 怎么定**:先用 §3.1 的算术强度 + 工作集判断自己是**带宽受限还是容量受限**,再分配 HBM/SRAM。片上 SRAM 的价值是**提高复用、减少回 HBM**:用 reuse 分析定 tile 大小 → 定 SRAM 容量与带宽(SRAM 很吃面积,够用即可)。片外内存按"GB/s 带宽需求"反推几代几栈。加速器内部多用 **scratchpad + 软件管理**(不做硬件 cache 一致性,省面积省复杂度);只有通用多核才需要 MESI 类协议。
- *LLM 卡*:decode 访存受限 → HBM 带宽是命门(HBM3E 每栈约 1.2 TB/s,选几栈看目标);权重 + KV cache 决定 HBM 容量;片上大 SRAM 缓 tile,把阵列利用率喂起来。

> 一句行话:"现代芯片性能往往不卡在算力,而卡在**喂数据**(memory wall)。"存储与带宽的架构常常比算力本身更决定成败。

### 3.5 片上互连 / NoC(Network-on-Chip)
- **要决策**:拓扑(总线 / crossbar / NoC 的 mesh、ring、tree);带宽、延迟、**QoS、仲裁**;协议(AXI、CHI 等),一致性 fabric vs 非一致性。
- **关键考量 & 怎么定**:先估**对分带宽(bisection bandwidth)**需求——画一张"通信矩阵",看哪些块之间有大流量(计算↔HBM、核↔核)。规模小用总线/crossbar;块多、流量大用 mesh/ring NoC(延迟敏感选低直径的 ring,带宽敏感选宽 mesh)。给 HBM 主干流量留**专用高优先级通道 + QoS**,别让控制流量饿死数据流量。
- *LLM 卡*:阵列↔HBM 是主干流量,NoC 的任务就是把 HBM 带宽**无损**送进阵列,否则再大的算力也会饿死。

### 3.6 数据流 / 数据通路(dataplane)
- **要决策**:数据怎么搬——**DMA 引擎**、缓冲/FIFO、流控、背压。
- **关键考量 & 怎么定**:用 **DMA + 双缓冲(ping-pong)**让"搬运与计算重叠",把访存延迟藏起来;端到端找最窄那一段(瓶颈就是吞吐上限),逐段把带宽配平;用背压/流控防止快段冲垮慢段。
- *LLM 卡*:权重/KV 从 HBM 经 DMA 流入 SRAM,与阵列计算重叠,调度让阵列尽量零空转。

### 3.7 IO 与连接
- **要决策**:高速 **SerDes**(PCIe、CXL、以太网、**UCIe** 芯粒互连);内存接口 PHY;低速外设(I2C/SPI/UART/GPIO)。
- **关键考量 & 怎么定**:主机接口带宽要 ≥ 喂数据需求,否则 PCIe 自己成瓶颈;多卡互联需求大就上专用高速链路。SerDes 是高难度模拟 IP,**优先买成熟 PHY**,代际选量产窗口里成熟可得的那一代。
- *LLM 卡*:PCIe/CXL Gen5/6 连主机;多卡 scale-out 需要高带宽的卡间/die 间链路(NVLink 类 / UCIe)。

### 3.8 时钟架构(clocking)
- **要决策**:时钟域划分、各域频率;**PLL/DLL**、**跨时钟域(CDC)**处理;**DVFS**(动态调压调频)。
- **关键考量 & 怎么定**:按频率/功耗需求切时钟域;**跨域信号必须做 CDC 同步**(漏一个就是极难定位的隐蔽硅 bug)。高算力块跑高频但受功耗约束,IO 按协议定频;DVFS 让闲时降频降压省电。
- *LLM 卡*:阵列一个高频域,HBM PHY、PCIe 各自独立域,边界处统一做 CDC。

### 3.9 功耗架构(power)
- **要决策**:电压域 / 功耗域划分;**门控**(时钟门控、电源门控);**PMU** 与低功耗状态(P-state/C-state);架构级**供电网络(PDN)**与散热预估。
- **关键考量 & 怎么定**:先做**功耗预算分解**——把总 TDP 按块切成配额(见 §6),每块自证不超。原则是"不用就关":细粒度时钟门控 + 粗粒度电源门控,闲态进低功耗态。**能效优化的重点是"少搬数据"**(数据搬运比计算更耗能),而不是一味堆算力。
- *LLM 卡*:几百瓦封顶意味着阵列算力上限其实是被功耗定的;靠数据复用 + 低精度把 perf/W 提上去。

### 3.10 复位与初始化架构
- **要决策**:复位域、复位时序、**启动流程(boot flow)**、安全启动。
- **关键考量 & 怎么定**:规划复位域 + 上电次序(power sequencing)+ boot flow——谁先复位、固件从哪加载、外设何时可用;整个流程要**可重复、可调试**(芯片回来 bring-up 时全靠它)。
- *LLM 卡*:上电 → 安全启动校验固件 → 初始化并训练 HBM/PHY → 驱动可见、可下发任务。

### 3.11 安全架构(security)
- **要决策**:**可信根(root of trust)**、安全 enclave、加解密引擎、密钥管理、隔离(类 TrustZone)、侧信道考量。
- **关键考量 & 怎么定**:**先定威胁模型**(防谁、防什么)再加机制,别无脑堆。常见基本盘:可信根 + 安全启动(防固件篡改)、加解密引擎、密钥隔离。
- *LLM 卡*:数据中心场景重点是固件完整性、(多租户下)模型与数据隔离;具体按客户合规要求定。

### 3.12 RAS(可靠性/可用性/可维护性)
- **要决策**:**ECC、奇偶校验**、错误检测与纠正、冗余;汽车/工业还要满足**功能安全(ISO 26262 等)**。
- **关键考量 & 怎么定**:按"出错后果多严重 × 出错率"定等级——大块 SRAM/HBM 必上 ECC,链路加 CRC/重传。数据中心要**可观测**(能报错、能定位、能隔离坏块)以维持大规模可用性;车规/工业另有功能安全硬性要求。
- *LLM 卡*:HBM、片上大 SRAM 上 ECC;链路 CRC;预留 RAS 遥测供运维。

### 3.13 可测性架构(DFT, Design-for-Test)
- **要决策**:扫描链(scan)、**BIST(MBIST/LBIST)**、JTAG/边界扫描,以及**覆盖率目标**。
- **关键考量 & 怎么定**:**架构期就要预留**——scan 链、测 SRAM 的 MBIST、JTAG,并定下覆盖率目标。不做的代价:测试成本飙升、坏片漏到客户、良率算不清。
- *LLM 卡*:海量片上 SRAM 必须 MBIST;大芯片靠 scan + 测试压缩来控制量产测试时间。

### 3.14 封装 / Chiplet 架构(现代重点)
- **要决策**:单片(monolithic) vs **chiplet(2.5D/3D)**;die 怎么切分、**die-to-die 接口(UCIe/BoW/自研)**、interposer/substrate、跨 die 散热。
- **关键考量 & 怎么定**:当芯片太大(接近 reticle 上限,约 858 mm²,即 26×33mm 曝光场)或良率太低 → 切成 chiplet 用 2.5D/3D + interposer 拼;HBM 几乎必然要用 **2.5D 硅 interposer** 与逻辑共封;die 间接口选 UCIe;别忘了算跨 die 的热与成本。这一层决定了**成本/良率/可扩展性**的大盘。
- *LLM 卡*:大算力 die + 多颗 HBM 经硅 interposer 做 2.5D 共封;算力部分还可拆成多 chiplet 以提良率、上规模。

### 3.15 软件/编程接口(常被硬件人忽略)
- **要决策**:编程模型、**寄存器映射(CSR)**、驱动模型、ISA 扩展。
- **关键考量 & 怎么定**:架构期就要和软件团队一起定:算子怎么映射到硬件、CSR 怎么排、驱动模型、要不要 ISA 扩展。**优先适配主流栈**(PyTorch + 主流编译器/runtime);自定义越多,软件负担越重,生态越难起来。
- *LLM 卡*:暴露给主流推理框架的算子接口 + 编译器把计算图下发到阵列;CSR 控制调度与电源。**"芯片再强,软件用不起来就是废铁"——生态兼容既是卖点也是生死线。**

### 3.16 第三方 IP 采购、集成与 qualification(buy 来的那一大半)
- **要决策**:哪些块**自研**、哪些**买现成 IP**(硬 IP / 软 IP);买谁的、买哪一版;以及怎么把它**集成进自己的芯片**、怎么确认它在你的工艺/场景下**真的能用**。现代 SoC 里第三方 IP(CPU/GPU 核、SerDes/PCIe/CXL/UCIe/HBM/DDR PHY、PLL、标准单元库、SRAM/ROM 编译器、foundation IP)常常占了 die 面积与项目风险的**一大半**——它不是小配角,而是一条独立工作流。
- **关键考量 & 怎么定**:
  - **选型**:按 **PPA + 目标 foundry/节点是否支持 + 可获得性 + license 成本 + 供应商口碑/路线图**筛;关键模拟 IP(SerDes/PHY)是高难硬 IP,**优先买成熟、已在目标节点流过片(silicon-proven)的**,别赌没人量产过的新版。
  - **软 IP vs 硬 IP**:软 IP(可综合 RTL,如 CPU 核)灵活、能跟着你的流程走,但要自己实现/签核;硬 IP(已做成版图的 GDS 黑盒,如 PHY/SRAM)拿来即用、PPA 有保证,但**不可改、必须按它的物理/供电/布局约束摆**。
  - **集成**:对齐接口(协议/时序契约)、**CDC**(IP 常自带时钟域)、**复位/上电次序**、功耗域(UPF 边界)、DFT(IP 的 scan/MBIST 怎么接进你的链)、地址映射;硬 IP 还要管好它的**模拟供电、参考时钟、布局禁区**。
  - **qualification(最易漏的一步)**:买来的 IP 要在**你的工艺角、你的用例**下验证可用——跑 IP 自带验证套件、做接口一致性/合规测试、确认 PPA 兑现、收集供应商 errata 与 silicon-proven 证据、确认出 bug 有支持渠道。**"买了"不等于"能用",qualification 才是。**
- **怎么入手**:架构期就定 make-vs-buy 清单、锁 IP 供应商与版本(它影响排期与 BOM);把 IP 当"带契约的黑盒",在自己这侧把接口/CDC/供电/DFT/UPF 都对齐;**第二供应源**(关键 IP/内存能否多源)也在这里评估,影响量产风险。
- *LLM 卡*:PCIe/CXL/UCIe/HBM PHY、PLL 这些高速模拟 IP 几乎都**外购成熟硬 IP**(自研不划算、风险高),按各自供电/参考时钟/布局约束集成进 die,接口做一致性测试、在目标节点做 qualification;自研的只留真正差异化的部分(脉动阵列、调度、片上网络)。

## 4. 贯穿始终、必须反复权衡的东西

上面是"有哪些内容",这里是"做每个决策时脑子里要挂着的标尺":

- **PPA 铁三角**:性能↑通常面积↑、功耗↑。架构就是在这三者间找点。
- **成本**:面积→die 成本、封装成本、掩模/NRE、IP 授权费。
- **进度 / time-to-market**:最优架构若做不完也是零分。
- **风险**:全新自研 IP vs 成熟复用 IP;激进架构 vs 保守可控。
- **可扩展性 / 产品家族**:这颗能不能裁/扩出一系列衍生品(架构期就要留好配置点)。
- **可验证性**:这套结构验证团队验得过来吗?复杂度爆炸会拖死验证。
- **可制造性 / 良率**:太大、太激进会掉良率。
- **工艺节点权衡**:先进节点性能/密度好,但贵、漏电、IP 不全、设计难度大。
- **买 vs 自研(IP reuse)**:复用省时间省风险,自研才有差异化。
- **灵活性 vs 效率**:可编程换弹性,专用化换能效/面积。

> 这些不是"选一个",而是**同时在脑子里转**,每个架构决策都在动这一组旋钮。

## 5. 架构设计是"怎么做"出来的(方法与工具)

它不是一次画完,而是**迭代收敛**:需求 → 探索 → 建模 → 评估 → 调整 → 收敛 → 定稿。常用手段:

- **解析模型 / 电子表格**:快速估 PPA、带宽、roofline——最先用、用得最多。
- **性能模拟器**:cycle-accurate 或 trace-driven,跑真实负载看吞吐/延迟(具体的保真度阶梯与工具清单见 §3.1)。
- **设计空间探索(DSE)**:系统地扫参数(核数、cache 大小、带宽、阵列尺寸……)找最优点。
- **虚拟原型 / SystemC-TLM**:给软件团队提前一个可跑的模型,软硬件并行开发、缩短上市时间。
- **功耗/面积估算模型**:基于工艺库外推(CACTI、McPAT、Accelergy 等)。
- **FPGA 原型 / 仿真器(emulation)**:后期更高保真验证(已偏微架构/RTL 期)。
- **对标(benchmarking)**:拿现有竞品/上一代做参照系——也是 §2.1 里定目标的同一把尺。

## 6. 架构阶段的交付物(交给下游什么)

- **架构规格文档(Architecture Spec)**:权威说明书。
- **顶层框图** + 各块接口定义。
- **memory map / 寄存器映射**。
- **PPA 预算分解**:把总目标拆成每个块的"配额"(blockA 多少面积/功耗/频率)。
- **DSE 结果**:用数据证明"为什么选这套"。
- **微架构规格**:每个块往下一层的实现要求,**这就是交给 RTL 团队的输入**。

## 7. 容易被忽视、但很关键的几点

1. **验证驱动架构**:在定结构时就想"这怎么验",别造出验不了的怪物。
2. **DFT/可测性要在架构期介入**,不是后面补——补不进去会掉良率、加测试成本。
3. **软件/固件团队必须早期参与**,接口(寄存器、编程模型)是架构产物。
4. **预留 margin 与配置点**:频率/带宽/功耗都要留余量;产品家族要留可裁剪的旋钮。
5. **bring-up 与可调试性**:芯片回来后怎么 debug?要有片上观测、调试接口——架构期就得想。
6. **数据搬运常比计算更耗能**:能效优化的重点常在"少搬数据/就近计算",而非堆算力。
7. **第二来源 / 供应链**:关键 IP、内存、封装能不能多源,影响量产风险。
8. **架构错误最贵**:这一步省下的几周,后面可能用几个月偿还——慢一点、建模充分一点是值得的。

## 8. 小结 + 下一步

**架构设计 = 在写 RTL 之前,用模型和探索,把"产品要什么"翻译成"由哪些块组成、怎么连、能不能在功耗/面积/成本内达到性能",并产出一份能指导后面所有工作的规格。** 它的灵魂是**在 PPA/成本/进度/风险的多重约束下做折中**,核心方法是**建模 + DSE 迭代收敛**;而对新手最实用的两条是——**先 roofline 砍掉 90% 的设计空间再上模拟器**、**从参考设计出发而不是白纸开始**。

下一步(第 ④ 步)是 **微架构设计**:把每个块的内部"怎么实现"定下来(流水线级数、cache 内部结构、状态机、缓冲深度……),它是架构到 RTL 之间的桥。
