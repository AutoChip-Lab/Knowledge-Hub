# Multi-Agent Knowledge · Step ⑫: Learning routes and references

> This chapter organizes the first eleven chapters into three learning paths: design, engineering, and research, and provides exercises, acceptance methods, and verifiable references. General single-agent content only appears as pre-knowledge when directly supporting team collaboration.

## 1. Multi-agent learning and referencing core terms

This chapter uses four terms to run through learning and acceptance: learning route determines the sequence, Baseline provides complexity comparison, Ablation checks whether the role or communication side really contributes, and Acceptance check determines whether the project reaches executable standards.

| Term | Working definition | Key idea |
|---|---|---|
| Learning path | Learning route | Arrange reading and practice order according to multi-agent concept dependencies. |
| Ablation | Ablation experiment | Delete a role, communication side or review link to measure whether it really contributes. |
| Baseline | Compare to the baseline | Determine whether a complex team is worth it with a single model, a fixed pipeline, or a simpler topology. |
| Acceptance check | Acceptance check | Executable standards to determine whether the collaboration system meets the standards. |

## 2. Three learning routes: design, engineering and research

The three routes are not mutually exclusive courses, but three starting points. First choose the main line based on the current goal, and then read the chapters it depends on; no matter which route you choose, you must ultimately be able to explain the team structure, operational evidence, and simple baselines.


### 2.1 Design route: team organization and collaboration mechanism

<div class="concept-card">
<div class="concept-line">① Panorama, definitions and necessary theories</div>
<div class="concept-line"> → ② Role and team design → ③ Task decomposition and planning</div>
<div class="concept-line"> → ④ Collaboration topology and Blackboard / Debate topic </div>
<div class="concept-line"> → ⑤ Communication protocol → ⑥ Routing and handover </div>
</div>

This route is suitable for people designing an Agent team for the first time. The goal is not to learn a framework API first, but to be able to explain: why multiple roles are needed, how tasks are distributed, along which edges information flows, who has the final decision-making power, and where to recover from failure.

### 2.2 Engineering Route: System Verification and Production Constraints

<div class="concept-card">
<div class="concept-line">⑤ Communication protocol → ⑥ Routing and handover</div>
<div class="concept-line"> → Blackboard / Debate Topic → ⑦ Observation, Evaluation and Debugging</div>
<div class="concept-line"> → ⑧ Safety and production → ⑨/⑩ Real case </div>
</div>

This route is suitable for people who want to bring multi-Agent systems online. Once completed, you should be able to save traces for each cross-role call, define schemas for handover packages and shared status, and set deterministic boundaries for permissions, budgets, retries, manual approvals, and rollbacks.

### 2.3 Research line: Effectiveness evaluation of complex collaboration

<div class="concept-card">
<div class="concept-line">① Necessary theory → ④ Collaboration topology</div>
<div class="concept-line"> → Debate / Dynamic Topology Special Topic → ⑦ Observation, Evaluation and Debugging</div>
<div class="concept-line"> → ⑪ Framework and project map → Paper and source code verification </div>
</div>

Research routes must include same-budget baselines, role or communication edge ablation, different random seeds, failure type analysis, and cost reporting. By only comparing the final scores of "One Agent" and "More Agents", it is impossible to determine whether the benefit comes from the organizational structure, more tokens, or more model calls.

## 3. Five progressive multi-agent exercises

The learning path provides a reading sequence, and the exercises are responsible for turning concepts into acceptable behaviors. The following five exercises progress from dual-role handoffs to routing, shared state, full teams, and production failures, retaining a simpler baseline at each step.


| Exercise | Collaboration mechanism to be implemented | Minimum acceptance conditions |
|---|---|---|
| Dual-role research review | Researcher produces evidence, Reviewer independently verifies | Uncited conclusions will be returned; the product version will be retained for each rework. |
| Supervisor Routing | Select Workers by Task, Capability, Permission, and Risk | Report accuracy on a fixed set of routes; hard rules cannot be overridden by the model. |
| Shared blackboard | Multiple roles share status by reference instead of full conversation | Can identify and handle concurrent writes, expired facts and conflicting conclusions. |
| Software factory or research team | Roles, task maps, handovers, and reviews form a complete closed loop | Each stage has input contracts, products, responsible persons, and acceptance gates. |
| Production drills | Injection timeouts, duplicate messages, overrides and supervisor failures | Can be recovered from checkpoints; side effects are not repeated; the entire link is auditable. |

Keep a simpler baseline for each exercise and record success rate, cost, latency, communication rounds, and human intervention rate. If a complex topology does not exceed the baseline within the same budget, the team should prioritize simplification.

## 4. Final project acceptance closed loop


The final project should deliver at least the following materials:

1. Task graph: includes dependencies, roles, acceptance conditions, timeouts and replanning triggers.
2. Topology diagram: Mark task dependencies, information dependencies and decision-making power respectively.
3. Protocol Schema: covering messages, handover packages, shared states, product references and idempotent keys.
4. Trajectory sample: contains at least one success, one rework, one unauthorized rejection and one failure recovery.
5. Evaluation report: includes simple baseline, same-budget comparison, ablation, cost and failure classification.
6. Security Checklist: Describes role permissions, data boundaries, manual approval, rollback, and audit retention policies.

The key to acceptance is not that "all agents say the task is completed", but that the final product passes independent inspection, process events can be traced to responsible roles and input evidence, and failed paths can be recovered without repeating side effects.

## 5. Basic papers and reviews of multi-agent

After completing the exercise and project acceptance design, the conceptual boundaries need to be checked against the original data. First, we start from the traditional multi-agent foundation and review to establish common issues such as environment, coordination and local information, and then enter the LLM multi-agent method.


- Shoham, Yoav; Leyton-Brown, Kevin. [Multiagent Systems: Algorithmic, Game-Theoretic, and Logical Foundations](https://www.masfoundations.org/).
- Wooldridge, Michael. [An Introduction to MultiAgent Systems](https://www.wiley.com/en-us/An+Introduction+to+MultiAgent+Systems%2C+2nd+Edition-p-9780470519462).
- Albrecht, Stefano V.; Christianos, Filippos; Schäfer, Lukas. [Multi-Agent Reinforcement Learning: Foundations and Modern Approaches](https://www.marl-book.com/).
- Chen et al. [A Survey on LLM-based Multi-Agent System](https://arxiv.org/abs/2412.17481), 2024.
- Han et al. [LLM Multi-Agent Systems: Challenges and Open Problems](https://arxiv.org/abs/2402.03578), 2024.

The first three provide traditional foundations such as environment, local information, games, and coordination; the last two put these issues back into the context of communication, illusion, scalability, and evaluation in LLM teams. When reading, you should distinguish between textbooks, formal papers, and preprint status.

## 6. LLM multi-agent framework and collaboration method

The foundational literature explains why multi-agent coordination is needed, and the work listed in this section further illustrates how LLM roles communicate, review, and form teams. When reading, you should also record the experimental settings, baselines and code entries.


- Li et al. [CAMEL: Communicative Agents for “Mind” Exploration of Large Language Model Society](https://arxiv.org/abs/2303.17760), 2023.
- Qian et al. [ChatDev: Communicative Agents for Software Development](https://arxiv.org/abs/2307.07924), 2023.
- Hong et al. [MetaGPT: Meta Programming for A Multi-Agent Collaborative Framework](https://arxiv.org/abs/2308.00352), 2023.
- Wu et al. [AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation](https://arxiv.org/abs/2308.08155), 2023.
- Chen et al. [AutoAgents: A Framework for Automatic Agent Generation](https://arxiv.org/abs/2309.17288), 2023.
- Du et al. [Improving Factuality and Reasoning through Multiagent Debate](https://arxiv.org/abs/2305.14325), 2023.
- Zhang et al. [AgentPrune: Reducing Redundancy in Graph-Based Multi-Agent Systems](https://arxiv.org/abs/2410.02506), 2024.

The framework name of the paper is not a substitute for design comparison. When verifying, record: whether the roles are fixed, whether the communication diagram is fixed, where the shared state is, who executes the termination condition, whether the baseline is the same as the budget, and whether the conclusion comes from the paper experiment or teaching inference.

## 9. Summary of the entire series: from team design to production verification

The value of multi-agent systems does not lie in "allowing more agents to participate", but in turning complex tasks into organizational processes that can be divided, communicated, verifiable, recoverable and governable. The most reliable learning endpoint is to be able to answer two questions at the same time: why the complex team outperforms the simple baseline, and when it no longer outperforms the baseline, which role, communication edge or layer of dynamic mechanism should be deleted.
