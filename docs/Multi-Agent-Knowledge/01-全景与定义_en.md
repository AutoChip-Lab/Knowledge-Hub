# Multi-Agent Knowledge · Step 1: Panorama and Definition

> Multi-agent is not about "opening more chat windows". It is a system design method that organizes tasks, roles, communication, environment, status and review.





## 1. Agent and core terms of multi-agent systems

When you first encounter the terms below, use these working definitions as a quick reference; later sections cover their properties and engineering implications.

| Term | Working definition | Key idea |
|---|---|---|
| Agent | Intelligent body | An operating unit that can choose actions based on goals, environmental observations, and available tools. |
| Multi-Agent System / MAS | Multi-agent system | A system in which multiple agents collaborate to complete tasks through communication or a shared environment. |
| Environment / State | Environment / State | External facts and task progress that multiple Agents work together. |
| Observation | Observation | The local state that an Agent can actually see. |
| Coordination | Coordination | Let interdependent actions form acceptable results according to constraints. |
| Planner | Planner | A person or module that breaks user goals into executable steps. |
| Router / Supervisor | Router / Supervisor | Decide to whom tasks are assigned, and when to upgrade, terminate, or reschedule. |

## 3. Minimal definition of Agent




Let’s first understand it with a definition that is small enough and can be expanded later:

> Agent is a computational entity that can choose actions based on goals in the environment and update subsequent behaviors based on observation results.

In LLM scenarios, Agents usually include:

| Parts | Function |
|---|---|
| Character description | Decide from what perspective it views the task |
| Goal | Decide what it is going to optimize |
| Tools | Decide what it can do to the outside world |
| Memory | Decide what history it can carry forward with it |
| Strategy | Determines how it plans, calls tools, stops, or calls for help |
| Observations | Tool results, environment status, other Agent messages |

A single Agent is usually:

<div class="concept-card">
<div class="concept-line">LLM + instructions + tools + memory + loop</div>
</div>

A minimal Python representation:

```python
from dataclasses import dataclass, field
from typing import Callable, Any

Tool = Callable[[dict], str]

@dataclass
class AgentSpec:
    name: str
    role: str
    goal: str
    tools: dict[str, Tool] = field(default_factory=dict)
    memory: list[str] = field(default_factory=list)

    def system_prompt(self) -> str:
        return f"""
You are {self.name}.
Role: {self.role}
Goal: {self.goal}
Available tools: {', '.join(self.tools) or 'none'}
Memory:
{chr(10).join('- ' + m for m in self.memory[-5:])}
""".strip()
```

<div class="code-explanation">
<div class="code-explanation-title">Python code description</div>
<p><strong> Purpose: </strong> uses <code>AgentSpec</code> to collapse the agent's name, role, goals, tools, and memories into a versionable object. <strong> Execution process: </strong><code>system_prompt()</code> Read these fields, only take the latest five memories and spell them into system prompts. <strong> Key points: </strong> This is just the specification layer. The real model calling, permission verification and loop control still need to be implemented by the runtime. </p>
</div>




This code is not yet a complete Agent, but it has already expressed the four core dimensions of "role, goal, tool, and memory".

## 4. Applicable conditions for multiple agents


First, use a single Agent as the default baseline. Only when the task truly requires division of capabilities, parallel work, independent review, or permission isolation, it is worthwhile to add multiple Agents.

Suitable for:
- Tasks naturally have multiple professional perspectives, such as "requirements, architecture, coding, testing, and security review."
- Multiple candidate solutions and reviews are required, such as design review, research review, and strategy selection.
- Long processes require stage gating, such as software development, data analysis, and contract review.
- Tool permissions need to be separated. For example, only the execution agent can run commands, and the review agent can only read results.
- Need an explainable organizational process, such as wanting to see what each character does.

Not suitable for:
- The tasks are short and can be completed in a single question and answer session.
- Taking apart multiple characters just to "look advanced".
- No explicit stopping condition.
- Without logs and reviews, it is impossible to judge whether multi-Agent is really better.
- Each Agent uses the same prompt words, the same tools, and the same context, just with different names.

A simple judgment function:

```python
def should_use_multi_agent(task: dict) -> bool:
    signals = 0
    signals += task.get("requires_multiple_expertise", False)
    signals += task.get("needs_independent_review", False)
    signals += task.get("long_horizon_steps", 0) >= 5
    signals += task.get("tool_risk_level", "low") in {"medium", "high"}
    signals += task.get("needs_parallel_research", False)
    return signals >= 2
```

<div class="code-explanation">
<div class="code-explanation-title">Python code description</div>
<p><strong>Purpose: </strong> Use five task signals to roughly judge whether it is worth introducing multi-agent. <strong> Execution process: </strong> Boolean conditions are accumulated by 0 or 1 in Python. Long processes and medium and high risks each contribute a signal; only when two signals are reached, <code>True</code> is returned. <strong> Key Points: </strong> Thresholds are a teaching example and production systems should be calibrated against real success rate, cost and latency data. </p>
</div>

The criterion is not "whether multiple roles can be separated", but "whether measurable gains in quality, speed, isolation or auditability can be achieved after the separation." If you can't tell where the revenue is coming from, you should continue to use a single agent.

## 5. Components of multi-agent systems




Multi-Agent System, referred to as MAS, can be understood like this:

> MAS is a system composed of multiple partially autonomous Agents that collaborate to complete tasks through communication or a shared environment.

It has more things than single Agent:

| New Question | Example |
|---|---|
| Collaboration topology | Who speaks first, who can call whom, and who has the final say |
| Communication protocol | Message format, round, status, reference, failure retry |
| Task allocation | How Planner divides tasks into different roles |
| Conflict Resolution | What to do when two experts come to different conclusions |
| Shared status | Which intermediate results can be seen by everyone |
| Cost Control | Multiple Agents will amplify tokens, tool calls, and delays |
| Security Boundary | Whether the permissions of each Agent are different |

A multi-agent system is:

<div class="concept-card">
<div class="concept-line">multi-agent (LLM + role-specific instructions + tools + memory + loop) + orchestrator + state managing + message passing</div>
</div>

## 7. Collaboration topology of multi-Agent team


**Topology** describes the connection and control relationships between Agents. It’s not just “who can chat with whom”, it also determines four things:

1. **How do tasks flow**: Are tasks delivered in a fixed order, or are they assigned dynamically by supervisors?
2. **How is the information visible**: Does the character only see the output of the previous step, or does it read and write the shared state together?
3. **Where does the decision-making power lie**: a single supervisor, a hierarchy of managers, market rules, or an independent referee making the final decision?
4. **How to return after failure**: Should the error go back to the previous step, return to the supervisor, write on the blackboard, or enter the next round of debate?

Therefore, the same number of agents does not mean that the system behavior is the same. Three agents can be arranged in a pipeline, or they can work around a supervisor, or they can read and write from the same shared workspace at the same time; the latency, cost, error propagation, and responsibility boundaries of these systems will be different.

Start by developing an intuition for six common shapes:

| Topology | Basic organization | Suitable mission characteristics | Main risks |
|---|---|---|---|
| Chain / Pipeline | The product is handed over to the next role in a fixed order | The steps are stable and the dependencies are clear | Upstream errors will be propagated along the chain |
| Supervisor system / Supervisor | Supervisor centrally assigns, collects and integrates results | Subtasks can be parallelized, but unified decision-making is required | Supervisor becomes a bottleneck or single point of failure |
| Hierarchy | The general leader manages members through multiple sub-team leaders | The task scale is large and the team needs to be hierarchical | Information may be lost between levels |
| Blackboard / Blackboard | Multiple Agents read and write the same shared state | Evidence and intermediate states need to be continuously summarized | Versions, permissions and conflict management are required |
| Debate | Multiple opinions respond to each other after several rounds, and then the referee makes a decision | High-risk judgment, comparison of mutually exclusive solutions | High communication cost, possible repeated idling |
| Market / Market | Candidate Agents bid for tasks based on ability, cost or load | There are many candidate roles and tasks need to be dynamically allocated | Scoring and authorization rules are difficult to design |

When choosing a topology, don't ask "which framework is the most popular" first, but first look at the task structure:

- There are many fixed steps, so start with the chain method first.
- Subtasks can be parallelized and need to be integrated uniformly, taking supervisory control into consideration.
- If the team is large and contains multiple sub-teams, consider the hierarchical structure.
- Multiple characters must continuously share evidence and progress, consider Blackboard.
- The risk of the conclusion is high, and it is necessary to actively expose counterexamples, join the debate and make independent referees.
- Tasks and loads vary greatly, requiring dynamic selection of personnel and consideration of market mechanisms.

Real systems often combine topologies. For example, a supervisor can assign tasks to three parallel sub-teams, use a chain process within the sub-teams, and then all teams share evidence through a blackboard, initiating debate only at key decision points. Chapter 4 will draw each of these topologies and further discuss costs, failure modes, state governance, and selection methods.

When reading subsequent topology diagrams, always check four questions: **Who can send what information to whom, who can modify the shared state, who can reject or terminate, and who is responsible for the final outcome**. If the diagram only has character names and connections but no rules, it is not an executable system design.

## 8. Differences between LLM multi-agent and traditional MAS




Traditional MAS often focuses on:
- Distributed decision-making.
- Auctions and negotiations.
- Formal agreement.
- Multi-robot control.
- Multi-agent reinforcement learning.

LLM-MAS pays more attention to:
- Character prompt words.
- Natural language communication.
- Tool call.
- Task decomposition and self-correction.
- Contextual windows and memory compression.
- Judging, debating, voting and refereeing.
- Observability and production safety.

The biggest changes are: **The communication language has changed from a strict protocol to a mixture of natural language + structured JSON. **

This makes the system easier to set up and easier to lose control of. Therefore, engineering needs to use schema, state machines, tests, and logs to tighten the boundaries again.

## 9. Minimum multi-Agent message collaboration example




The following is not a call to a real LLM, but a demonstration of the system structure.

```python
from dataclasses import dataclass

@dataclass
class Message:
    sender: str
    receiver: str
    kind: str
    content: str

class EchoAgent:
    def __init__(self, name: str, role: str):
        self.name = name
        self.role = role

    def handle(self, msg: Message) -> Message:
        return Message(
            sender=self.name,
            receiver="supervisor",
            kind="result",
            content=f"[{self.role}] received: {msg.content}",
        )

agents = {
    "researcher": EchoAgent("researcher", "collect evidence"),
    "critic": EchoAgent("critic", "find risks"),
}

task = Message("user", "researcher", "task", "Compare AutoGen and MetaGPT")
research_result = agents["researcher"].handle(task)
critic_result = agents["critic"].handle(
    Message("supervisor", "critic", "review", research_result.content)
)

print(research_result)
print(critic_result)
```

<div class="code-explanation">
<div class="code-explanation-title">Python code description</div>
<p><strong> Purpose: </strong> Use a minimal program that does not rely on a real large model to demonstrate the collaborative structure of "researcher output, supervisor transfer, and critic review". <strong> execution process: </strong><code>Message</code> saves the sender, receiver, type and body, and two <code>EchoAgent</code> processes the tasks and review messages in sequence. <strong>Key points: </strong>The example focuses on responsibilities and message boundaries, not model capabilities; the real system also needs to add routing, verification, logging and failure handling. </p>
</div>




This example shows three things:
- Agents do not necessarily communicate with each other directly, they can communicate with each other through supervisor.
- Messages include sender, receiver, kind, and content.
- "Research" and "Review" are different responsibilities, not just two model calls.
