# 专题《Debate 辩论拓扑、置信度与裁判可靠性》教学讲稿

## 讲授目标

本专题讲清多候选、Rubric、交叉评审、投票、Judge、可执行测试、少数意见、置信度路由和辩论停止条件。建议讲授 70～90 分钟。

## 一、开场：质量来自独立检查，不是更多赞同

【页面提示：展示候选到最终答案流程和辩论时序图。】

正确流程是先独立生成多个候选并保存证据，再交叉评审；发现阻塞问题就退回修订；满足标准后通过投票、测试或 Judge 聚合，同时保留异议和未解决风险。

如果候选在生成前互相看到完整思路，它们容易迅速趋同；三个相同答案可能只是同一种偏差重复三次。第一轮独立性是获得多样性的关键。

## 二、多候选生成

`generate_candidates()` 让多个 Agent 独立调用 `solve()`，把作者和答案绑定。真实系统还应保存模型版本、Prompt、证据和成本，才能比较候选来源。

多候选适合开放问题、高风险决定或评分器较可靠的任务。简单确定性问题应优先用工具和测试，不必为“多样性”支付额外成本。

## 三、评审 Rubric

`RubricScore` 把正确性、证据、完整性、安全和清晰度分开，再按权重求总分。Rubric 把“我更喜欢这个答案”变成可讨论标准。

每个分值必须有锚点示例，例如正确性 0 分代表核心结论错误，4 分代表关键主张均有证据。没有评分说明，不同 Reviewer 的 3 分不可比较。权重也应体现任务风险，高风险任务可提高安全权重。

## 四、交叉评审

`cross_review()` 让候选作者之外的 Reviewer 检查结果，跳过自评。它可以发现盲点，但成本按候选数乘评审者数增长，因此可先用便宜检查筛掉明显失败，再对少数候选深评。

评审应输出具体问题、证据位置、严重度和修复建议，而不是“看起来不好”。同时防止 Reviewer 接触候选身份后产生品牌或角色偏见。

## 五、投票、Judge 与测试

`majority_vote()` 是最简单基线，只统计答案频次；它没有处理平票、答案归一化和成员相关性，不适合高风险裁决。

`judge_select()` 汇总多个评分并选择平均分最高候选。平均值能缓解评分数量不同，但仍需处理异常值、偏差和空评分。

对代码、查询或计算结果，优先使用可执行测试。`select_by_tests()` 先从通过测试的候选中选风险最低者，全部失败时才比较部分得分。测试覆盖不足时，“通过”仍不代表完全正确，所以还需需求和安全评审。

## 六、保留少数意见

`ConsensusResult` 同时保存最终答案、获胜候选、少数意见、未解决风险和置信度。共识不是把失败候选删除，也不是假装全体一致。

少数意见在高风险场景特别重要：多数人忽略的边界条件，可能正是事故来源。最终报告应说明选择理由和未解决问题，让后续人工知道不确定性在哪里。

## 七、置信度驱动下一跳

`route_after_consensus()` 在置信度低于 0.65 时请求人工，有未解决风险时进入风险评审，其余才输出。阈值必须用历史评测校准，不能凭直觉设定。

置信度不是装饰。要检查校准：声称 80% 把握的一组结果，是否大约有 80% 正确？来自相同模型的多票也不能简单累加成更高置信度。

## 八、最新研究与停止条件

【页面提示：展示 2024～2026 年研究表和 `should_continue_debate()`。】

近期研究提醒我们，普通多轮辩论并不稳定优于独立候选多数票。有效增益更依赖初始多样性、真实新证据和校准置信度。教学时应先建立“独立候选 + 多数票”基线，再要求辩论证明额外调用确实提高质量。

停止条件至少看三个信号：轮数上限、答案与置信度是否已经趋同、这一轮是否产生新证据。达到三轮、已经稳定趋同，或连续一轮没有新证据时，应停止并进入测试、Judge 或人工流程。

Judge 也必须单独评测。它可能偏好更长、更自信、与自身风格相近的答案。可执行测试优先、匿名候选、固定 Rubric、Judge 委员会和人工抽检都是降低风险的方法。

## 九、专题复习与收尾

请学生比较多数投票、Judge 和测试的适用场景，解释候选为何要独立，Rubric 为何要有分值锚点，以及少数意见为什么必须进入最终对象。

本专题结论：高质量不是靠 Agent 数量或讨论轮数，而是靠独立候选、明确标准、可执行证据、可靠聚合、适时停止和诚实保留不确定性。
