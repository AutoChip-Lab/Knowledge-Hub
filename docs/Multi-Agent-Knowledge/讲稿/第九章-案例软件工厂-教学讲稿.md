# 第九章《案例：软件工厂》教学讲稿

## 讲授目标

本章以 ChatDev 1.0 和 MetaGPT 为例，观察软件多 Agent 项目怎样表达公司、角色、阶段、SOP 和产物。建议讲授 70～90 分钟。

## 一、开场：两个软件工厂怎样组织协作

学习框架案例时，第一步不是背架构图，而是确认版本和来源。本章使用 ChatDev 的 `chatdev1.0` 分支及论文，也参考当前 README 对 2.0 / DevAll 演进的说明；MetaGPT 部分使用官方仓库、论文和 `software_company.py`、`team.py`、角色源码。

讲解时需要区分项目实现与教学抽象：前者给出版本和代码位置，后者用于解释结构，不能写成源码逐行行为。

## 二、ChatDev 1.0：配置驱动的虚拟软件公司

ChatDev 1.0 把软件团队表达成 CEO、CPO、CTO、Programmer、Reviewer、Tester 和创意设计等角色，让它们在需求、设计、编码、评审、测试和文档阶段协作。

它最值得学习的地方是：大量组织逻辑写在配置中。ChatChain 定义阶段链，RoleConfig 定义角色身份，PhaseConfig 定义每个阶段谁和谁对话。

### ChatChain 阶段链

【页面提示：展示 DemandAnalysis 到 WareHouse 的流程图。】

流程从 CEO 与 CPO 的需求分析开始，接着由 CEO 与 CTO 选择语言，CTO 与 Programmer 进入编码，再进行代码补全、评审和测试。评审或测试失败会回到 Coding，而不是直线向前；通过后生成环境文档、用户手册以及 WareHouse 中的项目与日志。

配置片段中的 `SimplePhase` 适合一次明确对话就能推进的阶段；`ComposedPhase` 适合内部需要多次循环的阶段。CodeReview 带 `cycleNum: 3`，说明运行器最多循环指定次数。网页中的两行是独立 JSON 对象片段，不是可以整体解析的数组。

### RoleConfig 与 PhaseConfig

角色至少有三层：名称、职责提示和阶段绑定。只有名称，只是多个改名机器人；职责说明它关注什么，阶段绑定说明它何时与谁协作。

PhaseConfig 进一步把交互方向变窄。例如代码评审拆成 Reviewer 给评论和 Programmer 按评论修改；测试也拆成错误总结与修改。单轮目标越明确，输出越容易验证。

### 输出产物

ChatDev 一次运行不仅给最后一句回答，还在 WareHouse 保存生成的软件文件、三类配置、时间戳日志和任务 Prompt。保存配置、过程和产物，才能复现“这份代码是怎样产生的”，也能定位问题发生在哪个阶段。

## 三、MetaGPT：把 SOP 写入 Team、Role 和 Action

MetaGPT 用“Code = SOP(Team)”概括其思路。README 中的 `generate_repo("Create a 2048 game")` 是高层入口：一句需求进入内部团队，最终返回生成仓库。短函数隐藏了角色、动作、消息和环境循环，所以阅读不能停在入口。

【页面提示：展示 MetaGPT 对象关系图。】

`software_company.py` 导入 TeamLeader、ProductManager、Architect、Engineer2 和 DataAnalyst，再创建 `Team(context=ctx)` 并用 `hire()` 加入角色。加入团队后，角色通过共享环境、消息订阅和动作协作，不是简单按列表依次调用。

`Team` 组织角色、SOP 与环境，通过 `run_project()` 接收需求，通过 `run()` 推进活动。ProductManager 的 name 和 profile 只是身份字段，真正行为还来自目标、Action 和所监听的消息类型。

这里必须把图中原来笼统的“协调与推进”讲开。当前 `Team` 默认使用 `MGXEnv`：用户需求先被发布成 Message，消息用 `sent_from`、`send_to` 和 `cause_by` 表达来源、接收者和触发动作；环境把消息放进角色各自的缓冲区，角色观察到与自己相关的消息后才执行。普通成员把产物路径、摘要和状态发回环境，反馈会经过 TeamLeader；TeamLeader 再用 `publish_team_message()` 把完整需求、约束和上游文件路径定向交给下一位成员。

用一个完整交接来讲：TeamLeader 先把原始需求交给 ProductManager；产品经理返回 PRD 路径；TeamLeader 根据反馈把计划步骤标记完成，再把需求和 PRD 路径交给 Architect；架构师返回设计路径；TeamLeader 再把需求、PRD 与设计交给 Engineer2。DataAnalyst 处理独立或混合需求中的数据部分。角色不一定全部出场，小任务可以直接派给工程师，纯数据任务可以直接派给数据分析师。

【讲授强调：完成有四层，不能混为一谈。】一次 Action 返回只表示一次角色反应结束；一项计划任务由 TeamLeader 根据成员反馈调用 `Plan.finish_current_task` 推进，也可以 reset/replace；团队循环在全部角色 idle 或轮次耗尽时停止，预算不足则异常退出；软件是否验收通过则需要测试、评审、CI、验证角色或用户按明确标准判断。idle 只说明当前没有待处理消息，不说明软件正确。当前 `software_company.py` 中 QaEngineer 的招聘代码仍被注释，更不能把团队停止等同于测试通过。

生产化时还应补一个显式验收契约：任务必须带必需产物和验收条件，成员返回状态与证据，由非产出者或 CI 验证；只有所有必需任务进入 verified 才宣布项目完成。这个规则是教学强化，不要说成 MetaGPT 当前通用 Team 的内置保证。

ProductManager 把用户需求转成 PRD、用户故事和产品分析；Architect 消费上游需求形成架构与接口；Engineer 关注任务分配、代码、修复与评审等动作。`watch/action` 结构把“收到哪类消息后可以做哪类动作”写入角色边界。

## 四、ChatDev 与 MetaGPT 对照

ChatDev 1.0 的核心隐喻是配置化虚拟公司，流程主要位于 ChatChain、RoleConfig 和 PhaseConfig；MetaGPT 是对象化 SOP 团队，核心位于 Team、Role、Action 和消息环境。

ChatDev 更适合理解配置驱动阶段链、对话方向和循环；MetaGPT 更适合理解组织流程怎样编码成 Python 对象、角色怎样订阅消息并执行动作。两者都把角色、阶段、消息和产物显式化，也都远比“几个模型自由聊天”复杂。

## 五、阅读路线、练习与自检

读代码建议从项目入口开始：ChatDev 先追 ChatChain，再追 Phase 与角色配置，最后看 WareHouse；MetaGPT 从 `generate_repo()` 追到 Team、角色和 Action。自建角色和流水线则明确标为教学练习。

让学生回答：ChatDev 的 SimplePhase 与 ComposedPhase 有何差别？RoleConfig 与 PhaseConfig 分别管什么？MetaGPT 的消息怎样从用户经过 MGXEnv 和 TeamLeader 到达专业角色，再携带产物状态返回？Action 返回、Plan 步骤完成、环境 idle 和项目验收通过为什么不能混为一谈？两者的流程表达为何一个偏配置、一个偏对象？

本章结论：软件工厂案例的关键不是模拟公司称谓，而是把职责、阶段、输入输出、返工和产物变成可运行结构。下一章用 AutoGen 官方文献综述示例观察研究团队。
