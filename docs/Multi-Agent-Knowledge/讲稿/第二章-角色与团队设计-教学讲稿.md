# 第二章《角色与团队设计》教学讲稿

## 讲授目标

本章讲清角色设计为何本质上是边界设计，怎样从业务能力形成少而不同的角色，并用输出契约、能力矩阵和交接关系组成团队。建议讲授 65～80 分钟。

## 一、开场：不要先给 Agent 起名字

“分析师、资深分析师、专家分析师”听起来像团队，职责却可能完全重叠。设计角色应从业务目标开始：列出需要的能力，合并相近能力，拆分高风险能力，再定义使命、职责、非目标、输入输出、工具和权限。

【页面提示：展示从能力到角色和角色交接图。】

先讲“从能力到角色”的四步简图：列出所需能力，决定合并或拆分，补齐角色规格，再做团队级校验。图中最重要的是返工线：团队校验发现能力缺口、职责重叠、越权或交接产物无人消费时，要返回第二步重新分组，不能只改角色名称。

再用图后的文字展开判断依据：能力清单还要包含任务知识、工具、数据、环境、审批和独立评审；强耦合、低风险能力可以合并，利益冲突、高权限或需要独立观点的能力应拆分；角色规格必须写清 mission、responsibilities、non-goals、输入输出和最小工具权限；团队校验则检查覆盖、重叠、权限与交接四项。

随后展示角色交接图：需求角色交付验收标准，架构角色交付设计决定，开发角色交付补丁，测试角色交付失败报告，安全评审者决定是否存在阻塞风险。团队不是角色名单，而是一条产物和责任的交接链。

## 二、完整角色模板

SecurityReviewer 的 YAML 不只写名字，还写使命、职责、非目标、输入契约、输出 Schema、工具和可读记忆。它能读文件和扫描依赖，但不负责实现功能，也不能自行批准发布。

讲解时强调三种边界：职责边界说明应该做什么；非目标说明不该做什么；权限边界决定实际上能做什么。三者必须相互印证，不能一边说“只评审”，一边授予部署工具。

## 三、Pydantic 输出契约与发布门禁

安全评审 Agent 的原始结果先进入 `SecurityReviewResult`。`Literal` 限定决定和严重度，`Field` 要求证据与建议非空，并把置信度限制在 0～1。`model_validate()` 成功，只说明字段、类型和取值合格；严重度写成 urgent、遗漏证据或置信度为 1.5 都会被拒绝。

`can_ship()` 是下一层业务规则：决定必须是 approve，并且不能存在 high 或 critical 风险。网页示例返回 False，因为决定是 needs_changes 且存在高风险。

这里最重要的区分是：Pydantic 判断“数据是否合格”，Supervisor 判断“业务是否放行”。结构校验不能替代发布政策。

## 四、角色数量：少而不同

增加角色只有在职责、工具或输出真的不同才有价值。`role_is_distinct()` 比较职责交集、工具差异和输出契约，用启发式方法发现改名复制。文本不同但语义相同的职责仍需人工审查。

角色过少会把冲突职责塞给同一个 Agent，例如开发者既写代码又批准自己的代码；角色过多则增加消息、等待和责任模糊。默认从三到五个真正不同的角色开始，再用运行数据决定是否拆分。

## 五、能力矩阵

能力矩阵把每个角色能否搜索、写文件、执行代码和批准放在同一张表中。Researcher 可以搜索但不写文件，Developer 可以写和执行但不批准，Reviewer 读取和检查但不实现。

矩阵的价值是暴露职责冲突和权限空缺。不过它不能只停留在文档中，必须转成工具授权策略，由执行端强制执行。

## 六、角色 Prompt 与最小交互

好的角色 Prompt 包含使命、上下文、职责、非目标、可用工具、输出契约、升级条件和停止标准；坏 Prompt 只说“你是一名世界级专家”，没有可检验要求。

`RoleAgent.run()` 用统一接口调用模型并按输出模型校验。设计结果交给开发，开发补丁交给评审；未通过门禁时，结构化 findings 回到开发者形成返工闭环。真实系统还要限制最大返工次数并保存每版产物。

角色之间应传递任务所需的最小结构化产物，而不是整个聊天历史。交接时说明目标、输入引用、约束、未决问题和期望输出。

### 协作角色的停止与交接边界

每个团队成员都必须有显式停止条件：输出契约已经满足、达到最大轮数、预算不足、工具连续失败、风险超过权限，或者需要把控制权交给另一角色。停止不能只依赖模型说“完成了”，而应由运行时检查。

同样，角色名不能代替输出契约。“架构师”“测试员”只是身份标签，下游真正依赖的是可解析字段、产物版本、验收结论和未解决风险。讲授时要求学生同时为每个角色写出职责、非目标、工具权限、输出契约、停止条件和交接目标。

## 七、反模式与推荐模板

常见反模式包括：只换名字不换职责；所有角色拥有全部工具；没有非目标；由同一角色实现并批准；输出全是自由文字；角色之间无限讨论；增加动态专家却没有预算和评测。

研究写作可用 Researcher、Synthesizer、FactChecker、Editor；软件开发可用 Product、Architect、Developer、Tester、SecurityReviewer；数据分析可用 DataCollector、Analyst、Statistician、Reporter；运维响应可用 Triage、Investigator、Operator 和 IncidentCommander。模板是起点，不是固定编制，应根据任务删减。

## 八、章末复习与收尾

请学生为安全评审角色指出使命、非目标、输入、输出和权限，解释为什么 approve 与无高风险要分别检查，并判断两个“分析师”是否真正不同。

本章结论：好团队不是角色多，而是边界清楚、能力互补、交付可验证、权限相互制衡。下一章把团队目标拆成能够分派、排序、验收和重规划的任务图。
