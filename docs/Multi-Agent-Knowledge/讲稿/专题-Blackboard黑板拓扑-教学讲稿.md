# 专题《Blackboard 黑板拓扑与共享工作空间》教学讲稿

## 讲授目标

本专题沿一条主线讲清传统 Blackboard 的运行闭环、经典多 Agent 证据、中央与分布式形态、三种现代 LLM 中央 Blackboard 实现，以及结构化卡片、角色视图、写入、冲突和评测。建议讲授 75～90 分钟。

## 一、开场：黑板保存团队状态，不保存全部聊天

【页面提示：先展示“任务或新证据 → 结构化黑板 → 不同角色视图”的总览图。】

群聊回答“谁对谁说过什么”，黑板回答“系统现在知道什么、哪些事实有争议、还有哪些动作没完成”。Agent 读取的是与角色和任务相关的小视图，写回的是可寻址卡片，不是完整对话副本。

请强调：消息是修改共享事实源的事件，黑板才是当前任务状态。若黑板只是把消息按时间堆起来，它仍然是聊天记录，并没有形成 Blackboard 拓扑。

这一页总览图是课程最终要实现的多 Agent 形态。此时不要马上解释卡片字段，先告诉学生：下一步会拿掉角色视图、权限和持久化，只保留传统架构的骨架。

## 二、先讲传统 Blackboard 的运行闭环

【页面提示：切到“分层黑板 → 黑板事件 → 条件匹配 → Agenda / KSAR → 控制选择 → 更新黑板”图。】

传统系统由四种对象组成：分层黑板保存不同抽象层级的部分解；Knowledge Source 用条件部分判断能否贡献，用动作部分修改黑板；KSAR 表示某个知识源在具体上下文中的一次候选激活；控制组件从 Agenda 选择下一步。

按图完整走一遍：第一，黑板新增或修改状态；第二，系统匹配知识源条件；第三，满足条件的具体激活进入 Agenda；第四，控制组件根据当前状态和资源选择一个 KSAR；第五，知识源执行并更新黑板。更新又触发下一轮，所以它是闭环，不是一次广播。

用 DVMT 的一个车辆监测节点做口头例子：本地黑板包含传感器信号、车辆类型和局部轨迹等不同层级的假设；多个知识源都可能响应，但本地控制器只执行当前最有希望的候选。随后再补充多 Agent 层：多个地理分布节点还要交换高层轨迹假设和局部计划，才能从各自的局部视图形成全局车辆轨迹。这样学生能同时看到节点内部 Blackboard 和节点之间协调，且不会把两层混成一层。

## 三、只保留与多 Agent 直接相关的经典论文

本节不要再用 Hearsay-II、Hayes-Roth 和 Nii 证明“多 Agent Blackboard”。它们是 Blackboard 背景资料，但没有回答多个自治问题求解节点如何协作。

先讲 DVMT 1983 和 Durfee、Lesser 1988。多个地理分布节点各自只拥有局部传感数据，每个节点内部是 Blackboard problem solver；节点之间交换高层假设和局部计划。Partial Global Planning 负责减少重复劳动、安排结果交换并处理计划偏差。这里的 Blackboard 是 Agent 内部架构，跨 Agent 协调另有一层。

再讲 1993 年负荷预测系统和 1995 年 Temporal Blackboard。前者明确把多个自主处理 Agent、Blackboard module、Knowledge Sources 和 control mechanism 组成一个真实应用；后者直接把 Blackboard 作为 Agent 通信框架，并处理时间信息、多路访问和一致性。

最后用 Nwana、Lee、Jennings 1996 收束。论文把 Blackboard negotiation 明确描述为由 Agent 取代 Knowledge Source 读写公共黑板，同时指出 DVMT 采用 peer-agent Blackboard，并讨论中央瓶颈、共同语义和集中控制风险。

讲完后让学生先分类：DVMT 是“每个 Agent 一个本地 Blackboard”，其余典型工作更接近“多个 Agent 一份共享 Blackboard”。只有先分清状态放在哪里，后面的卡片、读取和调度才不会混乱。

## 四、先选择中央共享还是分布式本地 Blackboard

【页面提示：依次展示中央共享图和分布式本地图，不要把两图叠在一起讲。】

第一张图只有一份权威 Blackboard。状态变化先形成全局 Agenda，随后经过“选择一个候选”的决策节点，每轮只选中一个具体 Agent；Agent 读取相关视图并把结构化结果写回同一份黑板。Agent A 和 B 是互斥的可能分支，不表示两个 Agent 一定并发。

第二张图中，Agent A 和 Agent B 各有自己的 Blackboard、Agenda 和本地控制闭环。两块本地 Blackboard 之间的虚线表示消息复制，不表示共同读写，也不存在第三块中央黑板。发送方只挑选高层假设或局部计划，接收方验证后写入自己的本地状态，因此两边可以暂时不一致。

不要按“是否联网”区分它们。中央共享 Blackboard 可以位于远程服务，两个分布式节点也可以在同一机器进程中。只问一个问题：团队共同修改一份权威状态，还是每个 Agent 各自拥有权威本地状态？

中央共享方案容易实现和审计，但有瓶颈、单点故障和全局权限压力；分布式方案保留局部自主性，却要承担视图不一致、通信延迟和重复劳动。

本页代码明确选择中央共享方案，但此时还不要马上进入 `BoardCard`。下一节先比较三种现代 LLM 实现，观察同一份中央状态怎样配合不同控制机制。若采用 DVMT 式方案，卡片应作为跨 Agent 交换的高层假设，每个 Agent 还要有自己的本地状态和控制循环。

## 五、三种现代 LLM 中央 Blackboard

先讲 Han 与 Zhang 2025。所有角色信息进入共享黑板，控制单元根据当前内容选择下一位行动 Agent，结果写回后重复执行，直到形成共识。它最接近“中央状态 + 中央选择”的经典闭环。

再讲 Salemi 等人 2025。Main Agent 把帮助请求发布到共享 Blackboard，Helper Agent 根据能力、可用性和成本自主决定是否响应；响应仍写回同一请求，Main Agent 决定采用或忽略。这里中央化的是状态和广播通道，不是强制任务分配。

最后讲 Flock。Agent 通过类型契约声明 consumes 和 publishes；类型化 Artifact 发布到中央 Blackboard 后，所有匹配订阅者自动触发，输出继续作为 Artifact 发布。它是“中央状态 + 类型订阅触发”，更接近事件驱动的 Blackboard 项目。

要求学生对照复述：三者都没有要求 Agent 直接调用彼此，但分别使用中央选择、自主志愿和类型订阅。中央 Blackboard 不等于中央 scheduler。

到这里再引入 `BoardCard`，并告诉学生：它只实现三种系统共同需要的公共状态单元，不是完整 Blackboard 系统。

## 六、卡片为什么需要来源、版本和有效期

`BoardCard` 保存类型、owner、内容、来源引用、标签、版本、替代关系、可见范围和时间字段。来源回答“这条信息从哪里来”，版本和 `supersedes` 回答“它更新了谁”，有效期防止过期事实继续进入角色视图。

生产系统还需要不可变事件日志和独立授权服务。不能只相信 Agent 自己写入的 `visibility`，也不能让修改卡片正文的操作抹掉历史责任。

## 七、把读取路径和调度路径分开

`board_view()` 依次按可见范围、有效期和标签过滤，再保留最新版本，最后限制返回条数。Developer 可以订阅 implementation 与 blocking，Reviewer 可以订阅 artifact、claim 与 risk。

这不是“给每个 Agent 取相同的向量 top-k”。不同角色做不同决定，需要不同物化视图。完整历史保留在审计存储，进入 Prompt 的只是当前决定所需的最小充分状态。

讲到这里要停下来区分两个问题。`board_view()` 回答“某个 Agent 能看到什么”，是读取路径；它不决定谁先执行。机会式控制回答“哪些 Agent 现在可以贡献、下一步选谁”，是调度路径。

调度路径从新卡片触发条件匹配开始，把具体候选调用放进 Agenda，再按依赖、预计贡献、成本和负载选择。执行结果回写黑板，重新开始闭环。不要把角色视图、向量检索或订阅本身说成机会式调度。

## 八、写入策略与持久化边界

OAuth 风险卡用稳定 ID、blocking 标签和设计文档来源，把评审结论变成 Developer 可订阅的正式状态。它比“安全有问题，请注意”更容易检索、更新和验收。

`should_persist()` 只让决策、风险以及有来源的稳定事实进入较长期存储；temporary 或明确过期的条目不持久化。白名单策略能防止临时推测和完整工具日志污染长期知识。

写入前要求学生检查五项：未来是否复用、来源是否可靠、何时过期、是否含敏感数据、是否与现有卡片重复或冲突。

写入不是流程终点。卡片落板后必须重新触发条件匹配和控制选择，否则系统会退化成只存档、不驱动求解的共享数据库。

## 九、冲突不能简单覆盖

两个 Agent 可能分别写入“支持 streaming”和“streaming 仍是实验功能”。正确做法是把冲突标为 disputed，比较来源、版本、时间和验证结果；能裁决时生成新版本并 supersede，不能裁决时同时保留并转 Reviewer 或人工。

不要只选自报置信度最高的卡片。高置信错误、过期官方文档和不同适用范围都会让这种策略失效。“仍有分歧”本身也是黑板必须保存的状态。

## 十、评测与收尾

比较 Blackboard、Supervisor 和全量群聊时，记录任务成功率、每个 Agent 读取 token、重复卡片率、过期事实使用率、冲突解决时间和越权读取次数。

黑板通常适合多源证据、开放专家池和长任务；固定短流程往往使用 Pipeline 更简单。收尾时不要用属性列表结束，而要让学生复述完整闭环：黑板变化产生候选激活，控制器选择下一步，知识源执行并更新黑板；卡片、视图、权限和持久化都服务于这个闭环。
