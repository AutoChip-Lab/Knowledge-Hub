# 第十一章《框架与项目图谱》教学讲稿

## 讲授目标

本章不是做热度排名，而是教学生用官方文档、仓库、源码与论文核验框架，并按 Agent/Tool/Team、阶段、SOP、Task、handoff 和状态图比较项目。建议讲授 70～90 分钟。

## 一、开场：先核验，再总结

【页面提示：展示公开项目核验流程。】

看到一个框架说法后，先找官方仓库或论文，再定位 README、文档和源码对象。能够直接找到代码或官方定义，才写成真实结构；找不到就标为教学抽象。不要把“参考某框架风格”的自造代码说成框架源码。

## 二、AutoGen：Agent、Tool、Team

AutoGen 的 `AssistantAgent` 承载角色、模型和工具，`FunctionTool` 包装 Python 函数，`RoundRobinGroupChat` 与 `SelectorGroupChat` 组织团队，`TextMentionTermination` 表达停止。

模型客户端与 Agent、团队与停止条件是独立组件，说明生命周期和编排没有被藏在一个类里。AutoGen 适合先理解“Agent 怎样调用 Tool，Team 怎样控制顺序和终止”。

## 三、ChatDev 与 MetaGPT

ChatDev 1.0 适合学习配置驱动的软件公司：ChatChain 描述阶段，RoleConfig 描述角色，PhaseConfig 描述阶段参与者，WareHouse 保存产物。当前 README 还说明经典 1.0 与后续更通用平台应按版本区分。

MetaGPT 适合学习 SOP 对象化：高层 `generate_repo()` 内部建立 Team，Role 绑定 Action 并监听消息，Environment 支持协作。它代表 SOP 驱动路线，不是自由聊天式团队。

## 四、CrewAI：Task 是一等对象

CrewAI 的核心对象包括 Crew、Agent、Task 和 Process。Crew 保存 agents、tasks、process，还可配置 memory、cache、planning、knowledge sources 和回调。流程可采用 sequential 或 hierarchical。

它适合学习任务编排，因为工作不仅以消息存在，Task 本身拥有明确位置。官方同时提供 YAML 和代码方式。阅读时要观察任务的输入输出和执行流程，而不是只看角色描述。

## 五、Swarm：最小 handoff

OpenAI Swarm 的核心抽象很少：Agent 带 instructions 与 functions，函数可以返回另一个 Agent 完成交接，运行循环负责模型调用、工具执行、控制权切换和上下文变量。

网页明确保留其 experimental、educational 定位，并提示生产选型应结合官方状态和后继方案。它适合拆解 handoff 概念，不应因代码短就直接当生产框架。

## 六、CAMEL 与 LangGraph

CAMEL 是角色扮演式协作的重要论文入口，通过角色、任务和对话约束推动合作式求解。本章不摘录未逐行核验的具体源码，而把它作为思想和论文源流。

LangGraph 用状态图表达工作流：state schema 定义共享状态，node function 执行步骤，edge 或 conditional edge 决定转移，compile 后 invoke 或 stream。它适合学习显式状态、分支、循环和恢复。本页同样不使用没有官方锚点的伪代码，建议从官方 quickstart 追踪。

## 七、怎样选择学习顺序

【页面提示：展示框架学习路线。】

想理解 Agent、Tool、Team，先看 AutoGen；想看软件公司阶段链，看 ChatDev 1.0；想看 SOP 与角色对象，看 MetaGPT；想看 Task 编排，看 CrewAI；想拆 handoff，看 Swarm；想看图状态机，看 LangGraph；想追角色协作论文源流，看 CAMEL。

选择依据是学习目标、控制流、状态需求、工具生态和生产约束，不是 star 数或流行度。框架版本变化快，引用时应带文档页面、仓库路径和版本信息。

## 八、章末复习与收尾

要求学生任选一个框架，给出官方文档、仓库和源码对象三类证据；再用“对话、阶段、SOP、Task、handoff、状态图”中的一个主轴说明它的定位，并明确哪些描述是官方实现、哪些是教学总结。

本章已经替换自造的 AutoGen 风格代码、CrewAI 伪代码和未核验 LangGraph 片段。结论是：框架不是知识本身，而是对角色、工具、任务、状态和控制流的不同封装。最后一章将把全套内容组织成学习路线、练习和综合验收。
