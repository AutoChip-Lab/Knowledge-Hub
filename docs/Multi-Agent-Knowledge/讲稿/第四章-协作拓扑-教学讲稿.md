# 第四章《协作拓扑》教学讲稿

## 讲授目标

本章帮助学生理解 Pipeline、Supervisor、Blackboard、Debate、Market、Graph 和动态拓扑，能够根据任务结构、风险和成本选择组织方式。建议讲授 70～90 分钟。

## 一、开场：拓扑是团队的组织形状

同样一组 Agent，连接方式不同，行为会完全不同。拓扑回答三个问题：谁能和谁交流，工作按什么顺序流动，控制权怎样转移。

【页面提示：展示常见拓扑图和选择决策树。】

链式适合固定阶段；星型由 Supervisor 集中调度；黑板让成员围绕共享工作区协作；辩论让多个候选互相质疑；市场让角色通过报价争取任务；图拓扑表达分支、循环和并行；动态拓扑允许运行时调整团队。

选择时先问：步骤是否固定？是否需要集中控制？是否依赖共享证据？是否需要多候选比较？是否要动态分配资源？是否存在复杂分支和回退？不要因为某种拓扑看起来高级就使用它。

## 二、Pipeline：固定阶段最清楚

Pipeline 让同一个产物依次经过需求、设计、实现、测试和评审。`Pipeline.run()` 把上一步产物传给下一步，并记录阶段。

优点是路径清楚、容易教学和审计；缺点是上游错误可能一路传播，固定顺序也难利用并行。每一步都必须定义输入输出契约并保存检查点，失败时才能回到正确阶段，而不是从头再来。

## 三、Supervisor：集中调度

Supervisor 根据状态选择下一个专家。关键词 `route()` 展示便宜、可测试的规则路由；`Supervisor.run()` 则在有限轮次内不断选择角色并更新共享状态，无法完成时写入明确错误。

它适合需要集中预算、权限和最终裁决的场景。风险是主管成为瓶颈，所有信息都经过它会增加上下文压力；主管判断错误也会把整个团队带偏。因此应优先使用确定性规则处理明确情况，把模糊分类交给模型，并设置轮数上限。

## 四、Blackboard：共享工作区

黑板拓扑让 Agent 不必两两传递完整消息，而是把证据、假设、任务和产物发布到共享空间。其他角色按类型查询需要的卡片。

最小 `Blackboard` 检查卡片是否包含 id、kind、owner 和 content；证据示例还带有来源和置信度。真实系统需要唯一 ID、版本、权限、并发控制、去重和归档。

黑板适合研究、诊断和机会式协作，但共享不等于无边界。所有角色都能任意改写同一内容，会造成来源丢失和冲突。

## 五、Debate：用分歧发现盲点

`debate()` 让多个 Agent 在三轮中读取问题和已有论点，再追加新论点，最后由 Judge 统一决定。它适合高不确定问题、多候选方案和独立审查。

辩论的价值来自不同证据和不同假设，不是让相同模型重复赞同。论点会随轮次膨胀，因此要限制长度、保持第一轮独立、定义 Judge 评分标准，并允许保留少数意见。简单事实题通常不值得付出辩论成本。

## 六、Market：用报价分配任务

市场拓扑中，候选 Agent 提交 `Bid`，说明预期质量、成本和理由，分配器按效用选择执行者。网页示例使用“质量减去成本惩罚”的分数。

市场适合能力和成本差异明显、任务动态到达的场景。它的难点是报价真实性：如果 Agent 可以夸大质量，就会获得更多任务。因此预期质量必须和历史实绩、校准误差和失败率结合。

## 七、Graph：表达分支、循环与合流

图拓扑把工作步骤表示成节点，把条件转移表示成边。当前节点执行后，运行时检查满足条件的出边并前进；没有合法出边要立即报错。

Graph 适合复杂工作流，也是工程上最通用的表达。生产图还需要检测多条边同时满足、无限循环、不可达节点、并行合流、持久化检查点和恢复。图的核心不是画线，而是显式状态怎样驱动转移。

## 八、动态拓扑：谨慎扩张

动态拓扑可以临时增加专家、删除冗余边或改变子团队。`can_create_agent()` 用 Agent 数量、预计成本和未解决任务数设置三重门槛，避免系统遇到问题就无限增加角色。

动态程度越高，复现、权限和成本越难控制。只有固定拓扑已证明不足，并且有评测、预算、角色白名单和图快照时，才值得引入动态变化。

## 九、章末复习与收尾

让学生为四个任务选拓扑：固定内容生产适合 Pipeline；多个专家由一个入口调度适合 Supervisor；共享证据不断累积适合 Blackboard；复杂状态分支和返工适合 Graph。要求每次同时说明收益和代价。

本章结论是：拓扑没有统一最优，只有与任务结构相匹配。先使用最简单、可测试的组织形状，再根据真实瓶颈升级。下一章将为这些连接定义可靠的通信协议。
