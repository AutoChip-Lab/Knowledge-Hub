# 第一章《全景与定义》教学讲稿

## 讲授目标

本章面向第一次接触 Agent 和多智能体系统的学习者。讲完以后，学生应该能够：

1. 区分普通模型调用、单个 Agent 和多智能体系统。
2. 说出一个 Agent 至少包含哪些部分。
3. 解释多 Agent 比单 Agent 多出了哪些工程问题。
4. 用环境、观察、行动、目标和协调解释协作为什么必要。
5. 根据任务特征，初步判断是否值得使用多 Agent。
6. 看懂本章的两个 Python 示例，并指出其中的角色、消息和协作顺序。

建议讲授时间为 45～60 分钟。讲稿中的“页面提示”供讲授者参考，其余文字可以直接作为口语讲解。

## 一、开场：多智能体不是多开几个聊天窗口

【页面提示：展示章节标题和开头引文。】

大家好，这一章是整套课程的入口。我们暂时不急着学习某一个框架，也不急着写复杂代码，先建立一张完整的地图：什么是 Agent，什么是多智能体系统，以及什么时候值得使用它。

很多人第一次听到“多智能体”，会把它理解成同时打开几个聊天窗口，让几个大模型轮流发言。这个理解只看到了“数量变多”，没有看到“系统是怎样被组织起来的”。

真正的多智能体系统更像一个项目团队。团队里不能只有几个人，还需要回答一系列问题：任务由谁拆分？每个人负责什么？成员之间怎样传递信息？发生分歧时听谁的？谁能执行高风险操作？怎样判断工作已经完成？出了问题能不能回看过程？

因此，本章最重要的一句话是：

> 多智能体不是把模型复制很多份，而是把任务、角色、通信、环境、状态和评审组织成一个可以运行的系统。

后面所有章节，其实都在展开这句话中的某一个部分。

## 二、先认识四个基础术语

【页面提示：展示术语表。】

在开始之前，我们先统一四个词的含义。

第一个词是 **Agent，也就是智能体**。先把它理解成一个能够围绕目标采取行动的运行单元。它不只是回答一句话，还能观察当前情况、选择下一步、使用工具，并根据工具结果继续行动。

第二个词是 **Multi-Agent System，简称 MAS，也就是多智能体系统**。它表示多个 Agent 通过通信或共享环境共同完成任务。这里的重点不是“多个”，而是“共同完成”和“有组织地协作”。

第三个词是 **Planner，也就是规划器**。用户通常只说最终目标，例如“帮我完成一个安全可靠的网站”。这个目标不能直接变成几十个具体动作，所以 Planner 负责把大目标拆成需求分析、架构设计、编码、测试和安全检查等步骤。

第四个词是 **Router 或 Supervisor，也就是路由器或主管**。它负责决定一项工作交给谁，下一步应该走向哪里，什么时候重试，什么时候升级给人，什么时候停止。

可以用一个公司项目来帮助记忆：用户是客户，Planner 像项目经理，Router 或 Supervisor 像现场调度者，各个 Agent 像不同岗位的成员，多智能体系统则是包含这些岗位、制度和工作流程的整个团队。

本章的学习顺序也很简单：先区分一次模型调用和一个 Agent，再区分单 Agent 与多 Agent，最后通过一个最小程序观察研究者和评审者怎样接力。

## 三、从用户目标到最终交付

【页面提示：展示“用户目标 → 任务边界 → 成功标准……”概念卡和全景流程图。】

现在我们沿着网页上的主流程，从头走一遍。

第一步是 **用户提出目标**。例如：“比较 AutoGen 和 MetaGPT，给出适合我们团队的选择建议。”

第二步不是马上调用模型，而是先明确 **任务边界**。我们要比较哪些方面？只比较功能，还是还要比较成本、学习曲线、部署方式和社区生态？不写边界，系统很容易越做越大。

第三步是写清楚 **成功标准**。例如，最终结果必须包含对比表、适用场景、风险说明和明确建议。成功标准决定了后面怎样验收。如果没有成功标准，Reviewer 也不知道什么叫“完成”。

第四步是评估 **预算和风险**。预算不只指钱，也包括时间、模型调用次数、token 和工具执行次数。风险则包括是否会修改文件、访问隐私数据、调用生产环境或影响真实用户。

完成这些准备后，系统才决定使用单 Agent 还是多 Agent。如果问题简单，一个 Agent 直接完成通常更省成本。如果任务需要多个专业视角、独立评审或权限隔离，才进入多 Agent 路径。

进入多 Agent 路径以后，Planner 拆分任务，Router 分配角色，各 Agent 使用工具并产出结果。之后，Reviewer、Tester 或 Judge 进行检查。如果没有通过，就把问题送回规划或执行阶段；如果通过，才交付最终结果，并保存运行轨迹，也就是 Trace，供以后调试和复盘。

请大家注意流程图里的回路：**检查失败以后不是硬着头皮交付，而是回到前面修正。** 这说明 Agent 系统不是一条只向前走的流水线，它需要反馈和纠错。

### 单 Agent 和多智能体系统的组成

网页上把单 Agent 写成：

> LLM + instructions + tools + memory + loop

这里逐项解释。

- LLM 是语言理解与决策能力的来源。
- Instructions 是角色、目标和规则。
- Tools 让它能搜索、读文件、运行测试或调用外部服务。
- Memory 让它能够保留任务进度或过去经验。
- Loop 让它根据观察结果继续下一步，而不是只调用一次模型。

多智能体系统在此基础上还要增加：多个 Agent、通信协议、协作拓扑、共享状态、集体决策机制和治理边界。

【页面提示：展示系统层级图。】

网页上的第二张图把整个系统分成八层。我们从上向下理解。

用户层提出目标；任务定义层明确范围、标准和约束；组织设计层定义角色、拓扑和路由；通信协议层规定消息怎样传递；执行工具层连接文件、浏览器和代码执行等外部能力；质量评审层进行测试和审查；治理审计层记录日志、限制权限并控制成本；最后才是结果交付层。

这张图想告诉我们的不是必须照搬八层架构，而是：**一个生产级多智能体系统绝不只有几个提示词。** 它同时是任务系统、协作系统、工具系统、质量系统和治理系统。

页面列出的传统 MAS 教材、LLM 多智能体综述、CAMEL、ChatDev、MetaGPT 和 AutoGen，是这张知识地图的延伸阅读入口。初学阶段不要求立刻读完论文，只要先知道它们分别代表传统理论、角色通信、软件团队、标准作业流程和对话编排等方向。

## 四、什么才算一个 Agent

【页面提示：展示“Agent 的最小定义”。】

接下来回答一个最基础的问题：什么才算 Agent？

如果我们把一段问题发送给模型，模型返回一段文字，这叫一次模型调用。它当然很有用，但还不一定是 Agent。

本章给出的最小定义是：

> Agent 是一个能在环境中根据目标选择行动，并根据观察结果更新后续行为的计算实体。

这句话有四个关键词。

第一是 **目标**。Agent 不是随机行动，而是在优化某个结果。

第二是 **环境**。它面对的不只是聊天内容，还可能面对文件、网页、数据库、代码仓库或其他 Agent。

第三是 **行动**。它可以回答，也可以调用工具、修改状态、请求帮助或交接任务。

第四是 **观察和更新**。行动产生结果以后，Agent 会把结果作为新信息，再决定下一步。如果没有这个反馈过程，就很难称为真正的运行循环。

在 LLM 场景里，一个 Agent 通常包含六个部分。

角色说明决定它从什么角度看问题；目标决定它要追求什么；工具决定它能对外部世界做什么；记忆决定它能够带着哪些历史继续行动；策略决定它怎样规划、调用工具、停止或求助；观测则是工具结果、环境状态和其他 Agent 消息。

可以把这六项套到一个“代码评审 Agent”上：它的角色是评审者，目标是找出缺陷，工具是读取文件和运行静态检查，记忆里保存已检查文件，策略是先看改动再查高风险区域，观测是扫描结果和测试结果。

### 讲解最小 AgentSpec 代码

【页面提示：展示 `AgentSpec` Python 示例。】

这段代码没有调用真实模型，它是在描述 Agent 的规格。

`@dataclass` 让我们可以用一组字段保存 Agent 信息。`name` 是名称，`role` 是角色，`goal` 是目标。`tools` 是一个工具字典，键是工具名，值是可以调用的函数。`memory` 是字符串列表，用来保存记忆。

`field(default_factory=dict)` 和 `field(default_factory=list)` 表示每个 Agent 默认拥有各自独立的空工具表和空记忆，而不是多个对象共享同一个可变容器。

`system_prompt()` 方法把名称、角色、目标、工具名和最近五条记忆拼成系统提示词。只取最近五条，是为了演示上下文需要控制规模，不能无限塞入历史。

这段代码最容易产生的误解是：定义完 `AgentSpec`，是不是就已经有一个能工作的 Agent？答案是否定的。它只是规格层。真实运行还需要模型调用、工具执行、权限检查、循环控制、错误处理和日志记录。

所以，我们可以把 `AgentSpec` 理解成一份岗位说明书，而不是已经坐在岗位上工作的员工。

## 五、什么是多智能体系统

现在把多个 Agent 放在一起。

多智能体系统可以定义为：由多个部分自治的 Agent 组成，它们通过通信或共享环境协同完成任务。

“部分自治”表示每个 Agent 可以在自己的职责范围内做决定，但它仍然受到系统目标、权限和协议约束。团队协作不是每个人想做什么就做什么。

多个 Agent 放在一起以后，会新增七类问题。

第一是 **协作拓扑**：谁可以和谁通信？是所有人互相交流，还是都通过 Supervisor？

第二是 **通信协议**：消息包含哪些字段？怎样区分任务、结果和错误？超时以后怎样重试？

第三是 **任务分配**：Planner 怎样把子任务交给合适的角色？

第四是 **冲突解决**：两个专家给出相反结论时，由谁判断？是投票、辩论，还是交给 Judge？

第五是 **共享状态**：哪些信息所有成员都能看到？不同 Agent 同时更新状态时怎么办？

第六是 **成本控制**：Agent 数量增加，会同时放大 token、工具调用次数和等待时间。

第七是 **安全边界**：研究者、编码者、评审者和部署者应该拥有相同权限吗？通常不应该。

因此，“我调用了三个模型”不自动等于“我构建了多智能体系统”。只有当这些模型拥有明确职责，并通过一定机制分工、通信、协调和验收时，才形成了系统意义上的协作。

## 六、局部观察和行动依赖为什么要求协调

【页面提示：展示环境状态、局部观察与联合行动图。】

多个 Agent 不能各做各的，根本原因有两个：它们通常只能看到环境状态的一部分，而且一个 Agent 的行动会改变其他 Agent 的工作条件。

请严格区分状态、观察和判断。状态是系统实际存在的完整事实；观察是某个角色此刻收到的部分；判断是它根据这些材料形成的结论。两个 Agent 即使使用同一模型，也可能因为观察不同而产生相反判断。

行动依赖可分为信息、顺序和资源三类。Writer 等待 Researcher 的证据是信息依赖；先写草稿再核查是顺序依赖；两个角色同时修改同一文件是资源依赖。这说明局部正确不等于整体成功。

目标关系还可能是合作、竞争或混合。Developer 和 Reviewer 都希望项目成功，但前者强调交付速度，后者强调风险。协调机制必须明确谁行动、行动前需要什么、冲突怎样处理、何时停止，而不是简单增加聊天轮数。

分析一个系统时，让学生回答五个问题：共同环境是什么、每个 Agent 看见什么、行动怎样互相影响、局部目标是否一致、谁控制顺序与终止。只要其中一项不清楚，系统边界就仍不完整。

## 七、什么时候应该使用多 Agent

【页面提示：展示适合与不适合的两组列表。】

多 Agent 不是默认答案。每增加一个 Agent，就增加一份上下文、一组消息、一次协调和更多失败可能。所以选择多 Agent 时，要问“复杂度是否换来了真实价值”。

以下任务通常比较适合。

第一，任务天然需要多个专业视角。例如做软件产品，需要需求、架构、编码、测试和安全评审。

第二，需要生成多个候选，再进行独立评审。例如研究综述、设计方案比较或策略选择。

第三，流程较长，需要阶段门禁。例如合同审阅必须先抽取条款，再识别风险，最后由授权人员确认。

第四，需要隔离工具权限。例如执行 Agent 能运行命令，Reviewer 只能读取结果，部署 Agent 才能接触生产环境。

第五，需要清楚展示每个角色做了什么，以便解释和审计。

相反，以下情况通常不适合。

任务很短，一次问答就能完成；为了显得高级而硬拆角色；没有停止条件，几个 Agent 会无休止讨论；没有日志和评测，无法证明效果真的更好；或者所有 Agent 使用完全相同的提示词、工具和上下文，只是换了名字。这些做法只会增加成本，不会产生真正的分工。

### 讲解判断函数

【页面提示：展示 `should_use_multi_agent()`。】

示例函数把五个任务信号放在一起：是否需要多种专业能力、是否需要独立评审、是否包含至少五个长流程步骤、工具风险是否达到中高等级，以及是否需要并行研究。

在 Python 中，`True` 可以按 1 参与加法，`False` 可以按 0 参与加法，因此每满足一个条件，`signals` 就增加 1。最后，至少出现两个信号才建议使用多 Agent。

这里的数字 2 不是普遍真理，只是一个教学阈值。真实项目应该通过实验比较单 Agent 和多 Agent 的成功率、成本与延迟，再校准阈值。

例如，“把一句中文翻译成英文”几乎没有这些信号，用单 Agent 即可。“读取大量资料，形成两个候选架构，再由安全专家独立审查”，至少命中多专业、独立评审和长流程三个信号，更适合多 Agent。

## 八、LLM 多智能体与传统 MAS 的差别

传统多智能体系统并不是在大模型出现以后才有。这个领域很早就研究多个决策主体如何在分布式环境里协作或竞争，常见问题包括分布式决策、拍卖和协商、形式化协议、多机器人控制以及多智能体强化学习。

LLM 多智能体保留了“多个主体协作”的核心问题，但工程关注点发生了变化。我们更常讨论角色提示词、自然语言通信、工具调用、任务分解、自我修正、上下文窗口、记忆压缩、评审、辩论、投票、裁判、运行观测和生产安全。

最大的变化，是通信从严格定义的符号或协议，变成了 **自然语言与结构化 JSON 的混合体**。

自然语言的优点是灵活。我们很容易让一个研究者向一个评审者解释复杂发现。缺点是含义可能模糊、字段可能缺失、结论可能前后矛盾。

JSON 的优点是程序可以稳定读取。缺点是它只能保证结构，不能自动保证内容真实。

所以现代 LLM 多智能体系统常采用一种组合：模型用自然语言理解复杂任务，用 JSON 或 Schema 交付关键状态；运行时再用状态机、测试和日志约束整个流程。

这里可以记住一个工程原则：**语言模型负责处理开放语义，确定性程序负责守住控制边界。**

## 九、看懂最小多 Agent 示例

【页面提示：展示 `Message` 和 `EchoAgent` 代码。】

最后，我们通过一段不调用真实大模型的代码，观察最小协作结构。

首先定义 `Message`。一条消息包含四个字段：`sender` 表示发送者，`receiver` 表示接收者，`kind` 表示消息类型，`content` 表示具体内容。

为什么不能只传一段字符串？因为系统需要知道消息来自谁、要给谁、它是任务还是结果。只有内容而没有信封，就像收到一封没有寄件人、收件人和主题的邮件。

接着定义 `EchoAgent`。构造函数保存名称和角色。`handle()` 接收一条消息，再返回一条新消息。新消息的接收者统一是 `supervisor`，类型是 `result`，内容中会写明本 Agent 的角色以及收到的原始内容。

然后创建两个 Agent。`researcher` 的职责是收集证据，`critic` 的职责是发现风险。虽然二者都使用同一个 `EchoAgent` 类，但角色不同。

任务消息由用户发送给研究者，内容是“比较 AutoGen 和 MetaGPT”。研究者处理后，把研究结果交回 Supervisor。Supervisor 再构造一条 `review` 消息，把研究内容转交给 critic。critic 完成评审后，也把结果返回 Supervisor。

这里的执行顺序是：

> 用户 → Researcher → Supervisor → Critic → Supervisor

这个例子没有真正研究 AutoGen，也没有真正找风险，它只是用回声行为突出系统骨架。重点有三点。

第一，Agent 不一定彼此直接通信，它们可以统一通过 Supervisor，这是一种星型协作结构。

第二，消息需要明确的发送方、接收方、类型和内容。

第三，研究与评审是两种不同职责。价值来自职责分离，而不是简单调用两次模型。

如果把它扩展成真实系统，还要增加模型调用、消息 Schema、路由规则、超时重试、权限校验、运行日志、输出评测和异常处理。这些正是后续章节要逐步补齐的内容。

## 十、整套手册的学习路线

## 十一、章末复习与参考答案

【页面提示：先展示问题，邀请学生口头回答，再给出参考答案。】

第一个问题：怎样区分普通模型调用、Agent 和多智能体系统？

参考答案是：普通模型调用通常是一次输入和一次输出；Agent 围绕目标运行，能够选择动作、调用工具、观察结果并继续循环；多智能体系统则组织多个 Agent，通过角色分工、通信、共享状态、路由和评审共同完成任务。

第二个问题：为一个任务列出采用或不采用多 Agent 的两个证据。

例如“完成大型代码库的安全改造”适合多 Agent，因为它需要编码和安全评审两种专业能力，而且高风险工具需要权限隔离。“把一句话改得更礼貌”不适合多 Agent，因为任务很短，也没有独立评审或工具隔离的必要。

第三个问题：最小示例中的任务、消息、角色和评审分别在哪里？

任务是“比较 AutoGen 和 MetaGPT”；消息由 `Message` 表示；角色是 `researcher` 和 `critic`；评审发生在 Supervisor 把研究结果以 `review` 消息交给 critic 的步骤。

## 十二、收尾

今天最需要带走的，不是某一段代码，而是一个判断标准：

> 多智能体系统的本质不是数量多，而是把复杂任务变成可分工、可通信、可评审、可追踪的组织过程。

如果一个 Agent 自己的职责、工具和停止条件都不清楚，复制十个这样的 Agent，只会把混乱放大。下一章进入角色与团队设计，把本章的环境边界、行动依赖和目标关系落实成明确职责与权限。
