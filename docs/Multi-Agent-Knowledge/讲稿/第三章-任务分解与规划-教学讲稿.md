# 第三章《任务分解与规划》教学讲稿

## 讲授目标

本章帮助学生把“列一个 TODO 清单”升级为生成可分派、可排序、可验收、可重规划的任务图，并理解 SOP、Tree of Thoughts、Graph of Thoughts 与重规划的适用边界。建议讲授 60～75 分钟。

## 一、开场：计划不是愿望清单

“调研、开发、测试、上线”看起来像计划，但每一步由谁完成、依赖什么、产出什么、怎样验证都不清楚。真正可执行的计划必须包含任务边界、依赖、负责人、输入输出、成功检查、风险和预算。

【页面提示：展示从目标到任务 DAG。】

流程从用户目标开始，先澄清范围和成功标准，再拆分候选任务、补齐依赖与负责人、检查是否存在环和遗漏，最后形成 DAG。DAG 表示有向无环图：方向代表依赖，无环保证任务不会互相等待。

执行过程中，节点成功会解锁下游；失败、假设失效、工具故障或预算紧张会触发重规划。重规划应复用已完成产物，而不是全部推倒重来。

### 贯穿案例：为 Flask API 增加 GitHub OAuth

先告诉学生 Planner 不写最终代码，它给团队生成工作单。Product Manager 先确认回调地址、用户字段和失败行为；Architect 基于需求设计 session、token 存储和错误处理；Developer 实现；Tester 验证成功与失败路径；Security Reviewer 检查 token、权限和敏感数据。任何门禁失败时，Planner 只调整受影响的节点。

讲解每一节时固定追问四句话：这一节产生什么对象？谁读取它？它怎样改变任务状态？失败后回到哪里？这样学生不会把数据类、Prompt 和排序函数看成彼此无关的代码片段。

## 二、结构化计划与真实示例

`SubTask` 保存 id、描述、负责人、依赖、输入、输出、成功检查、风险和状态；`Plan` 保存总目标、假设、子任务与整体完成定义。

关键字段是 `success_check`。没有它，“完成”只能由 Agent 自己宣布；有了它，运行时可以检查测试、文件或评审状态。`depends_on` 则让系统能够排序和并行。

要逐个说明字段的消费者：调度器读取依赖，路由器读取负责人，Reviewer 读取成功检查，治理层读取风险，消息和日志用任务 ID 关联事件。结构化计划的价值不是“JSON 比文字高级”，而是不同系统组件可以对同一份计划作出一致动作。

GitHub OAuth 示例先用 T1 消除需求中的 TBD，再由依赖 T1 的 T2 设计认证流程，整体完成还要求测试、安全评审和文档。假设“已有用户表”“允许增加依赖”被单独记录，一旦失效就触发重规划。

## 三、Planner Prompt 应要求什么

Planner Prompt 不应只说“请分解任务”，而应提供目标、范围、不可违反的约束、可用角色和工具、预算以及输出 Schema，并要求指出假设、依赖、验收方式和风险。

Planner 的产物仍需程序校验：ID 是否唯一、负责人是否存在、依赖是否指向有效任务、是否有环、是否存在永远无法满足的完成条件。模型负责提出计划，运行时负责验证计划可执行。

这里可以让学生扮演运行时：看到一份计划后，不评价内容写得是否漂亮，只检查 owner 是否存在、依赖是否合法、验收是否可执行、高风险任务是否有独立评审。只有通过这些检查，任务才进入队列。

## 四、依赖排序

`topo_sort()` 用深度优先遍历先访问依赖，再加入当前任务；`temp` 集合记录当前递归路径，如果再次遇到其中的节点，就发现循环依赖。

例如 A 依赖 B、B 又依赖 A，两个任务都会永久等待，必须在执行前拒绝。教学代码还应补充不存在的依赖 ID 检查，并保留同一层无依赖任务的并行机会。拓扑排序给出合法顺序，不意味着所有任务必须串行。

回到 OAuth 案例，确认需求和调研 GitHub 平台限制可以并行；认证流程设计要等待需求；实现要等待设计和平台限制；测试与安全评审都等待实现，但二者可以并行。让学生画出这张图，比让他们背诵拓扑排序算法更重要。

## 五、SOP：固化稳定经验

SOP 是标准作业流程。网页 YAML 把小功能交付固定为需求、设计、实现、测试和评审阶段，每个阶段声明负责人、输入和输出。

SOP 适合高频、稳定、风险明确的工作，优点是可重复和可审计；缺点是面对新问题可能过于僵硬。配置只描述契约链，运行引擎仍要验证产物、设置门禁并处理失败。

## 六、Tree of Thoughts 与 Graph of Thoughts

ToT 适合解空间大、第一条思路不一定可靠的任务。`tree_search()` 从当前前沿生成多个候选，应用到状态并评分，每层只保留得分最高的有限宽度继续扩展，本质上是束搜索近似。

搜索能否胜过单一路径，取决于生成器是否有多样性、状态转移是否正确、评分器是否可靠。简单任务不应为了“搜索”而付出多倍调用成本。

GoT 把树放宽成一般图：不同思路不仅能分叉，还可以合并、复用，并通过反馈边再次修改已有结论。例如安全、成本和兼容性三个 Agent 的局部分析，可以合并成一个方案，再把 Reviewer 意见连回相关节点继续修订。

讲授时只需强调取舍：ToT 适合比较相对独立的候选路线；GoT 适合局部结果需要反复组合的复杂任务，但图状态、去重、停止和成本控制更困难。还要提醒学生，GoT 组织的是候选思路，任务 DAG 组织的是可执行工作，二者不能混用。

## 七、重规划与常见错误

`needs_replan()` 在阻塞问题出现、工具连续失败、关键假设失效或预算消耗超过八成时返回真。触发后要保存已完成节点、重新评估剩余依赖和目标，不要从零开始。

重规划要区分“执行失败”和“计划失效”。一次网络超时通常由执行器重试；官方 API 已弃用则说明原设计假设失效，需要 Planner 作废受影响的设计和实现节点。每次重规划都应记录触发原因、失效产物、受影响下游和新计划版本。

常见错误包括任务太大、任务过碎、依赖缺失、成功标准含糊、负责人不存在、循环依赖、忽略风险、规划后永不更新，以及频繁重规划导致没有稳定执行窗口。

可以用三个检查问题收束：这个子任务能否由一个角色独立领取？结果能否被验证？失败时能否知道影响哪些下游节点？

## 八、章末复习与收尾

请学生解释 TODO 与任务图的差别，指出 GitHub OAuth 示例中的假设和依赖，说明拓扑排序为何要检测环，并比较 SOP、ToT 与 GoT：SOP 复用稳定流程，ToT 比较树状候选，GoT 允许思路合并与反馈。

本章结论：规划的产物不是漂亮的大纲，而是一份可分派、可排序、可检查、可恢复和可重规划的任务图。下一章决定这些任务和角色怎样连接成 Pipeline、Supervisor、Blackboard 或 Graph 等协作拓扑。
