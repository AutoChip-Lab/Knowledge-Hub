# 第十二章《学习路线与引用》教学讲稿

## 讲授目标

本章负责收束整套纯多智能体课程。讲完以后，学生应能根据设计、工程或研究目标选择章节顺序，为练习设置简单基线、可观察指标和失败注入，并从论文结论追到官方项目中的角色、消息或工作流对象。建议讲授 60～75 分钟。

本章不再单独讲通用的单 Agent 循环、工具调用或 Agent benchmark。只有当这些前置知识直接服务于团队协作、路由、共享状态或评审时，才在讲解中提及。

## 一、开场：学习路线由问题决定

【页面提示：展示三条纯多智能体学习路线。】

不要把十二章主线和拓扑专题当成只能从头读到尾的清单。不同学习者要回答的问题不同：团队设计者关心“为什么这样分工”，生产工程师关心“怎样验证和约束”，研究者关心“复杂协作是否真的优于简单方案”。

路线不同，但三者共享一个原则：框架名称不能代替系统设计，更多 Agent 也不能自动证明系统更强。每条路线最后都要回到任务、角色、消息、共享状态、评审证据和治理边界。

## 二、三条纯多智能体路线

设计路线从第一章的边界与必要理论开始，再设计角色与任务图，然后选择拓扑、定义协议并完成路由交接。Blackboard 共享状态与 Debate 评审机制放在协作拓扑专题中深入学习。讲授重点是每一层怎样给下一层提供约束。

工程路线从第五章通信协议进入，再读第六章路由与交接、Blackboard / Debate 专题、第七章观测和第八章安全，最后用第九、十章案例对照。完成后，学生应能为跨角色调用定义 Schema、Trace、权限、预算、重试、审批和恢复路径。

研究路线从第一章必要理论和第四章拓扑开始，接着读 Debate、Dynamic Topology 专题、第七章评测和第十一章框架图谱，再进入论文和源码。研究路线必须强调同预算基线、消融、多个随机种子、成本报告和失败类型分析。

请学生为自己的目标选择一条路线，并回答：为什么先读这些章节？跳过的章节何时需要补？最终用什么产物证明自己真的掌握？

## 三、五个递进练习

第一个练习是双角色研究评审。Researcher 产出主张和证据，Reviewer 独立核验；没有引用的结论必须退回，每次返工保留产物版本。这个练习验证职责分离、消息契约和质量门禁。

第二个练习是 Supervisor 路由。准备研究、编码、测试、安全和人工审批等任务样例，测路由准确率、人工升级率和高风险误放行率。规则路由应作为基线，模型不能覆盖权限和风险硬约束。

第三个练习是共享黑板。多个角色只通过结构化状态和 Artifact 引用协作，故意注入并发写入、过期事实和冲突结论，观察系统怎样保留来源、版本和未解决状态。

第四个练习是软件工厂或研究团队。要求每个阶段都有负责人、输入契约、产物和验收门；出现评审驳回时必须回到正确角色，而不是由主管静默改写结果。

第五个练习是生产化演练。注入超时、重复消息、越权请求、预算耗尽和主管故障，验证检查点恢复、副作用去重、人工升级、回滚和审计链。

所有练习都要先写预期输入、可观察事件和验收条件。还要保留一个更简单的基线，记录成功率、成本、延迟、通信轮数和人工介入率。程序跑完不等于练习通过。

## 四、最终项目与验收闭环

【页面提示：展示最终项目验收清单。】

最终项目可以是多智能体技术调研助手、软件修改团队或芯片设计任务编排器，但至少要交付六类材料。

第一，任务图，包含依赖、角色、验收条件、超时和重规划触发器。第二，拓扑图，分别画出任务依赖、信息依赖和决策权力。第三，协议 Schema，覆盖消息、交接包、共享状态、产物引用和幂等键。

第四，轨迹样例，至少包含一次成功、一次返工、一次越权拒绝和一次故障恢复。第五，评测报告，包含简单基线、同预算对比、消融、成本与失败分类。第六，安全清单，说明角色权限、数据边界、人工审批、回滚和审计保留策略。

验收时不要问“所有 Agent 是否都说完成”，而要问：最终产物是否通过独立检查？每个关键决定能否追到责任角色和输入证据？失败恢复会不会重复副作用？复杂团队在相同预算下是否优于简单基线？

## 五、论文与项目引用怎样使用

多智能体理论可以从 Shoham 与 Leyton-Brown、Wooldridge 和多智能体强化学习教材进入。LLM 多智能体的边界与开放问题可结合 LLM-MAS 综述阅读。

协作方法和框架论文可以追 CAMEL、ChatDev、MetaGPT、AutoGen、AutoAgents、多智能体辩论和 AgentPrune。阅读时记录它究竟研究角色、通信、SOP、动态团队、辩论还是通信剪枝，不要把所有工作都笼统归为“多 Agent 框架”。

项目入口包括 AutoGen、CAMEL、ChatDev、MetaGPT、CrewAI、LangGraph 和 Swarm。先用官方文档定位概念，再进入仓库确认分支、文件路径、对象名和版本。论文主张、当前代码和教学解释必须分别标注。

本章刻意不列通用的 ReAct、Reflexion、WebArena、SWE-bench 等单 Agent 基础引用。它们可能是有用的前置阅读，但不属于这套“只讲多智能体”的核心引用表。

## 六、怎样讲授与评审这套课程

教学时先要求学生画角色、消息和状态，再允许使用框架。每新增一个角色，都要说明它提供哪项独立能力；每新增一条通信边，都要说明接收者为何需要这条信息；每引入动态机制，都要说明预算、回退和复现策略。

评审作业时优先检查边界和证据，而不是界面效果。角色名字是否对应不同职责？交接包是否足以继续工作？共享记忆是否保留来源？Judge 是否有明确 Rubric？权限是否在执行端检查？Trace 是否能够重建失败？

如果复杂系统没有在同预算下超过基线，应鼓励学生删掉角色、通信边或动态机制。简化不是失败，而是用评测证明哪些组织成本没有产生价值。

## 七、全系列收束

请学生最后完成四个口头问题：能否按设计、工程和研究目标各给出一条路线？能否为一个练习写出基线、指标和失败注入？能否解释同预算比较、消融和失败分类为什么缺一不可？能否从论文结论追到官方项目中的对应对象？

整套课程可以压缩成一条链：明确目标，设计角色，生成任务图，选择拓扑，定义协议，路由交接，共享状态，独立评审，记录轨迹，评测安全，持续简化。

真正成熟的多智能体系统，不是 Agent 数量最多，而是分工最有理由、通信最有边界、失败最可诊断、结果最可验证。能够用这四个标准审视系统，就已经获得了独立设计多 Agent 工程的基本方法。
