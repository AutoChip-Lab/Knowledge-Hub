# 第五章《通信协议》教学讲稿

## 讲授目标

本章用“一份长报告跨 Agent 流动”的案例，讲清消息、任务、产物、确认、幂等、引用、上下文预算、状态机和协议测试。建议分两课时，共 90～120 分钟。

## 一、开场：一句“报告写好了”为什么不够

人能根据上下文理解“写好了”，系统却还需要知道：谁完成的、交给谁、属于哪个任务、报告在哪里、是哪一版、是否需要确认、重复发送会不会再次发布。

【页面提示：展示长任务主线。】

用户把主题交给 Client Agent；Client 发送 Message；Research Agent 创建长期 Task，持续产出证据；Writer 生成 Artifact；Reviewer 返回结论；最终客户端收到稳定引用。消息负责通信，Task 负责长期状态，Artifact 负责大产物，三者不能混成一段文本。

### 从局部循环到团队循环

单个 Agent 常见的是 `task → action → observation → final`。多智能体团队则反复经过 `route → message → agent action → artifact/state update → review → reroute`。每个成员内部仍有自己的局部循环，但跨角色边界必须通过协议。

讲授时强调五个新增问题：谁能接收哪类任务，消息是否合法，产物属于哪个版本，交接后谁拥有控制权，失败后由谁升级。模型可以提出 handoff 或 final，运行时仍要检查接收者、Schema、预算、权限和终止条件。

## 二、一条消息的四层结构

教学 JSON 包含 `message_id`、`trace_id`、`task_id`、发送者、接收者、类型、载荷、产物引用、确认要求和幂等键。

【页面提示：展示四层结构图。】

路由层关心 sender、receiver 和 trace；语义层关心 kind、payload 和任务状态；上下文层关心 task、reply_to 和 artifact_refs；可靠性层关心时间、Schema 版本、ACK 和 idempotency key。

`ProtocolMessage` 用 Pydantic 做基础字段校验，再用 `model_validator` 检查条件规则，例如 artifact_ready 必须包含产物引用，review_result 必须包含决定。它不是某个公开标准的原样模型，而是演示“模型提出消息，运行时决定是否合法”。

## 三、从人类意图到可投递消息

不要跳过六步：识别意图，选择消息类型，构造载荷，绑定任务和引用，校验 Schema 与权限，最后才投递。自然语言“请审阅报告”需要变成明确的 `review_request`，并携带报告版本和审阅标准。

信封负责路由和可靠投递，载荷表达业务含义，Task 保存长期生命周期，Artifact 保存大内容。路由器不应解析整份报告，Reviewer 也不应依赖队列内部字段。

网页引用公开 A2A 类规范的对象关系来说明：一次请求消息可以创建长期 Task，Artifact 可能在任务完成后才出现。请求送达不等于任务已经完成。

## 四、发布确认、消费 ACK 和业务完成是三件事

【页面提示：展示 ACK 时序图。】

先纠正三个容易混淆的词。Publisher confirm 是 Message Bus 对发布方说“我已经接管这条消息”；consumer ACK 是消费者对 Bus 说“这次投递已经处理到可持久恢复的边界”；业务完成则是 Task 进入 COMPLETED，Artifact 已经持久化。三个信号方向不同、责任不同。

时序图中必须先幂等创建 `task-2048=WORKING`，确认 Task Store 已持久化，再由 Research Agent ACK 请求。如果先 ACK 再创建 Task，Agent 在两步之间崩溃时，Bus 已删除消息而 Task 尚不存在，任务会永久丢失。

报告完成时也不能直接发送 `artifact_ready` 就算结束。先写 Artifact 并得到引用与内容哈希，再把 Artifact 绑定 Task 并提交 COMPLETED，最后发布完成事件。Client 对完成事件返回的 consumer ACK 只说明通知已经记录；它查询 Task、读取并验证 Artifact 后，才完成业务验收。

如果把 ACK 当作业务成功，系统会在 Agent 刚接管任务时就错误标记完成。反过来，如果 ACK 丢失，消息可能重投，所以请求接管、完成事件记录和有副作用的工具都必须幂等。

## 五、幂等关闭重复副作用窗口

幂等不是简单缓存。客户端发布报告成功后，ACK 可能在网络中丢失；同一请求再次到达时，系统必须返回原结果，而不是再发布一次。

`execute_once()` 先把工具名和规范化参数计算成指纹，再检查幂等键。不存在时原子写入 `IN_PROGRESS`；已成功且指纹相同则返回原引用；处理中则拒绝重复执行；同一键对应不同指纹则判定冲突。

关键在“原子保留”。普通内存字典不能抵抗多个并发请求。生产系统还需定义失败是否可重试、处理中记录何时回收，以及外部工具本身是否支持幂等。

## 六、引用让结论回到证据

引用至少回答对象是什么、对象中的哪一部分、哪个版本以及谁引用。Artifact 清单用任务 ID、版本、媒体类型、URI、哈希和 citations 稳定描述报告。

消息只携带 Artifact URI，接收者按权限读取全文，再按 locator 回查来源。`sha256` 只能证明读取内容与发送方引用的字节一致，不能证明内容真实；真实性仍需要身份、访问控制和来源核验。

## 七、上下文预算：提供决策材料

【页面提示：展示 48k 历史压缩成 2.5k Context Packet 的图与预算表。】

压缩不是简单截取最后若干条。先保留目标和硬约束，再保留当前产物摘要、未解决风险和高价值证据；完整日志留在 Artifact Store，只放引用；最后明确说明省略了什么。

网页给出目标、硬约束、主张摘要、高风险引用、最近修改、产物引用和省略说明的预算分配。核心原则是：接收者需要足够做当前决定的信息，而不是整个团队的全部历史。

## 八、状态机：消息类型不等于任务状态

消息类型描述这一次通信在做什么，如 question、answer、progress；任务状态描述长期任务处在哪个阶段，如 SUBMITTED、WORKING、INPUT_REQUIRED、COMPLETED。

状态图中，question 让任务进入 INPUT_REQUIRED，answer 让它回到 WORKING；只有产物就绪且验证通过才能 COMPLETED。`validate_event()` 按当前状态拒绝非法事件，例如等待答案时不能突然 final。

合法事件还不等于可以随意写状态。生产系统应由统一 reducer 或事务记录旧状态、事件和新状态。

## 九、协议测试与公开规范

协议测试不能只验证对象能创建，还要验证错误消息被拒绝、非法状态没有变化、重复请求没有产生第二次副作用。网页示例分别测试条件字段、状态拒绝和幂等调用计数。

协议变更应依次通过 Schema、路由、状态机、幂等、兼容性和回放测试，再进入发布。教学模型应与 A2A、MCP 或框架自己的正式对象区分，引用公开标准时必须注明哪些字段是简化示例。

## 补充讲授：怎样解释真实项目支撑

【页面提示：展示“架构使用附录”表格。】

讲这张表时不要只念项目名称，要让学生判断证据对应到哪一层。JADE/FIPA 证明结构化 Agent 消息有经典 MAS 实现；CloudEvents 证明通用事件信封有成熟规范；A2A 的规范、样例和 TCK 直接支撑 Message、Task、Artifact、状态机与兼容测试；RabbitMQ 只证明发布确认和消费确认这两个中间件机制。

随后专门解释“无”的含义。四段式引用协议和固定字段的 Context Packet 没有找到完全相同的公开实现，所以它们仍是教学架构。幂等账本虽然与 A2A 的 messageId 去重相关，但“幂等键、载荷哈希、结果记录”这一整套结构也不能宣称已被 A2A 采用。没有精确证据时写“无”，比拿相邻概念充数更可靠。

## 十、练习、复习与收尾

练习可以不调用模型：模拟报告发布成功但 ACK 丢失，重新发送相同幂等键，验证工具只执行一次；再模拟同一键携带不同载荷，验证系统拒绝。

章末请学生区分 Message、Task 和 Artifact，解释 ACK 与完成、缓存与幂等、摘要与引用、消息类型与任务状态的差别。

本章结论：可靠通信不是“把字符串发过去”，而是让消息可路由、任务可跟踪、产物可引用、重试无重复副作用、状态变化可验证。下一章讨论系统怎样选中正确接收者，并让交接后的角色真正接得住任务。
