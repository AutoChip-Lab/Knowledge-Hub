# 第七章《观测、评测与调试》教学讲稿

## 讲授目标

本章讲清事件日志、Trace ID、质量与成本指标、路由回归、离线回放和故障分类，使学生能够回答“系统为什么失败”。建议讲授 60～75 分钟。

## 一、开场：没有事件，就只有一段神秘结果

【页面提示：展示 Trace 事件链。】

多 Agent 运行应被拆成事件：收到任务、创建计划、选择路由、启动 Agent、开始工具调用、工具结束、产物生成、评审完成、任务结束。事件带时间、角色、类型和结构化数据，按顺序连成 Trace。

只保存最终答案时，我们无法知道错误来自规划、路由、模型、工具还是评审。观测的目标不是记录得越多越好，而是让关键决定和状态变化可重建。

## 二、事件日志与 Trace ID

`Event` 统一保存时间、trace、角色、事件类型和 data；记录器自动生成时间戳并追加事件。生产后端要持久化、明确时区并对敏感字段脱敏。

同一个 `trace_id` 贯穿 Planner、Developer 和 Tester。查询它即可看到计划产生、文件修改和测试失败的顺序。子任务还可有 span 或 parent id，既保持全链路关联，又能区分并行分支。

【页面提示：展示按时间从前往后定位首个异常事件的流程图。】

先用最终失败锁定对应的 Trace，再从 `task_received` 开始按时间从前往后比较每个事件的预期与实际。时间上最早的偏离点才是首要排查对象；后续错误可能只是它的传播结果。只有当前候选原因经最小重放无法复现时，才向时间更早的上游事件回溯。

## 三、评测指标

先区分两个用途：Trace 用来解释一次运行为什么失败，指标用来汇总一批运行并比较两个系统版本。评测的输入包括固定任务集与验收标准、每次运行的 Trace 和结果标签，以及模型、Prompt、工具、预算等运行条件；输出应是带分组结果、基线差值和失败 Trace 链接的评测报告。

以一次端到端任务运行作为基本统计单位。基线和候选版本必须使用相同任务集、验收标准、预算与重复次数，并按任务类型和风险等级分组，避免简单任务比例变化制造虚假提升。

最终质量看任务成功率和首次通过率；协作过程看路由准确率、工具失败率与 Reviewer 缺陷拦截率；效率看每次成功成本、轮数与 p95 延迟；治理看人工升级率与危险结果放行率。人工升级率不是单纯越低越好，高风险任务主动交给人类可能正是正确行为；Reviewer 拦截率也必须与误拒率一起看。

`cost_per_success()` 用所有成功和失败尝试的总成本除以成功次数。如果没有成功返回无穷大。把失败成本计入，才能真实反映“获得一次成功要花多少”。

阅读报告时先看安全闸门，再看任务质量，然后比较成本与延迟，最后用路由、工具和评审指标解释变化，并回到对应 Trace 定位首个异常事件。下一节用路由回归测试展示如何把 `handoff_accuracy` 落成可重复运行的质量闸门。

## 四、路由回归测试

路由回归测试把 `handoff_accuracy` 落成固定实验。案例不能只有任务文本，还要包含案例编号、风险等级、允许去向和来源；它应覆盖清晰职责、边界歧义、超出能力、权限审批与历史事故。部分任务可以有多个安全去向，所以使用 `allowed_routes`，而不是假设每条任务只有唯一角色。

`evaluate_router()` 记录逐例预测、总体准确率和高风险错误。每次修改路由规则、能力表、Prompt 或模型版本，都对基线与候选版本运行同一案例集。发布时要求历史事故案例全部通过、高风险误路由为零、重要任务类别不低于基线；总体准确率不能掩盖高风险错误。

路由回归只能发现哪条案例退化，无法解释原因。失败案例的输入、配置和中间状态要进入下一节的回放包。

## 五、可回放调试

Trace 记录当时发生了什么；回放包还要保存重新构造某一步所需的输入、配置、依赖结果和状态快照。完整回放包包括 trace/span/step 标识、组件版本、实际输入输出、工具或 API 结果、共享状态与 Artifact 哈希，以及随机种子、预算、超时和权限策略；敏感信息必须先脱敏。

回放分三层：纯回放直接返回保存结果；单步重执行固定上游输入，只运行可疑组件；全链重执行从任务起点重新运行。调试默认从纯回放或单步重执行开始，有副作用的工具必须替换为 Stub、只读模式或沙箱。

`ReplayStore` 按 `trace_id + step_id` 保存组件、请求、响应与配置，`replay()` 只返回深拷贝，不执行组件。实际调试先定位首个异常事件，纯回放确认失败稳定存在，再一次只改变一个变量重执行可疑步骤；确认修复后，将最小回放包加入回归集，并把已验证原因交给故障分类统计。

## 六、故障分类

常见故障包括规划错误、路由错误、上下文缺失、Schema 错误、工具故障、权限拒绝、状态冲突、无限循环、评审漏检和成本耗尽。

故障分类决定修复位置。规划错误应修 Planner 或验收规则；工具超时应修执行与重试；权限拒绝可能是正确护栏，不应通过扩大权限“修复”；Judge 漏检要改 Rubric 和评测，而不是让 Writer 更自信。

## 七、章末复习与收尾

让学生为一次“测试失败但最终被批准”的运行画事件链，指出至少三个可能故障点；解释 Trace ID 与普通日志行的差别；计算包含失败尝试的每次成功成本；说明回放为何要避免重新执行副作用。

本章结论：多 Agent 的可调试性来自统一事件、贯穿全链的 Trace、分层指标、固定回归集和可控回放。下一章把这些观测能力用于权限、安全、审批、成本控制和回滚。
