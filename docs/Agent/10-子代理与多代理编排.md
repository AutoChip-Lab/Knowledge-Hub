## 0. 先看全景:从「一个 agent」到「一群 agent」

到这里为止,这门课讲的一直是**「一个」agent 的故事**:它有一个模型当大脑(第①篇)、一个循环转起来(第②篇)、一套工具伸手去做(第③篇)、一段上下文当工作台(第④篇)。哪怕讲规划(第⑤篇),讲的也还是**同一个** agent 自己给自己列计划、自己反思。

这一篇,我们第一次让**多个** agent 同台。核心问题很朴素:**当一个 agent 忙不过来、或者它的上下文快被搅乱时,怎么把活拆给别的 agent 去干?**

但开篇就得先纠正一个几乎人人都会犯的直觉——

> **反直觉钩子:你以为拆出子代理(subagent)主要是为了「并行、更快」。其实在真实的 harness 里,它的第一价值是「上下文隔离」,并行只是顺带的红利。**

这句话是整篇的题眼。记住它,后面 3.1 会把它掰开讲透。

```
前 9 篇讲的                        这一篇讲的
┌────────────┐                   ┌────────────┐   ┌────────────┐
│  一个 agent │      →           │  主 agent   │──▶│ 子 agent A │
│  自己转圈   │                   │ (orchestrator)│  └────────────┘
└────────────┘                   │            │──▶┌────────────┐
                                 └────────────┘   │ 子 agent B │
   一个循环、一段上下文             谁调度谁、       └────────────┘
                                 状态怎么传        各有独立上下文
```

## 1. 子代理与多代理编排到底是什么

先给两个一句话定义,先直觉、后术语。

> **子代理(subagent)= 主 agent 派生出来的、拥有一段全新独立上下文窗口的「下属 agent」;它自己转一圈循环,把活干完后只把一段「摘要」回传给主 agent。**

> **多代理编排(multi-agent orchestration)= 让多个 agent 按某种结构(主管 / 群集 / 流水线)分工协作——约定清楚「谁调度谁、状态怎么在它们之间传」。**

打个比方:主 agent 像个**带团队的项目负责人**。它自己脑子(上下文)里只装着「项目全局」;遇到一件需要埋头钻研、会翻一大堆资料的脏活,它不亲自下场把自己脑子搞乱,而是**叫一个下属**去干——下属拿个全新的、空白的脑子专心做,做完只回来汇报一句结论。负责人的桌面(上下文)始终干净。

这和第⑤篇的「规划」是两码事,别混:**规划**是一个 agent 在**自己脑子里**把任务拆成步骤;**子代理**是把某个步骤**整个甩到另一个脑子里**去执行。规划回答「分几步」,编排回答「这步交给谁、它的脑子和我的脑子怎么隔开」。第⑧篇已经把「子代理是一种上下文工程手段」点到了——这一篇把它当主角讲透,并往上再走一层到「多个 agent 怎么组织」。

## 2. 前提:什么时候一个 agent 不够用

不是所有任务都需要拆多 agent。拆之前,先确认你确实撞上了下面这两个前提之一,否则多此一举(甚至更糟,见第 5 节的坑)。

**前提一:单 agent 的上下文会被「探索」污染。**
回到那个贯穿全课的场景——**让 agent 修某开源仓库里的一个 bug**。为了定位问题,它可能要 Grep 半天、Read 五十个文件、试三条错误的思路。这些**探索性读取和失败尝试**全都堆进它那一段上下文里。等它终于找到病根,准备动手改代码时,上下文已经被几万 token 的「垃圾」塞满了——真正重要的信号被稀释,模型开始犯迷糊。这正是第④篇讲的「上下文腐坏」。

**前提二:任务天然可以分解并行。**
有些活彼此独立,没有先后依赖。比如「同时评估三个候选修复方案」「把测试、文档、类型检查三件事分头做」。这类任务(呼应第⑤篇讲的可分解规划)天生适合几个 agent 各干一摊、最后汇总。

> **一句话:上下文要保持干净 → 拆子代理做隔离;任务天然能并行 → 拆多 agent 抢时间。两个前提都不满足,就老实用一个 agent。**

## 3. 核心内容:从子代理到编排结构

### 3.1 子代理 = 上下文隔离(隔离 > 并行)

先把开篇那句反直觉的话兑现。以 **Claude Code 的 Agent 工具**为范例(它就是靠这个工具派生子代理的):当主 agent 调用 Agent 工具派出一个子代理时,这个子代理——

- 跑在一段**全新、隔离的上下文窗口(fresh, isolated context window)**里,和主 agent 的上下文完全分开;
- 有自己独立的转录记录,官方称之为 **sidechain(旁链)转录**——它读了什么、试错了什么,全记在这条旁链上,**不进主 agent 的历史**;
- 干完后,**只把一段文字摘要(text summary)回传给父级**,而不是把它经历的一切原样倒回去。

举个最能说明问题的例子(直接来自官方对该模式的描述):你让一个「搜索代码库」子代理去定位 bug 相关的文件。它**读了 50 个文件**、Grep 了十几个关键词、排除了一堆无关路径——这些全发生在它自己的隔离上下文里。最后它只回传**一小段摘要**:「病根在 `parser.py` 的 `tokenize()`,第 88 行边界判断漏了空串;相关测试在 `test_parser.py::test_empty`。」

主 agent 的上下文因此**只增加了这一小段结论,而不是 50 个文件的原文**。它拿着这段干净结论继续干活,脑子始终清爽。

```
                    ┌───────── 主 agent 的上下文(保持干净)─────────┐
                    │  目标 · 计划 · ……                            │
                    │        │ 派生                    ▲ 只回传     │
                    └────────┼────────────────────────┼───────────┘
                             ▼                         │ 一小段摘要
                    ┌────────────────────────┐         │
                    │  子代理:全新隔离上下文   │─────────┘
                    │  Read×50 · Grep · 试错   │
                    │  (这些全留在 sidechain)  │
                    └────────────────────────┘
```

看清楚这张图你就懂了:**哪怕子代理和主 agent 用的是同一个模型、串行执行、一点都不「并行」,它依然极其有用**——因为它把「脏活的过程」挡在了主上下文之外。这就是「隔离 > 并行」。

> **关键心态:子代理首先是一道「上下文防火墙」,其次才是一台「并行加速器」。它买的第一份东西,是让主 agent 的脑子不被探索过程搅浑。**

### 3.2 委派的经济学:活儿大量流经子代理

一旦一个 harness 认真用起子代理,一个反常识的现象会出现:**主 agent 反而变「瘦」了,大部分实际工作量沉到了被委派的子代理身上。**

有一篇关于「用自然语言描述 harness」的研究(**NLAH**,arXiv 2603.25723,Pan 等人)给出了一个具体测量:**在某一个特定配置下——具体是 TRAE 这套 harness、跑在 Full IHR 模式时——约 90% 的 token / 调用量,是流经被委派的子代理的**,而不是主 agent 自己消耗的。

这个数字要**戴着限定说**,不能当普适定律:

> **⚠️ 限定:这个「约 90%」是特定配置(TRAE harness、Full IHR)下的一次测量,不是所有 agent 的普遍规律。别把它当成「任何多 agent 系统都会 90% 走子代理」的定律来引用。**

但它揭示的**方向**是真实且有教益的:当委派成为主力模式,主 agent 越来越像一个**调度台**——它自己不干重活,只负责派活、收摘要、做决策。理解这一点,你才明白为什么「子代理的上下文管理」在成熟 harness 里是头等大事,而不是边角料。

> 一句话:**委派用重了,主 agent 就从「干活的人」变成「派活的人」;绝大部分 token 会流向下属,而不是留在主循环里。**

### 3.3 多代理的三种结构模式:谁调度谁、状态怎么传

有了「子代理」这块砖,往上一层就是**用什么结构把多个 agent 组织起来**。业界反复出现的是三种拓扑,我们用 **LangGraph** 的图式编排(第⑰篇会深挖它的运行时)来做落地范例。

先交代 LangGraph 的心智模型:它把 agent 工作流建模成一张**有向图**——**节点(node)**是一步计算(一次 LLM 调用、一次工具调用、一段路由逻辑),**边(edge)**定义控制流走向,而一个**带类型的状态对象(state)**贯穿所有节点、在它们之间传递。这就把「谁调度谁、状态怎么传」这两个问题,变成了图上明确可画的东西。

| 结构模式 | 谁在调度 | 状态怎么传 | 适合什么 |
|---|---|---|---|
| **主管(supervisor)** | 一个中心 supervisor agent 决定每一步派给哪个专家 | 各专家干完把结果交回 supervisor,由它汇总、再决策 | 职责清晰、需要中心把关的任务 |
| **群集(swarm)** | 没有中心;agent 之间**直接把控制权交接(handoff)** 给对方 | 一个共享会话状态,记着「当前活跃的是哪个 agent」 | 会话式、需要在专家间灵活流转 |
| **流水线(pipeline)** | 无智能调度,按固定顺序一节接一节 | 上一节的输出即下一节的输入 | 步骤固定、可预测的处理链 |

LangGraph 把前两种做成了**开箱即用的预置模式**:`langgraph-supervisor` 包提供 `create_supervisor()`,`langgraph-swarm` 包提供 `create_swarm()`;两者都用 `create_handoff_tool()` 来实现「把控制权交给另一个 agent」这个动作。swarm 的共享状态里专门有个字段记录 `active_agent`(当前活跃 agent),这样一次运行就能**从上次活跃的那个 agent 接着跑**。

```
主管(supervisor)              群集(swarm)                 流水线(pipeline)
      ┌─────┐               ┌─────┐  handoff              ┌───┐  ┌───┐  ┌───┐
   ┌──│ 主管 │──┐            │ A   │◀────────▶│ B  │        │ 甲 │─▶│ 乙 │─▶│ 丙 │
   ▼  └─────┘  ▼            └──┬──┘          └──┬─┘        └───┘  └───┘  └───┘
┌────┐      ┌────┐            └──────┬────────┘             固定顺序,无中心调度
│专家A│      │专家B│              共享状态(记 active_agent)
└────┘      └────┘
中心把关、可控               灵活流转、无瓶颈
```

一句提醒:LangGraph 官方把这些多 agent 拓扑做成**独立的、可选的小包**,而不是焊死在核心里——核心只管图、节点、状态和检查点。这本身也是一种设计取舍:核心保持轻,拓扑按需装。(注:`langgraph.prebuilt` 里的老 `create_react_agent` 等预置件自 1.0 起已标记弃用,逐步迁往 `langchain.agents.create_agent`;据官方,向后兼容维持到 2.0——这是截至 2026 年中的状态。)

> 一句话:**主管靠一个中心大脑派活、汇总;群集靠 agent 之间直接 handoff、共享一个记着「谁在当值」的状态;流水线干脆不调度,按固定顺序传。选哪个,取决于你需要多少中心把关、多少灵活流转。**

### 3.4 委派 vs 单体:什么时候该拆,什么时候别拆

再看一个把「委派」做成一等公民的范例——**OpenHands**(前身 OpenDevin,第⑯篇细讲)。它的多代理是靠**委派动作**实现的:一个通用的 `CodeActAgent`(编码主力)在遇到需要浏览网页的子任务时,会发出一个 `AgentDelegateAction`,把这段活委派给专门的 `BrowsingAgent`;后者干完,以 `AgentDelegateObservation` 的形式把结果交回。这是「通用协调者 + 专职专家」这个经典结构的一个干净实例。OpenHands 还有个 Planning Agent(beta,据官方 2026 年 3 月更新):在 Plan 模式下它只有只读工具、只能写一份 `PLAN.md` 并请求批准后才动手——本质上就是把「规划」这个职责隔到一个受限的 agent 里(和第⑤篇、第⑨篇都能接上)。

那么,**到底什么时候该拆多 agent,什么时候单 agent 更好?** 给你一张可操作的判断表:

| 该拆多 agent,当…… | 该守单 agent,当…… |
|---|---|
| **职责能清晰切分**(定位、修改、审查各是一摊) | 任务小、步骤连贯,拆开纯属折腾 |
| **上下文需要隔离**(探索会污染主脑,见 3.1) | 各步骤高度依赖同一份实时状态 |
| **子任务天然可并行**(呼应第⑤篇) | 强顺序依赖,拆了也并行不起来 |
| 某类活需要**受限权限**(如只读的规划 agent) | 协调开销 > 拆分带来的收益 |

注意最后一行的默认值:**协调本身是有成本的**。多引入一个 agent,就多一层「派活—等结果—解析摘要」的往返,还多一处可能对不上的状态。所以判断法则很简单——

> **关键心态:默认单 agent。只有当你能明确说出「我拆这个子代理,是为了隔离上下文 / 分离职责 / 抢并行」中的哪一条,才拆。说不出理由的拆分,是在给自己加复杂度。**

### 3.5 编排的代价:天下没有免费的下属

拆多 agent 不是白拿的。把这五笔账摆到台面上,你才不会被「多 agent = 更聪明」的幻觉骗了:

- **上下文碎片化。** 全局信息被切散在各个 agent 的隔离上下文里。子代理 A 知道的事,主 agent 和子代理 B 未必知道——隔离的另一面就是「看不见彼此」。
- **摘要会丢信息。** 3.1 里那个「50 文件 → 一小段结论」的漂亮压缩,反过来说就是**有损**的:子代理凭一段摘要交差,万一它漏掉了主 agent 后面才会用到的关键细节,主 agent 无从知晓,也无处追问。摘要越短,隔离越干净,丢信息的风险也越大——这是同一枚硬币的两面。
- **协调与调试变难。** 单 agent 出问题,你顺着一条转录往下读就行。多 agent 出问题,你得在**好几条转录(含各条 sidechain 旁链)**之间来回对,还要搞清是谁在哪一步把状态传歪了。
- **状态一致性变脆。** 多个 agent 各自持有一份状态视图,一旦谁改了共享状态、别人没同步,就会「各说各话」。
- **成本随 agent 数上升。** 每个 agent 都要独立组装上下文、独立调模型。多一个 agent,多一份 token 账单;层层委派时(见 3.2 的 90% 现象),这笔账会放大得比你想象的快。

> 一句话:**每拆出一个 agent,你都在用「上下文碎片化 + 摘要丢信息 + 协调调试更难 + 状态更脆 + 成本更高」这五样,去换「隔离 + 分工 + 并行」。买之前,先看清标价。**

## 4. 关键心态:多 agent 不是「越多越聪明」

新手看到「多代理」,第一反应常常是「agent 越多,系统越聪明,像开了个 AI 公司」。这是最需要掰正的误解。

多 agent **本质上不买「更高的智力」**——它买的是三样很具体的东西:**上下文隔离、职责分离、并行**。而它付出的代价是同样具体的:**协调开销与状态一致性**。它不会让每个 agent 变聪明,只会改变「活怎么在几个同样聪明(或同样会犯错)的 agent 之间分配」。

这和第①篇那句总纲一脉相承:**agent 强不强,主要看 harness 怎么组织,而不是堆数量。** 多加一个 agent 和多加一个 CPU 核心是一个道理——**能不能提速,取决于任务能不能拆、拆的收益抵不抵得过协调开销**,而不是核数越多越好。

> **一句话记住这一章的取舍:多 agent 主要买「隔离 + 分工 + 并行」,代价是「协调 + 一致性」。默认一个 agent,确有上面三条理由之一时再拆——拆得有据,而不是拆得热闹。**

## 5. 常被忽视的坑(多代理最容易翻车的几处)

1. **为并行而并行。** 看到「多 agent」就想并行提速,却忘了 3.1 的真相——**隔离才是主价值,并行只是红利**。把强顺序依赖的任务硬拆成并行,结果是各 agent 互相等、状态对不齐,比串行还慢还乱。**先问「我是为隔离、为分工,还是真为并行」,答不出就别拆。**
2. **子代理摘要丢掉关键上下文。** 子代理只回传一段摘要(3.1),这既是优点也是陷阱:它可能自作主张地**略掉了主 agent 后面才需要的细节**,而主 agent 拿着一段「看起来很完整」的摘要,压根不知道自己缺了什么。**给子代理明确的「摘要里必须包含什么」的约定**(如「务必回传涉及的文件路径和行号」),别让它凭感觉裁。
3. **多 agent 各自为政、状态打架。** 几个 agent 各持一份状态视图,谁改了共享状态没同步,就会「各说各话」,产出互相矛盾的结果(A 以为文件已改、B 还在读旧版)。**要么用一个明确的共享状态(如 LangGraph 那种带类型、有明确合并语义的 state)当唯一事实源,要么严格约定谁能写、谁只读。**
4. **层层委派,token 悄悄爆炸。** 委派可以嵌套:主 agent 派子代理,子代理又派孙代理……每一层都要独立组装上下文、独立烧 token。结合 3.2 那个「约 90% token 流经子代理」的现象,**深层委派的成本会涨得比直觉快得多**。给委派**设深度上限**,并像第⑦篇讲的那样给整体套上步数 / 预算的刹车。
5. **把本该单 agent 的事过度编排。** 一个连贯的小任务,被拆成「规划 agent → 编码 agent → 审查 agent」三个角色互相传话——协调开销远超收益,还平添了三处可能出错的接缝。**记住第 4 节:默认单 agent。** 多 agent 是撞上明确前提(第 2 节)时才动用的重武器,不是默认架构。
6. **调试时忘了还有 sidechain。** 多 agent 出了问题,只盯着主 agent 的转录看,会一头雾水——因为真相可能藏在某条**子代理的 sidechain 旁链**里(3.1)。排查多 agent 故障,**第一步就是把相关子代理的转录也翻出来**,别只看主线。

## 6. 小结 & 下一步

这一篇你需要带走的:

- **子代理 = 主 agent 派生的、有独立上下文的下属;干完只回传一段摘要。** 它的第一价值是**上下文隔离**(隔离 > 并行)——以 Claude Code 的 Agent 工具 + sidechain 为范例:读 50 文件,只回传一小段结论,主上下文保持干净。
- **委派用重了,主 agent 会变成调度台**:大量 token 流向子代理(NLAH 在 TRAE / Full IHR 特定配置下测得约 90%——这是配置特例,不是定律)。
- **三种编排结构**:主管(中心派活汇总)、群集(agent 间 handoff + 共享 active_agent 状态)、流水线(固定顺序);LangGraph 用节点 / 边 / 状态把「谁调度谁、状态怎么传」画成图。OpenHands 用 `AgentDelegateAction` 把委派做成一等公民。
- **默认单 agent**;确有「隔离 / 分工 / 并行」之一的理由再拆。多 agent 买的是这三样,代价是**协调开销 + 状态一致性 + 成本 + 更难调试**。
- 多 agent **不是越多越聪明**——它改变的是「活怎么分」,不是「谁更聪明」。

我们已经把一个 harness 内部的核心机制讲完了:循环、工具、上下文、权限、以及这一篇的多 agent 协作。可这些能力还都是 harness **内置**的。真实世界里,你总会想让它接上一个内部 API、一个私有工具、一段团队规范——**怎么在不改 harness 源码的前提下,给它「长出」新能力?** 这就是下一篇的主题:harness 如何通过插件、钩子、MCP 等机制,长出一整个**扩展生态**。
