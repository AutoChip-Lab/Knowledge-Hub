# 《Agent 与 Agent Harness》教学站 · 撰写规范（唯一事实来源）

> 这是所有章节作者(含 subagent)与审稿人必须遵守的规范。目标:**31 章由不同人/agent 写出来,读起来像一个人写的**,且质量对标姊妹站《芯片设计完整流程》。

---

## 0. 读者与总基调

- **读者**:团队新人 + 不太懂 agent 的工程师。**默认对方聪明但零基础**——不假设他知道 ReAct、MCP、compaction 是什么,但也不把他当外行糊弄。
- **语气**:第二人称「你」,像一个资深工程师坐在旁边、边画边讲。温暖、直接、有判断力。
- **两条铁律**:
  1. **先给直觉,再给术语。** 每个新概念先用一句大白话/类比说清「它到底在解决什么问题」,再引出正式名字。术语第一次出现给中英文(如 「上下文压缩(context compaction)」)。
  2. **诚实,不吹。** 讲清 trade-off、失败模式、局限。对产品不做营销式吹捧;把「宣传口径」和「已验证事实」分开写。**宁可说「资料不足以确认」,也不编。**
- **越读越有兴趣**:靠的是「原来如此」的顿悟感——多用「为什么这么设计」「如果不这样会怎样」「这里有个反直觉的点」,而不是罗列名词。

- 首个 `#` H1 会被脚本移除并替换成漂亮的 doc-hero,所以**每章必须以 H1 开头**。
- H2 用 `## 数字. 标题`;H3 用 `### 数字.数字 标题`(核心节子小节)。TOC、图标、secmap 链接都依赖这个编号。
- **blockquote(`>`)会自动变成 💡 高亮卡**——用来放「关键心态/一句话金句」,别滥用(每章 3-6 个为宜)。

## 2. 写作手法(让它不枯燥)

1. **一条贯穿全篇的具体场景**。每个概念章选一个统一的小场景反复回扣。**基础篇(1–5章)统一用同一个场景**:
   > 「让一个 agent 去修复某开源仓库里的一个 bug:读 issue → 找到相关文件 → 改代码 → 跑测试 → 提 PR。」
   讲工具时用它、讲记忆时用它、讲规划时用它。案例章则用该产品**自己的**真实工作流当例子。
2. **对比法**。新概念常常靠「和它容易混的东西划界」讲清:agent vs chatbot vs workflow;harness vs 框架 vs 模型;RAG vs 长上下文;等等。
3. **反直觉钩子**。主动抛「你可能以为 X,其实 Y」。例:「你可能以为 agent 强主要靠模型强,其实同一个模型换个 harness,SWE-bench 能差一大截。」
4. **给可操作的判断法则**。像姊妹站那样给「怎么入手/经验法则/什么时候该用」,而不是只下定义。
5. **数字与实例落地**。凡能给具体数(上下文窗口多大、多少个工具、多少 token)就给,并标注来源。
6. **每节收口**。较长的节末尾用一句 `> 一句话:…` 收束。

## 3. 准确性纪律(第一优先级)

- 关于**真实产品/论文**的每一个非常识性断言,必须能追溯到 `research/` 下对应的 **verified 简报**;简报没有的、或标注低置信度的,要么不写,要么显式写「据其官方文档…(截至 2026年中)」并降低确定性措辞。
- **绝不臆造**组件名、API 名、版本号、benchmark 分数。拿不准就查简报或省略。
- 技术在快速演进:涉及版本/最新特性时,用「截至 2026 年中」限定,避免写死。
- 概念章的经典出处(ReAct、Reflexion、Anthropic「Building effective agents」、MCP 规范等)在合适处点名,给读者顺藤摸瓜的线索,但**不要堆参考文献**。

## 4. 篇幅与密度

- 概念章(1–11):约 12–18 KB markdown(≈ 300–500 行)。
- 案例章(12–19):约 14–20 KB。
- 融会章(29–31):约 12–18 KB;第 31 章(芯片 agent 落地)可更长、更具体。
- **信息密度**对标 03-架构设计.md:每一段都要有增量信息,不灌水、不重复上一章已讲透的内容(可以用「(见第 X 章)」互链)。
