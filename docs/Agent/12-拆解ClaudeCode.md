## 0. 全景:从「讲原理」到「拆真家伙」

前面十一篇,我们把 agent 拆成了五件套(①–⑤),又把包在外面的 harness 拆成了循环、上下文、权限、子代理、扩展这几层(⑥–⑪)。全是「原理」。从这一篇起,我们换一种学法:**把当今主流的 agent / harness 一个个搬上手术台,对着前面那张「组件地图」逐一验尸——看真实产品到底怎么落地这些概念、又各自做了哪些取舍。**

第一个上台的,是 **Claude Code**。选它打头,有三个理由:它是把「terminal 里的自主编码 agent」这条路走得最完整的产品之一;它的组件几乎和我们⑥–⑪讲的地图一一对得上,是最好的「对照教材」;而且——**你现在读到的这整个网站,就是用 Claude Code 一手搭出来的**。所以这一篇,某种意义上是它的「自我拆解」。

```
第⑥篇给的 7 层栈(harness 地图)          这一篇在 Claude Code 上逐层验证:
   User / Interfaces                 →   CLI / IDE 插件 / SDK / 网页,共用一个引擎
   Agent loop                        →   gather→act→verify 的简单循环
   Permission system                 →   7 种模式 + deny-first + auto 分类器
   Tools                             →   ~42 个内置工具 + MCP
   State & persistence               →   CLAUDE.md / MEMORY.md / 只追加 JSONL 会话
   Execution environment             →   shell / 文件系统 / 沙箱 / 云 VM
```

> 读这一篇的正确姿势:**别把它当产品说明书背,而是拿它当「组件地图的第一次实战对照」。** 每拆一个部件,回想一下「这对应⑥–⑪里哪个概念、它做了什么不一样的选择」。

## 1. Claude Code 是什么、谁做的、现在什么状态

一句话:

> **Claude Code 是 Anthropic 官方的「智能体编码工具(agentic coding tool)」——一个能读代码库、改文件、跑 shell 命令、用 git、搜网页、派生子代理、连外部服务的 AI harness,全程用自然语言驱动。**

几个关键事实(据 Anthropic 官方文档,截至 2026 年中):

- **谁做的**:Anthropic——也就是 Claude 模型背后的公司。所以它是「模型厂商亲自下场做 harness」的代表作,这一点后面讲设计取舍时很关键。
- **什么时候**:2025 年 2 月 24 日随 Claude 3.7 Sonnet 作为「研究预览」推出;2025 年 5 月 22 日随 Claude Opus 4 / Sonnet 4 正式 GA(general availability)。截至 2026 年中仍在高频迭代(CLI 处于 2.1.x 系列,几乎天天有小版本)。
- **跑在什么模型上**:截至 2026 年中,Claude Sonnet 5 是免费/Pro 版默认;Opus 4.8 是 Max/Team/Enterprise 的旗舰;100 万 token 上下文在旗舰 Opus 与 Sonnet 5 上提供。会话中可用 `/model` 随时切换。
- **在哪用**:一个引擎,多个「门面」——终端 CLI(交互式 + 无头 `-p`)、VS Code / Cursor / JetBrains 插件、独立桌面 App、浏览器版(claude.ai/code)、Slack、GitHub Actions / GitLab CI,以及一套可编程的 **Agent SDK**(Python + TypeScript)。

**别和它的「近亲」搞混**:Claude Code 不等于「Claude 网页版聊天」,也不等于「Anthropic 的 Messages API」。Messages API 是光秃秃的模型接口——你自己写循环、自己派发工具;Claude Code 则是把循环、工具、上下文管理、权限、会话存储**全都替你跑好了**的那层 harness。用⑥篇的话说:**Messages API 给你「模型」,Claude Code 给你「模型 + harness + scaffolding」的整套。**

## 2. 一次任务在它体内怎么流动

拆组件之前,先看一遍**血液怎么流**——这是理解任何 harness 最快的路。设想你在项目里敲下一句:

> 「测试挂了,帮我修好。」

Claude Code 内部大致这样走一圈(对照②篇的「单步」和⑦篇的「循环工程」):

```
你的 prompt
   │
   ▼
① 提交拦截    UserPromptSubmit 钩子可先介入(注入上下文 / 拦截)
   │
   ▼
② 组装上下文  系统提示 + CLAUDE.md(根+懒加载子目录) + 自动记忆 + 已加载 skill 描述 + 会话历史
             └─ 若上下文吃紧 → 触发 5 段式压缩(便宜→昂贵)
   │
   ▼
③ 调模型      整段喂给 Sonnet/Opus → 流式吐出「文字 + tool_use 块」
   │
   ▼
④ 权限闸门    每个工具调用:先 deny-first 规则,再看当前模式,必要时过 auto 分类器
             └─ 放行的调用触发 PreToolUse 钩子(可阻断/改写)
   │
   ▼
⑤ 执行工具    如 Bash 跑 `npm test`(输出默认截到 3 万字符)→ PostToolUse 钩子 → 结果作为 tool_result 回填
   │
   ▼
⑥ 循环        模型读到测试报错 → Grep 找源码 → Read 文件 → Edit 改一处 → 再跑测试 …
             (一圈圈 gather→act→verify,自己纠错,可链几十步)
   │
   ▼
⑦ 收尾        模型判断做完 → Stop 钩子可否决停止 → 展示终稿(必要时 git 提交)
```

三个值得当场记住的细节:

- **你随时能踩刹车**:按 Esc 取消正在跑的工具;直接打字纠正,会在下一个步骤边界被读进去。**人始终在环里,这是它和「全托管云 agent」的关键分野。**
- **每一步都留痕**:整轮对话写进「只追加」的 JSONL 文件,每次编辑还单独快照——所以你能 resume / fork / 连按两次 Esc 回退。
- **累活可以外包**:遇到「把整个代码库摸一遍」这种会污染上下文的脏活,它会派一个**子代理**去干,只把摘要拿回来(见 3.5)。

> 一句话:**Claude Code 的一次任务,就是⑦篇那个 while 循环踩着④⑧的上下文、过着⑨的权限闸门、必要时叫⑩的子代理帮忙——把「修好测试」这个模糊目标,磨成几十步真实动作。**

## 3. 拆开看它的组件

现在对着⑥篇的 7 层栈,逐个部件拆。

### 3.1 Agent loop:一个「简单到反直觉」的循环

Claude Code 的核心循环,官方文档描述成三个交融的阶段:**收集上下文(gather context)→ 采取行动(take action)→ 验证结果(verify results)**,循环往复直到完成。就这么朴素。(对源码的社区分析把它的实现称作 `query.ts` 里的 `queryLoop()` 异步生成器、一个 ReAct 式 while 循环——这个内部命名来自第三方分析、非官方文档,当轶事听即可。)

这里有个贯穿全篇的反直觉主线:**循环本身极其简单,复杂度几乎全在循环外面的「harness 工程」里。** 规划、调试、多文件重构这些看起来很「智能」的行为,是从「简单循环 + 精心设计的工具/上下文/权限」里**涌现**出来的,而不是循环本身有多聪明。这正好印证②⑦两篇的说法。

### 3.2 工具集:~42 个内置工具 + MCP

截至 2026 年中,官方「工具参考」表列出约 **42 个内置工具**(数字会随 CLI 版本增减,当快照看)。粗略分类:

- **看与找**:Read、Grep、Glob、LSP。
- **动手改**:Edit、Write、NotebookEdit。
- **跑命令**:Bash、PowerShell。(Bash 默认超时 2 分钟、可配到 10 分钟;输出默认截到 3 万字符、硬上限 15 万,由 `BASH_MAX_OUTPUT_LENGTH` 控。)
- **上网**:WebSearch、WebFetch。
- **派活与编排**:Agent(子代理)、Task*(任务表)、Workflow、Skill、ScheduleWakeup 等。
- **接外部**:MCP 相关工具(ListMcpResources、ReadMcpResource…)。

一个特别能体现「上下文是稀缺资源」的设计:**MCP 外部工具的定义默认「延迟加载(deferred)」,只有当模型用 `ToolSearch` 按需检索到时才把 schema 装进上下文。** 换句话说,harness 宁可让模型「先搜一下有没有这个工具」,也不肯一上来把几百个工具塞满窗口——这和⑧⑪讲的思路完全一致。(顺带一提:我此刻能调用的很多工具就是这样「按需解锁」的。)

### 3.3 上下文与记忆:三层 + 5 段压缩

对照④⑧两篇,Claude Code 的记忆是清晰的三层:

| 层 | 是谁写的 | 存哪 | 生命周期 |
|---|---|---|---|
| **CLAUDE.md** | 人写的项目指令 | 项目根 + 子目录(懒加载) | 持久,每次会话开始读入 |
| **自动记忆 MEMORY.md** | agent 自己写的经验 | `~/.claude/…` | 半持久;会话开始加载前 200 行 / 25KB(先到者为准) |
| **对话历史** | 循环产生 | 会话 JSONL | 易失,会被压缩「吃掉」 |

当上下文快满,harness 会跑一条 **5 段式压缩流水线**:**预算削减 → snip → microcompact → 上下文坍缩 → auto-compact**,严格「从便宜到昂贵、从无损到有损」逐级触发——先清掉旧的工具输出(无损、免费),实在不够了才让模型总结历史(有损、花钱)。**为什么按这个顺序?** 因为总结是最后的手段:它会丢信息、还要额外调一次模型。这条流水线是「把上下文当预算精算」最教科书式的实现(见⑧篇)。

### 3.4 权限与沙箱:一根从「事事问你」到「全自主」的滑杆

这是 Claude Code 最见功力、也最值得抄作业的一层(对照⑨篇)。

- **7 种权限模式**(据对源码类型的分析):`plan、default、acceptEdits、auto、dontAsk、bypassPermissions`,外加一个给子代理用的内部 `bubble`。在 CLI 里,**Shift+Tab 循环切换 4 档**:默认 → 自动接受编辑 → 计划模式 → **Auto 模式**(一个研究预览阶段的分类器)。这根滑杆的两端,正是①篇讲的「半自主 ↔ 全自主」。
- **deny-first**:默认拒绝,规则**显式放行**——安全的默认姿态。
- **审批疲劳与自动分类器**:Anthropic 自己的分析发现,用户对约 **93% 的权限弹窗直接点了放行**。这说明**纯靠人工逐个审批是不可靠的**(全成了橡皮图章),于是有了 auto 模式那个 ML 分类器来当「人工全审」和「全自主」之间的中间档。(分类器内部怎么判定并未公开,别把它想象成有精确规则。)

> 反直觉的点:**加一堆权限弹窗,并不等于更安全。** 当人开始无脑放行,弹窗就退化成「安全剧场」。真正的安全来自「默认拒绝 + 分层信任 + 智能判定」,而不是「问得多」。

### 3.5 子代理:首先是「上下文隔离」,其次才是并行

Claude Code 的 **Agent 工具**能派生子代理,而它最关键的性质不是「并行」,是**隔离**:子代理跑在一个**全新的、独立的上下文窗口**里,有自己的 sidechain 转录,干完**只把一段文字摘要回传父级**。

举个例子:「在代码库里找出所有调用了这个废弃 API 的地方」——这活儿要读几十个文件,读完全塞进主上下文就污染了。派个子代理去读,父上下文里最后只多出一段「共 7 处,分别在 A/B/C…」的摘要。**主循环的工作台始终干净**。这正是⑩篇「子代理 = 上下文隔离手段」的活样板。

### 3.6 扩展生态:hooks / MCP / skills / SDK

对照⑪篇,Claude Code 对外的插槽相当齐全:

- **hooks**:约 **31 种生命周期事件**(数字随版本涨,早期分析是 27)× **5 种实现类型**(command / http / mcp_tool / prompt / agent)。组织用它把安全、质量门、审计**编码进 agent**——比如用 PreToolUse 钩子在危险命令执行前一票否决。
- **MCP**:作为 MCP client 连接外部工具/数据源(stdio / SSE / HTTP);配合 3.2 说的延迟加载省上下文。
- **skills**:可复用的技能包(Skill 工具),把「做某类事的专门知识/流程」模块化。
- **Agent SDK**:Python `claude-agent-sdk` 与 TS `@anthropic-ai/claude-agent-sdk`(2025 年 9 月从「Claude Code SDK」改名,以示不止用于编码),还能通过环境变量把后端切到 Amazon Bedrock、Vertex、Azure Foundry 等第三方。

## 4. 它与众不同的设计决策

拆完零件,退一步看它的「设计哲学」——这些取舍才是真正值得学的东西。

- **极简循环,极重 harness(minimal loop, maximal harness)。** 循环刻意做得简单,工程重量几乎全压在确定性的基础设施上(权限、压缩、会话存储、hooks)。有一个流传很广的社区源码估算:**约 1.6% 的代码是 AI 决策逻辑,约 98.4% 是运行基础设施**(务必当「社区估算、非官方精确指标」看)。它把⑥篇那句「harness 比模型更重要」演到了极致——**可靠性和安全,来自确定性工程,而非模型天赋。**
- **只追加的 JSONL 会话。** 顺序写、从不改——换来审计、resume、fork、rewind;代价是没法对历史做复杂查询。一个「用最笨的存储换最强的可恢复性」的清醒取舍。
- **处处懒加载。** 子目录 CLAUDE.md、MCP 工具 schema 都是「用到才加载」,因为上下文窗口是**绑定约束(binding constraint)**——这个词你在④⑧篇已经反复见到。
- **子代理隔离即信任边界。** 独立上下文 + 独立转录 + 只回摘要,把探索性读取和失败尝试**挡在父上下文之外**,顺带形成一道干净的信任边界。
- **hooks 作为一等策略。** 把「组织想强加的规则」做成正式扩展点,而不是逼人 fork 源码——这是它能被企业采纳的关键。

> 一句话:**Claude Code 的设计信条是「让循环笨一点、让 harness 强一点」。** 它赌的是:agent 的可靠性,更多来自你能确定性地控制的那 98%,而不是模型即兴发挥的那 2%。

## 5. 局限与坑

诚实地说几条边界(也提醒你别神化它):

1. **它不是「云托管、发完就走」的 agent。** 它默认跑在你本地、直接碰你的文件系统,强调**人可随时打断**。这是优点也是约束:要的是「你盯着它干活」,不是「你派它去、明天来收结果」——那种形态更像 Devin(见⑯篇)。
2. **全自主档位的风险要自己扛。** `bypassPermissions` / full-auto 一开,爆炸半径就交给你了。deny-first 是默认,但一旦你亲手关掉护栏,后果自负(见⑨篇的坑)。
3. **各种「数字」是快照,不是常量。** 42 个工具、~31 个 hook 事件、5 段压缩……都随 2.1.x 高频版本变。教学/引用时务必带「截至某版本」,别当成稳定事实。
4. **auto 分类器是研究预览、内部不透明。** 它怎么判、误报漏报多少,官方没公开。别在它身上编精确机制。
5. **上下文再大也不是无限记忆。** 100 万 token 也会满,压缩就会有损——长任务里「它忘了早先某个决定」是真实存在的失败模式(见④⑧)。

## 6. 一句话记住它 + 横向对比

> **Claude Code = 一个「极简循环 + 极重 harness」的本地自主编码 agent:把权限、上下文压缩、会话持久化、hooks 这些确定性工程做到极致,让人始终在环、随时可打断。它是「模型厂商亲自造 harness」的样板,也是我们这张组件地图最贴合的对照物。**

和后面几篇要拆的产品先摆个对照(细节各见其章):

| 对比对象 | 形态差异 |
|---|---|
| **GitHub Copilot** | Copilot 以 IDE 内的补全/聊天为主;Claude Code 是终端原生、跨整个代码库的自主 agent,有真 shell/git/web、子代理与 hooks。 |
| **Cursor(⑭)** | Cursor 是 AI 原生 IDE(VS Code 分叉),内联 diff/可视化编辑最强;Claude Code 更吃「终端流」与长程自主。常见搭配:Cursor 日常编辑 + Claude Code 啃复杂任务。 |
| **OpenAI Codex CLI(⑬)** | 都是终端 agent;Claude Code 的权限模型更成熟、有显式 CLAUDE.md 记忆、庞大 hook 系统、子代理与大上下文。 |
| **Devin(⑯)** | Devin 是云端沙箱托管 agent(每任务一台 VM);Claude Code 跑本地(可选云)、直碰文件系统、暴露每一步、任何时刻可人工打断。 |
