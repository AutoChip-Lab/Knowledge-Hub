## 0. 先看全景:从「零件」到「能跑的机器」

前五篇,我们把一个 agent 彻底拆成了零件:**模型**(想)、**循环**(转)、**工具**(做)、**上下文/记忆**(记)、**目标/终止**(何时收手)。第 ⑤ 篇结尾我们留了个钩子:**这五件套,靠什么粘在一起、协调着跑起来?**

答案就是这一篇的主角——**agent harness**。

打个比方:前五篇讲的是发动机、油箱、方向盘、刹车这些**零件**;从这篇起,我们讲的是把这些零件装进车架、接上线路、让它真的能上路跑的那套**底盘 + 布线 + 控制系统**。零件再好,没有这层骨架,它只是一堆躺在地上的东西。

```
板块一(①-⑤):零件           板块二(⑥-⑪):把零件组装成机器
┌──────────────────┐        ┌────────────────────────────────┐
│ 模型 · 循环 · 工具  │  ───▸  │  ⑥ 总纲:harness 是什么(你在这)  │
│ 上下文 · 目标/终止  │        │  ⑦ 循环/提示/工具分发            │
│  (五个零件)        │        │  ⑧ 上下文压缩                    │
└──────────────────┘        │  ⑨ 权限与沙箱                    │
                            │  ⑩ 子代理                        │
                            │  ⑪ 扩展生态                      │
                            └────────────────────────────────┘
```

这一篇是**整个板块二的地图**。读完你会知道:harness 到底指什么、它凭什么常常比模型更能决定成败、它由哪几层组成——而每一层,就是接下来 ⑦–⑪ 各自要放大来讲的那一章。

> 全篇继续沿用第 ① 篇那个跑腿例子:**让 agent 去修好某开源仓库里的一个 bug——读 issue → 找文件 → 改代码 → 跑测试 → 提 PR。** 这一次我们换个视角:不看模型「想」得怎么样,而看**包在它外面的那层代码**是怎么让这一切真的发生的。

## 1. Harness 到底是什么

一句话:

> **Agent harness(智能体运行时骨架)是包在大模型外面、负责「执行」的那层运行时代码——它是「一切不是模型的东西」。**

模型只会做一件事:给它一段文字,它吐回一段文字。它**读不了你的文件、跑不了你的测试、发不出 PR**。它甚至记不住上一句话说了什么——每次调用,它都是一张白纸。那么,一个 agent 里「读文件、跑测试、记历史、拦越权、一圈圈转下去」这些**真实发生的事**,是谁在干?

全是 harness 在干。

要把这层讲清楚,业界有一个已经收敛的心智模型,来自 **HuggingFace 术语表**(《Harness, Scaffold, and the AI Agent Terms Worth Getting Right》,2026-05)。它把一个 agent 拆成**三个盒子**:

```
      ┌─────────────────────────────────────────────────────────┐
      │                        Agent                            │
      │                                                         │
      │   Model              Harness             Scaffolding    │
      │  ┌──────┐          ┌──────────┐         ┌──────────┐   │
      │  │无状态 │          │  执行层   │         │  行为层   │   │
      │  │纯文本 │  ◂─调用─▸ │ 跑循环    │  ◂─配置─ │ 系统提示  │   │
      │  │函数  │          │ 派发工具  │         │ 工具描述  │   │
      │  │      │          │ 决定停不停│         │ 上下文策略│   │
      │  └──────┘          └──────────┘         └──────────┘   │
      │   「想」              「让它动」            「怎么动」      │
      └─────────────────────────────────────────────────────────┘

              Agent = Model + Harness + Scaffolding
```

- **模型(Model)**:一个**无状态的纯文本函数**——文字进,文字出。调用之间没有记忆、没有循环。它能*表达*「我想调某个工具」的意图,但**没法真的执行**。
- **执行层(Harness)**:负责**跑起来**——调用模型、接住模型吐出的工具调用请求、真的去派发执行、再决定「这一圈完了没、要不要再来一圈」。用 HuggingFace 术语表的话说:**「让 agent 真的跑起来的那个东西,就是 harness。」**
- **行为层(Scaffolding,脚手架)**:负责**定义行为**——系统提示、工具描述、怎么解析模型的输出、上下文/记忆用什么策略。它不「跑」任何东西,它**塑造**模型会怎么表现。

三者的分工可以用一句话拧清楚:

> **Model 决定「聪明不聪明」,Scaffold 决定「被教成什么样」,Harness 决定「能不能真的动起来、动的时候守不守规矩」。** 模型再聪明,没有 harness,它连一个文件都碰不到。

**scaffold 和 harness 的边界**最容易混,记住一个时间轴就行(这是 OpenDev 论文里给的干净说法):**scaffold 管的是「第一句 prompt 之前」——怎么把这个 agent 构造出来;harness 管的是「之后」——运行起来之后发生的一切。** 一个是搭台子,一个是唱戏。(需要说明的是:很多博客和论文把这两个词当同义词混用;这个「行为 vs 执行」的清晰切分是 HuggingFace/OpenDev 这一派的定义偏好,**尚未成为全行业严格遵守的共识**。本课统一采用这个切分,因为它最好教。)

## 2. 前提:它到底解决什么问题

回到那个白纸一样的模型。它是个**无状态的纯函数**:`f(文本) → 文本`。你要把它变成一个「**会动手、会记事、会守规矩、能连轴转**」的 agent,靠的全是外面这层。逐条看它补齐了什么:

| 模型天生缺什么 | harness 补上什么 | 对应第 ① 篇的哪个零件 |
|---|---|---|
| 只会说,不会做 | 接住 `tool_use` 请求,**真的去执行**并把结果递回 | 工具(手脚) |
| 一问一答就结束 | 一个 **while 循环**,让它一圈圈转到收工 | 循环(心跳) |
| 调用之间无记忆 | 攒历史、**压缩**、按需检索、落盘 | 上下文/记忆 |
| 不知道何时该停 | 终止判定 + 步数/token 安全上限 | 目标/终止 |
| 一执行就可能闯祸 | 权限门、沙箱、越权拦截 |(第 ⑨ 篇专讲) |

看这张表你会发现一件事:**第 ① 篇讲的「五件套」,除了「模型」那一件,其余四件「让它真的发生」的部分,本质上都是 harness 干的活。** 前五篇讲的是「概念上有这么几个零件」,这一篇开始讲的是「**工程上,把这些零件焊成一台能跑的机器,那层代码长什么样**」。

> 一句话:**模型是那台无状态的引擎,harness 是让引擎变成一辆能上路的车的一切——底盘、传动、油路、刹车、仪表盘。**

## 3. 核心:把 harness 这层骨架看清楚

这一节是本篇的心脏。我们分五步:先把三分法讲细(3.1),用一个反直觉的「比例数字」把你对 agent 的直觉翻个个儿(3.2),摊开 harness 的**七层栈**当作后面 ⑦–⑪ 的导览地图(3.3),再拿**硬证据**证明「harness 常比模型更重要」不是口号(3.4),最后划清几条最容易踩线的界限(3.5)。

### 3.1 三分法细讲:三个盒子各管什么

把 3.1 想成一次「职责对表」。同一次修 bug,三个盒子分别在忙什么:

- **Model(想什么)**:读到测试报错,判断「这可能是 `parser.py` 里的边界情况,我要先读它,再改第 42 行」。它**只产出意图**——一段文字 + 一个「我要调 Read 工具,参数是 `parser.py`」的结构化请求。到此为止,什么都没真的发生。
- **Harness(让它发生)**:接住这个请求 → 先过一道**权限检查**(这个读操作允许吗?)→ 允许就**派发**给真正的文件读取代码 → 把读到的内容(可能太长,还要**截断/落盘**)**塞回**给模型 → 记进历史 → 判断「还有工具要调吗?有 → 再转一圈;没有 → 收工」。**这一整套编排,就是 harness。**
- **Scaffolding(怎么想)**:在这一切开始*之前*就已经写好了——系统提示告诉模型「你是个严谨的工程师,改代码前先读懂上下文」、工具描述告诉它「Read 工具能干嘛、参数怎么填」、上下文策略决定「历史攒到多长就得压缩」。它是**模型行为的模具**,构造期就定型,运行期基本不变。

**边界再钉一次**:同样是「工具」这个词——工具的**描述文本**(告诉模型有这么个工具、怎么用)属于 scaffold,是构造期的事;工具被调用时**真正的派发与执行**属于 harness,是运行期的事。一个是说明书,一个是流水线。

> 一句话:三分法不是学术洁癖。它让你在 debug 一个「不好用的 agent」时,能精准定位——是**模型笨**(换模型)、**教得差**(改 scaffold 的提示/工具描述)、还是**执行层烂**(修 harness 的循环/权限/上下文管理)?这三处的修法完全不同。

### 3.2 比例震撼:1.6% 的「聪明」,98.4% 的「骨架」

先给你一个会颠覆直觉的数字,再解释。

社区对 **Claude Code** 抽取源码的一个估算是:**大约 1.6% 的代码是 AI 决策逻辑,大约 98.4% 是运行基础设施(也就是 harness)。** 这个数字被论文《Dive into Claude Code》(arXiv 2604.14228)一字不差地引用,但论文自己也注明:它出自「对抽取源码的社区分析」,而**这个分析怎么算出来的、按什么口径(代码行?文件数?哪一份抽取?),并没有公开**。

> ⚠️ 务必按简报口径标注:这是一个**社区估算,不是官方精确指标**。别把「1.6%」「98.4%」当成 Anthropic 发布的准确度量——它是个**说明方向**的插图式数字,不是可以拿去做严格计算的测量值。

即便打了这么重的折扣,它想戳破的那个直觉依然成立:**大多数工程师默认「LLM 就是 agent」**——以为一个 agent 里绝大部分的精妙都在模型里,外面那层只是薄薄一层胶水。**真相恰好反过来:** 一个生产级 agent 里,压倒性多数的代码,是在处理**模型碰不到的那些事**——怎么派发工具、怎么在上下文快满时压缩、怎么拦住危险操作、怎么把会话存下来能续能回滚。

```
你以为的 agent:                     真实的 agent:
┌─────────────────────┐            ┌─┐← ~1.6% 模型决策逻辑
│                     │            ├─┴──────────────────────┐
│      巨大的模型       │            │                        │
│                     │            │   ~98.4% harness       │
│  ┌──┐ 薄薄一层胶水    │            │  循环·工具派发·上下文     │
│  └──┘               │            │  ·权限·会话·子代理·扩展   │
└─────────────────────┘            └────────────────────────┘
       (直觉)                    (社区对 Claude Code 源码的估算)
```

**为什么这么设计?** 因为可靠性和安全性,来自**确定性的基础设施**,而不是靠模型「这次别犯错」的运气。你没法让一个概率机器保证不 `rm -rf` 掉不该删的东西——但你可以在它外面写一层**确定性的权限门**,把危险操作拦死。这正是第 ⑨ 篇的主题。

### 3.3 Harness 全景:七层栈(也是本板块的导览地图)

那 98.4% 到底在忙什么?最常被引用的参考分解,是 arXiv 2604.14228 给出的 **7 层栈**(从对 Claude Code 及两个独立开源系统 OpenClaw、Hermes Agent 的对比中提炼)。我把它画成一张图,**每一层旁边标了它对应本板块的哪一章**——这就是你读 ⑦–⑪ 的地图:

```
┌─────────────────────────────────────────────────────────────────┐
│ ① User(用户)         提交任务、批准权限、审阅结果               │
│         │                                                        │
│ ② Interfaces(接口)   CLI / 无头模式 / SDK / IDE——都汇入同一循环  │
│         │                                                        │
│ ③ Agent loop(循环)   组装上下文→调模型→解析→有工具调用?→派发→  │  ▸ 第 ⑦ 篇
│         │             收结果→再来一圈;无工具调用或触上限则终止    │  ▸ 第 ⑧ 篇(上下文压缩)
│         ▼                                                        │
│ ④ Permission(权限)   先拒后允的规则评估 + 模糊情况的判定         │  ▸ 第 ⑨ 篇
│         │                                                        │
│ ⑤ Tools(工具)        内置工具注册表 + 按需发现的 MCP 工具        │  ▸ 第 ⑦ 篇(工具分发)
│         │                                                        │
│ ⑥ State/persistence   只追加的会话记录、CLAUDE.md 层级、检查点、  │  ▸ 第 ⑧ 篇(记忆)
│         │             子代理侧链                                 │  ▸ 第 ⑩ 篇(子代理)
│         ▼                                                        │
│ ⑦ Execution env(执行环境)  shell / 文件系统 / MCP 连接 / 沙箱   │  ▸ 第 ⑨ 篇 · ⑪ 篇
└─────────────────────────────────────────────────────────────────┘
```

七层里,**第 ③ 层(循环)是引擎室**——第 ⑦ 篇会把它整个拆开,连带讲「工具怎么被分发」「系统提示怎么被组装」;它转起来后,上下文会不断膨胀,**第 ⑧ 篇**讲怎么**压缩**(据论文,Claude Code 用的是一条「最便宜的手段先上」的多阶段压缩流水线);**第 ④/⑦ 层**的权限与沙箱是**第 ⑨ 篇**;**第 ⑥ 层**里那个「子代理侧链」牵出**第 ⑩ 篇**的子代理;而让外部工具/能力接进来的扩展机制,是**第 ⑪ 篇**。

> 一句话:别被「七层」吓到。它就是一张地图——你现在只需要认得每层的门牌号,知道进哪扇门能找到哪一章。⑦–⑪ 会带你逐扇门推进去。

### 3.4 为什么 harness 常比模型更重要(硬证据,不是口号)

「harness 比模型重要」听着像营销话术。但它有**可引用的硬证据**,不是拍脑袋。

**证据一:同一个模型,只换 harness,分数差一大截。**
Claw-SWE-Bench(arXiv 2606.12344,350 个 issue / 8 种语言 / 43 个仓库)在**同一个模型 Qwen 3.6-flash** 上跑了五套不同的 harness:

| harness | Pass@1 |
|---|---|
| GenericAgent(最弱) | **38.6%** |
| …(中间三套)… | 介于两者之间 |
| OpenClaw(最强) | **66.0%** |

**模型一个字没换,光换外面这层骨架,Pass@1 就从 38.6% 拉到 66.0%——差了 27.4 个百分点。** 论文自己说,这个差距大到**足以重排一张排行榜的结论**,相当于跨了一整个「模型档位」。同一篇里还有个更极端的对照:固定用 GLM 5.1 做后端,同一套 OpenClaw,**配一个简陋适配器只有 19.1% Pass@1**(约 69% 的补丁根本打不上),换成**完整适配器直接 73.4%**(打不上的不到 1.5%)。**同一个大脑、同一套 harness,光是接线(adapter)糙一点,连补丁都贴不到位。**

**证据二:在长程任务上,harness 的方差能盖过模型的方差,甚至颠倒排名。**
论文《Stop Comparing LLM Agents Without Disclosing the Harness》(arXiv 2605.23950)提出了一个叫 **「约束绑定论题」(Binding Constraint Thesis)** 的框架:对于能力相近的前沿模型,在**长程任务**上,**harness 造成的差异可以大于模型造成的差异**——大到能**颠倒模型的排名**。结论很尖锐:任何不披露所用 harness 的排行榜,都是**「不完整、甚至可能误导人的」**。

**证据三:一个字的权重都不动,光改 scaffold 就能涨分。**
据 HuggingFace 术语表,仅靠 scaffold 层面的改进(提示、工具描述、上下文策略),就能在 **SWE-bench Verified 上带来约 10–20 分的提升,而模型权重丝毫未变**。

> 反直觉钩子回扣第 ① 篇:你可能以为「agent 强主要靠模型强」。**其实同一个模型换个 harness,SWE-bench 能差出一整个模型档位。** 模型定的是天花板,但你实际站在哪一层,是 harness 决定的。

### 3.5 划清界限:三个容易撞名的东西

「harness」这个词周围有几个近义词/撞名词,不划清界限,你会一直迷糊。

- **harness ≠ agent 框架(LangChain / LlamaIndex 这类)。** 框架是 **SDK 级的积木**——给你现成的链、内存、工具封装。harness 是把这些积木**组装成一个完整生产系统**的东西:接上安全、权限、持久化、可观测、多代理协调。一句话:**框架是 harness 的「输入」,不是 harness 本身。** 你用 LangChain 攒出来的,更接近 harness 的一部分零件,而不是那台完整的机器。
- **harness ≠ 评测 harness(eval harness,如 lm-evaluation-harness)。** 这是**最坑的撞名**。评测 harness 是跑一批固定测试场景、记录指标的工具——它**不编排实时的工具执行、不管理生产会话**。它和本课讲的 agent harness,除了共用「harness」这个词,几乎毫无关系。HuggingFace 术语表专门把这个「术语撞车」列为已知的混淆源。
- **harness ≠ scaffold ≠ model。** 3.1 已经钉过:model 是想的、scaffold 是构造期塑造行为的、harness 是运行期执行的。

> 一句话:下次有人跟你说「我们的 harness 分数很高」,先问一句——你说的是哪个 harness?**跑 agent 的那个,还是跑评测的那个?** 这一问,能省掉一半的鸡同鸭讲。

## 4. 关键心态:harness engineering 正在成为新的 prompt engineering

行业里有一条被反复提起的演进线索:**prompt engineering(提示工程)→ context engineering(上下文工程)→ harness engineering(骨架工程)**。

- 早期,大家拼的是**怎么写提示词**——同一个模型,提示写得好不好,差别巨大。
- 后来,拼的是**怎么管上下文**——放什么、不放什么、怎么压缩(第 ④、⑧ 篇)。
- 而现在,越来越多人认为,真正拉开差距的是**怎么设计整套 harness**——循环怎么转、工具怎么派、权限怎么设、上下文怎么裁、子代理怎么切。

> ⚠️ 诚实标注:「harness engineering 正在取代 prompt engineering」是一个截至 2026 年中被业界广泛采用的**共识叙事框架**(HuggingFace、MongoDB、Addy Osmani 等都在讲),**方向上很有支撑,但它是一个叙事框架,不是一条被测量证明的铁律。** 把它当「趋势判断」,别当「物理定律」。

即便如此,它带来的**一等决策**是实打实的:

> 关键心态:**选/造哪一套 harness,和选哪个模型,是同等重要的第一等决策。** 3.4 的证据已经说明——一个平庸模型配一套好 harness,可能干得过一个强模型配一套烂 harness。所以做一个 agent 产品时,「用什么模型」和「用什么 harness」应该被摆在同一张决策桌上,而不是把后者当成事后再补的「工程细节」。

---

## 5. 常被忽视的坑(最容易栽的四处)

1. **以为「换个更强的模型就万事大吉」,忽视 harness。** 这是最常见的错。模型是天花板没错,但 3.4 的 Claw-SWE-Bench 已经证明:**同一个模型,harness 差一截,分数能差出一整个模型档位。** 一个跑不好的 agent,九成情况下问题出在循环、上下文管理、工具设计、权限恢复这些 harness 环节,而不是「模型不够强」。**先怀疑骨架,再怀疑大脑。**
2. **拿「没公开 harness」的 benchmark 分数横向比模型。** 这正是「约束绑定论题」(arXiv 2605.23950)的核心警告:长程任务上,harness 的方差能盖过模型的方差,甚至**颠倒排名**。看到「模型 A 比模型 B 高 3 分」的排行榜,**先问它俩用的是不是同一套 harness**;若没披露,这个比较可能是无意义的,甚至是误导。
3. **把评测 harness(eval-harness)当成 agent-harness。** 术语撞车害人。看到「harness」二字先分清:是**跑 agent 的运行时骨架**,还是**跑评测的记分工具**?两者除了名字毫无关系(见 3.5)。混了,你会以为读懂了一篇论文,其实完全会错意。
4. **以为 harness 只是「胶水 / 脏活」,不值得投入。** 恰恰相反——那 98.4% 的「脏活」(社区估算,见 3.2)才是可靠性、安全性、可续跑性的来源。**可靠性来自确定性的基础设施,不来自模型的运气。** 把 harness 当二等公民,你造出来的 agent 就会在 demo 里惊艳、在生产里翻车。

## 6. 小结 & 下一步

这一篇你需要带走的:

- **agent harness = 包在模型外面、负责执行的那层运行时代码 =「一切不是模型的东西」。** 用三个盒子记牢:**Agent = Model(想)+ Harness(让它动)+ Scaffolding(怎么动)。**
- 模型是个**无状态纯文本函数**;是 harness 让它变成「会动手、会记事、会守规矩、能连轴转」的 agent。
- **比例震撼**:社区对 Claude Code 源码的估算是 ~1.6% 决策逻辑 / ~98.4% harness(社区估算,非官方精确指标)——翻转「模型就是 agent」的直觉。
- **硬证据**:同一模型换 harness,Claw-SWE-Bench 上 Pass@1 从 38.6% 到 66.0%(差 27.4 pp);「约束绑定论题」说 harness 方差能颠倒模型排名;光改 scaffold 就能在 SWE-bench Verified 上涨约 10–20 分。
- **划界**:harness ≠ agent 框架(那是它的输入积木)≠ eval-harness(术语撞车,别混)。
- **心态**:harness engineering 正成为新的 prompt engineering(截至 2026 中的业界共识框架);**选/造哪套 harness,和选哪个模型同等重要。**
- **七层栈**就是你读 ⑦–⑪ 的地图:循环 → 上下文压缩 → 权限沙箱 → 子代理 → 扩展生态。

下一篇,我们直接钻进七层栈的**引擎室——第 ③ 层「agent loop」**,把这台 harness 发动机拆开:一次任务进来,循环怎么组装上下文、怎么调模型、怎么把工具调用**分发**出去、系统提示是怎么拼起来的、它又凭什么知道「这一圈该停了」。
