## 0. 全景:把「零件」和「样板」熔成一套「方法」

到这里,你已经走过了一段不短的路。板块二(⑥–⑪)给了你 harness 的**零件清单**:循环、系统提示、工具分发、上下文压缩、权限沙箱、子代理、扩展生态。板块三、四(⑫–㉘)带你把这些零件在**真实产品**上逐个看了一遍——Claude Code 怎么做压缩、Codex 怎么做沙箱、SWE-agent 怎么做工具界面、UCAgent 怎么做错误遏制。

这一篇要做的,是**把「零件」和「样板」熔成一套「方法」**:当轮到你自己动手设计(或挑选)一个 harness 时,该怎么想、按什么顺序做决定、哪些取舍是关键、哪些坑是前人反复栽过的。

```
板块二:零件         板块三/四:样板              板块五:方法(你在这)
┌──────────┐      ┌──────────────┐         ┌────────────────────┐
│ 循环·提示 │      │ Claude Code   │         │ ㉙ 怎么设计一个       │
│ 上下文·权限│  +   │ Codex·Cursor  │   ═══▶   │    好 harness(方法论)│
│ 子代理·扩展│      │ Devin·UCAgent…│         │ ㉚ 怎么评测它好不好   │
└──────────┘      └──────────────┘         │ ㉛ 落到完整案例上     │
   「有哪些件」        「别人怎么装」              └────────────────────┘
```

这不是一份「照抄就能用」的模板——**没有放之四海皆准的 harness**。它是一套**决策框架**:帮你在「为这个具体任务」的前提下,做出一串彼此自洽的取舍。

> 读这一篇的正确姿势:**把它当成一张「设计检查单」。** 每一节回答一个「你必须拿主意」的问题;把这些主意串起来,就是你的 harness 设计。

## 1. 什么才算「好」的 harness

先破一个误解:**好 harness 不等于「功能多」。** 一个塞满了工具、模式、插件的 harness,如果不可控、不可测、上下文一团糟,它是个烂 harness。

一句话立标准:

> **好的 harness,是在「给定任务、给定风险档」下,能让一个会犯错的概率模型,稳定、可控、可审计地把事做完的那套运行时。** 衡量它的不是「能干多少花活」,而是六个字:**可靠、可控、可观测。**

把这句话拆成六根可操作的标尺,它们贯穿本篇:

| 标尺 | 在问什么 | 主要由哪一层决定 |
|---|---|---|
| **可靠性** | 同一个任务反复跑,成不成、稳不稳? | 循环 + 错误恢复(⑦) |
| **可控性** | 人能不能随时看懂、随时叫停、随时纠偏? | 权限 + 审批 + 可打断(⑨) |
| **上下文效率** | 长任务里它会不会「记不住」或烧爆预算? | 上下文管理(⑧) |
| **安全性** | 它会不会闯祸、被骗、越权、外泄? | 权限沙箱 + 轨迹级防护(⑨) |
| **可扩展性** | 别人能不能不改源码就加能力、加规则? | hooks/MCP/skills(⑪) |
| **可观测性** | 出了问题,能不能追溯它到底干了什么? | 转录/日志(⑪㉚) |

**注意这六根标尺常常互相拉扯**:自主度调高(少问人)→ 可控性下降;工具加多(能力强)→ 上下文效率与安全下降;沙箱收紧(更安全)→ 能干的活变少。设计 harness 的本质,就是**在这六根标尺之间,为你的具体任务找一个自洽的平衡点**——这正是第①篇那句「架构是在冲突约束里找折中」在 harness 上的回声。

## 2. 前提:先回答「为谁、多自主、多大风险」

在动任何一个组件之前,先把三个前置问题答清楚——**它们像三个旋钮,拧到哪一档,后面所有设计都跟着变。**

- **任务形态**:是「步骤能事先枚举」的活(那也许根本不需要 agent,一个 workflow 更省更稳——回第①⑤篇那把尺子),还是「步骤事先无法枚举」的活(如修 bug、做验证)才值得上自主循环?
- **自主档位**:你要的是「人盯着、每步可打断」的半自主(Claude Code/Cursor 那档),还是「派出去、几小时后收结果」的高自主(Devin/OpenClaw 那档)?**自主度越高,你在上下文、终止、安全上要花的功夫就越重。**
- **风险档位**:它能碰到的东西,搞砸的代价有多大?改一个玩具仓库,和碰生产数据库、动真实资金、跑不可逆命令,是完全不同的风险档——**风险档直接决定你权限/沙箱那一层要收多紧。**

> 一句话:**先定「任务 × 自主 × 风险」这三个旋钮,再谈组件。** 跳过这一步直接堆组件,是新手最典型的错——你会造出一个「什么都能干、但不知道为谁而造」的四不像。

## 3. 设计方法论:按这个顺序拿主意

这一节是本篇的核心。我们按一个**有依赖关系的顺序**走一遍:每一小节对应一个必须拿的主意,并把前面案例里的真实做法当「可抄的样板」。

### 3.1 从任务倒推,而不是从功能正推

**最重要的一条原则,也最反直觉:别从「我想让它会什么」开始,要从「这个任务真正需要什么」倒推。**

Anthropic 在《Building Effective Agents》里给过一条朴素到容易被忽视的建议:**从最简单的方案起步,只有当简单方案顶不住时,才往上加复杂度。** 能用一次 LLM 调用解决的,别上工具;能用固定 workflow 解决的,别上自主循环;能用单 agent 解决的,别上多 agent。

**每加一层复杂度,你都在用「可控性 + 可预测性」换「灵活性」。** 这笔交易只在任务真的需要那份灵活时才划算。倒推法能救你于「过度工程」——它是本篇 §5 好几个坑的总根源。

> 一句话:**先问「这活最少需要什么」,而不是「我能给它装什么」。** harness 设计里,克制是一种美德。

> 💡 「最简单的方案」有时根本不是写代码。如果任务的步骤大体能事先枚举、只是需要把几个 LLM 调用和外部系统串起来,Dify、n8n 这类低代码/可视化编排就够用了——拖几个节点比自建一套 harness 省得多。把它们放在你的选项清单里,是对「倒推法」的一种诚实。

### 3.2 循环要笨,安全上限要硬

定下「确实需要 agent 循环」之后,循环本身**越简单越好**。

回忆⑦⑫篇那个反复出现的主线——**「极简循环,极重 harness」**:Claude Code 的循环就是朴素的「收集上下文 → 行动 → 验证 → 重复」,复杂度全在循环外的确定性工程里。你不需要一个精巧的控制流状态机;**复杂行为是从「简单循环 + 好工具 + 好上下文」里涌现的,不是从复杂循环里设计出来的。**

但简单循环必须配**硬性的安全上限**:步数上限、token 预算上限、墙钟时间上限、以及明确的「做完了」判定。这是 AutoGPT 时代最惨痛的教训(⑰篇)——**没有硬刹车的自主循环,会在死胡同里反复烧钱**。Hermes 默认约 90 轮上限、各家都有迭代/预算封顶,不是保守,是保命。

> 一句话:**让循环笨,让刹车硬。** 循环负责「转」,安全上限负责「别转到天荒地老」。

### 3.3 把上下文当预算,从第一天就精算

这是决定「能不能干长活」的命门,也是新手最容易低估的一层。

核心心态(⑧篇):**上下文窗口是绑定约束(binding constraint),不是无底缓冲区。** 哪怕 100 万 token,长任务也会填满;填满之后要么截断丢信息、要么压缩(有损)。所以从设计第一天起,就要有一套**上下文预算策略**:

- **压缩要分级、按便宜→贵触发**:抄 Claude Code 的 5 段式思路——先清旧工具输出(无损、免费),实在不够才让模型总结(有损、花钱)。别一上来就粗暴截断或全量总结。
- **该长期记住的,搬出易失的对话历史**:用记忆文件(CLAUDE.md / AGENTS.md 一类)存持久约定,用外部存储 + 检索(RAG)存大知识。这件事可以做到多彻底?Letta(MemGPT)干脆把记忆做成运行时的一等公民,让 agent 自己在有限窗口和外部记忆之间分页调度——记忆不再是附着在提示里的字符串,而是运行时管理的资源。
- **脏活外包给子代理保持主上下文干净**:让「读 50 个文件」的探索在子代理的独立上下文里发生,只把摘要收回来(⑩)。
- **对缓存友好**:把稳定的系统提示 + 工具定义做成长而不变的前缀,让 prompt cache 反复命中——Codex 和 Manus 都靠这个把长历史的成本压到接近线性(⑧⑬⑰)。

> 一句话:**上下文是你最贵、最稀缺的资源。** 一个不管上下文的 harness,做玩具没问题,一上真实长任务就崩。

### 3.4 权限与沙箱要分层,安全是轨迹级的

风险档越高,这一层越是重头戏(⑨篇)。有几条被反复验证的原则:

- **默认拒绝(deny-first)**:规则显式放行,而不是默认全开。
- **审批 + 沙箱是两根正交的轴**:审批管「要不要问你」(软同意),沙箱管「操作系统允不允许」(硬边界)。Codex 把这两根轴分得最清——先用 Seatbelt/Bubblewrap 把「最多能碰什么」框死,再谈问不问。**硬边界不该依赖模型听话。** 沙箱这根轴还能推到极致:E2B 把每次执行放进用完即弃的云端 microVM 里,让「炸了也无所谓」变成默认姿态;想连身份、网关一起托管,还有 AWS Bedrock AgentCore 这类把运行时整层外包出去的路子。
- **在 schema 层就杜绝,胜过运行时拦**:OpenDev 那个漂亮做法——让规划子代理**根本看不到写工具的定义**,于是「写」在架构上就不可能,而不是靠运行时检查去拦(⑦⑨)。
- **警惕审批疲劳**:Claude Code 的数据显示用户对约 93% 的弹窗直接放行——**问得多 ≠ 更安全**,弹窗多到人麻木就成了「安全剧场」。所以要有分级信任 + 智能判定,而不是无脑弹窗。反过来,当风险档确实高、你就是想「每步都过一遍」时,Cline 的 Plan/Act 双模式给了另一种答案:先在 Plan 模式里把方案谈定,再进 Act 模式逐步执行、逐步审批——把「先想清楚再动手」做成了显式的运行时状态。
- **安全是轨迹级的,不是终点级的**:HarnessAudit 的洞见(⑨)——一个 agent 可能给出正确的最终答案,却在**中途**越了权。只看最终输出的评估是瞎的;要审的是整条轨迹。

> 一句话:**让它动手,又不让它闯祸,必须一起设计。** 而且安全的上限由 harness 定,不由模型定。

### 3.5 工具要少而精,为模型而设计

工具是 agent 的手脚,但**不是越多越好**(③⑬篇)。

- **面窄、可审计**:Codex 收敛到「一把 shell + 一种 apply_patch 补丁」,而不是几十个细粒度文件工具——面越窄,模型越不容易用错,审批越容易判断爆炸半径。
- **为模型设计接口(ACI)**:SWE-agent 的核心洞见(③⑯)——工具是给**模型**用的界面,不是给人用的。错误信息要可读、反馈要简洁、操作要防呆。同一个底层能力,ACI 设计得好不好,SWE-bench 分数能差一大截。
- **按需加载,别撑爆上下文**:MCP 工具定义延迟加载、用到才装进上下文(⑪)——工具再多,也别让它们的 schema 常驻占预算。工具接口要不要整个押注 MCP?Goose 就是把 MCP 当原生底座的一个极端——工具几乎全走 MCP 扩展接进来,好处是接生态省心,代价是你的能力边界跟着 MCP 生态的成熟度走。

> 一句话:**给模型「一把好用的瑞士军刀」,而不是「一整面五金墙」。**

### 3.6 子代理:先为「隔离」,再为「并行」

什么时候该拆多 agent?记住⑩篇那个反直觉的排序:**子代理首先是上下文隔离手段,其次才是并行手段。**

该拆的信号:某段工作会往主上下文里灌大量探索性内容(读一堆文件、试一堆失败方案)、职责清晰可切、或天然可并行。这时派个子代理去,独立上下文里干,只回摘要——主循环保持干净。

**别为并行而并行**:多 agent 会带来上下文碎片化、摘要丢信息、协调与调试变难、成本随 agent 数上升。默认单 agent,确有隔离/职责/并行的需要,再拆。

真要拆,「怎么组织这群 agent」本身就有好几种范式可挑,各押一个不同的抽象:CrewAI 押「角色 + 委派」,AutoGen 把「对话即编程」当核心,MetaGPT 把人类团队的 SOP 固化进流程,OpenAI Agents SDK 用 handoff 在 agent 间转交控制权,LangGraph 则把整个协作画成一张可控的状态图。选范式的前提仍是上面那句——先确认你真的需要多 agent,再谈用哪套编排。

> 一句话:**多 agent 买的是「隔离 + 分工 + 并行」,付的是「协调 + 一致性 + 成本」。默认单 agent。**

### 3.7 留好扩展点与观测点

最后一层,决定 harness 的**长期生命力**(⑪篇)。

- **扩展点**:hooks(让组织把安全/质量/审计编码进 agent)、MCP(标准化接工具)、skills(可复用能力)——它们让别人不 fork 你的源码就能扩展你。生态是护城河;而 OpenClaw/Hermes(⑱)更进一步,连「别的 agent」都能当扩展点来编排。**别忘了「换模型/换供应商」也是一种长期扩展点**:OpenCode 用 client/server 拆分把接口和 provider 解耦、做到 provider 无关,Vercel AI SDK、Strands Agents 也都把「底层换成哪家模型」抽象成可替换的一层——押死单一供应商,是给未来的自己埋雷。
- **观测点**:从第一天就把轨迹、工具调用、成本、错误**记全**(append-only 转录是个好范式)。没有可观测性,agent 就是个黑盒——出了问题无法排障,也无法评测(㉚篇)。

> 一句话:**扩展点决定它能不能长成生态,观测点决定它能不能被调试和信任。**

## 4. 关键心态:三条贯穿全篇的设计哲学

如果把上面七小节压成三句话,请带走这三条:

- **极简循环,极重 harness。** agent 的可靠性,来自你能确定性控制的那 98%(权限、压缩、存储、观测),而不是模型即兴发挥的那 2%。别把智能都指望在循环里,把功夫下在循环外。
- **harness 常比模型更重要(⑥)。** 同一个模型换套 harness,干活能力能差一大截。所以「选/造哪套 harness」是和「选哪个模型」同等重量的一等决策——别只盯着模型排行榜。
- **克制优先,按需加码。** 从最简单起步,每一层复杂度都要用「它顶不住了」来正当化。过度工程不是能力的体现,是没想清楚任务的体现。

> 一句话总纲:**好 harness 不是「最强大的」,是「最适配这个任务、这个风险档、这个自主度的」——而适配,来自克制和取舍,不是堆料。**

## 5. 常被忽视的坑与最佳实践

把全站散落的坑收拢成一张「设计前必看」的清单——它们几乎每一条都真实地坑过人:

1. **没有硬性终止/预算上限。** 自主循环最致命的坑:方向错了还闷头转,烧 token 到天荒地老(AutoGPT 教训)。**最佳实践:步数、token、时间三重封顶 + 明确的「做完」判定,从第一天就加。**
2. **把上下文当无底缓冲区。** 无脑塞满窗口 → 成本爆炸、注意力稀释、关键信息被挤掉。**最佳实践:分级压缩(便宜→贵)、记忆文件外置、子代理隔离、对缓存友好——把上下文当预算精算。**
3. **审批疲劳当成安全。** 弹窗弹到人麻木、无脑放行,等于没有权限。**最佳实践:deny-first + 分级信任 + 智能判定 + 缓存已批准项,而不是「事事都问」。**
4. **只看最终输出,忽视中途越权。** agent 可能答对却在轨迹中途干了坏事。**最佳实践:审整条轨迹;把危险动作在 schema 层就杜绝(让模型压根看不到该工具)。**
5. **为并行而并行地上多 agent。** 徒增碎片化、协调成本与调试难度。**最佳实践:默认单 agent;只在「上下文要隔离/职责要切分/可真并行」时才拆。**
6. **把「框架」当「harness」。** LangChain/LangGraph 这类是 SDK 积木(⑥⑰),是 harness 的**输入**,不是一套能直接上生产的完整 harness;拿它当 harness 用,会发现权限、观测、会话持久化这些生产必需件全得自己补。**最佳实践:分清「积木」和「成品」,该补的生产件别指望框架送。**
7. **过度工程:一上来就套最复杂的架构。** 用重型规划/多 agent/一堆插件去干一件简单事。**最佳实践:从最简单起步,顶不住再加(§3.1)。**
8. **没有可观测性。** agent 变黑盒,出问题无法追溯、无法评测。**最佳实践:全量记录轨迹/工具调用/成本,append-only,从第一天就有。**
9. **装一堆没人维护的插件/MCP。** 每个都是攻击面和上下文负担。**最佳实践:按需加载、精选、可审计;工具面越窄越好。**
10. **用了不公开 harness 的 benchmark 分数去横比模型。** 长程任务上 harness 方差可超过模型方差(Binding Constraint Thesis,⑥),不披露 harness 的比较是误导。**最佳实践:比模型时固定并披露 harness——这也是㉚篇的主题。**

---

## 6. 小结 & 下一步

这一篇你需要带走的:

- 好 harness 的标尺是 **可靠、可控、可观测**(六根细标尺互相拉扯),不是「功能多」。
- 动组件前先拧三个旋钮:**任务形态 × 自主档 × 风险档**。
- 设计顺序:**从任务倒推 → 笨循环硬刹车 → 上下文当预算 → 权限沙箱分层 → 工具少而精(ACI)→ 子代理为隔离 → 留好扩展与观测点。**
- 三条哲学:**极简循环极重 harness、harness 常比模型重要、克制优先按需加码。**
- 十个坑,几乎每个都对应一条「从第一天就该做对」的最佳实践。

方法论有了,但一个绕不开的问题是:**你怎么知道自己设计的 harness 到底好不好?怎么公平地比较两套 harness、两个模型?** 这就要谈评测——而评测里全是陷阱。

> 👉 继续读 **第 ㉚ 篇:怎么评测一个 Agent**。
