## 0. 先看全景:从「工作台」到「怎么管这张工作台」

第 ④ 篇给过你一个结论:**上下文窗口(context window)是 agent 的工作台,也是它最硬的那条约束。** 模型每次「想」,只能看见你摆在这张台面上的东西;台面就那么大,一长任务就撑满,而且越满越贵、越满越慢、越满越容易糊涂。那一篇讲的是**概念与直觉**——窗口是什么、记忆有哪几种、RAG 和长上下文各解决什么。

这一篇接着往下走一层:**harness 到底用什么具体机制,在这张工作台快满的时候「腾地方」,又不把关键东西一起扫进垃圾桶。** 这是第 ④ 篇的工程续集——从「上下文是稀缺资源」这句口号,落到「一行行代码怎么把它当稀缺资源来管」。

```
第 ④ 篇(概念)                     第 ⑧ 篇(harness 机制,你在这)
───────────────                   ─────────────────────────────
上下文窗口是工作台        ───▸      台面快满了,怎么按「便宜→贵」逐级腾地方(压缩流水线)
窗口有限、越长越贵越慢    ───▸      哪些该搬去「窗外」长期存(记忆文件 / 文件系统)
记忆分种类               ───▸      怎么在腾地方时不弄坏缓存(KV-cache 命中率)
                        ───▸      怎么用子代理把脏活隔离在窗外(细节留 ⑩)
```

老场景照旧,一路回扣它:**让一个 agent 去修某开源仓库里的一个 bug——读 issue → 找相关文件 → 改代码 → 跑测试 → 提 PR。** 只不过这次它不是修一个五分钟的小 bug,而是啃一个要读几十个文件、跑几十轮测试、干上一两个小时的硬骨头。**任务一长,上下文管理就从「可有可无的优化」变成「能不能干完的前提」。**

## 1. 上下文管理到底是什么

一句话定义:

> **上下文管理(context management)= harness 在每一次调用模型之前,主动地裁剪、总结、搬运对话历史,让「此刻最相关」的信息留在窗口里,让不相关的旧东西腾出去。**

注意三个词:**每一次**、**主动**、**此刻最相关**。

- **每一次**:这不是任务结束才做一遍的收尾,而是**每一轮循环、每次调模型前**都要过一遍的日常动作(回顾第 ② 篇的循环:想一步 → 做一步 → 看结果 → 再想,每一「想」之前都要先把台面收拾好)。
- **主动**:模型自己不会收拾台面。它只会一股脑往历史里追加——又读了一个文件、又跑了一次测试、又贴了一大段报错。**是 harness 这层代码替它做减法。**
- **此刻最相关**:两小时前读的那个文件、十轮前那次跑挂的测试全文,现在多半没用了;但「issue 到底要我修什么」「我已经确认过哪些文件无关」这类事实,得一直留着。管理的本质,是**在有限台面上做一道「留谁、扔谁、扔之前要不要先记下来」的取舍题**。

和第 ④ 篇划清界线:④ 讲的是「有哪几种记忆、RAG 是什么」这些**概念地图**;⑧ 讲的是「harness 具体拿什么机制去执行」这些**工程动作**。你可以把 ④ 当成「认识工作台」,把 ⑧ 当成「工作台整理术的操作手册」。

> 关键直觉:**上下文管理不是「把历史存好」,而是「每次调模型前,决定这一台面该摆什么」。** 存是记忆的事,摆是管理的事——这一篇讲的是「摆」。

## 2. 前提:为什么长任务必然撞上这堵墙

在往下讲机制之前,先把三个**已经在第 ④ 篇讲透、这里只做地基**的前提钉牢——它们合在一起,就是「长任务必然要做上下文管理」这个结论的三条腿:

1. **LLM 调用之间是无状态的。** 模型不记得上一次调用发生了什么。它这一轮能「记住」的一切,全靠 harness 把历史重新塞回窗口里喂给它。**历史不是模型的记忆,是 harness 每轮重新递上去的一叠纸。**(见第 ④ 篇)
2. **窗口有限,且越长越贵越慢。** 截至 2026 年中,旗舰模型(如 Claude 的 Opus 旗舰与 Sonnet 5)已能上到 100 万 token 上下文——听起来很大,但填得越满,每一轮的费用和延迟就越高,而且注意力会被稀释(下节讲)。**「窗口大」不等于「随便塞」。**
3. **长任务的历史只增不减。** 每读一个文件、每跑一次测试、每贴一段 stack trace,都往历史里堆料。一个要干两小时的修 bug 任务,天然会产出**远超窗口容量**的历史。

三条腿一拼,结论无法回避:

> 一句话:**无状态 + 窗口有限 + 历史只增不减 = 任何够长的任务,都一定会撑满窗口。** 于是问题不是「要不要管理上下文」,而是「撑满那一刻,harness 拿什么招接住」。

## 3. harness 怎么管上下文:五个机制

这一节是全篇的重心。上下文管理不是一招,而是一套配合打的组合拳。我们从「窗口内怎么腾地方」(3.1、3.2),讲到「把该长期记的搬去窗外」(3.3、3.4),再到「用子代理把脏活隔离在窗外」(3.5)。

### 3.1 为什么必须管理:窗口是「绑定约束」,以及上下文腐化

先说清楚「不管会怎样」,你才会认真对待「怎么管」。

第 ⑥ 篇提过一个词:**绑定约束(binding constraint)**——在一堆约束里,真正卡住你、决定成败的那一条。对一个跑长任务的 agent 来说,**上下文窗口往往就是那条绑定约束**。它能不能啃下一个要读几十个文件的 bug,常常不取决于模型聪不聪明,而取决于 harness 能不能在有限台面上一直摆着「对的东西」。这也回扣了第 ① 篇那个反直觉钩子:**同一个模型,换个更会管上下文的 harness,干长活的能力能差一大截。**

不管的话,会撞上两种「上下文腐化(context rot)」:

- **旧内容挤占。** 台面就那么大,两小时前读过、现在无关的文件全文还赖在上面,新的、真正相关的信息就挤不进来——你被迫要么截断、要么撑爆。
- **注意力稀释。** 就算硬塞得下,窗口越长,模型的注意力越容易被无关内容分走,关键指令反而被淹没。业界管这叫**「lost in the middle」**——塞在长上下文中段的信息最容易被模型忽略(见第 ④ 篇)。**台面越乱,模型越找不到重点。**

```
不管理:                          管理后:
┌─────────────────────────┐      ┌─────────────────────────┐
│ issue 原文               │      │ issue 原文(留)          │
│ 文件A全文(早没用了)     │      │ 「A、B 已排除无关」(总结) │
│ 文件B全文(早没用了)     │  ─▸  │ 当前假设 + 正在改的文件C   │
│ 20 轮前的测试全文报错     │      │ 最近一次测试结果(留)      │
│ ...台面塞满、重点被淹...  │      │ ...台面干净、重点突出...   │
└─────────────────────────┘      └─────────────────────────┘
```

> 一句话:**不管上下文,agent 不是「装不下」就是「找不到重点」——两种死法,后一种更阴险,因为它看起来还在跑,只是越跑越蠢。**

### 3.2 压缩流水线:从便宜到昂贵、从无损到有损,逐级触发

这是本篇的技术核心。我们以 **Claude Code 的压缩流水线(compaction pipeline)** 为范例——据其技术分析(截至 2026 中),它在**每次调模型前**评估 token 预算,如果逼近上限,就按下面五级**从便宜到昂贵**依次触发,前一级没腾够地方,才动用下一级:

```
预算充足 ──▸ 不做任何事,直接调模型
     │
     ▼ 预算紧张,逐级触发(便宜 → 贵、无损 → 有损)
┌───────────────────────────────────────────────────────────┐
│ ① 预算削减 Budget Reduction  收紧本轮可用预算,先给压缩留出余量  │ 最便宜
│ ② 清理 Snip                 截掉过旧的历史碎片(可丢的旧片段)   │   │
│ ③ 微压缩 Microcompact        清掉旧工具输出(读过的文件、旧报错) │   │ 无损为主
│ ─────────────────────────────────────────────────────────  │   ▼
│ ④ 上下文坍缩 Context Collapse 折叠大段历史(仍可恢复的投影式)   │
│ ⑤ 自动压缩 Auto-Compact      让模型把整段历史"总结"成一段摘要   │ 有损、花钱、最后一招
└───────────────────────────────────────────────────────────┘
```

关键不在于记住这五个名字,而在于**理解它为什么按「便宜 → 贵」排序**。想清楚这条排序,你就抓住了整套设计哲学:

- **前几级几乎无损、几乎免费。** 比如「微压缩」主要是**清掉旧的工具输出**——你 20 轮前 `Read` 出来的那个文件全文、那次跑挂的测试的完整 stack trace,现在早没用了。丢掉它们**不损失任何还需要的信息**(模型已经从中得到了它要的结论),而且**是纯本地操作、不用再调一次模型**,所以又安全又便宜。这类操作先上。
- **最后一级有损、花钱。** 「自动压缩」是**让模型把一大段历史读一遍、总结成一小段摘要**——它是这五级里**唯一要额外调一次模型**的(花钱、花时间),而且**必然有损**:总结就意味着丢细节,万一丢的恰好是关键事实(比如「我们早就确认过 bug 不在 `parser.py`」),后面模型可能又绕回去踩一遍坑。所以它是**最后一根救命稻草,不到万不得已不动**。

把这条排序翻译成一句设计原则:

> **能不花钱就不花钱,能不丢信息就不丢信息;先扔「明确没用的垃圾」(无损),实在不够了才「压缩还有点用的东西」(有损)。** 反过来做——一上来就让模型总结全历史——既贵又容易误伤,是新手最常犯的错。

回到修 bug 的场景:agent 读了 12 个文件、跑了 8 轮测试,台面快满了。理想的 harness 会先**把那 12 个文件的全文清掉**(它需要的结论早已吸收),把 8 轮测试**只留最近一两轮**——光这一步就腾出大半台面,而且一个字的「有用信息」都没丢。只有当这些无损手段都腾不够时,才轮到「请模型把前一小时的经过总结成三段话」。**顺序,就是这套流水线的灵魂。**

### 3.3 记忆文件:把「该长期记住的」搬出易失的对话历史

压缩流水线解决的是「窗口内怎么腾地方」。但有些东西**根本不该待在易失的对话历史里**——对话历史会被压缩、会被截断、会随任务结束而蒸发。真正该长期记住的知识(项目怎么构建、测试怎么跑、这个 agent 上次踩过什么坑),应该搬到**窗外的文件**里,需要时再加载。这就是**记忆文件(memory file)** 作为**外部记忆(external memory)** 的思路(第 ④ 篇讲过「外部记忆」这个概念,这里讲 harness 具体拿哪些文件来做)。

以 Claude Code 为例(截至 2026 中),它有一套分层的记忆文件:

| 记忆文件 | 谁来写 | 何时加载 | 装什么 |
|---|---|---|---|
| `CLAUDE.md` / `AGENTS.md` | 人写(项目指令) | 每次会话开始读根目录文件;**子目录里的文件懒加载**——只有当 agent 真的碰到那个目录下的文件时才读进来 | 项目约定:构建命令、代码风格、"改这个模块前先看那个" |
| 自动记忆 `MEMORY.md` | **agent 自己写** | 会话开始时加载**前 200 行或 25KB(以先到者为准)** | agent 攒下来的经验教训:上次发现的坑、这个仓库的隐藏习惯 |
| 对话历史 | 循环自动追加 | 一直在窗口里(直到被压缩/截断) | 本次任务的临时过程——**易失,随时可能被 3.2 的流水线清掉** |

这张表的精髓是**「持久度」由高到低的分层**:项目指令最稳(人维护)、自动记忆半稳(agent 维护、只加载开头一截)、对话历史最易失。**上下文管理的一半功夫,就是判断一条信息该待在哪一层。** 一条「这个仓库的测试必须用 `make test` 而不是 `pytest`」的事实,如果只待在对话历史里,压缩一来就没了、下次会话更是从头再踩;把它写进 `CLAUDE.md`,它就成了**每次都自动摆上台面的常识**。核心心态很简单:**别让「该记一辈子的知识」漂在「随时会被扫掉的对话历史」里**——稳的搬进记忆文件,临时的留在历史里让它自然被压缩。(第 ⑫ 篇拆 Claude Code 时会把这套三层记忆再展开一次。)

⚠️ 一个诚实的限定(截至 2026 中):自动记忆只加载**前 200 行 / 25KB**。这既是好事(不让记忆文件无限膨胀反过来吃满窗口),也是个坑——**写进去但排在 200 行之后的东西,等于没写**(见 §5)。

### 3.4 文件系统即记忆 + 复述法:Manus 的两招,和那个 10× 的数字

如果说记忆文件是「结构化的窗外存储」,那 **Manus** 团队公开的实践,把「窗外」这件事推到了更彻底:**把整个文件系统当成一块无限、持久的外部记忆。**

据 Manus 团队(联合创始人、首席科学家 Ji Yichao 的公开文章)介绍,他们的做法有三个值得学的点:

- **文件系统即无限记忆 + 可恢复压缩(restorable compression)。** 长内容(比如一个网页的全文、一份大文件)**不直接堆在上下文里**,而是写到沙箱的文件系统上;上下文里只留一个「指针」(URL 或文件路径)。要压缩时,**丢掉正文、留住指针**——需要时随时能按路径把正文重新读回来。**这是「有损压缩」里最聪明的一种:表面上丢了,实际上可恢复,几乎无损。** 这和 3.2 里 Claude Code「总结全历史」的有损压缩形成对比——一个把信息搬到窗外可召回,一个把信息揉成摘要难还原。
- **复述法(recitation):用 `todo.md` 把注意力拉回目标。** 长任务跑到第 40 步,最初的目标很容易被埋在上下文最开头、被模型的注意力冷落(又是 lost-in-the-middle)。Manus 的招法是维护一个 `todo.md` 任务清单,并**在上下文的末尾反复"复述"当前进度**——把「我到底在干嘛、干到哪了」不断挪到模型注意力最集中的位置。**这是用极低的成本,对抗长任务里的"目标漂移"。** 回到修 bug:一个跑了一小时的 agent,靠不断复述「☐ 让 test_parse 变绿 ☑ 已排除 parser.py」,就不容易忘记自己最初是来干嘛的。
- **KV-cache 命中率是最重要的生产指标。** 这是 3.3、3.4 里最反直觉、也最省钱的一条。据 Manus 团队公开的数字:同一个模型(Claude Sonnet),**命中缓存的输入 token 约 \$0.30/百万,未命中约 \$3.00/百万——差约 10×**;而 agent 的输入输出比常高达 **约 100:1**(读的远多于写的)。这意味着:**你怎么组织上下文,直接决定成本是 1 份还是 10 份。** 命中缓存的前提是**前缀稳定、只在末尾追加**——所以「在历史中间插改一段」「每轮重排上下文」这类看似无害的操作,会把缓存全部击穿,让成本暴涨。

这条给上下文管理加了一个此前没提过的维度:**管理上下文不只是「装不装得下」,还得「改动方式对缓存友不友好」。**

> 关键心态(标注为 Manus 团队公开数字):**KV-cache 命中与否,价差约 10×。** 所以好的上下文管理会像"守着一条流水线"一样只在末尾追加、保持前缀稳定——为了省下那 10 倍的钱,宁可绕开"重排历史"这种看着更整洁的做法。

⚠️ 补一句沿革(截至 2026 中):Manus 的核心是**单个 agent 循环 + 一条 append-only 的事件流**(平均约 50 次工具调用),而不是一上来就多 agent 分工——它的多 agent 并行(Wide Research)是后来(1.6 版)才加的。别把它讲成「一个复杂多智能体系统」;它恰恰是**「靠把上下文工程做到极致,让单循环也能干长活」** 的典范。

### 3.5 子代理隔离:把脏活丢到窗外,只回传一段摘要

最后一招,是把上下文管理和「分工」结合起来:**子代理(subagent)隔离。**

设想修 bug 时你需要「搞清楚这个函数在整个仓库里被谁调用」——这可能要读 50 个文件。如果让主 agent 亲自读,这 50 个文件的全文会**全部涌进主上下文**,瞬间把台面冲垮(还得靠 3.2 去费劲清理)。更聪明的做法是:**把「读 50 个文件」这件脏活丢给一个子代理**,让它在**自己独立的、全新的上下文窗口**里读个够,读完**只回传一段两百字的摘要**给主 agent。

```
主上下文(要保持干净)
     │  「去查清 foo() 被谁调用」
     ▼
┌──────────────┐   读 50 个文件(脏活全在这个窗口里)
│   子代理      │   ┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄
│ (独立上下文)  │   50 个文件全文都留在子代理这边
└──────┬───────┘
     │  只回传:「foo() 被 3 处调用:A/B/C,其中 B 传了 None」
     ▼
主上下文(依然干净,只多了一段摘要)
```

从上下文管理的视角看,子代理的价值**不只是并行,更是「上下文隔离」**:它把「大量探索、失败尝试、无关全文」这些**会污染主台面的脏活**,统统关在一个隔离的窗口里,主上下文只收干净的结论。**这等于给主 agent 装了一个"承包商":脏活外包出去,自己只看验收报告。**

这里只讲它作为「上下文管理手段」的这一面。子代理还有并行、专精、信任边界等更多用途,以及它自己的坑(比如摘要丢了主 agent 后来才需要的细节)——**完整拆解留给第 ⑩ 篇。**

---

## 4. 关键心态:把上下文当「预算」精算,而不是当「无底缓冲区」

如果这一篇只让你带走一句话,是这句:

> **Production 级的 agent,把上下文窗口当成一笔要精算的"预算",而不是一个能随便往里塞的"无底缓冲区"。**

新手默认后者:反正窗口有 100 万 token,先全塞进去再说。老手默认前者:**每一轮,每一个 token 都在和别的 token 抢台面**;塞进去一段没用的旧文件全文,就等于挤掉了一段可能有用的新信息、稀释了一分模型的注意力、还可能击穿缓存多花 10 倍钱。于是老手会像管现金流一样管上下文——**该扔的立刻扔(无损优先)、该搬窗外的搬窗外(记忆文件 / 文件系统)、该外包的外包(子代理)、绝不为"看着整洁"去干击穿缓存的傻事。**

这带出全篇最想让你记住的一个判断:**很多时候,一个 agent 能不能干"长活",卡点不在模型智力,而在上下文管理。** 一个绝顶聪明的模型,配一套只会「塞满 → 撑爆」的 harness,连一个要读 30 个文件的 bug 都啃不下来;一个中等模型,配一套会「无损压缩 + 外部记忆 + 复述目标 + 缓存友好」的 harness,却能稳稳跑完两小时。这正是第 ① 篇「同模型不同 harness 差一大截」那句话,在「长任务」这个维度上的具体兑现。

> 一句话:**别问"这个模型够不够聪明去干这个长活",先问"我的 harness 会不会管上下文"。** 后者常常才是那条绑定约束。

## 5. 常被忽视的坑

1. **无脑塞满窗口。** 「窗口有 100 万 token,先全塞进去」是最常见的错。塞满不仅贵、慢,更会**稀释注意力**——关键指令被淹在垃圾里,模型越读越蠢(§3.1 的「注意力稀释」)。**怎么避:** 把「进不进窗」当成每轮都要做的主动决策,默认少塞,只摆此刻相关的。
2. **有损压缩丢了关键事实。** 「自动压缩」让模型总结历史,**总结必然丢细节**。万一丢的是「我们早确认过 bug 不在 `parser.py`」,模型会绕回去重踩,白烧一轮又一轮。**怎么避:** 把有损压缩放在最后一级(§3.2 的排序),优先用无损手段;把「必须永久记住」的结论**主动落到记忆文件**,别指望它在摘要里活下来。
3. **记忆文件过期误导。** 记忆文件是双刃剑:一条过时的「用 `pytest` 跑测试」会让 agent **信心满满地走错路**——错误的记忆比没有记忆更危险,因为它带着权威感。还要记得自动记忆**只加载前 200 行 / 25KB**(§3.3):排在后面的等于没写。**怎么避:** 记忆文件要能被审查、被更新;别把它当只写不读的日志。
4. **缓存不友好,成本悄悄飙 10 倍。** 「每轮把上下文重排得更整洁」「在历史中间插改一段」——这些看着无害的操作会**击穿 KV-cache**,让本该 \$0.30 的 token 变成 \$3.00(§3.4,Manus 公开数字)。这类成本泄漏**不报错、不告警**,只在账单上默默翻倍。**怎么避:** 前缀稳定、只在末尾追加;把「缓存友好」当成上下文设计的硬指标。
5. **压缩时机太晚,污染已经发生。** 等窗口 99% 满、模型已经开始被无关内容带偏,才想起来压缩——晚了。此时台面早已乱到影响判断,压缩只是给一堆已经腐化的历史"擦桌子"。**怎么避:** 压缩要**前置、常态化**(每轮 pre-check,§3.2),在腐化发生之前就持续腾地方,而不是等撑爆了才抢救。
6. **该外包的脏活留在主上下文里干。** 让主 agent 亲自读 50 个文件、亲自跑一堆探索性尝试,把主台面冲得一塌糊涂。**怎么避:** 大规模探索、失败率高的脏活,丢给子代理在隔离窗口里干,主上下文只收摘要(§3.5,详见第 ⑩ 篇)。

## 6. 小结 & 下一步

这一篇你需要带走的:

- **上下文管理 = harness 在每次调模型前,主动裁剪 / 总结 / 搬运历史,让此刻最相关的信息留在窗口里。** 它是第 ④ 篇「上下文是稀缺资源」的工程续集(④ 讲概念,⑧ 讲机制)。
- **不管理会腐化两次:旧内容挤占(装不下)、注意力稀释(找不到重点)**——后者更阴险。
- **压缩流水线按「便宜 → 贵、无损 → 有损」逐级触发**(以 Claude Code 五段式为范例):先清明确没用的旧工具输出(便宜、无损),实在不够才让模型总结历史(花钱、有损)。**排序,就是灵魂。**
- **把该长期记的搬出易失的对话历史:** 记忆文件(`CLAUDE.md` / `MEMORY.md`,分持久度三层)、文件系统即记忆 + 可恢复压缩(Manus)、`todo.md` 复述法把注意力拉回目标;并牢记 **KV-cache 命中与否价差约 10×**(Manus 公开数字),缓存友好是硬指标。
- **子代理是"上下文外包":** 把弄脏台面的脏活丢到隔离窗口,只收摘要(详见第 ⑩ 篇)。
- **心态:把上下文当预算精算,不当无底缓冲区。** 能不能干长活,常卡在这里,而非模型智力。

你现在会造工具、会管上下文、能让 agent 干长活了——但一个能跑 shell、能改文件、能连外网的循环,威力和风险是一体两面。它可能帮你修好 bug,也可能删掉不该删的东西,或被一段恶意文字骗着去干坏事。**「让它动手」和「不让它闯祸」必须同时设计。**

> 👉 继续读 **第 ⑨ 篇:权限与安全** —— harness 怎么给这台越来越能干的机器,装上刹车和护栏。
