## 0. 全景:回到命令行,和一个只干「读图—改代码—提交」三件事的老手结对

⑫⑬ 拆的是终端里的自主编码 agent(Claude Code、Codex CLI),⑭ 拆的是长在编辑器里的 Cursor。这一篇我们把镜头拉回**最朴素的那一格**——命令行里的**开源结对编程器(pair programmer)**,代表作是 **Aider**。

为什么值得单独拆它一章?因为 Aider 比上面那几位都**老**(2023 年就有了),而且它没有靠「模型越来越强」来解决问题,而是**发明了一个至今仍被反复致敬的机制:repo map(仓库地图)**——用一张一千来 token 的「代码结构缩略图」,让 agent 在**根本不读完整个仓库**的前提下「看懂」一个大代码库该从哪下手。这是④篇「上下文工程(context engineering)」最漂亮、最可教的一个真实案例:不是把整个代码库塞进窗口,而是**算出哪些符号最相关,只塞那一小撮**。

```
你可能以为大代码库的问题这么解:                 Aider 实际这么解:
   把整个 repo 塞进 100 万 token 窗口       →    tree-sitter 解析出所有符号
   (贵、慢、还是会满、还稀释注意力)              用图排名(类 PageRank)算「谁最相关」
                                                只把 ~1k token 的缩略图塞进去
```

> 读这一篇的正确姿势:**别只记住「Aider 是个开源结对工具」,要记住它把④篇那句「上下文是稀缺资源」当成了立身之本——repo map 就是「选择性上下文(selective context)」这个抽象概念长出来的一具真实身体。**

篇末我们再把 **Google Gemini CLI** 摆到对面——同样是 CLI agent,但它走的是另一条路(ReAct 循环 + 百万上下文 + 内置工具),两相对照,你会更清楚「CLI agent」内部其实有好几种不同的活法。

## 1. Aider 是什么、谁做的、现在什么状态

一句话:

> **Aider 是一个开源、终端原生的「AI 结对编程 CLI」——你在本地 git 仓库里跑起它,用自然语言跟一个大模型聊你想怎么改,它把模型给出的编辑落到你的文件上,并且默认每改一次就自动 git 提交一次。**

几个关键事实(据官方文档,截至 2026 年中):

- **谁做的**:由 **Paul Gauthier** 发起,用 **Python** 写成,pip / uv 即可安装。它是一个**无常驻进程的 stateless CLI**——没有后台守护进程,「状态」全交给 git 历史保管。这一点后面讲设计时是关键。
- **什么状态**:2023 年首发;截至 2026 年中,PyPI 上最新版是 **v0.86.2(2026 年 2 月 12 日)**(v0.86.0 是 2025 年 8 月那版,之后有 .1 / .2)。**注意**:它 2024–2025 年那种「大约两周一发」的高频节奏,到 2026 年已明显放缓成「偶尔发一版」——所以别把「频繁迭代」当成它现在的标签。GitHub 上大约 4.7 万 star(2025 年中的快照,现在多半更高;别把精确数字当常量)。
- **跑在什么模型上**:**模型无关(model-agnostic)**是它的核心卖点——通过 `litellm` 这层抽象,同一个 Aider 可以对接 OpenAI、Anthropic、Google Gemini、DeepSeek、Ollama、OpenRouter、xAI 等一大票后端。**自带模型钥匙(BYOM,bring your own model)**,你用谁家的模型、花谁的钱,自己定。

**别和它的「近亲」搞混**:Aider 不是 IDE 插件(那是 Copilot、Cursor 干的活),它是**纯终端**的;它也不是 Claude Code / Codex CLI 那种「有一个自主 shell 工具、能自己反复跑命令」的 agent。用⑫篇的话讲——Claude Code 的循环会自己决定去 `Bash` 里跑 `npm test`;Aider 的「跑命令」则是**你触发、你配置**的(下面 3.4 会讲清这条界线)。

> 一句话:**Aider 的定位是「结对(pair)」,不是「代跑(delegate)」——它假设你坐在旁边、盯着每一步,而不是你派它出门、明天回来收结果。**

## 2. 一次任务在它体内怎么流动

老规矩,先看血液怎么流。设想你在一个几百个文件的 Python 项目里跑起 Aider,把两个你已经看好的文件加进会话,然后敲下:

> 「把 `parser.py` 里的配置解析改成支持 YAML。」

Aider 内部大致这样走一圈(对照②篇的「单步」和④篇的「上下文工程」):

```
你在 git 仓库里跑起 aider,把 parser.py 加进会话
   │
   ▼
① 建/更新 repo map    tree-sitter 扫描所有被 git 跟踪的文件 → 抽出定义与引用
                     → 建成依赖图(引用方文件 ──▸ 定义方文件)
                     → 跑「个性化 PageRank」,偏向你加进会话的文件
                     → 挤出一张 ~1024 token 的结构缩略图(--map-tokens 可调)
   │
   ▼
② REPL 读入你的话     在 `>` 提示符读到「支持 YAML」这句
   │
   ▼
③ 组装上下文         系统提示(含当前编辑格式的说明) + repo map
                     + 你显式加进来的文件全文 + 对话历史
   │
   ▼
④ 调模型            经 litellm 发给你配的模型(Claude / GPT / Gemini …)
   │
   ▼
⑤ 模型按格式产出改动  用当前编辑格式回一段改动
                     (whole / diff / udiff / architect …见 3.2)
   │
   ▼
⑥ 应用编辑          解析并落到文件;匹配不上时走「模糊回退」
                     (空白归一化、上下文放宽,尽量把 diff 贴对位置)
   │
   ▼
⑦ 可选:跑测试/lint   若你配了 --test-cmd / --lint 或敲了 /test:自动跑
                     → 失败的报错以 tree-sitter 格式化的形式贴回给模型,迭代修
   │
   ▼
⑧ 自动提交          先把你原有的「脏改动」单独提交一次(保护你的手工活)
                     再把 AI 的这次编辑用一条「小模型生成的信息」提交
                     /undo 一键回滚,/diff 看改了啥
```

三个值得当场记住的细节:

- **第①步是它和所有人不一样的地方**:别人一上来是「读文件」,Aider 一上来是「先给整个仓库画张地图」。这张图不含代码正文,只含**结构**(哪有什么函数/类、谁调用谁),所以极省 token。
- **git 是它的「撤销键」**:⑧步不是可选的花活,而是它的**安全网核心**——每次 AI 改动都是一个独立 commit,`/undo` 就是一次 git 回滚。它没有像 Claude Code 那样自造一套 JSONL 会话存储,而是**直接复用你早就有的版本控制系统**当回滚/审计机制。
- **它不会自己乱跑命令**:⑦步的测试/lint 是**你配置、你授权**才发生的,不是循环自作主张地开 shell。这条界线是它和⑫⑬那类「自主 shell agent」的根本分野。

> 一句话:**Aider 的一次任务,是「先看图(repo map)→ 挑相关文件 → 让模型按一种精确的编辑格式产出改动 → 落盘 → git 提交」——一条极其克制、每一步都留痕、随时能 `/undo` 的流水线。**

## 3. 拆开看它的组件

现在对着⑥–⑪的组件地图,把 Aider 逐个零件拆开。它的招牌零件排在最前。

### 3.1 repo map:招牌中的招牌——用「图排名」在有限窗口里塞进「最相关的那一小撮」

先说它解决的问题(④篇的核心痛点):**一个几千文件的仓库,根本塞不进上下文窗口;就算塞得进,也又贵又慢,还会把模型的注意力稀释掉。** 那怎么让 agent「知道该改哪、该看谁」?

Aider 的答案分三步(全部可对源码 `aider/repomap.py` 核实):

1. **解析(parse)**:用 **tree-sitter** 把每个被 git 跟踪的文件解析成语法树,抽出里面的**定义(哪定义了 `parse_config`)和引用(谁调用了它)**。tree-sitter 是纯 Python、pip 装得上、还能给出完整签名——早年 Aider 是用 ctags 干这活的,后来换成了 tree-sitter,信息更丰富。
2. **建图 + 排名(rank)**:把这些符号关系建成一张图(引用方文件指向定义方文件),然后跑一遍 **个性化 PageRank(personalized PageRank)**——对,就是当年 Google 给网页排名那个算法的变体。它的妙处在于**能传递「间接重要性」**:一个被很多地方引用的核心文件,哪怕你没提它,也会因为「大家都指着它」而排名很高,自动被认为「相关」。
3. **裁剪(budget)**:按 token 预算(默认约 **1024 token**,`--map-tokens` 可调、会动态扩)把排名最高的一小撮符号,压成一张紧凑的结构缩略图,塞进上下文。

有意思的是它的**排名权重**(核实自源码,比外界常引的版本更细):对你**亲口提到**的标识符 ×10;对「起得好」的名字(长度 ≥ 8 的 snake / kebab / camelCase)×10;对**来自你已加进会话的文件**的引用 ×50;反过来,对下划线开头的私有标识符、以及**在超过 5 个文件里都有定义**的「太普通」的名字 ×0.1 降权;引用次数还做了 `sqrt()` 缩放,防止某个被引用几百次的名字一家独大。

你品这套权重——它其实是在把「什么叫相关」翻译成可计算的规则:**你关心的、名字有信息量的、和你正在改的东西挨得近的,加权;到处都是的、私有的噪音,降权。** 这就是④篇「上下文工程」最教科书的一次落地。

> 反直觉的点:**解决「大代码库放不进窗口」的正道,往往不是把窗口撑更大,而是算出该放什么。** repo map 用一千来 token 干了「假装读完了整个仓库」的活——这比堆上下文长度优雅得多,也是为什么它到今天还被反复致敬。

### 3.2 编辑格式(edit formats):把「让模型可靠地落编辑」当成一门可度量的工程

这是 Aider 第二块硬核招牌,也是新手最容易低估的一环。核心洞察是:**不同模型「把改动写出来」的失败方式不一样**,所以「让模型用什么格式回复改动」本身就是一个要调的参数。Aider 文档里至少有**六种**具名编辑格式(不是外界常说的四种):

| 编辑格式 | 怎么改 | 适合/意图 |
|---|---|---|
| `whole` | 整个文件重写一遍 | 最简单最可靠,但小文件也得吐全文,**token 最贵** |
| `diff` | git 冲突风格的 SEARCH/REPLACE 块 | 主力格式,只发要改的段落 |
| `diff-fenced` | 路径放在代码围栏**内部**的 diff 变体 | 专治 Gemini 爱把路径放错位的毛病 |
| `udiff` | 简化版 unified diff、**不带行号** | 专门对付 GPT「偷懒」(用占位注释省事) |
| `editor-diff` / `editor-whole` | 给 architect 模式下的 editor 模型用的精简格式 | 见 3.3 的分工 |

`udiff` 那段故事特别能说明问题:GPT-4-Turbo 有个著名毛病,改代码时爱写 `# ... 其余不变` 这种偷懒占位。Aider 专门设计了 `udiff` 格式(而且**故意不带行号**,因为「GPT 处理源码行号的能力糟透了」)。效果是可度量的——在一个 89 题的重构/防偷懒基准上,把成功率从 **20% 拉到 61%**,「偷懒占位」的任务从 12 个降到 4 个。

> 一句话:**「让模型用什么格式回复」不是细节,是可靠性工程。** 换个输出契约(output contract),同一个模型的行为能差三倍——这是 agent 新手最常低估的一课。

这里的权衡也很清楚:`whole` 最可靠但最费 token;`diff` 系列省 token 但对模型「贴对位置」的能力要求更高(所以才需要 3.1 里说的**模糊回退**兜底)。**可靠性 ↔ token 成本**,你按模型脾气挑格式,而且**换格式不用动 agent 循环本身**。

### 3.3 architect / editor:一条「谁想 / 谁写」分家的最小两模型流水线

2024 年 9 月,Aider 加了个 `architect` 聊天模式,思路极简却很有代表性:**把「想怎么改」和「精确地把改动写成正确的 diff」拆给两个模型**。

- **architect(架构师)模型**:通常用一个强推理模型,负责**想清楚该做什么**——用自然语言描述改动方案,不管格式。
- **editor(编辑)模型**:通常用一个**更便宜**的模型,负责**把方案落成格式正确的编辑**(用上面 3.2 那对 `editor-*` 精简格式)。

为什么值得拆开?因为「推理」和「按格式精确输出」是两种不同的能力,让最擅长推理的模型专注想、让便宜模型专心格式化,**既提质又省钱**。官方验证过的一个组合:o1-preview 当 architect + o1-mini 或 DeepSeek 当 editor,在其基准上到过 **85.0%**(截至该文发表时的快照,数字会随模型换代波动)。

> 一句话:**architect/editor 是「任务分解 + 专家分工」的最小样板**——一个 agent 里的两个角色,一个决定「做什么」、一个负责「怎么写对」,顺带省下一笔钱。这正是⑩篇多角色思路的一个轻量前身。

### 3.4 git 集成:把版本控制直接当成 agent 的安全网

Aider 不自造状态系统,它**把 git 本身当撤销/审计机制**:

- **每次 AI 编辑 = 一个 commit**,提交信息由一个**便宜的「弱模型」**生成(这个弱模型只看 diff 和聊天历史,不看全上下文,省钱)。
- **`/undo` = 一次 git 回滚**;`/diff` 看这轮改了什么。
- **脏改动先提交(dirty-commit-first)**:如果你自己手上还有没提交的改动,Aider 会**先把你的活单独提交一次**,再动手改——这样万一 AI 改砸了,你 `/undo` 掉的只是 AI 那一层,**你的手工活绝不会被卷进去丢掉**。

这套设计的高明之处在于:它**复用了每个开发者本来就有、本来就信任的基础设施**,而不是发明一套谁也没审计过的新存储。天然可回滚、天然可审阅、天然和你的 PR 流程接得上。

### 3.5 模型无关与 `/` 命令:一个薄而通用的外壳

- **模型无关(model-agnostic)**:靠 `litellm` 一层抽象接住几乎所有主流后端(OpenAI / Anthropic / Gemini / DeepSeek / Ollama / OpenRouter / xAI …),BYOM。这让 Aider 天生**不被任何一家模型厂商绑定**——和 Claude Code(只吃 Claude)、Gemini CLI(只吃 Gemini)是鲜明对比。
- **聊天模式**:`code`(默认,会改文件)、`ask`(只问不改)、`architect`(两模型)、`help`(查自己文档)。
- **`/` 命令**:`/add` `/drop` 手动管理「哪些文件进上下文」、`/run` 跑个 shell 命令并可把输出喂给模型、`/test` `/lint` 跑测试与静态检查、`/undo` `/diff` 管 git……

**这里要划一条准确的界线(别踩雷)**:Aider 的「跑命令」——`/run` 和配置好的 `/test` / `--lint` / `--test-cmd`——是**用户触发或用户配置**的,失败信息会以 tree-sitter 格式化的报告贴回给模型迭代。但这**不是**一个像 Claude Code 那样「循环自己决定反复开 bash」的自主工具调用回路。把 Aider 说成「有自主 shell 的 agent」是不准确的。

> ⚠️ 一个常被以讹传讹的点:网上流传「Aider 内置 MCP 客户端」——**这在 Aider 官方文档里查不到确证**。那些「Aider MCP」项目其实是**第三方桥接/封装**(让别的 MCP 客户端去调 Aider,或反过来),不是 Aider 官方内建的 MCP-client 能力。除非官方 changelog 明确写了,**别把它当成 Aider 的既有特性来教。**(MCP 是什么见③篇。)

## 4. 它与众不同的设计决策

拆完零件,退一步看它的「设计哲学」——Aider 有一套非常一致、非常「工程师」的价值观:

- **repo map:用「相关性算法」换「上下文长度」。** 别人比谁窗口大,Aider 比「谁更会挑」。个性化 PageRank 让重要文件不必被点名就能浮上来。**trade-off**:图排名是**结构层面**的相关,它猜「哪些符号可能相关」,不保证一定对——偶尔它会漏掉一个语义上相关、但结构上不显眼的文件,这时就得你 `/add` 手动补。
- **可插拔编辑格式:把可靠性做成可度量、可调的旋钮。** 不同模型不同脾气,换格式不动循环。**trade-off**:格式越省 token(diff 系列)越依赖「贴对位置」,所以要有模糊回退兜底;`whole` 最稳但最贵。
- **git-first 的安全网:复用而非重造。** 把回滚/审计外包给 git,顺带天然接进你的 PR 流程。**trade-off**:git 是顺序追加的,你没法对「agent 的历史决策」做复杂查询(和⑫篇 JSONL 会话那个取舍异曲同工)——但换来的是零学习成本的可恢复性。
- **模型无关 + 基准驱动。** BYOM 让它不被任何厂商绑定;而且它**长期维护一个公开基准**(polyglot 排行榜),用真实分数驱动设计决策——这在 agent 工具里是相当负责任的做法。
- **重度自举(dogfooding)。** Aider 很大一部分代码是 Aider 自己写的——**但这个数字波动极大**:某些版本高达 88%(v0.86.0)、62%(某个 main 分支快照),但也有版本只有 21%(v0.85.0)。诚实的说法是「通常一版 ~70–80%,但逐版波动很大」。**它本身就是一堂课:自报指标要带着怀疑读。**

> 一句话:**Aider 的信条是「克制 + 复用」——不撑窗口而算相关性、不造存储而用 git、不锁模型而做抽象层。** 它证明了一件事:一个 agent 工具的核心价值,可以来自一个聪明的**上下文机制**,而不是最新最强的模型。

## 5. 局限与坑

诚实地说几条边界(也提醒你别把它和自主 agent 搞混):

1. **它是「结对」,不是「代跑」。** Aider 的设计假设是「你盯着、你显式加文件、你授权跑测试」。它没有一个自主 shell 循环去自己反复试错——想要「派出去、自己啃几十步」的形态,那是⑫⑬(Claude Code / Codex)或⑯的 Devin,不是 Aider。用错了预期,会觉得它「不够聪明」,其实是你用错了工具。
2. **repo map 是「结构相关」,不是「语义全知」。** 图排名会漏掉结构上不显眼、但语义上关键的文件;它也会被命名习惯差的代码库拖累(名字起得烂,权重就抓不准)。遇到它没找对地方,**手动 `/add` 才是正解**,别指望它每次都自己猜中。
3. **各种数字都是快照,不是常量。** 版本号(v0.86.2)、排行榜分数(gpt-5 88.0% / 225 道 Exercism 题,截至 2026 中)、自举比例(21%–88% 剧烈波动)——**引用时务必带「截至某版本/某快照」**,别当稳定事实背。
4. **「Aider 有 MCP 客户端」是未经证实的传言。** (见 3.5 的⚠️卡)官方文档无确证,别把第三方桥接当官方特性写进教案。
5. **模型无关是双刃剑。** BYOM 给你自由,也把「挑对模型 + 挑对编辑格式」的责任推给了你。用错格式(比如让一个爱偷懒的模型走 `diff` 而不是 `udiff`),可靠性会肉眼可见地掉。**默认值不一定是你这个模型的最优值。**

---

## 6. 一句话记住它 + 横向对比(附:另一种 CLI 形态 Gemini CLI)

> **Aider = 一个开源、模型无关、git-first 的终端结对编程器:招牌是 repo map(用 tree-sitter + 图排名,在一千来 token 里塞进「最相关的那一小撮」)和可插拔编辑格式(把「让模型可靠落编辑」做成可度量的工程)。它赌的是「聪明的上下文机制 + 复用 git」,而不是「最强的模型 + 自主 shell」。**

### Google Gemini CLI:同为 CLI,另一种活法

把 **Gemini CLI** 摆到 Aider 对面,你会看清「CLI agent」内部的路线分歧。它是 Google 2025 年 6 月开源(Apache 2.0)的终端编码 agent,核心是一个 **ReAct(reason-and-act,推理—行动)循环**(见②篇),特点包括:**内置工具**(文件读写、shell、Google 搜索接地、web-fetch)+ **MCP** 扩展、**100 万 token 上下文**(已确证)、以及和 CLAUDE.md 同构的 **`GEMINI.md`** 分层上下文文件(全局 → 项目 → 子目录)。它的「跑命令」是**真的自主 shell 工具**——这和 Aider 那条克制的界线正相反。

> ⚠️ 关键限定(截至 2026 中,必读):**Gemini CLI 正在对消费者「日落」。** 2026 年 5 月 19 日(Google I/O)Google 宣布把它转向一个**闭源、Go 写的「Antigravity CLI」**后继品;2026 年 6 月 18 日起,Google 的托管后端**停止为免费版、AI Pro/Ultra 订户及个人 Code Assist 用户服务**——只有 Code Assist Standard/Enterprise 组织(或付费 Cloud API key)还能用。那个 Apache-2.0 的 GitHub 仓库仍在、还在发版(如 v0.49.0,2026 年 6 月),但**个人用户已经没法再用 Google 的模型免费跑它了**。所以把它当「一个历史上很重要、正在转型」的工具来学,而不是「现在推荐你去装」的工具。

顺带一个安全教训(见⑨篇的坑):Gemini CLI 的沙箱**默认是关的**,2025 年被 Tracebit 报告过一个提示注入(prompt injection)漏洞——恶意 README 里藏指令 + 用 `;` 绕过命令白名单——2025 年 7 月 25 日在 v0.1.14 修复。这生动说明了「沙箱默认关」和「命令白名单」为什么都是危险的偷懒。

### 横向对比表

| 对比对象 | 形态与设计侧重的差异 |
|---|---|
| **Aider(本篇)** | 开源、模型无关(BYOM)、git-first、**克制的用户触发式命令**;招牌是 repo map + 可插拔编辑格式。假设「你盯着结对」,不是自主代跑。 |
| **Google Gemini CLI** | 同为开源 CLI,但走 **ReAct 循环 + 真·自主 shell 工具 + 内置 Google 搜索接地 + MCP + 1M 上下文 + GEMINI.md**;绑 Google 模型。**⚠️ 消费者端 2026 年中已日落转向闭源 Antigravity。** |
| **Claude Code(⑫)** | 也是终端、也吃 git,但**只用 Claude、闭源**,有成熟权限模型、庞大 hooks、子代理、自主 shell 循环。Aider 比它早出现,且自带基准与编辑格式体系。 |
| **OpenAI Codex CLI(⑬)** | 终端自主 agent,强调长程自主;Aider 相比之下更「人在环里、每步 git 留痕」的结对姿态。 |
| **Cursor(⑭)** | AI 原生 IDE(VS Code 分叉),GUI + 内联补全最强、绑自家订阅/模型;Aider 纯终端、BYOM、无 GUI。常见搭配:Cursor 日常编辑 + Aider 啃 git 化的批量改动。 |

**怎么选(经验法则)**:想**不被模型厂商绑定、要 git 化的可回滚工作流、又愿意自己盯着结对**——Aider;想要**一个引擎多门面、权限/子代理/hooks 齐全的自主 agent**——回看⑫;想要**云端托管、发完就走**——那是⑯的地界。

下一篇起,我们离开「终端/编辑器里的结对与自主编码工具」,走进**更自主、更框架化、更领域化**的拆解——看当 agent 被放进一个云端沙箱、或被抽象成一套可编排框架时,前面这张组件地图又会怎么被重新排列。
