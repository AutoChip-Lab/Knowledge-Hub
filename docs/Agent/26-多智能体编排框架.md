## 0. 全景:当「一个 agent」变成「一群 agent」

前面拆的东西,主角基本都是**一个** agent:Claude Code 是一个陪你写代码的 agent,自主 SWE 是一个自己跑几十步的 agent。就算讲子代理(见「子代理与多代理编排」那篇),也是「一个主 agent 临时喊几个帮手」。

这一篇换个视角:**如果从一开始,你要设计的就不是一个 agent,而是一支队伍呢?** 一个当产品经理、一个当架构师、一个当程序员、一个当测试;或者一个当「用户」下指令、一个当「助手」执行;或者干脆放一堆 agent 在一个「消息池」里,谁被点到谁说话。

这类专门用来**组织多个 agent 协作**的东西,就是本篇的主角——**多智能体编排框架(multi-agent orchestration framework)**。

```
   单 agent(⑫⑬⑯…)                     多智能体编排框架(本篇)

   你 ──▶ [一个 agent] ──▶ 结果          你 ──▶ ┌─ 编排层(谁下一个说话?)─┐
              │                                    │   ┌──────┬──────┬──────┐  │
              ├─ 工具                              │   │agentA│agentB│agentC│  │
              ├─ 记忆                              │   └──┬───┴──┬───┴──┬───┘  │
              └─ 循环                              │      └──── 消息/交接 ────┘  │
                                                   └──────────── 结果 ──────────┘
   一个脑子转几十圈                            多个脑子,靠「怎么协作」这层黏合
```

放进整张地图看:它们和「讲 LangGraph 的那篇」是**同一层**的东西——都是**编排框架(orchestration framework)**,坐在「LLM 提供方」之上、「你的应用」之下,自己不产生智能,只负责**把 LLM 的调用组织成一套协作流程**。区别在于:LangGraph 把协作画成一张**显式状态图**(节点/边/条件),而本篇这几家,各自押了另外几种「怎么让多个 agent 配合」的范式。

> 读这一篇的正确姿势:**别记 API,记每一家赌的是哪种「协作范式」。** 同样是「让一群 agent 干活」,有人赌角色分工(role-based)、有人赌自由对话(conversation)、有人赌固化流程(SOP)、有人赌点对点交接(handoff)——这几个词,就是本篇的骨架。

## 1. 这一类到底是什么、边界在哪

先给一句话定义:

> **多智能体编排框架 = 一套用来「定义多个 agent、并决定它们何时、以什么方式互相协作」的软件层。** 它不是模型、不是成品 agent,而是「组织 agent 的规则 + 运行这套规则的引擎」。

三条边界,帮你把它和邻居划清:

- **它不是模型。** 这几家几乎都**模型无关(provider-agnostic)**:CrewAI、AutoGen、MetaGPT、smolagents 等都通过一个模型抽象层(常见是 LiteLLM 这类多 provider 网关)接 OpenAI / Anthropic / 本地 Ollama 等,自己一行模型都不训。智能来自底座 LLM,框架只管「怎么把这些调用编排起来」。
- **它不是单 agent harness。** ⑫⑬ 那种是「把一个 agent 跑好」;这里是「把 N 个 agent 的**关系**管好」。多出来的核心难题是:**谁先说话?谁把活派给谁?一个 agent 的输出怎么变成另一个的输入?什么时候整支队伍算干完了?** 这几个问题的不同答法,就是不同框架的灵魂。
- **它和「子代理」不完全是一回事。** 子代理(subagent)通常是「主 agent 临时外包一段脏活、拿回摘要」(见讲子代理的那篇);而这里的「多智能体」往往是**从设计之初就并列存在的一组角色**,协作是主结构而非临时手段。当然两者会重叠——不少框架里,「委派给子 agent」正是它们实现协作的方式之一。

**为什么值得单独一章?** 因为「多」不是「一」的简单叠加。你可能以为:三个 agent 就是把一个 agent 的活分三份、快三倍——**恰恰相反**。多个 agent 各自持有自己那份上下文,token 成本至少翻几倍(三个 agent 至少 3×);一环的错误会被下游放大成「级联幻觉(cascading hallucinations)」;调试从「看一个 agent 的 trace」变成「查 agent 之间互相给错了什么指令」。**协作本身就是新的一类工程问题**,而这几家框架,就是对这个问题的几种不同答卷——多 agent 不是「更多的 agent」,是「多出了一整层叫『协作』的问题」。

## 2. 一次任务在这类系统里怎么流动

抽象说没意思,挑两个范式差得最远的代表,各走一遍生命周期,你立刻就懂「范式差异」是什么意思了。

**代表一:CrewAI(角色分工 / role-based)——像一条流水线上的一队员工。**

设想你丢进去一句「帮我写一篇关于某新技术的深度调研报告」,再配一支两人 Crew(团队):一个 `研究员`、一个 `写手`。

```
  crew.kickoff(inputs={...})
     │
     ▼
① 拼提示     取「研究员」的 role/goal/backstory + 任务描述 + 可用工具 → 组成一个 ReAct 提示
     │
     ▼
② ReAct 循环  LLM 产出 Thought → Action(选工具名)→ Action Input(JSON 参数)
     │        └─ CrewAI 后端解析这段输出,自己调对应的 Python 工具函数(LLM 不直接跑代码)
     │           工具返回值作为 Observation 回注 → 循环,直到 Final Answer 或撞 max_iter(默认 20)
     ▼
③ 交接       「研究员」任务的输出,自动成为「写手」任务的 context(sequential 流水线)
     │
     ▼
④ 第二棒      「写手」拿着上一步的调研结果,重复 ②,产出报告
     │
     ▼
⑤ 收尾       返回 CrewOutput(最终结果 + 各任务输出 + token 用量)
```

注意 ③ 那个**交接**:sequential 模式下,前一个任务的输出**自动**变成后一个任务的上下文,像流水线一样往下传。这就是「角色分工」范式的心跳——**活儿沿着组织结构流动**。

**代表二:AutoGen(自由对话 / conversation)——像一群人围着一张桌子开会。**

同样一个任务,在 AutoGen 里长得完全不同。你把几个 agent 塞进一个 **Team**(团队/群聊),设好终止条件,然后:

```
  team.run(task="...")                任务作为第一条消息进入「共享消息上下文」
     │
     ▼
① 选发言人   编排器决定「谁下一个说话」——这是整个循环的核心分歧点:
     │        · RoundRobin:按固定顺序轮流
     │        · Selector:用一个 LLM 动态挑下一个发言者
     │        · Swarm:上一个 agent 用 HandoffMessage 显式把话筒交给指定 agent
     ▼
② 该 agent 说一句  拿共享上下文调 LLM → 产出 文本 / 工具调用 / 代码块
     │
     ▼
③ 执行 + 广播  若有工具/代码,交执行器跑,结果作为「新消息」回灌;新消息广播给全体
     │
     ▼
④ 查终止     每个 agent 响应后检查一次终止条件(如有人说了 "TERMINATE"、或到消息上限)
     │        └─ 没满足 ──▶ 回到 ①,再选下一个发言人
     ▼
   满足 ──▶ 返回 TaskResult(完整消息序列 + 停止原因)
```

看出差别了吗?CrewAI 是**活儿在固定角色间往下传**(流水线);AutoGen 是**一堆 agent 共享一个对话、靠一个「谁下一个说话」的调度策略往前推**(圆桌会议)。**同一个任务,两种世界观。** 这就是「编排范式」四个字的分量。

> 一句话:看一个多 agent 框架,先问它这个问题——**「下一个动作是谁发起的、凭什么?」** 答案是「按角色顺序」还是「按对话动态」还是「按固化 SOP」还是「按显式交接」,决定了它是哪一派。

## 3. 拆开看

现在把三家深拆上手术台(CrewAI / AutoGen / MetaGPT),每家对着上面那条生命周期,看它在哪一环下了重注;最后一节做「群像横扫」,覆盖其余四家。

### 3.1 CrewAI:灵魂是「角色 / 任务 / 委派」——把 agent 当有职位的员工

**它是谁**:CrewAI 是一个用 Python 写的、基于「角色扮演」的多智能体编排框架,由 João (Joe) Moura 于 2023 年在巴西圣保罗创立(公司 CrewAI, Inc.,Moura 任 CEO)。**License 是 MIT**(开源核心)。官方一句话定义:「Framework for orchestrating role-playing, autonomous AI agents」。截至 2026 年中,它**活跃**:开源框架于 2025-10-20 发布 1.0.0,此后约每月一个大版本,最新为 v1.15.1(2026-06)。GitHub 官方 README 显示约 54.9k stars。

它的心智模型很好记——**Agent 像有职位的员工,工作沿组织层级流动**。核心是一个**双层框架**:

- **Crews(团队)**:高层抽象。给 agent 配 `role`(职位)/ `goal`(目标)/ `backstory`(背景故事)+ 工具 + 任务,让它们「自治协作」。适合探索性、需要 agent 自己决定怎么做的场景。
- **Flows(工作流)**:低层、确定性的控制层。事件驱动的 Python 类,用装饰器(`@start` / `@listen` / `@router` / `@or_` / `@and_`)精确控制「什么时候跑什么」,可把多个 Crew 和直接的 LLM 调用串成生产级流水线,还能用 `@persist` 把状态落 SQLite、从某个 state 恢复或 fork。

**先给直觉**:Crew 是「给 agent 自由发挥、你别管太细」;Flow 是「你精确编排、agent 只在你指定的节点上干活」。一个偏自治、一个偏确定——这个「双层」正是 CrewAI 的卖点:探索用 Crew,上生产用 Flow 把 Crew 串起来。

**控制回路怎么跑**(Crew 层):`crew.kickoff()` 启动后,每进入一个 Task,就把 role/goal/backstory + 任务描述 + expected_output + 可用工具拼成一个 **ReAct 提示模板**,交给 **AgentExecutor** 跑 Thought → Action → Action Input → Observation 的循环——**关键点:是 CrewAI 后端解析 LLM 的输出、自己调对应的 Python 工具函数,LLM 不直接执行代码**,这天然多了一层隔离。循环直到 `Final Answer`,或撞上 `max_iter`(默认 20)/ 超时上限。

任务之间怎么交接,取决于 **Process(执行策略)**:

- **`Process.sequential`**:任务按列表顺序跑,前一个的输出**自动**成为下一个的 context(就是 §2 那条流水线)。
- **`Process.hierarchical`**:任务不预先分派,由一个 **Manager**(`manager_llm` 或自定义 `manager_agent`)分析任务列表,把每个任务**动态委派**给最合适的 agent,审阅其输出、判断够不够好、要不要重派——模拟公司层级。委派靠内建的 delegation 工具实现(`allow_delegation`,默认 False)。

**记忆**:1.x 引入统一 `Memory` 类(取代早期 short/long/entity 三件套),默认后端 **LanceDB**,落盘 `./.crewai/memory`。检索用一个**复合打分**——`语义相似度×0.5 + 时间衰减×0.3 + 重要性×0.2`(30 天半衰期),存储时用 LLM 推断 scope/重要性。这是「长任务撑爆上下文」的一个具体工业答案(见讲上下文压缩的那篇)。

**一个必须点出的坑**(这也是 CrewAI 教学时最诚实的地方):有较权威的批评(Towards Data Science)指出**层级 Manager–Worker 模式与文档不完全一致**——实测中编排逻辑偏弱,容易退化成「所有任务顺序跑」,出现调错 agent、输出被覆盖、延迟/token 膨胀。教学时应把它标为「**文档承诺 > 实际行为**」的注意点。

> CrewAI 的赌注:**上手快就是最大生产力。** 普遍评价它的学习曲线在同类里最平缓(CrewAI 最易 > AutoGen > LangGraph 最陡),做出可跑 demo 约 2–3 人日。代价是控制力/灵活度最低、调试难,而且多 agent 各持全上下文 → token 成本翻倍。**它用「灵活度」换「开发速度」——这笔账划不划算,取决于你的任务多确定。**

### 3.2 AutoGen / AG2:灵魂是「对话即编程」——让 agent 围着桌子开会

**它是谁**:AutoGen 是微软研究院(AI Frontiers 实验室)做的**事件驱动多智能体「对话」框架**——把复杂任务拆给多个可对话(conversable)的 agent,让它们通过异步消息互相协作/交接。原始论文获 COLM 2024(及 ICLR'24 LLM Agents Workshop 最佳论文)。代码 **MIT**、文档 CC-BY-4.0。

**但它的 2026 现状是本篇最需要讲清的一件事——这个名字被拆成了两条继承线:**

- **原版 AutoGen 已进入「维护模式(maintenance mode)」**:仓库明文只收 bug/安全补丁,不再有新功能。原因是 2025-10-01 微软宣布把 **AutoGen + Semantic Kernel 合并**成统一的 **Microsoft Agent Framework(MAF)**,作为「企业级后继者」,MAF 于 2026-04-03 发布 1.0 GA,微软推荐新用户直接上 MAF。
- **社区分叉 AG2 仍在活跃开发**:由原核心作者 Chi Wang、Qingyun Wu 等维护(Apache-2.0),2024-11 从 AutoGen 分叉,最新 v0.14.0(2026-06),有明确的 v1.0 路线图。

一句话:**「原创者出走成立 AG2、微软把品牌并入 MAF」——AutoGen 既没死也没被收购,而是被拆成了两条线。** 选它的人现在多半要在 **AG2(社区继续)vs MAF(微软企业级)** 之间做迁移决策——这个治理动荡本身就是它最大的痛点。截至 2026 年中,`microsoft/autogen` 约 59.5k stars,`ag2ai/ag2` 约 4.7k。

**招牌思想:对话即编程(conversation programming)。** AutoGen 最初的立论是——把多 agent 协作抽象成**消息对话**,用自然语言 + 代码定义每个 agent 的行为和对话模式。相比 LangGraph 的「显式状态图」,它更偏「让 agent 自由对话、由 LLM 动态选人」:灵活,但更难精确控制。

**v0.4 的一次彻底重写**(理解现在的 AutoGen 必须知道):从 v0.2 的「单进程对话循环」改成了 **Actor 模型 + 异步消息总线**,换来分布式、跨语言(Python + .NET/C#)、事件驱动、和内建 **OpenTelemetry** 可观测性。代价是**破坏性重写**,迁移有成本,社区一度分裂。分四层:Core(要控制)/ AgentChat(要快)/ Extensions(要集成)/ Developer Tools(AutoGen Studio 低代码 GUI + AutoGen Bench 评测)——按需选层。

**控制回路的心脏,是「谁下一个说话」这个调度策略**(就是 §2 讲的那条)。整个循环本质是:**共享消息上下文 + 一个「谁下一个说话」的调度策略 + 每步可执行工具/代码 + 终止条件把关**。不同 **Team** 类型只是替换了「选发言人」这一步:

- **RoundRobinGroupChat**:按固定顺序轮流,每次发言广播给全体。
- **SelectorGroupChat**:每条消息广播后,用一个 ChatCompletion 模型**动态挑**下一个发言者。
- **Swarm**:agent 用 **`HandoffMessage`** 显式把控制权**交给指定 agent**(去中心化的点对点交接)。
- **MagenticOneGroupChat**:由一个 Orchestrator 用「双账本」机制驱动(见下)。

工具/代码执行走 `CodeExecutorAgent` + 可插拔后端(本地命令行 / Docker 容器 / Azure 代码执行器,官方推荐容器隔离);人类可通过 `UserProxyAgent` 当作团队里的一个「agent」参与(每步/从不/终止时几种模式)。状态可 `save_state`/`load_state` 序列化,支持暂停-恢复长任务。

**Magentic-One——AutoGen 的规划范例**:这是用 AgentChat + Extensions 搭出的一个多 agent 团队(arXiv:2411.04468),默认所有 agent 用 GPT-4o。它的 Orchestrator 领导 4 个专用 agent——**WebSurfer**(网页交互)、**FileSurfer**(读本地文件)、**Coder**(写代码)、**ComputerTerminal**(跑代码/装库),用一套**双账本机制**做规划:

- **Task Ledger(外循环)**:记事实、猜测、计划。
- **Progress Ledger(内循环)**:自反思进度、给各 agent 派活、判断是否完成/是否卡住。卡住时,外循环**重写 Task Ledger、重新规划**。

据其原文,Magentic-One 在 GAIA / AssistantBench / WebArena 上取得「与当时 SOTA 统计上相当」的表现,且不修改核心 agent 能力。(注:各基准的**精确准确率**在原文以柱状图呈现、正文无文字数值,底稿标注为未找到公开文字数据——所以这里只给定性结论,不编数字。)

> AutoGen 的赌注:**协作应该像对话一样自由,调度可以交给 LLM。** 这让它极灵活(换个「选发言人」策略就换了一种协作拓扑),但也更难精确控制、更难保证不跑偏。而它 2026 年的处境提醒我们另一件事:**框架的「治理稳定性」也是一种技术选型指标**——一个被拆成两条线的名字,迁移成本要提前算进去。

### 3.3 MetaGPT:灵魂是「SOP 固化」——把一支 LLM 团队装进一家 AI 软件公司

**它是谁**:MetaGPT 是一个「把整套 GPT 角色组织成一家 AI 软件公司」的多智能体框架——输入一句话需求,产出 PRD / 系统设计 / 任务拆分 / 代码 / 测试。出品方是 **DeepWisdom(深度赋智)**,创始人吴承霖(GitHub: geekan)是论文通讯作者。**License MIT**。仓库已从 `geekan/MetaGPT` 组织化到 `FoundationAgents/MetaGPT`,约 69.2k stars,是 GitHub 上最靠前的「AI 软件公司」类项目之一。

**2026 现状有个反差信号**:框架**活跃**(main 分支持续 commit),但 GitHub Releases 的正式 tag 停在 v0.8.2(2024-03),发版节奏在 2024 年后放缓;团队重心转向**研究产出**(Data Interpreter / AFlow / SPO / AoT 等陆续中稿 ICLR/NeurIPS/ICML)和**商业产品 MGX**(mgx.dev,2025-02 上线的 no-code Web 产品)。

**招牌哲学一句话:`Code = SOP(Team)`。** 把人类的标准作业流程(SOP,standard operating procedure)固化成一串 prompt 序列,套在一支 LLM 团队上。

**为什么这么设计?一个反直觉的洞察。** 你可能觉得「让几个 agent 自由讨论,总能碰出好方案」——MetaGPT 的立论恰恰相反:**朴素地把多个 LLM 串成对话链,会产生「级联幻觉」**——一环的逻辑错误或跑题会被下游放大,最终产物逻辑不自洽、跑不起来。它的解法不是让 agent 更自由,而是**用结构约束协作**:

- **结构化通信 > 自由对话**:这是 MetaGPT 最标志性的设计。ChatDev 那类让 agent 用自然语言对话协作,MetaGPT 坚持让 agent 交换**文档 / 图 / 接口定义**这种结构化交付物,显著降低跑题与幻觉累积。取舍是:更工程化、更可控、可执行性更高,但也更「死板」、强依赖 SOP 设计质量。

**核心抽象是一个五层对象模型**(自底向上):

| 组件 | 职责 |
|---|---|
| **Action** | 最小执行单元 = 一次结构化 LLM 调用(专属 prompt 模板 + 结构化输出解析)。如 `WritePRD`、`WriteDesign`、`WriteCode`、`RunCode`。 |
| **Role(智能体)** | `name + profile + goal + constraints` + 一组 Action + 私有 memory。行为遵循 **`_observe → _think → _act`** 循环;`_watch` 声明它订阅哪些上游消息类型。 |
| **Message** | 结构化消息,带 `cause_by`(由哪个 Action 触发)/ `send_to` 等元数据,是「结构化通信」的载体。 |
| **Environment(含共享消息池)** | 所有 Role 共享的**全局发布-订阅消息池**。Role 把产物 publish 进池子,其他 Role 按自己订阅的消息类型过滤订阅,**而不是点对点对话**——避免信息过载。 |
| **Team** | 顶层编排器:`hire(roles)` 组队、`invest(budget)` 设预算、`run_project(idea)` 注需求、`run(n_round)` 逐轮驱动。内含 `CostManager` 做预算控制(超预算抛 `NoMoneyException`)。 |

**控制回路怎么跑**(以 `metagpt "写一个 2048 游戏"` 为例):`Team.hire([ProductManager, Architect, ProjectManager, Engineer, QAEngineer])` 组队,`run_project(idea)` 把需求包成一条 Message publish 到消息池。然后 `Team.run(n_round)` 进入**逐轮循环**,每轮检查环境是否 idle、预算是否超限,再让池中每个 Role 执行一次 `_observe → _think → _act`——`_observe` 只按自己 `_watch` 的订阅取上游依赖已满足的新消息(比如 Architect 只有看到 PM 产出的 PRD 才被激活),`_act` 产出结构化交付物再 publish 回池,触发下游。交付物链条是一条清晰的软件公司流水线:

**Product Manager**(WritePRD → PRD)→ **Architect**(WriteDesign → 文件列表/接口定义/时序图)→ **Project Manager**(拆成任务列表)→ **Engineer**(WriteCode → 实现)→ **QA Engineer**(测试用例)。

**可执行反馈闭环(executable feedback)——Engineer 的关键机制**:它不止写代码,还会**写并运行单元测试 / 执行代码(`RunCode`)**,拿到运行结果;失败就带着错误信息 debug 重写,**最多重试 3 次**直到通过。这是相对纯对话式框架的关键差异——**用「能不能真跑通」来给「逻辑幻觉」兜底**。(论文报告此机制在 MBPP 上带来约 5.4% 的绝对提升;单个 SoftwareDev 任务在 GPT-4 下平均约 $1.09 成本。)

**交接靠消息池的发布-订阅**:上游产物落池即触发订阅它的下游角色(隐式交接),Team/Environment 只负责逐轮 tick 与预算/归档,不需要中央调度器逐条派单。

MetaGPT 还有一条**研究驱动的演进线**值得点名——它发现固定 SOP 对「软件开发」这类流程清晰的任务收益大,但对流程不确定的开放任务反而是约束。于是催生了 **Data Interpreter**(面向数据科学的单智能体,用**动态分层图规划**边写代码边执行自纠错)和 **AFlow**(ICLR 2025 Oral,把 workflow 优化重述为「代码化 workflow 的搜索问题」,用蒙特卡洛树搜索变体自动生成 agentic workflow)。这条从「固定 SOP」到「自动生成 workflow」的弧线,本身就是一堂关于「协作流程该写死还是该学出来」的课。

> MetaGPT 的赌注:**对流程清晰的任务,固化的人类 SOP 比自由发挥更靠谱。** 它不让 agent 自由讨论,而是逼它们像真实软件公司一样一环环交付结构化产物、并用「代码能否跑通」当客观兜底。收益是可控、可执行、成本可预算;代价是死板、强依赖 SOP 质量,且对开放任务水土不服——这也正是它后来去做「动态图 / 自动 workflow」的动机。

### 3.4 群像横扫:另外四家,各占一句灵魂

深拆到此为止。剩下四家不逐一开膛,但每家点清「定位 / 独特点 / 开源闭源 / 状态」——它们分别代表了几种值得知道的变体。

**OpenAI Agents SDK——极简、代码优先,把「交接」降维成一次工具调用。** OpenAI 官方维护的**轻量多智能体编排框架**(MIT,2025-03-11 发布,是实验框架 Swarm 的生产就绪继任者)。核心原语刻意做得少而正交:`Agent`(LLM + 指令 + 工具 + 交接 + 护栏)、`Runner`(执行 agent 循环的引擎)、`Handoffs`、`Guardrails`(带 tripwire 断路的输入/输出安全校验)、`Sessions`(跨轮会话记忆)、`Tracing`(默认开启)。**最妙的一招:handoff 对 LLM 就是一个工具调用(`transfer_to_<agent>`)**——把多 agent 协作降维成「一个特殊工具」,模型天然会用,不需要额外的调度器或消息总线;代价是编排「分散」在各 agent 的 handoff 列表里,而非集中式图。截至 2026 年中约 27.6k stars,周级/双周级发版,**活跃**;2026 年新增了 **Sandbox Agents(容器化沙箱)** 和 Codex 风格 harness(带断点续跑)。注意别混淆:它没改名,2025-10 推出的上层产品套件 **AgentKit**(可视化 Agent Builder 等)是包着它的一层,二者是包含关系而非替代。**选它常因为:已在 OpenAI 生态里,想要最少胶水 + 官方支持。**

**Google ADK——代码优先、企业向,「确定性图 + LLM 自由推理」显式二分。** Google Cloud 出品的开源 agent 开发框架(**Apache-2.0**,2025-04 在 Google Cloud NEXT 发布),是 Google 内部 Agentspace、CES 等产品用的同一套框架。它的独特点是**把「确定性编排」和「LLM 动态路由」显式分开**:既有 `LlmAgent`(LLM 驱动决策),又有 **Workflow Agents**(`SequentialAgent` / `ParallelAgent` / `LoopAgent`,由**代码**控制顺序、不靠 LLM)。**版本断代必须讲清**:1.x(2025)是「层级式执行器 + Runner 事件循环」,其控制回路是 Runner 与 Agent 之间的 **yield-暂停-恢复**协程循环(状态只有在 yield 出事件、被 Runner 提交后才持久化/可见);2.0(2026-05 起 GA)**彻底重构为 graph-based Workflow Runtime**——Agent/Tool/Function 都当图里的节点、用边连成显式路径,原生支持 routing / fan-out-in / loop / retry / HITL / 嵌套 workflow。**这是破坏性变更**(API、存储、异常语义都变,存储不兼容被 Google 标为「数据丢失」级风险),截至 2026 年中 2.0 仅 Python/Go GA。约 20.4k stars,**活跃**。配套的 **A2A(Agent2Agent)协议**管「跨进程/跨厂商的 agent 之间怎么对话」(2025-06 捐给 Linux Foundation,与 Anthropic 的 MCP 互补——MCP 管 agent↔工具,A2A 管 agent↔agent)。**选它常因为:想吃 ADK + MCP + A2A 这套互操作生态 + Vertex 托管部署。**

**CAMEL-AI——研究基因的「agent 社会」,原点是学术论文而非工程编排。** 开源多 agent 框架 + 研究社区(**Apache-2.0**;起源于 KAUST 研究团队,原始 CAMEL 论文 arXiv:2303.17760 发在 NeurIPS 2023;与商业公司 Eigent 深度绑定)。它和前几家最大的不同:**原点是「寻找 agent 的 scaling law」这个研究议程**——用大规模模拟 agent 之间的对话来研究涌现行为、顺便**合成对话数据**。招牌技术是 **role-playing + inception prompting**:两个 agent(一个「AI User」下指令、一个「AI Assistant」执行)在最小人工干预下自主推进任务。它提供两套抽象覆盖两类需求——轻量的 `RolePlaying`(两 agent 自治对话,适合数据生成/研究)和层级的 `Workforce`(Task Planner + Coordinator + Worker Nodes,带**自动失败恢复**:某 subtask 失败就重试/重新规划/递归再分解/动态新建 worker)。还有**一等公民级的双向 MCP 支持**(既能当 client 连外部 MCP server,也能把任意 CAMEL toolkit 反向暴露为 MCP server)。约 17.3k stars,**活跃**(最新 v0.2.90,2026-03);上层应用 OWL(GAIA benchmark 均分 69.09,README 口径)被 NeurIPS 2025 接收。**选它常因为:要做合成数据 / agent 社会仿真 / 研究**——它诚实地记录了 role flipping、复读指令、无限循环等自治对话的已知失败模式,生产落地需额外 harness 兜底。

**smolagents——极简到 ~1000 行,信仰「code-as-action」。** Hugging Face 出品的极简(barebones)Python agent 库(**Apache-2.0**,2024-12-31 首发,是被弃用的 `transformers.agents` 的官方继任者)。核心信仰一句话:**让 LLM 把动作写成可执行的 Python 代码(code-as-action),而不是写成 JSON 工具调用**。它的 `CodeAgent` 把动作生成为 Python 代码片段(还有传统的 `ToolCallingAgent` 走 JSON)。**为什么押代码?** 官方援引 CodeAct(arXiv:2402.01030)等论文论证:代码天生**可组合**(函数嵌套/循环/条件)、能直接**用变量管理复杂对象**、更通用,而且**LLM 训练语料里代码远多于 JSON**;官方称 `CodeAgent` 比传统工具调用**约少 30% 步骤**(即约省 30% LLM 调用,基于一个未正式发表的内部 benchmark,应视为官方声明而非独立复现)。多 agent 靠**层级式 managed agents**——一个 manager `CodeAgent` 通过 `managed_agents=[...]` 持有子 agent,像调用工具一样在代码里调它们。**安全模型值得单独记**:默认的 `LocalPythonExecutor` 是自研解释器(不用原生 `exec`,加载 AST 逐操作执行、默认禁 import 除非白名单、总操作数封顶防死循环),但官方明确警告**它不是安全边界**——真正的隔离要用远程执行器(E2B / Docker / Modal / Blaxel)。约 27.8k stars,**活跃**(每 ~12-15 天一版,最新 v1.26.0)。**选它常因为:要一个能一眼读完源码、抽象压到最薄、且 Hub 原生可分享的最小 agent 库。**

## 4. 它们与众不同的设计决策与取舍

退一步看,这七家其实都在回答同一个问题——「**多个 agent,怎么协作才不散架?**」——但下注的位置差得很远。可以把它们的答案沿一条谱系排开:

**从「自由」到「约束」的谱系。** 一端是**极度自由**:AutoGen 让 agent 自由对话、由 LLM 动态选发言人;CAMEL 让两个 agent 在最小干预下自主对话。另一端是**极度约束**:MetaGPT 用固化的 SOP + 结构化交付物把每一步钉死;Google ADK 2.0 用显式的图/节点/边换可预测、可重放。CrewAI 则**横跨两端**——Crew 偏自由、Flow 偏约束,让你按场景选。这条谱系的核心张力是:

> **自由 = 灵活但难控;约束 = 可控但死板。** 没有免费的午餐。AutoGen 的自由对话能适配你想不到的协作拓扑,但也更容易跑偏、更难调试;MetaGPT 的固化 SOP 让软件开发任务收敛得又快又稳,但换个开放任务就水土不服。**你要先想清楚你的任务「流程有多确定」,再选框架的位置。**

**「交接(handoff)」这件事,有几种截然不同的实现——这是本篇最该记住的横切面:**

- **靠角色顺序**:CrewAI sequential(上一任务输出自动成下一任务 context)、MetaGPT SOP(角色链)。活儿沿固定结构流动。
- **靠中央协调器派活**:CrewAI hierarchical 的 Manager、CAMEL Workforce 的 Coordinator、MetaGPT 的逐轮驱动。有个「主管」看谁合适就派给谁。
- **靠 LLM 动态选人**:AutoGen 的 SelectorGroupChat。每一步问一次模型「谁下一个说话」。
- **靠点对点显式交接**:AutoGen Swarm 的 `HandoffMessage`、OpenAI Agents SDK 的 handoff(降维成一次工具调用)。一个 agent 主动把话筒交给另一个。
- **靠发布-订阅消息池**:MetaGPT 的 Environment(产物落池、订阅者自动激活)、CAMEL 社会层的消息传递。没有中央调度,靠「谁关心什么」自组织。

**还有一个所有多 agent 框架共有的、绕不开的取舍——上下文与成本。** 多个 agent 各持自己那份历史,token 成本天然翻倍。于是每家都得给出「上下文节流」的答案:CrewAI 的复合打分记忆检索、MetaGPT 的订阅式过滤(只 observe 自己 watch 的消息)、OpenAI Agents SDK 的 Sessions(可 `limit=N` 只取最近 N 条)、CAMEL 的 `message_window_size` 滑窗 + `prune_tool_calls_from_memory`、smolagents 的 step_callbacks 手动裁剪。**没有一家能靠「把所有历史塞给所有 agent」活下去**——上下文工程在多 agent 场景下不是可选项,是生存必需。

> 一个贯穿的反直觉:**这几家没有一家是靠「训一个更强的模型」赢的。** 它们全是 provider-agnostic 的编排层,拼的是「怎么组织协作」——角色分工、自由对话、SOP、handoff、图。这正是「harness / 编排比模型更重要」这句话在多 agent 场景下的复现:同一个底座 LLM,换个编排范式,协作的稳定性天差地别。

## 5. 常被忽视的坑

这个流派最容易「看 demo 上头、上生产翻车」。诚实列几条(机器会渲染成 ⚠️ 警示卡):

1. **「多 agent = 更快 / 更强」是最贵的误区。** 你以为把活儿分给三个 agent 就快三倍——真相是**成本先翻几倍**(每个 agent 各持全上下文,三 agent 至少 3× token),而**质量未必更高**:一环的错误会被下游放大成级联幻觉(MetaGPT 立框架就是为了对付这个)。**怎么避**:先问「这个任务真的需要多个 agent 吗?」很多任务一个 agent 带几个工具就够了。别为了「多 agent」而多 agent——它是一种有明确成本的架构选择,不是免费的性能加速。

2. **文档里的编排 ≠ 实际跑出来的编排。** 最典型的是 CrewAI 的层级 Manager–Worker 模式——有权威批评实测它容易退化成「所有任务顺序跑」,出现调错 agent、输出被覆盖、延迟/token 膨胀,与文档承诺不完全一致。**这不是黑某一家,而是所有「让 LLM 当协调器」的框架的通病**:调度逻辑本身也是 LLM 生成的,一样会出错。**怎么避**:别信文档里那张漂亮的架构图,一定要开日志看**实际的 agent 调用序列和消息流**,验证它真按你以为的方式在协作。

3. **框架的「治理稳定性」是被忽视的选型维度。** AutoGen 是最鲜活的教训:一个名字被拆成 AG2(社区)和 MAF(微软企业)两条继承线,原版进维护模式。MetaGPT 的正式发版停在 2024-03、重心转向研究与商业产品。**你今天基于某框架写的代码,一年后维护者可能已经换人、换架构、甚至换名字。** ADK 1.x→2.0 的破坏性变更(存储不兼容、被标「数据丢失」级风险)是另一面。**怎么避**:选型时把「维护者是谁、发版节奏、有没有破坏性重写史、迁移路径清不清晰」和技术能力放在同等位置评估;别只看 star 数和 benchmark。

4. **本地代码执行默认不是安全边界。** smolagents 官方**白纸黑字警告**:`LocalPythonExecutor` 再怎么加 AST 逐操作检查、禁 import、封顶操作数,**也没有任何本地 sandbox 能完全安全**(比如授权了 Pillow 就能生成海量大图撑爆磁盘)。CrewAI 1.x 干脆**移除了 `CodeInterpreterTool`**(安全/精简考量)。**怎么避**:凡是让 agent 真跑代码/命令的场景,生产环境必须上真正的隔离(远程沙箱 / Docker / 云 VM),别指望框架自带的本地执行器兜底——它主要防手滑,不防恶意(见讲权限与沙箱的那篇)。

5. **benchmark 数字横向不可比,且多为厂商自报。** 本篇出现的分数(Magentic-One 的 GAIA/WebArena、OWL 的 GAIA 69.09、smolagents 的「约少 30% 步骤」、MetaGPT 的 MBPP 增益)——**每一个都要问清「什么模型、哪个变体、什么时候、谁测的」**。smolagents 那个 30% 来自未正式发表的内部 benchmark;CrewAI 官网的「63% Fortune 500」「每月 4.5 亿 agentic workflow」是公司自述、无第三方审计;CAMEL 的「可模拟 ~1M agent」是营销措辞。**怎么避**:把「厂商声称」和「已验证事实」严格分开——引用任何数字都带上来源和限定,别拿两家的裸数字直接排名。

6. **别把「一群 agent 讨论出结果」当成「结果是对的」。** 自治对话有一堆已知失败模式:CAMEL 的原始论文就明确记录了 role flipping(角色互换)、assistant 复读指令、flake replies、无限消息循环——这些不是 bug,是自由对话范式的固有风险(CAMEL 的 issue #3362 还记录了「工具调用不收敛」)。**怎么避**:凡用自由对话/动态选人的编排,一定要设**硬性终止条件**(消息上限、预算上限如 MetaGPT 的 `NoMoneyException`、轮数上限)当刹车;并且**产出必须人工 review**——多 agent 让你从「干活」解放到「审活」,但没把你从「负责」里解放出来。

## 6. 一句话记住 + 对比表

> **多智能体编排框架 = 定义一群 agent、并决定它们何时以何种方式协作的软件层。看一家框架,先问「下一个动作是谁发起的、凭什么」——答案是角色顺序(CrewAI)、自由对话(AutoGen)、固化 SOP(MetaGPT)、还是点对点交接(OpenAI Agents SDK)。七家没有一家靠更强的模型赢,拼的全是「怎么组织协作」这一层。**

七家横向对齐(数字均为截至 2026 年中的快照,star 数与 benchmark 会持续漂移):

| 系统 | 定位 | 内核(协作范式) | 开源/闭源 | 最适合谁 | 状态 |
|---|---|---|---|---|---|
| **CrewAI** | 角色扮演多 agent 编排框架 | 角色/任务/委派 + 双层(Crews 自治 / Flows 确定性) | 开源(MIT) | 想最快搭出可跑的角色分工团队 | 活跃(1.x,约每月一版) |
| **AutoGen / AG2** | 事件驱动多 agent「对话」框架 | 对话即编程 + 「谁下一个说话」调度(RoundRobin/Selector/Swarm) | 开源(AutoGen MIT / AG2 Apache-2.0) | 想要灵活对话式协作、能接受治理动荡 | 原版维护模式;AG2 社区活跃;微软主推 MAF |
| **MetaGPT** | 「AI 软件公司」多 agent 框架 | SOP 固化 + 结构化通信(消息池发布-订阅) | 开源(MIT) | 一句话→完整项目脚手架/数据分析 | 框架活跃;正式发版放缓、重心转研究+MGX |
| **OpenAI Agents SDK** | 轻量代码优先编排框架 | handoff 降维成工具调用 + Runner 循环 | 开源(MIT) | 已在 OpenAI 生态、要最少胶水 | 活跃(周级发版,含 AgentKit 上层) |
| **Google ADK** | 代码优先、企业向 agent 框架 | 确定性图/节点 + LLM 动态路由(显式二分) | 开源(Apache-2.0) | 要 ADK+MCP+A2A 生态 + Vertex 托管 | 活跃(2.0 破坏性重构,Python/Go GA) |
| **CAMEL-AI** | 研究向「agent 社会」框架 | role-playing + inception prompting;Workforce 层级协作 | 开源(Apache-2.0) | 做合成数据 / agent 社会仿真 / 研究 | 活跃(NeurIPS 血统,OWL 上层应用) |
| **smolagents** | 极简 code-as-action agent 库 | 动作写成可执行 Python 代码 + 层级 managed agents | 开源(Apache-2.0) | 要一眼读完源码的最小 agent 库 | 活跃(每 ~12-15 天一版) |

**怎么选**:想**最快搭一个角色分工的团队**、能接受灵活度换速度,看 CrewAI;想要**灵活的对话式协作**、且愿意在 AG2/MAF 之间做迁移决策,看 AutoGen 一系;想**一句话生成整个软件项目 / 用固化 SOP 约束协作**,看 MetaGPT;已在 **OpenAI 生态**、想要最薄的胶水,看 OpenAI Agents SDK;要**企业级部署 + 确定性图 + 跨厂商互操作**,看 Google ADK(配 A2A/MCP);要**做研究 / 合成数据 / agent 社会仿真**,看 CAMEL-AI;要**极简、可魔改、Hub 原生的 code-as-action 库**,看 smolagents。

对照「讲 LangGraph 的那篇」:LangGraph 把协作画成一张**你手写的显式状态图**(节点/边/条件),控制力最强、也最重;本篇这七家则各自押了另一种范式——从 CrewAI 的角色流水线,到 AutoGen 的自由对话,到 MetaGPT 的固化 SOP。**它们和 LangGraph 拼在一起,才是「多 agent 编排」这块拼图的全貌**:一头是「你精确画图」,一头是「让 agent 自由对话」,中间散落着角色、SOP、handoff、消息池各种折中。

> 最后一句:**多 agent 不是终点,是一个有明确成本的架构选择。** 先确认你的任务真的需要一支队伍、而不是一个带够工具的 agent;确认了,再回到本篇那个问题——「下一个动作是谁发起的、凭什么」——挑一种最贴合你任务「流程确定性」的协作范式。选对范式,比选对框架更重要。
