## 0. 全景:不写代码,也要把 agent 搭出来

前面几篇拆的东西,有一个共同的隐含前提:**你会写代码。** Claude Code 要你敲命令、Aider 要你在终端里对话、LangGraph 那一类框架更是直接给你一堆 Python 积木让你自己拼 agent 循环。讲 LangGraph 的那篇说得很清楚:**框架是 harness 的「原料」,你得自己写代码把它搭成一个能跑的东西。**

可现实里有一大批人——运营、产品经理、业务分析师、也包括「懒得为每个想法都从零写一遍胶水代码」的工程师——他们想搭 agent,但**不想(或不必)写那一堆接线代码**。于是出现了这么一类东西:**你在浏览器里拖几个方块、连几根线,一个能调模型、查知识库、调工具、还能对外发布成 API 的 agent 就搭出来了。**

这就是本篇的主角——**低代码 / 可视化 agent 平台(low-code / visual agent platform)**。

```
       代码优先(你自己写循环)          低代码平台(你拖方块连线)
                                                          ┌── 部署/运维/多租户
   LangChain / LangGraph  ─────────▶  Dify / n8n / Coze / Langflow  ── 日志/追踪(可观测)
   给你积木,自己拼 harness              给你画布,连线即 harness    └── 一键发成 API / MCP
        │                                       │
   表达力最强、门槛最高                    门槛最低、开箱带运维
        │                                       │
        └──────── 同一条谱系:从「写 agent」到「画 agent」────────┘

                       托管 SaaS(你连搭都不用搭)
                       ↑ 更上层:ChatGPT、Devin 这类成品(见前面几篇)
```

把它摆进全站地图:讲工具的那篇讲过「工具是为模型设计的接口」、讲扩展生态的那篇讲过「插件 / MCP 怎么把外部能力接进来」、讲子代理编排的那篇讲过「多 agent 怎么交接」。**本篇要看的,是把这些东西全部「画」到一张画布上、再配一套部署运维的那一类系统。** 它们不是新的 agent 循环范式——循环还是你熟悉的那个 ReAct / tool-calling 环——而是把「怎么组装、怎么上线、谁来运维」这半张图给端了。

这一篇深拆两家形态最典型的:**Dify**(LLM 原生、RAG + Agent + Workflow 一体的 LLMOps 平台)和 **n8n**(从通用自动化长出来、AI 只是其中一类节点的工作流引擎)。再用一个「群像」小节横扫另外两家:字节的 **Coze** 和 IBM 系的 **Langflow**。

## 1. 这一类到底是什么、边界在哪、为什么值得单独一章

先给一句话定义:

> **低代码 / 可视化 agent 平台,是把「agent 的组装 + 部署 + 运维」搬到一张可视化画布上的宿主系统:你用拖拽连线代替写胶水代码,平台替你管好执行引擎、沙箱、日志、多租户和对外接口。**

拆开这句话,三个关键词划出它的边界:

- **可视化编排(visual orchestration)**:核心交互是「拖一个节点、连一根线」。一个节点是「调 LLM」「查知识库」「调工具」「走条件分支」,线代表数据/控制怎么从一个节点流到下一个。你不是在写 `agent.run()`,你是在**画一张图**。
- **低代码(low-code),不是无代码**:简单场景纯拖拽就够,但每家都留了一个**「代码逃生舱」**——一个能跑 Python/JS 的 Code 节点。你可以不写代码,但**卡住时总能写**。这条底线很重要:它决定了平台的表达力天花板。
- **平台 / LLMOps,不只是库**:它比 LangChain 那种「纯库」靠上一层——自带 UI、日志、追踪(tracing)、多租户、把应用**一键暴露成 REST API 或 MCP server**。「LLMOps」这个词的意思是:不光帮你把 agent 搭出来,还帮你把它**运维起来**(观测、管模型、管凭据、管版本)。

**它跟前面几篇的东西怎么划界?** 用三句话对齐:

- **和代码框架(LangChain / LangGraph)的界**:框架给你积木、要你写代码;平台给你画布、要你连线。框架表达力无上限但门槛高、还得自己搭运维;平台门槛极低、开箱带运维,但表达力被画布约束。**这是本篇最核心的一组取舍,后面反复回扣。**
- **和托管成品(Devin、ChatGPT 那种)的界**:成品是「你连搭都不用搭,直接用」;平台是「你还得自己搭一个 agent 出来,只是搭得很轻松」。平台的产出是**你的 agent**,成品的产出是**别人替你做好的 agent**。
- **和裸 agent 循环的界**:平台内部跑的还是你熟悉的那个循环(ReAct / 原生 tool-calling,见讲核心循环和讲规划推理的那两篇)。**平台没有发明新的 agent 循环,它发明的是「循环外面那一圈」——组装方式、执行引擎、运维面。**

为什么值得单独一章?因为**这是绝大多数「非 AI 原生公司」真正把 agent 用起来的入口。** 一家已经有一堆业务系统的公司,想给客服加个 AI、想让运营自动跑个数据流水,几乎不会从零写 LangGraph——他们会打开一个可视化平台,把现成的连接器和一个 LLM 节点连起来。**这一类系统的采用量,可能比所有编码 agent 加起来都大。**

> 一句话:**这一类平台卖的不是「更强的 agent」,而是「更低的搭建门槛 + 一整套开箱即用的运维」。** 它们赌的是:大多数人缺的不是模型,是把模型接进业务、还能上线运维的那一整套脚手架。

## 2. 一次任务在这类系统里怎么流动

可视化平台有一个所有人第一次看都会踩的**认知陷阱**:画布上那张图,和 agent 内部那个循环,**是两个不同层次的东西**。看懂这两层的嵌套关系,就看懂了这一整类系统。

**外层是「图执行」——确定性的、你画出来的。** 数据从一个节点流到下一个节点,走哪条边由你画的连线和条件分支决定。这一层是**你**在控制流程。

**内层是「agent 循环」——非确定性的、LLM 驱动的。** 当图执行走到一个「Agent 节点」时,控制权临时交给 LLM:它自己决定调哪个工具、调几次、什么时候收手。这一层是**模型**在控制流程。

拿 Dify 走一遍最直观。设想你在画布上搭了这么一条流程:「用户提问 → 查公司知识库 → 交给一个能调工具的 Agent 节点 → 输出」。一次请求进来是这样流动的:

```
  用户请求(或直接 POST 到应用自动生成的 REST 端点)
     │
     ▼
① 入队       所有节点任务进一个统一队列,调度器管依赖与顺序(不是直接递归跑)
     │        worker pool 按负载自动伸缩,并行分支的状态由队列统一管
     ▼
② 知识检索   Knowledge Retrieval 节点:向量 / 关键词(BM25)/ 混合检索,取 top-k
     │
     ▼
③ Agent 节点 ←──── 控制权在这里从「图」交给「LLM」───────┐
     │                                                      │
     │   初始化:装 instructions + query + 工具集 + 记忆     │
     │   设 Maximum Iterations(防死循环的硬上限)           │
     │        │                                             │
     │        ▼                                             │
     │   ┌─▶ 调 LLM ─▶ 要调工具?─是─▶ 执行工具 ─▶ 结果回灌 ┘
     │   │                │                    (再调 LLM)
     │   └── 否,给最终答案 ◀┘
     │        (Function Calling 或 ReAct 策略,可插拔)
     ▼
④ ResponseCoordinator 把多节点的流式输出按正确顺序拼给用户
     │
     ▼
⑤ 全程落库:日志 / 追踪 / iteration count / reasoning trace(可观测)
```

注意 ③ 里那个内环——它就是你在讲核心循环那篇学过的 observe→think→act,只不过现在它**被包在一个节点里**,而这个节点又**被包在一张你画的图里**。这就是「双层嵌套」的完整样子。

**如果流程里没有 Agent 节点会怎样?** 那它就退化成一条**纯确定性的工作流**——每一步调什么、按什么顺序调,全是你预先画死的,LLM 只在某个 LLM 节点里生成一段文本,不做任何自主决策。**这恰恰是这类平台最微妙、也最容易被忽视的一点:同一张画布,既能画「死板的流水线」,也能画「自主的 agent」,还能两者混着画。** Dify 官方把这件事叫「让工作流学会自主推理」——工作流是骨架,Agent 节点是骨架上那几处「交给模型自己拿主意」的关节。

n8n 的流动骨架几乎一样,只是外层平台回路更重(它天生是个分布式自动化引擎):触发器(Chat / Webhook / 定时)进来 → main 进程把任务**入 Redis 队列**、自己不执行 → 空闲的 worker 抢到任务、从数据库拉工作流定义 → 图执行走到 **AI Agent 节点** → 节点从它挂着的一堆**子节点**(一个语言模型、若干工具、可选记忆)现场组装出一个 agent → 跑内层 tool-calling 循环 → 结果沿连线继续走下游节点。

> 一句话:**看这类系统,先分清你看的是哪一层。** 外层图是你画的确定性流程,内层循环是 LLM 的自主决策。平台的全部本事,就是把这两层干净地嵌套起来,再在外面套上「触发→队列→worker→落库」的工业化执行壳。

## 3. 拆开看

现在把两家主角搬上手术台深拆,最后用一个「群像」小节横扫另外两家。每家都对着上面那条「双层嵌套」的骨架,看它在哪一层、下了什么样的重注。

### 3.1 Dify(LangGenius):LLM 原生,RAG + Agent + Workflow 一体的画布

**它是谁**:Dify 是一个**可视化、自托管的 LLM 应用与 agentic 工作流开发平台**,由 **LangGenius, Inc.**(总部在加州 Sunnyvale)出品,2023 年 3 月上线。截至 2026 年中,GitHub 上约 **148k star**(核对时点 2026-06/07),最新稳定版 **1.15.0(2026-06-25)**;2026 年 3 月刚完成一轮 **$30M 的 Pre-A 融资**,是这个赛道商业化很积极的一家。它的官方仓库描述已经从早期的一串功能罗列,收敛成一句 "Production-ready platform for agentic workflow development."——**「生产级」和「agentic workflow」两个词,就是它的自我定位。**

**先说许可这个坑,因为很多人会踩**:Dify 用的是 **Dify Open Source License**——基于 Apache 2.0 **加两条附加条款**:(a) 不得拿 Dify 去运营**多租户 SaaS 服务**(除非另买商业授权);(b) 不得移除界面上的 Dify LOGO 与版权。所以严格讲,**它不是纯 OSI 意义上的开源,是「source-available + 商业条款」。** 自己团队内部用、私有部署,完全免费;但你想拿它包装成一个对外卖的多租户 SaaS,就踩线了。

**它下的重注:把 RAG、Agent、Workflow、人工审批全塞进同一张画布,而且是 LLM 原生的。** 别家(尤其 n8n)是「通用自动化平台顺手长出了 AI 能力」,Dify 反过来——**它从第一天就是为 LLM 应用生的**。这体现在两个地方:

**第一,它有两条不同的回路,而且刻意让你两条都能用。**

- **宏观:队列式图引擎(Queue-based Graph Engine,1.9.0 起)。** 这是它 2025 年 9 月的一次架构重写。旧模型是「递归直接执行」,新模型是**所有节点任务先进一个统一队列**,由调度器管依赖和顺序,worker pool 按负载自动伸缩。好处很实在:并行分支不再各自维护「难以复现」的状态,状态由队列统一管,并行执行的报错少了。它还配了个 **ResponseCoordinator** 专门保证多节点流式输出的时序、一个 **CommandProcessor** 支持运行中 pause/resume/terminate、并且**可以从任意节点开始执行**(partial run / resume / 调子图)。
- **微观:Agent 节点内部循环。** 前面 §2 画的那个内环就是它。有意思的是它的**推理策略(Agent Strategy)是可插拔的**:可以用 **Function Calling**(把工具定义直接交给模型原生的 function-calling,适合 GPT-4、Claude 这类强模型),也可以用 **ReAct**(用结构化 prompt 逼出显式的 Thought→Action→Observation,适合 function-calling 弱的模型、且推理链透明)。官方说这让 Dify 成了「AI 推理策略的试验台」。这正好回扣讲规划推理那篇讲的两种范式——**在 Dify 里,你能一个开关切换它们。**

**第二,RAG 是它的一等公民,不是外挂。** 它自带一个知识库子系统:向量库通过 `VectorFactory` 工厂模式抽象,**支持 20+ 种实现**(默认 Weaviate,另有 Qdrant、Milvus、pgvector 等),检索支持向量 / 关键词(BM25)/ 混合三种。讲 RAG 那篇讲的东西,在 Dify 里是画布上一个开箱即用的节点。

**它的工业化底座**(对着讲 harness 核心组件那篇扫一遍):

- **执行环境 / 沙箱**:一个独立的 **DifySandbox**(用 Go 写),隔离执行 Code 节点(Python/Node.js)和 Template Transform,网络可开关;再配一个 **ssrf_proxy** 防止沙箱里的代码或 HTTP 节点向内网乱发请求(SSRF 防护)。这是标准的「代码逃生舱要关进沙箱」的做法,回扣讲权限沙箱那篇。
- **上下文 / 记忆**:Agent 节点用 **TokenBufferMemory** 按 token 预算保留最近 N 条消息;长期知识不塞进对话记忆,而是走 RAG 检索注入——**「短期靠 buffer、长期靠检索」这条分工,和讲上下文记忆那篇讲的完全一致。**
- **工具 + MCP**:50+ 内建工具 + 一大堆插件工具。而且 **v1.6.0 起做了双向 MCP 支持**(协议版本 2025-03-26):既能当 MCP client 去调外部 MCP server(Linear、Notion、Zapier 等),也能把**自己的 Dify 应用暴露成一个 MCP server** 给 Claude Desktop、Cursor 调用。讲扩展生态那篇讲的 MCP,在这里是双向打通的。
- **人在环(HITL)**:**Human Input 节点(v1.13.0,2026-02)**。工作流能在关键决策点**原生暂停**、推一个可定制表单给人(web 里展示或邮件发出),人可以 review、改变量、点 Approve/Reject/Escalate 决定走哪条分支。关键在于**引擎持久化这个暂停态**——默认能挂起长达 **7 天**(`HUMAN_INPUT_GLOBAL_TIMEOUT_SECONDS` 默认 604800 秒)再可靠 resume。这不是把进程挂着等,是把状态落盘、随时能捡回来,回扣讲权限那篇的「破坏性动作要人确认」。
- **插件运行时(v1.0.0 起)**:模型、工具、策略全从核心解耦成**插件**,升级插件不用动整个平台大版本。本地部署时插件作为**子进程**、经 stdin/stdout 管道通信,由一个 plugin daemon 管生命周期。**注意一个安全上的取舍:插件的隔离不走运行时沙箱,而是靠「加密签名 + marketplace 预装审核」**——这和 Code 节点走 DifySandbox 是两套不同的隔离机制,别混。

**Dify 的灵魂一句话**:它是「LLM 原生」的低代码平台——RAG、Agent、确定性工作流、人工审批这四样东西,在同一张画布里可以自由组合。它赌的是:一个团队想把 LLM 用到生产里,缺的是把这四样拼起来、还能观测和运维的那整套东西,而不是缺某一样。

### 3.2 n8n:从通用自动化长出来,AI 只是画布上的一类节点

**它是谁**:n8n(读作 "n-eight-n")是一个 **fair-code 的可视化工作流自动化平台**,由柏林的 n8n GmbH 出品,创始人 Jan Oberhauser 在 2019 年 6 月单人把它推上 GitHub。截至 2026 年中,GitHub 约 **195k star**(2026-07-03 读数),最新版到 **2.28.x**(2025-12 发的 2.0 是一次「安全硬化」大版本);2025 年 10 月刚拿了 **1.8 亿美元 C 轮、估值 25 亿美元**(领投 Accel,NVIDIA 创投臂 NVentures 等跟投)。

**它和 Dify 最根本的区别,一句话:Dify 是「为 LLM 生的平台」,n8n 是「通用自动化平台顺手接了 LLM」。** n8n 的老本行是 iPaaS(集成即平台服务)——就是 Zapier / Make 那种「某系统触发一个事件 → 自动去另一个系统做个动作」的跨系统自动化。它画布上有几百个现成的应用连接器(Slack、Postgres、Google、HTTP……),AI Agent 只是**后来加进来的又一类节点**。

**这个出身决定了它的两个独门设计:**

**第一,agent 只是图上一个节点,于是 agent 能顺手调你所有的业务系统。** 这是 n8n 最强的一张牌。别的平台你想让 agent 调一个内部 CRM,得自己写个工具封装;在 n8n 里,那个 CRM 早就是画布上一个现成节点了。它的杀手锏叫 **Workflow Tool**——**把整条 n8n 工作流打包成一个工具**给 agent 调。于是一个 agent 无需写任何胶水代码,就能间接调用平台上全部几百个集成。**「已经有一堆系统、想加个 AI」的公司,选它就是因为这个。**(它列的客户口径里有 Volkswagen、Delivery Hero、KPMG、Twitch 等,未逐一独立核实。)

**第二,类型化的 AI 连接(typed AI connections)——本篇最值得记住的一个设计。** 普通节点之间走 `main` 连接、传 JSON 数据。但 AI 节点用的是**专用类型的连接**:AI Language Model、AI Memory、AI Tool、AI Retriever、AI Embedding……每种连接一个颜色、一个类型。

先给直觉:你搭一个 agent,最容易犯的低级错是**把组件插错口**——把一个向量库接到了本该接语言模型的地方。在纯代码框架里,这种错要到**运行时**才爆。n8n 的做法是**用类型系统在编辑期就挡掉**:一个「期望语言模型」的口子,你**根本连不上**一个 vector store。

这背后是一个很深的 harness 哲学分野:

> **n8n 的赌注是:agent 的组装应该像插硬件一样「插错就插不进去」,而不是像写代码一样「插错了跑起来才报错」。** 「一切皆自然语言」的 agent(全靠 prompt 描述)把组装的正确性交给运气;n8n 把它交给类型系统。

具体到 AI Agent 节点本身:**自 v1.82.0 起,它只保留一种 agent 类型——Tools Agent**(旧版那些五花八门的 agent-type 选项都被砍了,这是一次「少即是多」的收敛)。它实现 LangChain 的 tool-calling 接口,**必须至少接 1 个 Tool 子节点、必接 1 个语言模型**,可选接 Memory 和 Output Parser。内层循环就是标准的 tool-calling loop,由 **Max Iterations(默认 10)** 兜底防死循环。

**它的工业化底座**(和 Dify 对照着看,能看出「重运维」的通用平台长什么样):

- **三种进程 + 两种执行模式**:进程分 **Main**(UI/API/编排/触发器)、**Worker**(queue 模式下真正执行)、**Webhook**(专收生产 webhook 立即入队)。执行分 **regular**(单进程直接跑)和 **queue**(分布式:Main 只入队、Redis + Bull 做队列、Worker 抢活执行)。这套东西是为「一个企业每天跑几十万次工作流」设计的,比 Dify 的队列引擎更偏「传统分布式任务系统」。
- **沙箱(Task Runners)**:隔离执行 Code 节点里的用户 JS/Python。**n8n 2.0 起默认开启**,而且是 **secure-by-default**——默认屏蔽环境变量访问、默认禁用那些能执行任意命令的节点。这是 2.0 那次「安全硬化」的核心。
- **记忆是可插拔子节点,不是隐式全局态**:Simple/Window Buffer Memory(进程内、**默认不跨会话持久化**)或外部后端(Postgres/Redis 等),靠 **session key**(会话/用户 ID)分区。**跨会话或跨 agent 的长期状态,n8n 不替你隐式管——你得显式存外部库、按 session ID 读写。** 这是它「显式接线」哲学的又一处体现:不给你魔法,给你可控。
- **多 agent 交接**:两种官方模式——**Router**(条件分支路由到各自带 agent 的子工作流)、**Orchestrator**(主 agent 用 **AI Agent Tool** 把别的 agent 当工具调)。回扣讲子代理编排那篇,但**状态不自动跨 agent 传**,得显式经数据库/传参共享。
- **人在环**:核心是 **Wait 节点 / 「Send and Wait for Response」**——暂停并落盘工作流,经 Slack/Telegram/Chat 发审批,人点批准后外部回调 `$execution.resumeUrl` 恢复。和 Dify 的 Human Input 节点异曲同工,都是「把暂停态落盘、等人、再 resume」。

**n8n 的灵魂一句话**:它把 agent 当成「一张持久化自动化图上的一个节点」,用类型化连接强制正确接线,让 agent 顺手调用你已有的几百个业务系统。它赌的是:agent 的价值不在 agent 本身有多聪明,而在它能不能接进你已有的那一大摊系统里干活。

### 3.3 群像横扫:Coze 与 Langflow

前两家把「LLM 原生平台」和「通用自动化平台」两个极点撑开了。剩下两家各自补上一块拼图——一个是「大厂开源、Go 写的高并发后端」,一个是「LangChain 的可视化 IDE、被 IBM 收进 watsonx」。

**Coze / Coze Studio(字节跳动)——大厂出品、Go 全栈、Apache 2.0 无附加限制的开源内核。** Coze(中文名「扣子」)是字节的可视化 bot/agent 搭建平台,托管 SaaS 于 2024 年 2 月上线;它的开源内核 **Coze Studio**(和 LLMOps 组件 **Coze Loop**)于 **2025-07-25** 才开源。它和 Dify/n8n 的三个独特点:

- **底座是一个独立的 Go 框架 Eino。** 这是 Coze 最值得记住的一点:上层是拖拽画布(前端 React + 字节开源的 FlowGram 画布引擎),**但真正跑 agent 循环、工具调用、工作流 DAG、RAG 的,是底下一个叫 `cloudwego/eino` 的 Go 框架。** 类比一下:**Eino 之于 Coze,约等于 LangGraph/LangChain 之于上层应用——只是用 Go 写、为字节的高并发后端场景打磨过。** Eino 提供三级编排原语(Chain 只前进 / Graph 带分支循环和全局 state / Workflow 是 struct 字段级映射的 DAG),一个 ReAct 式的 `ChatModelAgent`,还有一套 ADK(Agent Development Kit)提供 Agent-as-Tool、Interrupt/Resume 的人在环 checkpoint、以及 Deep Agent / Supervisor 等预置多 agent 模式。截至 2026 年中,`coze-dev/coze-studio` 约 **21k star**、`cloudwego/eino` 约 **12k star**(GitHub API,2026-07-03)。
- **许可最宽松:Apache 2.0,没有 Dify/n8n 那种商业附加条款。** 可商用、可二开、可再分发,几乎无限制——**这是自建团队相对 Dify(禁多租户 SaaS)、n8n(禁转售托管)的关键卖点。**
- **但开源版是「阉割版」,这是最大的坑。** 开源的 Coze Studio **明确不含多 agent、语音 I/O、多模态生成、应用 UI builder**(已交叉两处社区来源)——这些重度能力仍锁在托管商业版里。多 agent 能力在 Eino 的 ADK 层和商业版/内测中的 Coze Space 里,开源 Studio 本体只做单体 agent。**「Apache 2.0 全开源」这句话为真,但你得清楚开源出来的是哪一部分。**

**Langflow(Logspace → DataStax → IBM)——LangChain 的可视化 IDE,一次搭建三种发布。** Langflow 是一个 **MIT 许可**、基于 Python 的可视化 low-code 平台,2023 年 2 月由巴西创业公司 Logspace 发布;2024 年 4 月被 DataStax 收购,2025 年 2 月 IBM 又宣布收购 DataStax——**所以 2026 年 Langflow 的最终母公司是 IBM,会被并进 watsonx 生成式 AI 产品组合。** 截至 2026 年中约 **151k star**(2026-07),最新 v1.10.1(2026-06)。它的三个独特点:

- **它基本就是「LangChain 的可视化 IDE」:连一个节点 ≈ 写一段 LangChain 代码。** 组件(Component)是图的节点、端口(Ports)按数据类型着色——**和 n8n 的类型化连接是同一个思路**:Message(靛蓝)、LanguageModel(紫红)、Tool(青)、Data(粉)等,只有同色端口能连,插错插不上。Agent 组件把规划和工具调用循环**下放给 LLM 的原生 tool-calling**,自己不造 ReAct 执行器。
- **最漂亮的差异化:一次搭建、三种发布。** 同一张 flow 可以**同时**以三种面孔对外:REST API(`/api/v1/run/$FLOW_ID`)、内置 MCP server 把 flow 暴露成一个 tool 给别的 agent 调、以及嵌入式聊天 widget——三者复用同一条执行回路,只是入口不同。**「搭一次,到处发」——这是它相对纯框架的核心卖点。**
- **一个诚实的模糊点**:它早期强绑 LangChain,现在把 **LangChain 降成了可选 bundle**、核心 Agent 组件不再强依赖它。代价是——**官方文档没有逐字说明「去 LangChain 化之后,核心 agent 循环到底由哪个库/执行器驱动」**;`max_iterations=15`、AgentExecutor 这些是 LangChain 的历史/通用口径,当参考、别当当前实现的确证。1.10 版本另做了不少生产化打磨(Redis job queue 支持多 worker、per-flow 向量库做长期记忆的 Memory bases、给 agent 沙箱化文件访问的 File System 组件),官方博客声称把内存占用降了约 89%(单方口径,未见独立复核)。

> 群像一句话:**Coze 赌「大厂级高并发 Go 后端 + 最宽松许可」,但开源版被裁了重度能力;Langflow 赌「LangChain 可视化 IDE + 一次搭建三种发布」,背靠 IBM watsonx。** 加上前面两家,四条路线各占一个生态位。

## 4. 它们与众不同的设计决策与取舍

退一步看,这四家其实都在回答同一个问题——「**怎么让一个不想写(或不必写)胶水代码的人,把一个能用的 agent 搭出来、还能上线运维?**」——但下注的位置差得很远。§3 已经给过每家的「灵魂一句话」,这里只补各自的**代价**(收益前面讲过了,取舍才是本节重点):

- **Dify(押 LLM 原生全家桶)的代价**:许可有商业条款(禁多租户 SaaS、禁去 LOGO);超复杂/高度定制的逻辑,画布不如纯代码灵活;多 agent 更偏「显式编排的图 + 分支」,缺成熟的自治多智能体原语。
- **n8n(押通用自动化 + 类型化接线)的代价**:长时自主、开放式的 coding-agent 式探索不是它的原生场景;多 agent 状态得手工经数据库管;它本质是自动化引擎,LLM 应用的「深度」(RAG 管线之类)不如 Dify。
- **Coze(押大厂 Go 后端 + 最宽松开源)的代价**:开源版功能被裁(无多 agent / 语音 / 多模态生成 / 应用 UI builder);组件多(Studio + Loop + Eino),学习和运维面较大。
- **Langflow(押 LangChain 可视化 + 一次搭建三种发布)的代价**:去 LangChain 化后核心循环实现表述变模糊;要对 agent 控制流做代码级细粒度掌控时,不如 LangGraph 那类代码优先框架。

一句话把四条路对齐:**Dify 改造「LLM 应用要用到的一切怎么在一张画布里凑齐」,n8n 改造「agent 怎么接进已有系统、怎么保证接线不出错」,Coze 改造「后端用什么写、许可能不能随便用」,Langflow 改造「搭好的东西怎么一次性发到三个地方」。**

但真正贯穿这一整类系统的、最需要你记住的一组取舍,是**「画布 vs 代码」**:

> 反直觉的点:**可视化不是「代码的简化版」,它是一组不同的权衡。** 画布买到的是低门槛、天生带运维、非工程师也能读改、组装错误能在编辑期挡掉;卖出去的是表达力的天花板——凡是画布节点表达不了的控制流,你都得退回到那个「Code 节点逃生舱」里写代码。**所以这四家没有一家是「纯无代码」——它们都留了逃生舱,因为它们都清楚:画布能覆盖 80% 的场景,但那 20% 永远得靠代码兜底。**

这也解释了它们和代码框架(讲通用 agent 与框架那篇里的 LangGraph 那一类)的分工:**不是谁取代谁,是同一条谱系的两端。** 原型阶段、业务集成、非工程团队,画布赢;需要对 agent 控制流做外科手术式的精细掌控,代码赢。成熟团队常常两头都用——画布搭外围流程,把最硬的那块逻辑塞进 Code 节点或干脆用框架另写一个服务、再当工具接回画布。

## 5. 局限与坑

这一类平台最容易被「拖拖拽拽就能搭 agent」的爽感冲昏头。诚实列几条最坑的(机器会渲染成 ⚠️ 警示卡):

1. **把「可视化」当成「没有天花板」——其实画布有表达力上限。** 拖拽爽在简单场景;一旦逻辑需要复杂的动态控制流、运行时生成代码、框架级的细粒度掌控,画布就变成一张越连越乱的意大利面。为什么致命:**你可能搭到一半才发现「这逻辑画布画不出来」,又不甘心推倒重来,于是塞进一堆 Code 节点,得到一个既失去代码可维护性、又失去画布可读性的四不像**。怎么避:动手前先判断任务是「集成 + 编排」为主(选平台)还是「复杂自主控制流」为主(选代码框架);Code 节点当逃生舱、别当主干。

2. **没看清许可,把带商业条款的平台包装成对外 SaaS。** 这四家的许可**天差地别**:Coze/Langflow 是干净的 Apache 2.0 / MIT,随便商用;但 **Dify 禁止你拿它运营多租户 SaaS、禁止去 LOGO**,**n8n 禁止你把它作为托管服务转售**。为什么致命:**内部用完全免费,一旦你把它包装成一个对外收费的多租户产品,就直接踩了商业条款的红线**——这不是技术问题,是法律问题,可能等产品都上线了才被发现。怎么避:**动工前把目标平台的 LICENSE 逐条读一遍**,尤其看清「self-host 免费」和「作为服务转售」是两码事;要做对外 SaaS,要么选 Apache/MIT 的、要么买商业授权。

3. **默认开源 == 全功能,结果开源版是阉割版。** 最典型是 Coze:开源的 Coze Studio **不含多 agent、语音、多模态生成、应用 UI builder**,这些锁在托管商业版里。为什么致命:**你照着官网/演示看中的能力,自托管一部署发现根本没有**,方案设计推倒重来。怎么避:**别看「产品」的功能表,去看「你要部署的那个开源仓库」的功能表**——两者往往不是一回事;先拿开源版跑通你的核心场景再规划。

4. **代码逃生舱是把双刃剑——它也是最大的安全面。** 每家都有能跑 Python/JS 的 Code 节点,这是灵活性来源,也是攻击面来源:任意代码执行、SSRF(向内网发请求)、环境变量/密钥泄露。为什么致命:**一个可视化平台常常被交给非安全背景的人使用,他们很容易在 Code 节点里写出能读到密钥、能打内网的代码而不自知**。怎么避:**别关掉平台默认的沙箱硬化**——n8n 2.0 默认开 Task Runner、屏蔽环境变量、禁危险节点;Dify 用 DifySandbox + ssrf_proxy;这些默认值是保护你的,擅自放开等于自拆防线(回扣讲权限沙箱那篇)。

5. **忽视「双层嵌套」,把确定性工作流当成了自主 agent(或反过来)。** 同一张画布既能画死板流水线、也能画自主 agent,新人常分不清自己搭的是哪种。为什么致命:**你以为搭了个「会自主决策的 agent」,其实只是一条把 LLM 当文本生成器的确定性流水线(没放 Agent 节点、模型不做任何工具决策);或反过来,你以为流程可控,其实关键分支交给了 LLM 自由发挥、结果不可复现**。怎么避:搭之前先想清楚每一步该「我画死」还是「交给模型拿主意」——确定性的用条件分支画死,需要自主的才放 Agent 节点。

6. **多 agent 的「交接」大多是显式接线,不是自治协商。** 这四家的多 agent(Dify 的 routing/handoff、n8n 的 Router/Orchestrator、Coze 靠 Eino ADK、Langflow 的 agent-as-tool)**基本都是「你显式编排的图 + 把子 agent 当工具调」**,而且**状态大多不自动跨 agent 传**(n8n 明确要你手工经数据库共享)。为什么致命:**你照着「多 agent 自治协作」的宣传去设计,却发现平台只给你「显式路由 + 手工传状态」,复杂的多 agent 状态管理全落回你自己头上**。怎么避:把这类平台的多 agent 当「显式编排的子流程」设计,别当「会自己商量着干活的一群自治体」;跨 agent 状态老老实实用外部库显式管。

## 6. 一句话记住 + 对比表

> **低代码 / 可视化 agent 平台 = 把「agent 的组装 + 部署 + 运维」搬上一张画布:你连线代替写胶水代码,平台替你管执行引擎、沙箱、日志和对外接口。四家各押一角——Dify 押「LLM 原生全家桶」,n8n 押「通用自动化 + 类型化接线 + agent 只是一个节点」,Coze 押「大厂 Go 后端 + 最宽松开源(但开源版被裁)」,Langflow 押「LangChain 可视化 + 一次搭建三种发布」。它们卖的都不是更强的模型,而是更低的搭建门槛 + 开箱的运维。**

四家横向对齐(数字为截至 2026 年中的快照,star 数随时点漂移):

| 维度 | Dify(LangGenius) | n8n(n8n GmbH) | Coze / Coze Studio(字节) | Langflow(→ IBM) |
|---|---|---|---|---|
| **一句话定位** | LLM 原生的 LLMOps / agentic workflow 平台 | 通用自动化引擎,AI 只是一类节点 | 大厂开源的 agent 搭建平台 | LangChain 的可视化 IDE + 发布层 |
| **内核 / 底座** | 队列式图引擎 + 可插拔 Agent 节点(FC/ReAct) | 三进程分布式引擎 + 类型化 AI 连接 + Tools Agent | 底层 Eino(Go 框架)跑循环/DAG/RAG | 组件图 + 端口类型系统,循环下放给 LLM tool-calling |
| **独特点** | RAG/Agent/Workflow/HITL 一体、双向 MCP、可插拔推理策略 | agent 顺手调几百个业务集成、类型化接线编辑期挡错 | Go 全栈高并发、许可最宽松 | 一张 flow 同时发成 API + MCP + 聊天 widget |
| **开源 / 闭源** | source-available(Apache 2.0 + 商业附加条款:禁多租户 SaaS、禁去 LOGO) | fair-code(Sustainable Use License,禁转售托管,非 OSI 开源) | 内核 Apache 2.0(最宽松),托管版为商业 SaaS | MIT(最干净) |
| **最适合谁** | 想把 RAG + Agent + 审批一体上生产、能接受商业条款的团队 | 已有一堆业务系统、想加 AI 而非从零搭 agent 平台的公司 | 要生产级高并发 Go 后端、许可无限制的自建团队 | 要快速原型 + 一键多渠道发布、或在 IBM/DataStax 生态里的团队 |
| **2026 状态** | 活跃加速;~148k★;1.15.0(2026-06);刚拿 $30M Pre-A | 高度活跃;~195k★;2.28.x;C 轮 $180M / 估值 25 亿 | 活跃;Studio ~21k★ / Eino ~12k★;2025-07 才开源,开源版功能被裁 | 活跃;~151k★;v1.10.1(2026-06);最终母公司 IBM(经 DataStax) |

**怎么选**:想要 **LLM 应用的深度**(RAG + Agent + 工作流 + 人工审批一体)且能接受商业条款,选 **Dify**;想让 agent **接进你已有的一大摊业务系统**、且看重「接线不出错」,选 **n8n**;想要**生产级高并发、许可完全无限制**的自托管后端(但先确认开源版有你要的功能),选 **Coze**;想**快速原型再一键发成 API/MCP/聊天**、或已在 IBM/DataStax 生态里,选 **Langflow**。而只要你需要对 agent 控制流做**代码级的外科手术**,就该退回到讲通用 agent 与框架那篇里的 LangGraph 那类**代码优先框架**——画布和代码,是同一条谱系的两端,不是替代关系。

对照前面几篇:讲编码 agent 的那几篇拆的是「一个 harness 驱动一个模型专攻写代码」,讲通用 agent 与框架的那篇拆的是「产品 / 框架 / 起源三个物种」,而**这一篇补上的是「让不写代码的人也能搭 agent」这块拼图**——它把 harness 里「怎么组装、怎么上线、谁来运维」的那半张图,变成了一张任何人都能上手拖拽的画布。

> 最后一句反直觉的收尾:**这四家没有一家在比谁的 agent 循环更聪明——循环还是那个你早就学过的 ReAct / tool-calling 环。它们比的是循环外面那一圈:组装门槛、运维能力、集成广度、发布方式、许可自由。** 这恰恰是讲「什么是 harness」那篇的主题在低代码世界里的回响——**agent 强不强,拼的从来不只是模型,更是模型外面那套让它能搭起来、跑起来、管起来的东西。**
