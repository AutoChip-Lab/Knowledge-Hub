## 0. 全景:前面全是「编码 agent」,这篇横看三个物种

从⑫到⑯,我们拆的每一个真家伙都长得很像:**都是「一个 harness 驱动一个模型,在代码库里读文件、跑命令、改代码、提 PR」**。Claude Code、Codex、Cursor、Aider、Devin——形态各异,但**都是编码 agent**。你现在对「编码 agent」这个物种已经很熟了。

这一篇要故意打破这个惯性。我们**横着**摆三个**根本不同物种**的东西上手术台,不是为了逐个夸,而是为了让你亲眼看到:**「agent」这个词底下,其实藏着完全不同的几类东西——它们的形态、循环范式、自主度、开放性都不一样。**

```
        产品                 框架                 起源
   ┌──────────┐        ┌──────────┐       ┌──────────┐
   │  Manus   │        │ LangGraph│       │ AutoGPT  │
   │ 你直接用 │        │ 你拿去搭 │       │ 全自主   │
   │ 的成品   │        │ 的积木   │       │ 的第一课 │
   └────┬─────┘        └────┬─────┘       └────┬─────┘
        │                   │                  │
   云虚机 agent        图执行引擎         目标→自生子任务
   上下文工程          节点/边/状态       死循环/烧钱教训
        │                   │                  │
        └───── 产品 ≠ 框架 ≠ 起源:同叫「agent」,是三种不同东西 ───┘
```

> 读这一篇的正确姿势:**别急着比谁强,先分清谁是谁。** 一个是你付费直接用的**产品**,一个是你写代码去搭 agent 的**框架**,一个是 2023 年的**历史起源**。把这三个的「类别」分清了,你对「agent 生态」的地图就立起来了。

## 1. 是什么:先把「三物种」的类别讲清楚

在拆细节之前,必须先做一件事——**划物种**。这是本篇最重要的一句话:

> **「产品」「框架」「起源」是三种完全不同的东西,虽然它们都被人叫作「agent」。** 混淆它们,是新人读 agent 生态时最常摔的跤。

用一句大白话对齐三个:

- **Manus = 一个产品。** 你花钱订阅、打开网页就能用的成品。它自己有一台云端虚拟机、自己的 agent 循环、自己的一套上下文工程。你给它一个目标,它跑完给你一个**成品**(报告、部署好的网站、数据文件)。你不需要写任何代码。**它是「harness + 模型 + 一切」都替你打包好了的最终形态。**
- **LangGraph = 一个框架。** 它本身**不是**一个能用的 agent。它是一堆积木——节点、边、状态、检查点——你(开发者)拿这些积木**自己搭**一个 agent 出来。用⑥篇的术语说:**框架是 harness 的「输入/原料」,不是 harness 本身。** 你可以用 LangGraph 搭出一个像 Manus 那样的东西,但 LangGraph 自己不是 Manus。
- **AutoGPT = 一个起源故事。** 它是 2023 年春天那个引爆全场的开源项目——第一次让大众看见「给 GPT-4 一个目标,它能自己生成子任务、自己循环干下去」。它今天技术上已经不重要了(甚至官方都不再维护那个经典版),但它作为**「全自主 agent 的第一课与教训」**,是绕不过去的一章。

> 一句话:**Manus 是你「用」的,LangGraph 是你「搭」的,AutoGPT 是我们「回顾」的。**

## 2. 各自怎么工作:三段生命周期

拆组件之前,先各看一眼「血液怎么流」。三个物种的工作方式差得越大,你越能记住它们是不同的东西。

**Manus:一次任务 = 一台云虚机跑一圈单-agent 循环。**

```
你给个目标(「帮我调研这五家竞品并写份报告」)
   │
   ▼
① 起虚机   一台云端 microVM 秒级启动(据 E2B 描述,发布期约 150ms 引导)
   │
   ▼
② 列清单   agent 写一份 todo.md,把目标拆成步骤(系统提示保持稳定,护 KV-cache)
   │
   ▼
③ 循环     读事件流上下文 → 选一个允许的工具 → 执行(browser_navigate / shell_exec …)
   │        → 把观测(含报错/堆栈)追加进上下文 → 在上下文末尾重抄一遍 todo.md → 重复
   │        (据其官方博客,平均一个任务约 50 次工具调用)
   ▼
④ 卸载记忆 中间数据写进虚机文件系统;超长内容「可恢复地压缩」(丢正文,留 URL/路径)
   │
   ▼
⑤ 交付     产物存在虚机里;异步——你可以关掉网页,跑完了通知你
```

**LangGraph:一次任务 = 一张有向图按「超步(superstep)」一步步推进。**

```
graph.invoke({"messages":[...]}, thread_id="abc")
   │
   ▼
超步1  Plan:  START 触发 → 路由到 agent 节点
       Execute: agent 节点读 messages 通道 → 调 LLM → 得到 AIMessage
       Update:  AIMessage 追加进 messages 通道 → 存检查点(checkpoint)
   │
   ▼
超步2  Plan:  条件边看 AIMessage 里有没有 tool_calls
              ├─ 有 → 路由到 tools 节点
              └─ 无 → 路由到 END(结束)
       Execute: ToolNode 跑工具 → 得到 ToolMessage
       Update:  ToolMessage 追加 → 存检查点
   │
   ▼
回到 agent 节点,带着工具结果再调一次 LLM …… 直到 LLM 不再要工具 → END
```

**AutoGPT:一次任务 = 一个「自己给自己派活」的递归循环。** 这是它的灵魂,也是它的命门:

```
你给:agent 名字 + 角色 + 最多 5 条目标(「调研公司 X 的前五名竞品,写份报告」)
   │
   ▼
构造 prompt:系统提示(角色+目标) + 21 条可用命令的说明 + 约束(~4000词短期记忆上限、
            「每条命令都有成本」) + 短期记忆 + 长期记忆命中 + 上一条命令结果 + 「只回合法JSON」
   │
   ▼
GPT-4 回一段 JSON:thoughts(reasoning/plan/criticism/…) + command(name, args)
   │
   ▼
命令执行器跑命令(google 搜索 / browse_website / 写文件 / 跑 Python / 派子 agent …)
   │
   ▼
结果 embed 存进记忆 → 回到上一步,GPT-4 自己决定「下一条命令干啥」
   │
   ▼
一直循环…… 直到 GPT-4 自己发出 task_complete   ← 注意:停不停,全靠模型自觉
```

三段流程摆一块,一条共同的血脉就冒出来了:**三家的循环内核都是「LLM 决定下一步动作」(ReAct 血统)——Manus 单-agent 循环、LangGraph 图上的 agent 节点、AutoGPT 递归自我派活,骨架各不相同,但都是「想→行动→观测→再想」。** 记住这条共性,§4 还会回来算它们真正的分野。

## 3. 拆三家

现在逐家拆。每家给一个像样的小拆解,拆完你应该能说清「它是什么物种、循环怎么转、最值得学的一招是什么」。

### 3.1 Manus:云端虚拟机 agent,靠「上下文工程」吃饭

**一句话:Manus 是一个通用自主 agent 产品——你给目标,它在一台云端虚拟机里(浏览器 + shell + 代码 + 文件系统)自己跑完,交回一个成品。**

几个关键事实(部分为厂商自述,已标注,截至 2026 年中):

- **谁做的**:Butterfly Effect(消费品牌又叫 **Monica**)。母公司在开曼群岛注册,团队 2022 年起于中国(武汉/北京),2025 年年中把总部/运营迁到新加坡。三位联合创始人:肖弘(CEO)、季逸超(首席科学家,那篇著名的「上下文工程」博客的作者)、张涛(产品)。**2025 年 3 月 6 日**邀请制内测发布,demo 上线不到 20 小时破百万观看,邀请码一度被炒到几千美元。
- **一段所有权风波**(值得知道,免得你读到旧新闻犯迷糊):Meta 在 2025 年底宣布收购(约 20 亿美元,2025 年 12 月 29 日交割);中国发改委 2026 年 1 月介入调查,4 月 27 日要求交易撤销;Meta 约在 2026 年 6 月切断了对 Manus 的数据访问,随后一个腾讯牵头的财团着手按原价回购 Meta 手中的股份,把控制权还给原班投资人。Manus 作为独立订阅产品继续运营。
- **底层跑什么模型**(**官方未按版本披露**):据联合创始人的公开说法和系统提示泄露,Manus 用的是 **Claude(发布期为 3.5 Sonnet v1,后测 3.7)+ 微调的 Qwen** 变体。**注意:Manus 自己不训练大模型,它是「消费模型」的一方**——它的核心竞争力不在模型,在「上下文工程」。

**拆最值钱的一层——上下文工程(context engineering,回扣⑧篇)。** 季逸超那篇博客是一份罕见的一线实战账本,里面几招直接就是⑧篇的活教材:

- **KV-cache 命中率 = 「最重要的生产指标」。** 为什么?因为 Claude Sonnet 命中缓存的 token 和没命中的差 **10 倍价钱**(据其博客:$0.30 vs $3.00 / 百万 token),而 agent 的输入:输出约 **100:1**(极度 prefill 密集)。为了让缓存尽量命中,Manus 做了三件事:**系统提示前缀保持稳定、上下文只追加不修改、序列化保持确定性**——任何一点抖动都会让后面整段缓存失效、成本飙升。这解释了它一堆看似奇怪的设计选择背后的经济学。
- **文件系统即记忆。** 上下文窗口再大也有限,Manus 把中间产物写进虚机文件系统当「无限记忆」,并用**可恢复压缩**——比如网页正文太长,就丢掉正文只留 URL,需要时再抓回来。**永不永久丢信息**(回扣④⑧的「文件系统当外部记忆」)。
- **todo.md 复述(recitation)。** agent 每跑几步,就在上下文**末尾重抄一遍**那份待办清单。这是对付「lost-in-the-middle(中间遗忘)」的土办法:把目标不断拉回模型注意力的最末端(也就是最「新鲜」的位置),防止它跑着跑着忘了要干嘛。
- **不动态删工具,改用 logit masking(掩码)。** 想限制某一步能调哪些工具?常规做法是把不该用的工具定义删掉——但那会**破坏 KV-cache、还让模型犯懵**。Manus 改成:工具定义一个不删,只在采样时用 logit 掩码「屏蔽」掉不该选的。这是「框架设计必须迁就推理机制」的经典一课。
- **错误留在上下文里。** 报错、堆栈不擦掉,当作模型的「隐式信念更新」——让它自己看见「刚才那样干失败了」。

工具方面:据 E2B 博客约 **27 个工具**(系统提示泄露版数到 **29**),按组前缀命名(`browser_*`、`shell_*`);浏览器自动化基于 **Playwright**,发布期用了开源的 **Browser Use** 库。虚机基础设施发布期用 **E2B 的 Firecracker microVM**(Ubuntu,秒级引导),**特意不用 Docker**(启动要 10–20 秒、还没有完整 OS)。

> 反直觉的点:**Manus 的护城河不是模型,是「怎么喂模型」。** 它公开信奉「**少结构,多智能(less structure, more intelligence)**」——不搞重型多-agent 编排,就一个单-agent 循环 + 一套精算到极致的上下文工程。团队自嘲这套框架重写了 4 次,戏称「Stochastic Graduate Descent(随机研究生下降)」。

**一个必须给你打的预防针**:网上有些第三方文章(包括一篇非-Manus-团队的 arXiv 学术论文)把 Manus 描述成一个有独立「Planner / Verification Agent」的重型多-agent 系统——**这是外部推断,不是官方描述,把它当事实是不对的**。Manus 团队自己的一手材料描述的是**单-agent、单上下文流**的循环。唯一被确认的多-agent 并行,是后来 v1.6 加的 **Wide Research**(扇出并行子任务)。这正是⑧篇「读资料要分清一手/二手、工程口径 vs 营销/学术口径」的活样本。

### 3.2 LangGraph:把 agent 建成一张「图」——它是积木,不是 harness

**一句话:LangGraph 是 LangChain 出的一个低层编排框架——它让你把 agent 建模成一张有向图:节点是计算步骤,边是控制流,一个带类型的「状态对象」贯穿全程,并且允许循环(cycle),从而能表达 agent loop。**

先把它的**物种**钉死,这是回扣⑥篇的关键:

> **LangGraph 是「框架/积木」,是 harness 的输入,不是 harness 本身。** 它不给你一个能直接用的 agent;它给你 `StateGraph`、`checkpoint`、`interrupt` 这些原语,让你**自己搭**一个 harness 出来。你和 LangChain 公司都可以用它搭产品,但它本身不是产品。这跟 Manus(成品)是根本区别。

拆它的核心积木(据官方参考,截至 2026 年中,LangGraph 已发布 1.0 稳定版、当前在 1.2.x 系列):

- **StateGraph(状态图)+ 一个 Pregel 运行时。** 你用 `add_node()` / `add_edge()` / `add_conditional_edges()` 搭图,`.compile()` 编出一个可运行对象。它的运行时是 **Pregel / BSP(整体同步并行)**模型:每个「超步(superstep)」分 Plan → Execute → Update 三段,节点在同一超步内**看不到彼此的写入**,只能看上一超步已提交的状态。**这条约束听着别扭,好处是给了你数据库事务般的语义**——并行节点安全、失败可回滚(一个并行节点挂了,这一超步的写入一个都不提交)。
- **状态与通道(channel):并行安全的秘密。** 状态是个带类型的对象(`TypedDict`/Pydantic),**每个字段绑一个「通道」,通道决定这个字段怎么合并写入**。比如默认 `LastValue`(取最新值);而 `BinaryOperatorAggregate`(**这才是真类名**,别写成坊间简称「BinOp」)配上 `operator.add`,就能让多个并行节点安全地往同一个 list 里追加而不打架。这套「每字段一个 reducer」是 LangGraph 能干净并行的核心机关。
- **持久化 / 检查点(checkpoint):内建在运行时里。** 配一个 `BaseCheckpointSaver`(开发用 `InMemorySaver`,生产用 `PostgresSaver`),它就在**每个超步后自动存状态**。这一下子解锁了:崩溃可恢复、多天长跑、时间旅行调试(time-travel)、以及——
- **human-in-the-loop(人在环中):`interrupt_before` / `interrupt_after`。** 在某个节点前后**暂停并存档**,外部代码用 `get_state` / `update_state` 检查和修改状态,再 `invoke(None, config)` 从断点续跑、**不重跑前面的节点**。这是把「人工审批」焊进 agent 骨架的正规做法(回扣⑨⑩)。
- **循环是一等公民。** 跟只能画 DAG(有向无环图)的系统不同,LangGraph **允许 cycle**——这正是「agent loop」能被画出来的前提;一个 `recursion_limit` 防止跑飞。
- **多-agent 拓扑是「选装库」。** LangGraph 不把某种多-agent 结构焊死在核心里,而是拆成独立包:`langgraph-supervisor`(一个主管 agent 派活给下属,回扣⑩)和 `langgraph-swarm`(平级 agent 互相「交接 handoff」)。想用哪种自己装哪种。

**一个必须讲的迁移坑(据素材):** 你在老教程里到处能看到的那个 `create_react_agent`(以及整个 `langgraph.prebuilt` 模块),在 **1.0 已被弃用(deprecated)**,官方推荐迁到 `langchain.agents.create_agent`;向后兼容维持到 **2.0**、届时移除。**这些 prebuilt 一直只是「薄薄的封装、可选的便利品」**——LangGraph 的本体是那些低层积木,prebuilt 只是帮你少写几行样板。别把「弃用一个 prebuilt」误读成「框架变天」。

> 反直觉的点:**LangGraph 故意做得「低层」。** 你可能觉得框架应该越高层越省事,但它赌的是另一头:**给你对节点/边/状态的直接控制,换取生产级的可靠性(持久化、HITL、可恢复)**——代价是学习曲线比「把 agent 当员工」的 CrewAI 之类陡。它更像给你一台机床,而不是一件家具。

### 3.3 AutoGPT:全自主 agent 的「起源故事」与第一堂教训课

**一句话:AutoGPT 是 2023 年 3 月 30 日发布的开源应用,第一次让大众看见「给 GPT-4 一个高层目标,它能自己生成子任务、用工具执行、观察结果、再迭代」——不需要人一步步喂 prompt。**

它的历史地位怎么强调都不为过:发布在 GPT-4 面世约两周后,几天内冲上 GitHub 趋势榜第一,**它让「autonomous AI agent(自主 AI agent)」这个词进入了大众词汇**,引爆了 2023 年春天那场「agent 淘金热」(AgentGPT、HuggingGPT、CAMEL、AutoGen、CrewAI、MetaGPT 一串跟着冒出来)。它的孪生极简版是 Yohei Nakajima 的 **BabyAGI**(约 100 行 Python,拆成执行/造任务/排优先级三个角色),两者独立地演示了同一个经典循环:**目标 → 造任务 → 执行 → 重排 → 重复**,成了这个模式的标准教学骨架。

拆经典版(AutoGPT "Classic", 2023)的组件——一个 GPT-4 之上的单-agent 递归循环(§2 的 ASCII 已给全景,这里补关键机关):

- **固定 21 条命令的动作空间**:google 搜索、Selenium/BeautifulSoup 浏览、文件读写、跑 Python 子进程、派子 agent(`start_agent`/`message_agent`/`delete_agent`)。空间小而明确、模型好选,但换新工具类型就得改代码/写插件。
- **双层记忆**:短期是 FIFO(v0.2.1 只保留前 9 条消息,受 ~4000 词上限约束);长期把结果用 `ada-002` embed、KNN top-10 检索,存本地库或 Pinecone。**后来团队判定这套过度设计了**——把向量库换成了「一个 JSON 文件 + NumPy 点积」,因为 agent 很少产生足够多的独立事实来撑起一个向量库(这是一记漂亮的「先剖析再优化」教训:瓶颈是 LLM 推理的秒级耗时,不是相似度搜索的毫秒级)。
- **严格 JSON 输出,带一个 `criticism`(自我批评)字段**:本意是让它一边干一边自省纠错——**但实践中自我批评往往空洞、并没能阻止它跑飞。**

**现在讲最重要的部分——它的著名失败模式,也就是本篇要送给你的「第一堂教训课」:**

- **无停止条件 → 死循环。** 整个循环**没有硬性终止条件**,全靠模型自己发 `task_complete` 喊停。给它一个含糊目标(「把竞品彻底调研一下」),它永远找不到满意的收手点,就无限地搜、重读、「改进」下去。有案例记录一次跑掉了 **300+ 次 API 调用、耗时约 2 小时**。
- **目标漂移(goal drift)。** 自己给自己派活派着派着,就偏离了最初目标,越干越野。
- **烧钱。** 递归地调 GPT-4,成本指数级堆积。这里有一条**说明性**(不是实测)的算式,却极其锋利:若每步可靠性 85%,一个 10 步任务的成功率只有 0.85¹⁰ ≈ **20%**——**这就是「长链全自主」最核心的反对论据**(85% 是教学假设,不是 AutoGPT 的实测数字,别当真实指标引用)。

**结局说明了一切:AutoGPT 自己「反水」了。** 2024 年它整个产品转向——抛弃递归自主循环,改成一个**低代码、可视化、块状 DAG 工作流搭建器**(人来定流程,agent 填格子)。这个转向,连同 AutoGen/CrewAI/MetaGPT 集体转向「有角色、有交接、有人在环」的结构化多-agent——**是整个行业用脚投票:大家要的是「可靠性」,不是「纯自主」。** 顺带一提,经典版今天已**明确不再做安全维护**(依赖不更新、issue 不修),别在生产里用它。

> 一句话:**AutoGPT 是「设个目标、走开、回来收结果」这个美梦的巅峰,也是它破灭的起点。** 它最大的价值不是能用,是它**用最戏剧化的方式暴露了全自主的所有坑**——后面每一个成功的 agent 框架,都是在「补 AutoGPT 的漏」。

## 4. 设计决策:三分物种 · 两种编排哲学

拆完三家,退一步,把真正值得学的取舍拎出来。本篇的所有张力,都能归到两条轴上。

**第一轴:产品 / 框架 / 起源的三分。** 这是本篇的骨架。同叫 agent,**你和它的关系完全不同**:
- 对 Manus(产品),你是**用户**——你付钱,它替你把一切(模型、循环、上下文、虚机)打包好,你只管给目标、收成品。你几乎不能改它内部。
- 对 LangGraph(框架),你是**开发者**——它给你积木,harness 得你自己搭。灵活度拉满,但你得会写代码、扛可靠性。
- 对 AutoGPT(起源),你更多是**学习者**——它教你「全自主为什么会崩」,后面每个成功的框架都是在补它的漏。

**第二轴:「少结构,多智能」vs「图式强编排」。** 这是 Manus 和 LangGraph 的正面对撞,也是⑤⑩两篇「规划/编排该多重」那个老问题的两个极端答案。
- **Manus 押「少结构」**:不搞重型编排,就一个单循环 + 极致上下文工程,赌「模型够聪明,别用死板结构捆住它」。省心、上限高;但可控性、可复现性弱,出了岔子难精确干预。
- **LangGraph 押「强结构」**:把控制流显式画成图、状态显式建模、每步存档。可控、可审计、可恢复、可人工插手;但要你预先设计 schema 和拓扑,啰嗦、上手陡。
- **没有谁对谁错**——这就是⑥篇那句「harness 比模型重要」在两个方向上的展开:一个把重量压在「聪明地喂上下文」,一个把重量压在「确定性地编排流程」。

> 一句话:**本篇三家,本质是在回答两个问题——「你和 agent 是什么关系(用/搭/学)」和「agent 该被多重的结构约束(少结构靠上下文 / 强结构靠编排)」。** 把这两个问题分清,三个物种的位置就都定下来了。

## 5. 局限与坑

这一节把三家踩过的、以及你读它们时最容易踩的坑,按严重程度排一排。

1. **【最大警示】把 AutoGPT 的教训当耳边风——迷信「全自主长链」。** AutoGPT 用最戏剧化的方式证明了:**没有硬停止条件的全自主循环,会死循环、会目标漂移、会指数级烧钱**;而「每步 85% 可靠、10 步只剩 20%」这条(说明性)算式,是长链全自主的死穴。**避坑**:给 agent 明确的成功/终止判据(回扣⑤),关键步骤留人在环(回扣⑨⑩),别信「设个目标走开就行」。整个行业(包括 AutoGPT 自己)都转向了「结构化 + 人在环」,这不是巧合。
2. **把「框架」当成「harness」用——低估了你自己要补的工作量(呼应⑥)。** LangGraph 给你的是**积木**,不是一个开箱即用的 agent。**误区**:以为 `pip install langgraph` 就有了个能干活的 agent。**真相**:你得自己设计状态 schema、画图、配检查点、写工具、扛可靠性——框架只是省了你造轮子,harness 还是你搭的。分不清「框架 vs harness」,预算和排期会大幅低估。
3. **把 Manus 的漂亮数字当成中立事实——它们多为厂商自述。** 「147 万亿 token」「8000 万台虚拟机」「GAIA 上碾压 Deep Research」之类,**大量来自 Manus 自己的发布/交易营销材料,未经独立审计**;GAIA 那组分数是发布期自报、且它未在标准条件下出现在公开榜单上。**避坑**:引用时务必标「厂商自述/发布期数据,截至某快照」,别当作已验证的中立基准(回扣本站「vendor 宣称与同行评审分开」的纪律)。
4. **把 Manus 想象成重型多-agent 系统——那是第三方推断,不是官方描述。** 一些二手文章(含一篇**非-Manus 团队**的 arXiv 论文)描绘了独立的 Planner / Verification Agent。**一手证据指向单-agent 单上下文循环**;唯一确认的多-agent 并行是后加的 Wide Research。**避坑**:分清一手(团队自述)/二手(外部分析)/学术推断,别把外部脑补当成产品内部真相。
5. **各种数字都是快照,不是常量。** Manus 的工具数(27/29)、版本里程碑,LangGraph 的版本与 API(`create_react_agent` 已弃用、2.0 才移除),AutoGPT 的 star 数……都随版本高频变动。**引用时一律带「截至某版本/2026 中快照」,别写死。**

## 6. 一句话 + 对比表

> **这一篇的三家,是三个不同物种:Manus 是你「用」的产品,LangGraph 是你「搭」的框架,AutoGPT 是我们「回顾」的自主起源与教训。** 分清「产品 / 框架 / 起源」这三种「同叫 agent、其实完全不同」的东西,你对 agent 生态的地图就从「一堆编码工具」升级成了「一张有维度的坐标系」。

把三家沿本篇的几条轴拉齐(细节各见其节):

| 维度 | Manus | LangGraph | AutoGPT |
|---|---|---|---|
| **物种** | 通用 agent **产品**(Monica,2025) | 编排**框架/积木**(LangChain) | 自主 agent **起源项目**(2023) |
| **循环范式** | ReAct 式单-agent 循环 + 上下文工程 | 图执行(节点/边/状态,Pregel 超步) | 递归自我派活循环(ReAct 血统) |
| **自主度** | 高(异步跑完交成品),但形态是「你给目标它执行」 | 由你搭图决定;内建 HITL 断点,可高可低 | **名义最高**(设目标走开),实则因无停止条件而失控 |
| **开放性** | 闭源商业产品(消费模型:Claude+Qwen) | 开源 MIT(平台层商业化) | 开源(Classic 及 Forge MIT;platform 目录 Polyform Shield) |

**和下一篇的钩子**:这三个物种里,LangGraph 那种「低层框架」和 Manus 那种「重上下文工程的单循环」,其实预示了 2026 年 harness 演化的下一步——**harness 开始「往上长」,甚至开始驱动别的 harness**。下一篇我们就去看这批新形态。
