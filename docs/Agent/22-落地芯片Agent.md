
如果你同时关心芯片流程和 agent/harness 设计,这两个方向合起来会形成一张完整地图:一边懂芯片流程,一边懂 agent;而这一篇,是它们的交汇点。读完你会有一张清晰的蓝图——不是「照抄就能跑」的代码,而是**一套「该怎么架构这个 agent」的设计主张**,以及对「现在能做到哪、做不到哪」的诚实判断。

## 1. 这个系统到底要做什么,以及它现在处在哪一档

先把野心和现实同时摆上桌。

**野心**:一个 agent,你给它一份产品需求(「做一颗 X 场景、Y 性能、Z 功耗预算的芯片」),它能自己(或在工程师协作下)走完规格 → 架构 → RTL → 验证 → 综合 → 物理 → 签核,最后吐出一份能送晶圆厂的 **GDSII**。

**现实**:必须泼一盆冷水。截至 2026 年中,一篇系统梳理这个领域的综述《The Dawn of Agentic EDA》(arXiv 2512.23189)给了一把清醒的尺子——**芯片设计自主性的 L0–L4 阶梯**:

```
L0/L1  工具辅助的人类设计(手工 & 算法脚本)     ← 传统 EDA
L2     AI 辅助的副驾驶(预测、补全)             ← 大多数「AI EDA」现状
L3     Agentic EDA(agent 编排工具跑流程)        ← 我们现实的目标区
L4     自主设计 4.0(自己开车,近乎无人)         ← 远方
```

综述给了一句必须记住的判断:**至今没有任何 LLM 驱动的 agent,完成过一次「完全无人工、工业级芯片」的流片。**(作为对比,像 AlphaChip 那样的强化学习系统已经在**局部环节**——布局——进了量产,但那不是「LLM agent 端到端」。)

所以,把目标定准:

> **这个案例现实中应该对准的,是一个 L3 的「Agentic EDA」系统——agent 编排 EDA 工具、自动化流程里最成熟的环节、在关键节点把人焊进环里,而不是一个 L4 的「按个按钮就流片」的黑魔法。** 目标是**放大工程师**,不是取代他们。这不是保守,这是㉙篇「从任务倒推、克制优先」原则在一个天价、不可逆领域的必然结论。

## 2. 为什么这件事格外难(它如何决定我们的每一个设计取舍)

芯片设计,是对 agent harness 最严苛的一场考试。它的四个特性,直接决定了后面每一个架构选择——所以先讲透。

- **每一步都贵、慢、且部分不可逆。** 一次真实的流片(mask + 制造)动辄几百万美元、几个月周期,错了几乎无法回头。一次全芯片仿真、一次布局布线,可能占用集群跑几小时到几天。**这意味着:反馈闭环的每一次「试错」都是有实打实成本的**——你不能像修 bug 那样让 agent 无限重试(回扣⑤㉙的「硬性预算上限」,在这里是保命而非省钱)。
- **正确性判据极严。** 软件错了打个补丁;芯片错了可能整批报废。功能验证因此吃掉整个开发周期的约 **70%**(UCAgent 论文数据,⑲)。**这意味着:验证不是一个环节,是贯穿始终的主线**,agent 系统必须把「怎么证明它是对的」放在和「怎么生成它」同等重要的位置。
- **工具链重且专有。** 综合、布局布线、时序签核、DRC/LVS,全靠一整套复杂的商业/开源 EDA 工具(Synopsys/Cadence/Siemens,或 OpenROAD/iverilog/Verilator)。**这意味着:agent 的「手脚」不是通用 shell,而是这些 EDA 工具——怎么把它们接成 agent 能调、能读懂反馈的工具,是核心工程(回扣③⑪)。**
- **HDL 数据稀缺、长程。** 大模型见过的 SystemVerilog(0.84 GB)比 Python(178 GB)少两个数量级(UCAgent 数据,⑲);一个真实项目跨越数周到数月、状态庞大。**这意味着:模型在 HDL 上天然弱(要靠 harness 补),且这是个极端长程任务(上下文与记忆是命门,回扣④⑧)。**

> 一句话:**芯片设计把 agent 的每一个弱点都放大了——它贵到不容试错、严到不容出错、重到工具难接、长到上下文必爆。** 正因如此,它是检验「你到底懂不懂 harness」的终极考场。这一篇后面的每个设计,都是在回应这四条约束。

## 3. 蓝图:怎么把它拆成一个 agent 系统

现在动手画草图。核心思路一句话:**用「阶段分解」当骨架,用「确定性 checker」当护栏,用「真实 EDA 工具的反馈」当闭环,把人焊在关键节点上。** 逐层看。

### 3.1 骨架:按流程阶段分解,一阶段一子-agent + 一道 checker

别想着造一个「无所不能的单体 agent」去端到端硬啃——那是㉙篇批判的过度自主。正确的骨架,是把姊妹站那 13 步(到 GDS 为止取前 10 步)**分解成一串阶段**,每个阶段一个**专职子-agent**(回扣⑩),阶段之间用一道**确定性检查器(checker)**把关:

```
需求 ─▸[规格 agent]─✓─▸[架构 agent]─✓─▸[RTL agent]─✓─▸[验证 agent]─✓─▸
     ─▸[综合 agent]─✓─▸[物理 agent]─✓─▸[签核 agent]─✓─▸ GDS
        每个 ✓ = 一道确定性 checker:这一阶段的产物没过硬性检查,就不许进下一阶段
```

这个「分阶段 + 每阶段确定性 checker」的模式,不是我拍脑袋——它正是 ⑲ 篇 **UCAgent** 的核心招式:把一个长任务切成 31 个细阶段,每阶段用一个**确定性检查器**(不是 LLM 当裁判)卡住,过了才前进。**为什么这招在芯片上如此关键?** 因为它做到了**错误遏制(error containment)**:概率模型会犯错,但只要每一阶段的产物都被一道确定性的硬检查拦住,错误就不会像单体长链那样滚雪球累积到最后一发不可收拾(回扣⑤「85% 每步 → 10 步只剩 20%」那条死穴)。**确定性 checker,是我们对抗芯片流程「不容出错」的主武器。**

### 3.2 手脚:EDA 工具即工具,编译/仿真反馈即闭环

agent 的动作,不是敲通用 shell,而是**调 EDA 工具**:仿真器(Verilator/VCS)、综合(Yosys/DC)、布局布线(OpenROAD/Innovus)、物理验证(DRC/LVS)。怎么把它们接成 agent 的工具?两条路(回扣③⑪):

- **脚本/CLI 封装**:像 ⑲ 的 **ChatEDA** 那样,让 agent 产出 Tcl/shell 脚本去驱动 EDA 工具——这是 EDA 世界的原生语汇。
- **MCP 标准化**:把每个工具封成一个 MCP server,agent 作为 client 按需调用——⑲ 的 **UCAgent** 就是这么做的(它甚至能被外部的 Claude Code / OpenHands 通过 MCP 反过来驱动)。这让整套系统可组合、可替换。

而**闭环的关键,是让反馈「可回填」**(回扣②那条铁律「没有 observation 就没有 agent」)。芯片里最珍贵的 observation,是**工具的真实报告**:编译错误、仿真波形/断言失败、覆盖率数字、时序 slack、DRC 违例。⑲ 的 **AutoChip** 用最小的方式证明了它的威力——一个 LLM + 一个 Icarus Verilog 编译器 + 把编译错误喂回去,光是「加上编译器反馈」这一下,Verilog 准确率就 **+24.20%**。

> 一句话:**在芯片 agent 里,EDA 工具的报告就是模型的「现实回音」。** 你的 harness 值不值钱,很大程度上取决于你把这些又臭又长的工具日志,提炼成模型能读懂、能据以纠错的 observation 的能力——这是 ACI(③)在 EDA 上的硬功夫。

### 3.3 复用现成拼图:把 ⑲ 的真实 agent 拼进对应环节

好消息是:你**不必从零造每个阶段的 agent**。⑲ 篇拆过的那些真实芯片 agent,恰好就是这条流水线各环节的现成拼图:

| 流水线环节 | 可复用/借鉴的真实 agent(⑲) | 招牌机制 |
|---|---|---|
| **RTL 生成** | AutoChip / ChipMATE | 编译器反馈迭代 / 多 agent 无金标交叉验证 |
| **功能验证** | UCAgent | Python-over-SV + 31 阶段 + 确定性 checker + VCLM 可追溯 |
| **全流程编排(RTL→GDS)** | ChatEDA(AutoMage) | NL→规划→Tcl→驱动 EDA 工具 |
| **物理设计参数调优** | ORFS-agent | 在 OpenROAD 里做参数寻优,可调用贝叶斯优化子工具 |
| **软硬件协同/系统级** | QiMeng | 三层:基础模型 + 硬件 agent + 软件 agent |

**你的工作,更多是「集成 + 编排 + 补缺」,而不是「重新发明」。** 这本身就是一个 agent 设计智慧:站在开放生态的肩膀上(回扣⑪「生态是护城河」、⑱ OpenClaw 那种「把别的 agent 当子代理编排」的元层思路)。

### 3.4 上下文与记忆:项目状态巨大,靠文件系统与结构化 artifact

芯片项目的状态是海量的:规格文档、多版 RTL、网表、覆盖率数据库、时序报告、版图……**全塞进上下文窗口是绝无可能的**(回扣④⑧,这里的矛盾比任何软件项目都尖锐)。对策直接抄成熟做法:

- **文件系统即记忆**(⑧,Manus 的招):所有中间产物(RTL、报告、网表)落盘,上下文里只留「指针 + 摘要」,需要时再取。
- **每阶段状态可序列化、可续跑**(⑲,UCAgent 的招):任一阶段崩了或要人介入,能从该阶段续跑而非从头再来——对动辄数周的流程,这是刚需。
- **跨阶段传递用结构化 artifact,不是自然语言全文**:架构 agent 交给 RTL agent 的,应是一份结构化的模块/接口规格,而不是一大段散文——减少歧义、省上下文、便于 checker 校验(呼应 UCAgent 的 FG/FC/CK 可追溯标注)。

### 3.5 权限、安全与人在环:天价、不可逆动作必须有人拍板

这是芯片 agent 相比编码 agent **最不一样、也最不能省**的一层(回扣⑨㉙)。因为这里的「爆炸半径」不是删错文件,是**烧掉几小时集群机时、甚至几百万流片费**。

- **成本护栏是第一位的**:每一次全芯片仿真、每一次 PnR 都是真金白银。harness 必须有**预算与审批闸门**——不是「问不问你」的 UX 问题,是「这次昂贵的 EDA 跑,值不值得、批不批」的经济决策。
- **分级自主,越接近流片越要人签核**:早期探索性的环节(比如 RTL 草稿、参数扫描)可以给 agent 较高自主度;越往下游(签核、流片这种**不可逆**动作),越要收紧到**必须人工签字**。这正是①篇「自主性阶梯」和⑨篇「按风险档分级权限」在芯片上的落地——**流片这个动作,永远不该交给一个概率模型独自拍板。**
- **轨迹级把关,不只看结果**(⑨,HarnessAudit 的洞见):一份最终「看起来对」的网表,可能是 agent 中途绕过了某个约束换来的。签核环节要审的是整条设计轨迹是否合规,而不只是终点产物。

### 3.6 编排与评测:主管 agent 调度,芯片指标当尺子

- **编排**:用一个**主管 agent**(⑩ 的 supervisor 模式)按流水线顺序调度各阶段子-agent,管好阶段间的 artifact 传递、失败重试(带预算上限)、和人在环的介入点。
- **评测**:怎么知道这个 agent 系统好不好?按㉚篇的纪律,**别只看一个数字**。芯片 agent 的「尺子」应该是一组真实工程指标:功能**覆盖率**(像 UCAgent 那样逼近 100%)、**PPA**(性能/功耗/面积)是否达标、**时序**是否收敛、**DRC/LVS** 是否 clean,以及终极问题——**它产出的 GDS,到底能不能真的流片、流出来能不能点亮。** 这些比任何 VerilogEval 分数都更接近「真本事」(而 VerilogEval 一类基准的水分,㉚篇已经警告过)。

> 一句话:**这套蓝图的灵魂,是「分阶段的专职 agent + 每阶段确定性 checker + 真实 EDA 反馈闭环 + 关键节点的人 + 成本护栏」。** 它不炫,但它诚实地对齐了芯片流程「贵、严、重、长」这四条铁律。

## 4. 关键心态:造一个「放大工程师」的 L3 系统,而不是一个 L4 神话

如果这一篇你只带走一段话,请带走这个心态:

> **这个案例追求的不是一个「取代芯片工程师」的自主神明,而是一个「把资深工程师放大十倍」的 L3 协作系统。** 它自动化掉繁重、可验证、迭代密集的环节(RTL 生成、验证、参数寻优),把工程师从体力活里解放出来,让他们专注在判断、架构决策、和关键签核上——人始终在关键环里。

把它拆成几条可操作的信念:

- **从最成熟的环节先自动化,别一上来全流程全自主。** RTL 生成和功能验证是当前 LLM agent 最成熟、反馈最清晰(编译/仿真/覆盖率)的两块——从这里切入,价值最高、风险最低。这是㉙篇「从简单起步、顶不住再加复杂度」的直接应用。
- **确定性 checker + 概率 agent,是黄金组合。** 让 agent 负责「生成与探索」(它的强项),让确定性检查器负责「把关与遏制错误」(概率模型的短板)。别指望模型自己保证正确性(⑲ UCAgent 的核心智慧)。
- **人在环不是妥协,是设计。** 在天价、不可逆的领域,把人焊在关键节点,是**成熟**的表现,不是能力不足。UCAgent 那个 15 万行的 PageTableWalker,正是靠 human-in-the-loop 才在 8 小时内做成的——它诚实地告诉我们现在的边界在哪。
- **harness 比模型更重要,在这里尤其成立(⑥)。** 芯片模型再强,没有一套把 EDA 工具接好、把上下文管好、把成本护住、把错误遏制住的 harness,它什么也交付不了。**真正的工程工作,80% 在造那套 harness。**

## 5. 常被忽视的坑

把「造芯片 agent」这件事上最容易栽的跟头列出来——它们几乎每一条都对应前面某一篇的教训:

1. **追求「一键流片」的 L4 幻想。** 现实是至今无人做到工业级无人流片(§1)。**避坑**:定 L3 目标,人在关键环;从成熟环节切入,别赌全流程全自主(㉙)。
2. **忽视 EDA 的成本护栏,让 agent 随意烧集群/烧流片费。** 一次昂贵的 PnR/仿真/流片错了代价巨大。**避坑**:把预算与审批闸门做成一等公民,昂贵动作要经济决策 + 人签核(⑨§3.5)。
3. **把不可逆动作(流片)交给概率模型独自拍板。** **避坑**:越下游越收紧自主度,签核/流片必须人工签字(①⑨)。
4. **跨阶段上下文爆炸。** 想把整个项目状态塞进上下文,必崩。**避坑**:文件系统即记忆、结构化 artifact 传递、每阶段可序列化续跑(④⑧⑲)。
5. **低估验证。** 验证占约 70% 工作量,却常被当成「RTL 写完顺手做的事」。**避坑**:把验证当贯穿主线,给它独立的、强的 agent + 覆盖率闭环(⑲ UCAgent)。
6. **迷信单点 benchmark(如某个 VerilogEval 分)当成「能造芯片了」。** 基准有水分、且离真实流片很远(㉚⑲)。**避坑**:用覆盖率/PPA/时序收敛/能否真流片当尺子。
7. **让概率模型自己保证正确性,不设确定性 checker。** 芯片不容错误累积。**避坑**:分阶段 + 每阶段确定性检查器做错误遏制(⑲)。
8. **从零重造每个环节,无视开放生态。** **避坑**:复用/编排 ⑲ 的现成拼图(AutoChip/UCAgent/ORFS-agent…),你的价值在集成与补缺(⑪⑱)。

## 6. 结语:两个站,一张地图,一个完整案例
