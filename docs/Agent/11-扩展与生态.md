## 0. 先看全景:前面拆的是内脏,这篇讲「插槽」

⑥ 到 ⑩ 这五篇,我们把 harness 的**内脏**一件件掏出来看过了:harness 到底是什么、循环怎么转、工具怎么派发、越权了怎么拦、活儿怎么分给子代理。那些都是 harness **对内**的机制——它自己怎么把一个任务跑完。

这一篇换个方向,看 harness **对外**的那一面:它给外人预留的**接口和插槽**。

为什么需要这一面?想想一个尴尬的事实:**没有任何一套 harness 能内建所有人想要的工具和规则。** 你们公司有个内部部署系统、一套「谁都不许 `git push --force` 到 main」的铁律、一个只有你们懂的代码规范——这些 Claude Code 的作者不可能预先知道,也不该塞进它的源码里。而你又不能为了加这几样就去 fork 整个 harness、然后永远追不上上游。

于是所有认真的 harness 都长出了一圈**正式的扩展点**:hooks、MCP、skills、slash 命令、插件,外加一套可观测性设施。**它们决定了这个 harness 能不能从「一个好用的工具」长成「一个别人愿意在上面盖楼的生态」。**

```
        ┌─────────────── harness 内核(⑥-⑩ 讲过)───────────────┐
        │   循环  →  工具派发  →  上下文管理  →  权限  →  子代理    │
        └───┬────────┬────────────┬──────────┬─────────┬────────┘
            │        │            │          │         │
         hooks     MCP         skills    slash 命令   插件      ← 这一篇讲的「插槽」
         (钩策略)  (接工具)    (装技能)   (存流程)   (换零件)
            │        │            │          │         │
            └────────┴──────┬─────┴──────────┴─────────┘
                            ▼
                     可观测性(转录 / 成本 / 轨迹)← debug 与评测的地基
```

读完这篇,你会明白**「扩展性」不是锦上添花的配置项,而是一套 harness 长期生命力的护城河**——以及每个扩展点各自会在什么地方咬人。这也是板块二的收尾,读完我们就该去拆真家伙了。

> 全篇继续用那个老场景回扣:**让 agent 去修某仓库里的一个 bug**。只不过这次我们要问的是:怎么在**不改 harness 源码**的前提下,给它加一条「危险命令直接拦」的规则、接一个「查内部 Jira」的工具、教它一套「你们团队专属的提 PR 流程」。

## 1. 「扩展性」到底是什么

一句话:

> **harness 的扩展性 = 它预留的一组正式扩展点,让组织和用户在不 fork 源码的前提下,给 agent 加能力、加策略、加流程、加工具。**

拆开看,「加」的其实是三类不同的东西,分别插在不同的插槽上:

- **加能力(工具)**——让 agent 能碰它原本碰不到的系统:你的 Jira、你的内部 API、你的数据库。这是 **MCP** 的地盘。
- **加策略(规则)**——在 agent 干活的关键节点上插一句「这里得听我的」:危险命令拦下来、每次改完文件跑一遍 lint、收工前必须过一道审计。这是 **hooks** 的地盘。
- **加流程 / 加知识(技能)**——把一套「怎么做某件事」的做法固化下来复用:你们团队专属的发布流程、代码评审清单。这是 **skills** 和 **slash 命令**的地盘。

而 **插件(plugin)**是更大的一层——它把上面这几样打包成一个可安装、可分发的整体,让「扩展」本身也能被别人一键装上。

这里有个容易忽略的分野,先点破:**扩展点是给「不改源码的人」用的**。改源码当然什么都能干,但那条路对绝大多数使用者(包括企业)是死路——你一旦 fork,就把自己钉在了那个版本上。**扩展点的全部意义,就是把「我想改它的行为」这件事,从「fork + 改代码」降级成「写个配置 / 装个包」。** 所以衡量扩展性,看的从来不是 harness 自己有多少功能,而是**别人能在不动它源码的情况下往里塞多少东西**。

## 2. 前提:为什么非有扩展性不可

要理解扩展点为什么是刚需,而不是可选项,只需接受一个前提:

> **需求是长尾的,而 harness 的源码是有限的。** 内建功能永远追不上所有组织的所有规矩和所有内部系统——差距不是「暂时的、迟早补齐」,而是**结构性的、永远存在**。

这带来两个直接后果:

**其一,不可扩展的 harness 会被自己的成功压垮。** 假设 Claude Code 不给你任何钩子,你想加「禁止 `rm -rf`」这条规则,唯一的办法是提 issue、等作者合并、等发版。你们公司几百条内部规矩,每一条都走这条路?不可能。没有扩展点,harness 要么功能贫瘠,要么被 fork 成一地碎片。

**其二,生态才是护城河,而生态只能长在扩展点上。** 一套 harness 真正的壁垒,往往不是它内核多聪明(内核就那么点代码,⑥ 篇讲过「~1.6% 是 AI 逻辑」的社区估算说法),而是**它周围长出了多少别人写的工具、skill、插件**。用的人越多、扩展越多,迁走的成本越高。这就是为什么大厂都在抢着**定标准**(MCP、ACP、agentskills.io):谁的插槽形状成了行业默认,谁就坐在了生态的中心。

对照修 bug 的场景:一个连「接你们内部测试系统」都做不到的 harness,在你们这儿根本跑不起真实任务——它连「反馈闭环」(① 篇讲的 agent 命根子)都接不上。**扩展性是 agent 能不能落进你真实环境的前提,不是奢侈品。**

## 3. 核心内容:五种插槽 + 一套仪表盘

下面把五个扩展点逐个拆开。全程以 Claude Code 为主要样本(它的扩展体系文档最全、最适合教学),旁边用 OpenClaw 和 Hermes 做对照——这两家只点到,深拆留给第 ⑱ 篇《新一代 harness》。

⚠️ 先立一条免责声明,后面所有数字都受它约束:**下面凡是「几种事件」「几种类型」这类计数,都是「截至 2026 年中」的一张快照。** 这些 harness 版本迭代极快(Claude Code 当时在 2.1.x 系列频繁发版),事件数、工具数每隔几版就变。**记住结构和意图,别背数字。**

### 3.1 hooks:把安全 / 质量 / 审计「编码」进 agent

先给直觉。agent 的循环(⑦ 篇)就像一条流水线,一圈一圈转:组装上下文 → 调模型 → 要用工具了 → 执行 → 收结果 → 再转。**hook(生命周期钩子)就是你在这条流水线的特定工位上,预先埋下的一段自己的代码**:「每当流水线走到『要执行工具』这一步,先停一下,让我这段代码看一眼,我说行才行。」

这个「工位」在术语里叫**生命周期事件(lifecycle event)**。以 Claude Code 为例,截至 2026 年中,它的官方文档列出**约 31 种**生命周期事件(那篇拆解它源码的 arXiv 论文当时数到 27 种——**这个集合是随版本增长的**,别当定数)。最常被用到的几个:

| 事件 | 触发时机 | 典型用途 |
|---|---|---|
| `UserPromptSubmit` | 你刚敲下 prompt、还没送进模型 | 注入额外上下文、或直接拦掉不该问的 |
| `PreToolUse` | 模型说「我要用某工具」、但还没真执行 | **拦危险命令**、改写参数、要人二次确认 |
| `PostToolUse` | 工具执行完、结果还没回给模型 | 跑 lint / 格式化、审计日志、质量门 |
| `Stop` | 模型判断「我干完了」、准备收工 | **否决收工**(比如「测试没绿不许停」) |

每个事件上,你可以挂**5 种实现类型**的钩子(截至 2026 中):`command`(跑个 shell 脚本/程序)、`http`(打个 HTTP 请求给你的服务)、`mcp_tool`(调一个 MCP 工具)、`prompt`(塞一段提示给模型)、`agent`(干脆派一个子代理去做质检)。从「一行 shell」到「一个会思考的验证 agent」,粒度任你选。

**为什么这件事这么重要?** 因为它是**组织把自己的安全、质量、审计规则「编码」进 agent 的地方**。回到修 bug 的场景:

- 在 `PreToolUse` 上挂一段脚本,匹配到 `rm -rf`、`git push --force`、`DROP TABLE` 就返回「阻断」——`agent` 再怎么被绕、被注入,也过不了这一关(注意:在 Claude Code 里,`PreToolUse` 钩子靠**退出码 2** 来阻断工具执行,这个约定 ⑨ 篇的 OpenDev 五层安全架构里也见过)。
- 在 `PostToolUse` 上挂 lint:每次 agent 一改完文件,自动跑一遍格式检查,不合规就打回去让它重改——**质量门变成了自动的**。
- 在 `Stop` 上挂一道检查:测试没全绿就否决收工,逼它接着干。

```
   模型: "我要执行 rm -rf /tmp/build && deploy"
                 │
                 ▼
        ┌──── PreToolUse 钩子 ────┐
        │  匹配 DANGEROUS 模式?    │
        │    ├─ 是 → 退出码 2 ✋阻断 │──→ 工具根本不执行,理由回给模型
        │    └─ 否 → 放行           │
        └──────────┬──────────────┘
                   ▼
              真正执行工具
                   │
        ┌──── PostToolUse 钩子 ───┐
        │  跑 lint / 写审计日志     │
        └──────────┬──────────────┘
                   ▼
              结果回给模型
```

这就是 hooks 的杀手锏:**它是确定性的、代码级的,不依赖模型「愿不愿意听话」。** 你不是在 prompt 里客气地写「请不要删库」(模型可能被注入骗着删),而是在流水线上焊死了一道闸——模型的意图必须**物理地**通过这道闸才能落地。

> 一句话:hooks 是 harness 里**「人写死的确定性规则」和「模型的概率性决策」交汇的那道闸门**——组织的安全、质量、审计,都焊在这里。

### 3.2 MCP:让工具生态从 N×M 变成 N+M

MCP(模型上下文协议,Model Context Protocol)这个概念,③ 篇讲工具时已经介绍过——它是一个**把「工具/数据源」标准化的开放协议**。这里只补 harness 这一侧的事:**harness 作为 MCP 客户端(client),去连接外部的 MCP 服务器(server),从而把 server 暴露的工具接进自己的工具箱。**

先说清它解决的痛点,一个乘法变加法的故事。假设有 N 个 harness(Claude Code、Codex、OpenClaw……)和 M 个工具/数据源(Jira、GitHub、Postgres、你的内部系统……)。没有统一协议时,想让**每个 harness 都能用每个工具**,你得写 **N×M** 个专用适配器——每加一个新工具,得给所有 harness 各写一遍。这是集成地狱。

MCP 把这层拍平:每个工具只需实现**一次** MCP server,每个 harness 只需实现**一次** MCP client,大家就都通了。**N×M 塌缩成 N+M。**

```
     没有 MCP:每对都要专门适配(N×M)      有 MCP:各连一次协议(N+M)

     Claude Code ─┬─ Jira                Claude Code ─┐
                  ├─ GitHub                Codex ──────┤
     Codex ───────┼─ Postgres             OpenClaw ───┤     ┌─ Jira
                  ├─ 内部系统                          ├─ MCP ─┼─ GitHub
     OpenClaw ────┴─ ...                              │  协议 ├─ Postgres
        每加一个工具 × 每个 harness         每个只连一次 ┘     └─ ...
```

**这正是「生态」二字的技术底座:** 因为有 MCP,你为修 bug 场景写的那个「查内部 Jira」的 server,今天给 Claude Code 用,明天换 Codex 照样能用——工具的价值不再被锁死在单一 harness 里。这也是为什么大家抢着让自己的协议成为默认。

但 harness 侧有个非接不可的现实问题,前面几篇反复强调过:**上下文窗口是最稀缺的资源(④ 篇)。** 每个 MCP 工具的定义(名字、描述、参数 schema)都要占 token,你要是把连上的几十上百个工具定义**全部塞进上下文**,还没干活就把窗口撑掉大半——模型光是「读工具说明书」就累趴了。

Claude Code 的解法很典型,叫**延迟加载(lazy loading)**:MCP 工具的定义**默认不进上下文**,而是**按需**用一个叫 `ToolSearch` 的元工具去检索——「我这一步好像需要查数据库,搜一下有没有相关工具」,搜到了再把那几个的定义拉进来用。(注意:这里的 `ToolSearch` 名字对你可能眼熟——本课这些 subagent 环境里就在用它按需取工具,是同一个思路的实现。)这样,baseline 的上下文开销跟你「装了多少工具」几乎脱钩。

> 一句话:MCP 让工具生态从 **N×M 变 N+M**——但工具越多,越要靠**延迟加载 / 按需检索**保护上下文,否则「能力越强」直接变成「越容易撑爆窗口」。

### 3.3 skills:可复用、可移植、甚至能自己长出来的「技能包」

hooks 加的是规则,MCP 加的是工具,**skill(技能包)加的是「做某件事的成套做法」**——它不是代码,更像**一份写给 agent 看的、结构化的操作手册**:遇到「发布一个版本」这类任务时,该按什么步骤、守什么规矩、用哪些工具。

先看两家的形态,一静一动:

**Claude Code 的 Skill / OpenClaw 的 `SKILL.md`——「装进去的静态技能」。** 你(或团队)预先写好一个技能包,agent 在需要时把它加载进上下文,照着做。OpenClaw 把 skill 明确定义为**指令包而非代码**(workflow、rubric、约束的自然语言描述),好处是**非程序员也能用中文/英文扩展 agent 的行为**,代价是不如代码工具精确。这里也藏着一个上下文技巧:skill 的**描述**常驻(让模型知道「有这么个技能」),**完整内容按需展开**——又是「延迟加载」那一套。

**Hermes(Nous Research)的自进化 skills——「自己长出来的动态技能」。** 这是 Hermes 最亮的一个设计,也是本篇唯一值得多说两句的对照点(深拆仍留第 ⑱ 篇)。Hermes 的**闭环学习回路(closed learning loop)**是这样转的:

```
  agent 干完一个非平凡的任务(比如踩了坑、被用户纠正、绕了 5 步以上)
                    │
                    ▼
      自动从这条「成功轨迹」里提炼出一个 skill,写成 Markdown 存盘
                    │
                    ▼
      下次遇到类似任务 → 分级取用(先看目录,再看正文,再看引用文件)
                    │
                    ▼
      用的过程中发现不对 → 就地打补丁改进这个 skill(patch)
                    └──────────────→(回到顶上,越用越顺)
```

用修 bug 的场景说:Hermes 头一回摸清「你们这个仓库的测试得先 `make setup` 再 `pytest -x`」,它会**自动把这段经验写成一个 skill**;下次再来修 bug,直接调出来照做,不用重新摸索。取用时同样是**分级披露(progressive disclosure)**——先只加载技能目录的元数据(省 token),真要用某个才展开正文。

更关键的一点:Hermes 的 skill 兼容一个**开放的 `agentskills.io` 标准**,意味着技能**跨 harness 可移植**——理论上你在一个 harness 里攒的技能,能搬到另一个支持该标准的 harness 上用。(⚠️ 据其官方资料,截至 2026 中,这个标准由谁创建 / 治理尚未完全确认;当作「Hermes 采用的一个开放标准」理解即可,别把它当成某家的私产或行业既定法律。)

> 一句话:skill 是把「怎么做一件事」从一次性的即兴发挥,变成**可复用、可分享、甚至能被 agent 自己写出来并持续改进**的资产——它让 agent 的经验能沉淀,而不是每次从零开始。

### 3.4 slash 命令 / 插件 / 元编排:从「存个流程」到「把别的 harness 当零件」

前面三种插槽解决「加规则、加工具、加技能」。这一节的三样,解决的是**「打包、分发、和更大的编排」**。

**slash 命令(自定义命令)——把一段常用流程存成一个词。** 最轻量的扩展:你把「修 bug 的标准套路」写成一个 `/fix-issue` 命令,以后一敲这个词,就等于把那一大段 prompt/流程展开出来。它本质是**给高频工作流起个短名**,和 skill 有点像,但更偏「即时触发的动作」。(你在本课环境里看到的那些 `/deep-research`、`/code-review`,就是这类东西。)

**插件(plugin)——把上面这一切打包成可安装的整体。** 单独的 hook、MCP server、skill、slash 命令都是零件;**插件是把它们装进一个盒子、让别人一键安装的分发单位**。OpenClaw 在这条路上走得最远:它提供了一套 **`AgentHarness` 插件 SDK**——第三方写一个插件,实现规定的接口(据其文档,核心是 `supports(ctx)` 判断「这个任务我接不接」和 `runAttempt(params)` 真正跑一轮),就能把自己的 harness 逻辑**插进 OpenClaw**,而不必改 OpenClaw 的核。这就是「让别人在你上面盖楼」的极致形态。

**元编排——把整个外部 harness 当子代理来驱动。** 这是 2026 年最有意思的一个模式,也是 OpenClaw 的招牌思路。前面说的都是「往一个 harness 里塞扩展」,而 OpenClaw 反过来问:**能不能把 Claude Code、Codex、Gemini CLI、OpenCode 这些「别人家的完整 harness」,当成我的子代理,统一调度?**

它靠的是一个叫 **ACP(代理客户端协议,Agent Client Protocol)**的开放标准——⚠️ 讲清出处很重要:**ACP 不是 OpenClaw 发明的,它是 Zed 公司搞的独立标准(2025 年 8 月发布,基于 JSON-RPC 2.0),定位是「AI 编码 agent 界的 LSP」。** OpenClaw 只是通过一个插件(`@openclaw/acpx`)**消费**这个标准,从而能把上面那些外部 harness 拉起来、当作可驱动的子代理。

```
        ┌──────────── OpenClaw(元编排 / 网关)────────────┐
        │      统一调度 · 多渠道送达 · 会话持久化           │
        └───────┬──────────┬───────────┬────────┬────────┘
                │ ACP      │ ACP       │ ACP    │ ACP  (Zed 的开放协议)
                ▼          ▼           ▼        ▼
          Claude Code    Codex    Gemini CLI  OpenCode   ← 每个都是完整 harness
          (当子代理)    (当子代理)  (当子代理)  (当子代理)     被当「零件」驱动
```

这就引出一个层级上的洞察,和 MCP 正好配成一对:**MCP 是「agent ↔ 工具」的协议,ACP 是「编辑器/编排器 ↔ agent」的协议**——一个往下接工具,一个往上接「把 agent 当被驱动的对象」。这套「用 harness 编排 harness」的元编排思路,后面拆 OpenClaw 时会专门深挖,这里点到为止。

**扩展点是有层级的**:slash 命令存一段流程,插件打包一整套扩展,而 ACP 这类协议把**整个 harness 变成可被编排的一块积木**——生态就是这样一层套一层长起来的。

### 3.5 可观测性:没有仪表盘,agent 就是个黑盒

前面四节都在「往 agent 里加东西」。这一节反过来:**怎么看清 agent 到底干了什么。** 这就是**可观测性(observability)**——转录、日志、成本与轨迹追踪。

它听起来不像「扩展点」,但它是**所有其他扩展点的前提**,也是后面评测(㉚ 篇)的地基。理由很简单:agent 是个**概率机器**(① 篇的关键心态),同一个任务跑两次路径都可能不同。当它跑偏了、烧了一堆钱、或者干出了不该干的事,你**唯一能复盘的依据,就是它留下的痕迹**。没有这些痕迹,agent 就是个你没法 debug、没法评测、没法信任的黑盒。

可观测性至少包含三样东西(⑦、⑩ 篇零散见过,这里归拢):

- **转录 / 日志(transcript)**——完整记下每一轮:模型想了什么、调了哪个工具、传了什么参数、拿回什么结果。Claude Code 把会话写成 `~/.claude/projects/` 下的 **append-only(只追加、从不改写)JSONL 文件**;Hermes、OpenClaw 则把它落进 **SQLite**(还带全文检索,能跨会话搜「我上次是怎么修这个的」)。append-only 是关键:历史不可篡改,才谈得上审计和精确复盘。
- **成本 / token 追踪**——每一步花了多少 token、多少钱。长任务和多子代理(⑩ 篇)最容易悄悄烧钱,没有成本表你根本不知道钱漏在哪。
- **轨迹追踪(trace)**——尤其是有子代理时,得能看清「主代理派了谁、那个子代理又干了什么」的完整调用树,否则多代理系统就是一团理不清的乱麻。

对照修 bug:agent 提了个 PR,但改动很奇怪。有转录,你能倒回去看清「它在第 7 步误读了某个测试的报错,从此走岔了」;没有转录,你只能对着一个莫名其妙的 diff 干瞪眼,连它**为什么**这么改都无从查起。

> 一句话:可观测性不是运维的奢侈品,而是**让概率性的 agent 变得可 debug、可审计、可评测**的地基——没有它,前面所有扩展点出了问题你都排不了障。

## 4. 关键心态:扩展点的设计,就是 harness 的护城河

把这五种插槽 + 仪表盘连起来看,你会得到本篇最重要的一个判断:

> **一套 harness 的长期生命力,取决于它的扩展点设计得好不好。内核可以被复刻,生态不能。**

这话背后有个反直觉的点,值得停下来想:**你可能以为,harness 之间竞争拼的是内核——谁的循环更聪明、谁的上下文管理更精。其实到了一定程度,内核会趋同(就那么点核心逻辑),真正拉开差距、锁住用户的,是它周围长出来的生态。**

- 谁的 hook 事件更细、更全,组织就越能把自己的规矩焊进去,**迁走的成本就越高**。
- 谁的 MCP / skill / 插件生态更繁荣,别人写的工具和技能就越多,**新用户一来就有一堆现成的东西可用**。
- 谁的协议(MCP、ACP、agentskills.io)成了行业默认,谁就**坐在了整个生态的收费站上**——这正是 2026 年各家抢着「定标准、开源协议」的深层动机:开源的是协议,圈住的是生态。

但护城河也有反面。扩展点是一把双刃剑:**它把权力交给了外人。** 你让别人不改源码就能往 agent 里塞规则、塞工具、塞代码——那「别人」里也包括写得烂的、不维护的、甚至有恶意的。所以扩展点设计的真功夫,不在「能扩展多少」,而在**「怎么在开放的同时守住安全和可控」**。下一节的坑,基本都长在这条张力上。

> 一句话:**内核决定 harness 好不好用,扩展点决定它能不能长久、能不能长成生态。** 前者是产品力,后者才是护城河。

## 5. 常被忽视的坑(扩展点最容易反噬的几处)

扩展点的每一个好处,反过来都是一个坑。逐条过一遍:

1. **hooks 写不好,反成安全漏洞。** hook 是你亲手往 agent 循环里插的一段**会被执行**的代码——它跑在你的机器上、你的权限下。一个从不可信来源装来的 hook,等于给了对方在你环境里执行任意代码的机会。而且 `PreToolUse` 拦截逻辑很难写全:命令行有一万种绕法(拼接、变量替换、shell 续行、命令别名……),你以为拦住了 `rm -rf`,它换个写法就溜过去了。**避:hook 逻辑当安全边界来写(白名单优于黑名单)、只从可信来源装、并给 hook 本身也上审计。**

2. **MCP 工具一股脑加载,撑爆上下文。** 兴冲冲接了 20 个 MCP server、几百个工具,结果每个工具定义都占 token,agent 还没开始干活,上下文就被「工具说明书」吃掉一大半——模型在几百个工具里挑花了眼,反而更容易调错、幻觉出不存在的工具。**避:默认延迟加载 / 按需检索(见 3.2 的 `ToolSearch`),只在需要时把相关工具拉进上下文,而不是开局全塞。**

3. **skills 无版本、无隔离,变成定时炸弹。** 一个到处传的 `SKILL.md` 里如果藏了坏指令(比如「顺手把某密钥发到这个地址」),而你的 harness 对 skill 不做隔离、不记版本、不校验来源,它就成了一个**社会工程学的投毒入口**。skill 越像「自然语言写的,谁都能改」,这个风险越高。**避:skill 要有来源标记和版本、可信 skill 与随手装的分层、对注入内容做扫描——把它当代码依赖一样管理,而不是当便签。**

4. **装一堆没人维护的插件,把 harness 拖垮。** 插件的开放性是双刃剑:装得越多,你的 agent 就越是一个由几十个陌生人代码拼起来的缝合怪。其中随便一个失修、有 bug、或与新版本不兼容,都能拖慢启动、引入崩溃、甚至打开安全缺口。OpenClaw 明确承认它的插件是**在进程内、故意不沙箱**运行的——「一个恶意的原生插件等同于在 OpenClaw 进程里任意执行代码」;一篇针对它的公开安全分析(据公开资料,截至 2026 中)报告了数百条安全提示(advisory),包括恶意 `SKILL.md` 投毒和执行白名单绕过。**避:把每个插件当一个信任决策来做,少即是多,定期清理没人维护的。**

5. **没有可观测性,agent 变黑盒,出事无从排障。** 这是最沉默、也最致命的一个坑——因为它平时不疼。直到某天 agent 干出了一件蠢事 / 危险事,你想复盘,却发现**什么痕迹都没留**:没转录、没轨迹、没成本记录。于是你既不知道它为什么这么干,也没法评测它到底靠不靠谱(㉚ 篇的评测,喂的就是这些痕迹)。**避:从第一天就把转录、成本、轨迹追踪打开,append-only 存好——它平时是保险,出事时是唯一的证据。**

## 6. 小结 & 收官板块二

这一篇,也是「Agent Harness」板块的收尾,你需要带走的:

- **harness 的扩展性 = 一组正式扩展点,让人不 fork 源码就能加能力、加策略、加流程。** 它是刚需不是奢侈品——需求是长尾的,内建功能永远追不上。
- **五种插槽各管一摊:** hooks 加**规则**(把安全/质量/审计焊进循环的关键节点)、MCP 加**工具**(N×M→N+M 的开放协议,靠延迟加载护上下文)、skills 加**技能**(可复用、可移植,Hermes 甚至能让 agent 自己长出来)、slash/插件加**打包与分发**、ACP 这类协议实现**把整个 harness 当积木**的元编排。
- **外加一套可观测性仪表盘**(转录 / 成本 / 轨迹),它是 debug、审计、评测的共同地基——没有它,概率性的 agent 就是个排不了障的黑盒。
- **核心心态:内核会趋同,生态才是护城河。** 扩展点设计得好不好,决定一套 harness 能不能长久;而每个扩展点的开放性,反面都是一个安全/上下文/维护的坑。

到这里,板块二(⑥–⑪)讲完了:我们从「什么是 harness」一路拆到了它的循环、工具、权限、沙箱、子代理,再到今天这层对外的扩展生态。**概念的骨架已经搭全——你现在手里有了一整套「看 harness 的透视镜」。**

下面,该拆真家伙了。

从 ⑫ 篇起,我们进入板块三:**把这些抽象组件,一个个放进真实产品里逐个拆解**——看 Claude Code、Codex、Cursor、Devin、OpenClaw、Hermes 这些活生生的 harness,各自是怎么把「循环 + 工具 + 上下文 + 权限 + 子代理 + 扩展」这套零件,拼成一个能干活的整体的。你会发现:同样的零件,不同的拼法,长出的是气质完全不同的 agent。
