## 0. 全景:agent 从终端搬进了你的编辑器

前两篇(⑫ Claude Code、⑬ Codex CLI)拆的都是**终端类 agent**——它们活在命令行里,你敲一句话,它在一个黑底白字的窗口里读文件、跑 shell、改代码。这一篇,我们换一个大流派:**编辑器类 agent(editor-native agent)——agent 不再住在终端,而是长在你的 IDE 里,盯着你的光标、你打开的文件、你整个代码库。**

这两派的分野,一句话就能抓住:

> **终端类 agent 是「你走进它的世界」(打开一个终端会话去指挥它);编辑器类 agent 是「它住进你的世界」(它就在你天天写代码的编辑器里,和你的光标、diff、语法高亮共生)。**

这个「住进来」不是表面装修。它带来一整套终端类给不了的东西:**实时的补全**(你打字它就在灰字里预测下一段)、**内联的 diff 审阅**(改动直接标在你正看的那行代码旁边,绿加红删)、以及**对「你此刻在看什么」的天然感知**。代价也很实在——它更黏在 IDE 内,长程自主和「发完就走」这类形态反而没有终端类顺手。

这一派里走得最靠前的样板,是 **Cursor**。拆它的同时,我们会横向带出另两个编辑器类代表:**GitHub Copilot**(从补全一路长成 agent 的元老)和 **Cline**(开源、可审计、Plan/Act 双模式的另一种活法)。三者会在 §6 用一张表对齐。

```
       终端类 agent(⑫⑬)              编辑器类 agent(本篇)
    ┌──────────────────────┐      ┌──────────────────────────┐
    │  你 → 打开终端会话     │      │  agent 住进你的 IDE       │
    │  盯着 stdout 流       │  vs  │  盯着你的光标/打开的文件   │
    │  长程自主、可无头运行  │      │  实时补全 + 内联 diff 审阅 │
    │  Claude Code / Codex  │      │  Cursor / Copilot / Cline │
    └──────────────────────┘      └──────────────────────────┘
```

> 读这一篇的正确姿势:**别只看「Cursor 有哪些按钮」,要看它把⑥–⑪那张组件地图,搬进「编辑器」这个约束里之后,哪些部件被重新设计了。** 你会发现「编辑器原生」这一个约束,逼出了好几个终端类根本不需要的组件(索引管线、apply 模型、shadow workspace)。

## 1. Cursor 是什么、谁做的、现在什么状态

一句话:

> **Cursor 是一个「AI 原生」的代码编辑器——它不是给 VS Code 装个 AI 插件,而是直接把 VS Code 的开源代码 fork 出来、把 AI 焊死在编辑器内核里,让补全、检索、多文件自主改动都是「原生功能」而非外挂。**

几个关键事实(据 Cursor 官方博客/更新日志与公开报道,截至 2026 年中):

- **谁做的**:**Anysphere, Inc.**(现在对外就叫 Cursor),2022 年由四位 MIT 学生创办(Michael Truell 任 CEO),总部在旧金山。(联创之一 Arvid Lunnemark 已于 2025 年 10 月离开、另起一家 AI 安全实验室。)
- **商业状态**:增长极猛——2025 年 11 月的 D 轮融资时宣布年经常性收入(ARR)突破 10 亿美元,到 2026 年初约 30 亿。**截至 2026 年中的一个大新闻:SpaceX 在 2026 年 6 月 16 日宣布以约 600 亿美元全股票收购 Cursor,预计 2026 年三季度完成——但截至本篇写作时,这笔交易尚待监管批准、还未正式交割。** 引用时务必带这个「已宣布、未完成」的限定。
- **它是个「分叉的 IDE」**:基于开源的 VS Code 代码库 fork 而来。这一条是理解它一切设计的地基——因为 fork 了整个编辑器,它才能直接钩住渲染层、文件系统、扩展宿主,做出插件做不到的事(见 §4)。
- **跑在什么模型上**:模型可选——你能挂 Claude、GPT、Gemini,也能用 Cursor 自研的 **Composer** 系列(Composer 1 于 2025 年 10 月 29 日发布;Composer 2 于 2026 年 3 月 19 日发布,以 Moonshot AI 开源的 Kimi K2.5 为底座续训 + 强化学习而来)。

**别把它和「VS Code + Copilot 插件」搞混**——这正是 Cursor 反复强调的分野。插件只能在 VS Code 给的官方接口里施展;Cursor 因为 fork 了整个编辑器,能做**语义索引、专用 apply 模型、隐藏窗口里跑 LSP 的反馈回路**这些插件够不着的东西。用⑥篇的话说:**Copilot 给你「IDE 里的一个 AI 面板」,Cursor 给你「一个 AI 就是内核的 IDE」。**

## 2. 一次任务在它体内怎么流动

拆组件之前,先看一遍**血液怎么流**。设想你在 Cursor 里,想做一个跨文件的改动,在 Agent 面板里敲下:

> 「把认证逻辑从 session cookie 改成 JWT,记得把用到的地方都改了。」

Cursor 内部大致这样走一圈(对照②篇的「单步」和④篇的「上下文与检索」):

```
你在 Agent 面板里的 prompt(可 @ 指定文件/文件夹/符号)
   │
   ▼
① 组装上下文   你的话 + 常驻规则(.cursor/rules) + 当前打开的文件
              + 会话历史 + @ 进来的东西 + 代码库检索命中的片段
              └─ 按「优先级」打包进 token 预算(高优先留、低优先超预算就丢)
   │
   ▼
② 调模型       整段喂给你选的 LLM(Claude / GPT / Gemini / Composer)
              → 吐出「计划 + 一串工具调用」
   │
   ▼
③ 工具循环     codebase_search(对着索引做语义检索)找相关代码
              → grep_search(精确匹配)→ read_file → run_terminal_cmd …
              └─ 可派一个 Explore 子代理,在自己的短上下文里并行地广搜
   │
   ▼
④ apply 改动   模型对每处改动只产出「意图/差异」,交给专门的 apply 模型
              → 由它把这个意图精确、快速地「套」成真正的文件重写
   │
   ▼
⑤ 校验         在隐藏的 shadow workspace 里跑语言服务器(LSP)
              → 把 lint / 类型错误喂回给 agent → agent 据此再改
   │
   ▼
⑥ 你审阅       改动以「内联 diff」呈现(绿加红删,就标在代码旁)
              → 你逐处 accept / reject;checkpoint 让你能一键回滚
```

三个值得当场记住的细节:

- **「@」是你手动喂上下文的方式**:你可以 `@` 一个文件、一个文件夹、一个函数,把「你确定相关」的东西钉进上下文——这是**人工检索**;而 `codebase_search` 是 agent 的**自动检索**(见 3.2)。两条路并存,正是编辑器类 agent 的典型上下文来源。
- **「改动怎么写进文件」被单独外包了**:注意第 ④ 步——大模型不直接逐字重写文件,它只说「我要改成什么样」,由一个**专门的 apply 模型**去精确落地(见 3.4)。这是 Cursor 最有教学价值的一刀切分。
- **你是最后一道闸**:改动默认以 diff 摆在你面前等你点头(对照⑨篇的权限)。**人在环、逐处审阅,是编辑器类 agent 的默认姿态**——它把「审批」做成了你最熟悉的那个 UX:看 diff。

> 一句话:**Cursor 的一次任务,是「检索先把相关代码喂饱上下文 → 前沿模型想清楚改什么 → 专用 apply 模型负责精确落地 → 隐藏窗口跑 LSP 验一遍 → 最后以 diff 交你审」。** 它把④篇的检索、⑨篇的审批,都翻译成了编辑器里看得见摸得着的动作。

## 3. 拆开看它的组件

现在对着⑥篇的组件地图,逐个部件拆。你会看到很多组件是「因为要长在编辑器里」才特意造出来的。

### 3.1 Tab 自动补全:它的看家 UX

先拆最不像 agent、却是 Cursor 起家立命的那一件——**Tab 补全**。你打字,它在灰字里预测你接下来要写的(不只是一行,而是多行、甚至跨文件的连锁改动),你按 Tab 就接受。

它的看点不在「会补全」——各家都会——而在**背后是一个用强化学习(RL)训出来的「要不要弹」的策略**。据 Cursor 自己的博客(截至 2026 年中):模型对「弹了被接受」给 **+0.75** 奖励、「弹了被拒」给 **−0.25**、「没弹」给 0;于是模型学会了**只有当它预测「被接受概率超过 25%」时才把建议露出来**。结果是**建议数少了约 21%、接受率却高了约 28%**——弹得更少,但弹的更准。这套策略靠每天超 **4 亿次真实请求**的信号持续更新。

> 反直觉的点:**补全体验变好,不是靠「多补几个建议」,而是靠「学会闭嘴」。** 一个总在你眼前闪、七成要被你拒掉的补全,是噪音不是助手。Cursor 把「该不该打扰你」这件事,直接做成了模型要优化的目标。

### 3.2 代码库索引与检索:把「相关代码」自动喂进上下文

这是编辑器类 agent 相对终端类最核心的一件武器,也是④篇 **RAG(检索增强,retrieval-augmented generation)** 概念最干净的落地样板。

问题很实在:你的代码库有几十万行,模型的上下文窗口再大也塞不下(见④⑧)。怎么让 agent「只看它该看的那几段」?Cursor 的答案是**索引 + 向量检索**:

- **建索引**:把代码库切成一块块**语法片段(syntactic chunk)**,给每块算一个 **embedding(向量)**,存进向量数据库。为了增量更新(你改了几个文件,不必全库重算),它用一棵 **SHA-256 哈希的 Merkle 树**来 diff——只有内容真变了的文件才重新切块、重算向量。
- **隐私姿态**:据其官方「安全索引」博客,**文件路径是加密的,源代码本身不落在服务器上**——服务端存的是向量和混淆过的元数据。
- **用索引**:agent 需要「找相关代码」时,发一个 `codebase_search`(**语义**检索,按意思找,而不只按关键词),把命中的片段喂回上下文;要精确匹配时再用 `grep_search`。

这就是 RAG 的教科书形态:**不把整个代码库倒进窗口,而是按需检索、只喂相关片段**。它和终端类 agent「用 Grep/Glob 现场翻文件」的思路互补——一个靠预建索引的语义相似度,一个靠即时的字符串匹配。

> 一句话:**代码库索引 = 给你的整个仓库配了一本「语义目录」;检索 = agent 每次只翻到它需要的那几页。** 这是编辑器类 agent 能「盯着整个代码库干活」而不撑爆上下文的底层功夫。

### 3.3 Agent / Composer 模式:多文件自主改动

Tab 是「你主导、它补一笔」;**Agent / Composer 模式**则反过来——**你给目标,它自主地跨多个文件改**。这才是 Cursor 里真正意义上的「agent loop」(对照②⑦篇):它自己决定先读哪个文件、搜什么、改哪几处、跑不跑测试,一圈圈 gather→act→verify,直到把你那个「改成 JWT」的模糊目标磨成几十处具体改动。

配套的还有一个 **Explore 子代理**:当 agent 需要「广撒网地搜一大片代码」时,可以派它在**自己独立的短上下文**里并行地搜,只把结果拎回来——这正是⑩篇「子代理 = 上下文隔离」在编辑器里的复用。而 `Composer` 既是这个 agent 模式的名字,也是 Cursor 自研模型系列的名字(Composer 1 / 2),两者容易混,记住:**Composer 首先指「那个多文件自主改动的 agent 模式」**,自研模型只是它可选的引擎之一。

### 3.4 apply 模型:为什么「把 diff 套进文件」值得单拆一个模型

这是全篇最该停下来细品的设计。你可能以为:大模型既然能想出「这行改成 JWT」,让它直接把整个文件重写一遍不就完了?**Cursor 偏不。** 它把一次改动拆成两个模型分工:

- **前沿模型(想「改什么」)**:Claude / GPT / Composer 这类聪明但**慢且贵**的大模型,负责推理出「意图」——这段逻辑该怎么变。
- **apply 模型(干「怎么精确写进文件」)**:一个**专门的、更小更快**的模型,负责把上面那个「意图/差异」精确无误地套进真实文件。据 Cursor 与 Fireworks 的博客(这套数字有据可查于 2024–25 年,到 2026 年中是否仍是同一实现未获官方确认,当快照看):它曾是一个在 Fireworks 上微调的 Llama-3-70B,用一种叫「speculative edits(投机式编辑)」的算法——**把文件里没改动的原文当作「草稿 token」直接复用,只在真正要改的地方才生成**——从而做到约每秒 1000 token、比朴素重写快十几倍。

**为什么值得单拆?** 因为「想清楚改什么」和「一字不差地写进文件」是两种截然不同的活:前者要智商(贵),后者要**精确 + 速度**(不能改错一个括号,还得快到你没感觉在等)。让一个又贵又慢的前沿模型去逐字重写整个大文件,是拿高射炮打蚊子——既烧钱,又慢,还容易在无关的行上手滑。**把「落地」独立成一个便宜、快、专精的模型,是延迟/成本/准确性的一次干净取舍。**

> 一句话:**Cursor 把「一次代码修改」拆成了「决策」和「执行」两个模型——聪明的负责想,快准的负责写。** 这是整个 agent 系统设计里「用专用小模型接住通用大模型的脏活」最漂亮的一课。

### 3.5 规则:`.cursor/rules`(项目级的行为约束)

对照⑫篇的 `CLAUDE.md` / ⑬篇的 `AGENTS.md`,Cursor 也有它的「项目宪法」:**`.cursor/rules`**(规则文件,后缀 `.mdc`)。你把「这个项目用 pnpm 不用 npm」「所有 API 返回都要走这个错误封装」这类**项目级约束**写进去,agent 干活时就会遵守。它也认 `AGENTS.md` 这类跨工具的约定。

有个和上下文工程相关的细节值得点出:据 Cursor 的设计,规则不是**每次都把全部规则统统塞进上下文**,而是让 agent 通过一个 `fetch_rules` 工具**按需拉取相关规则**——避免规则一多就把上下文撑爆。这和 3.2 的检索、⑧篇「上下文是稀缺预算」是同一个思路:**能按需加载的,就别常驻。**

### 3.6 background / 后台 agent:把长任务异步跑起来

前面几件都是「你盯着它、实时审」的同步形态。但 Cursor 也长出了**异步的一面**:**background / cloud agent**——把整个 agent loop 搬到一台**隔离的云端 VM** 里跑,它在那边独立地跑终端命令、浏览、改代码,你则从 IDE、网页、Slack 或手机上**监控**进度,干完再把结果拉回本地。

顺着这条线,Cursor 2.0(2025 年 10 月 29 日)加了**并行多 agent** 的界面(用 git worktree 或远程 VM 同时跑多个 agent);Cursor 3(2026 年 4 月 2 日)进一步推出「Agents Window」,把「编排一队 agent、在本地与云之间来回交接」做成一等公民。**这一步很关键**:它说明编辑器类 agent 也在往「后台自主、批量并行」靠拢——这恰恰是它原本相对终端类/云端类偏弱的地方(见 §5)。

## 4. 它与众不同的设计决策

拆完零件,退一步看它的「设计哲学」——这些取舍才是真正值得学的东西。

- **Fork 整个编辑器,而不是做插件(fork, don't plug in)。** 这是 Cursor 一切设计的地基。因为掌控了渲染层、文件系统、扩展宿主,它才能做出插件做不到的东西——比如 **shadow workspace**(一个隐藏的窗口,在里面对 AI 提议的改动跑语言服务器/LSP,把 lint 和类型错误喂回 agent,而**不污染你正在编辑的界面**)。**取舍**:代价是它得永远追着上游 VS Code 更新跑,维护负担不小。
- **把「改什么」和「怎么写进文件」拆成两个模型(见 3.4)。** 用一个便宜、快、专精的 apply 模型接住前沿模型的脏活——这是 agent 系统里延迟/成本权衡的经典范式,值得抄进任何自建 agent。
- **索引 + 语义检索作为一等上下文来源。** 编辑器类 agent 天然「盯着整个代码库」,靠的就是这条预建索引 + 向量检索的管线(3.2)。这是它和终端类「现场 Grep」最本质的分工。
- **审阅 UX 原生化成「内联 diff」。** ⑨篇讲的「权限/审批」,在 Cursor 里不是一个个弹窗,而是**你最熟悉的 diff**——绿加红删,标在代码旁,逐处点头。**这是编辑器类相对终端类最大的体验优势:可视化强、审阅直觉、改动看得见。**

但天下没有白拿的优势。把这些优势翻个面,就是它相对终端类 agent 的**取舍**:

> **编辑器类 agent 的强,强在「可视化 + 人在环审阅」;它的弱,也弱在「太黏在 IDE 内」。** 你很难像终端类那样,把 Cursor 塞进一个无头的 CI 流水线、或「发一个任务、明天来收 PR」——它的整套价值(diff 审阅、光标感知、内联补全)恰恰依赖于「你正坐在编辑器前」。background agent(3.6)就是它在补这块短板,但那和 Copilot 的云端 coding agent 相比仍是新路(见 §6)。

## 5. 局限与坑

诚实地说几条边界(也提醒你别神化它):

1. **它天然「黏」在 IDE 里,不擅长「发完就走」。** 内联 diff、光标感知、实时补全这些看家本领,全建立在「你正坐在 Cursor 前审阅」之上。要「派个任务、明天收 PR」这种异步托管形态,它的 background agent 还年轻,不如 Copilot 那套「云端 + GitHub Actions + 自动开 PR」成熟(见 §6)。
2. **索引不是免费的,也不是实时精确的。** 语义检索靠预建的向量索引,索引有同步延迟、切块和 embedding 都可能「找偏」。**别把 `codebase_search` 当成能读懂全库的上帝之眼**——它只是「相似度检索」,该 `@` 钉死的关键文件还得你手动钉(见 3.2)。
3. **apply 也会「套错」。** 把 diff 精确套进文件是概率过程,偶尔会套到错的位置或漏套。所以那道**内联 diff 审阅是必须认真看的闸**,不是走过场——尤其大改动、多文件时(见 3.4)。
4. **各种「数字/组件」是快照,不是常量。** Tab 的 RL 参数、apply 模型是不是还是那个 Llama-3-70B、索引的同步节奏、泄露系统提示里的「12 个工具」……都随 2.0/3.0 的大重写在变(那份泄露提示本身是 2025 年初的快照)。教学/引用时务必带「截至 2026 年中」,别当稳定事实。
5. **模型可选是双刃剑。** 能挂 Claude/GPT/Gemini/Composer 很自由,但也意味着**同一个任务换个模型,行为和质量会差一截**;而且用自研 Composer 才最能吃到 Cursor 内部为它做的优化——「用哪个模型」本身成了一个你得管的变量。
6. **收购悬而未决,产品形态可能变。** SpaceX 的收购截至 2026 年中**只是宣布、尚未交割**;交割后公司/产品结构如何变,是个开放变量。别把当下的形态当成板上钉钉的长期状态。

## 6. 一句话记住它 + 编辑器类横向对比

> **Cursor = 把 agent 焊进编辑器内核的 AI 原生 IDE:靠「代码库索引 + 语义检索」喂饱上下文,靠「前沿模型想 + apply 模型写」拆分职责,靠「内联 diff」把审阅做成你最熟的 UX。它是「编辑器类 agent」这一大流派最完整的样板,长处是可视化与人在环审阅,短处是黏在 IDE 内、不擅长发完就走。**

编辑器类内部也不是一种活法。把三个代表放一起看,分野立现——

**GitHub Copilot** 是这一派的元老,它的独特在于**沿着一条「补全 → Chat → Agent 模式 → 云端 coding agent」的完整进化链**长起来,且**深度焊在 GitHub 平台上**。它的 agent 有两副面孔:**Agent 模式**是 IDE 内**同步**的自主编码(你在 Copilot Chat 面板里看着它读文件、改、跑测试,危险命令默认要你批准);**Coding Agent** 则是**异步**的——你把一个 GitHub issue 指派给 `@copilot`,它在 GitHub Actions 驱动的临时环境里克隆仓库、检索代码、干活,最后**开一个 draft PR** 交你审。这条云端异步链上,它下了不少安全功夫:防火墙默认开(白名单式网络访问)、默认只给只读的 GitHub token、单次会话有 **59 分钟硬上限**、每个任务只开一个 PR、CI/CD 默认不在 agent 的 PR 上自动跑。**它赌的是「把 agent 缝进 SDLC(issue→PR→Actions→review)」,而不是做最强的交互编辑体验。**

**Cline** 走的是**另一条价值观截然不同的路:开源、可审计、你掌控一切**。它是个 **Apache-2.0 开源的 VS Code 扩展**(也有 JetBrains 插件、CLI、SDK),最鲜明的两点:一是 **BYOK / 模型无关**——你自带 API key,想挂 30 多家 provider 里的哪个都行,连本地模型都成,没有厂商锁定;二是 **Plan / Act 双模式**——Plan 模式**只读**(它只探索代码、提方案,禁止改文件),你满意了再切到 Act 模式让它动手,两个模式还能配不同模型(便宜的规划、聪明的执行)。它的审批做得极细:**8 个可勾选的自动批准类别**(读项目文件/读所有文件/改项目文件/执行安全命令……)外加一个「YOLO 模式」总开关,把⑨篇的「事事问你 ↔ 全自主」这根滑杆摊成了一排你能逐个拨的开关;配合影子 Git 检查点,任何改动都能一键回滚。**它赌的是「把控制权和透明度全交给你」,用开源和逐步审批换掉 Cursor 那种「闭源、黏 IDE」的黏性。**

三者(加上⑫的 Claude Code 作终端类对照)用一张表对齐:

| 维度 | **Cursor** | **GitHub Copilot** | **Cline** | (对照)Claude Code ⑫ |
|---|---|---|---|---|
| **核心形态** | AI 原生 IDE(fork VS Code),补全 + Agent 模式 | GitHub 平台内的插件:Agent 模式(同步)+ Coding Agent(异步) | 开源 VS Code 扩展(+ JetBrains/CLI/SDK),Plan/Act 双模式 | 终端原生 CLI agent(无 GUI) |
| **上下文来源** | 语义索引 + 向量检索(codebase_search)+ `@` 手动钉 + `.cursor/rules` | 工作区结构摘要 + 代码检索/RAG + 打开的文件 | Plan 模式主动读/搜代码库 + `.clinerules` + `ContextManager` 记账 | Grep/Glob 现场翻 + `CLAUDE.md` + 自动记忆 |
| **自主度** | 半自主为主,内联 diff 逐处审;background agent 补异步 | 同步档「你看着它」;异步 coding agent「发完就走」(59 分钟封顶) | 逐步审批(8 类可自动批 + YOLO),Plan 先看后动 | 一根从「事事问你」到全自主的滑杆(见⑨) |
| **开放性** | 闭源商业;模型可选(Claude/GPT/Gemini/自研 Composer) | 闭源商业;模型可选菜单(GPT-4.1 已于 2026 年 6 月退役,当快照看) | Apache-2.0 开源;BYOK、30+ provider、可挂本地模型 | 闭源;主要跑 Anthropic 自家 Claude 模型 |
| **审阅 UX** | 内联 diff(绿加红删,标在代码旁)+ checkpoint 回滚 | IDE 内 diff / PR 审阅(异步档走 GitHub PR review) | diff + 影子 Git 检查点(三种恢复路径) | 终端里展示改动 + 只追加 JSONL 可 resume/fork/rewind |

一句话把这一派收口:

> **编辑器类 agent 的共同信条是「让 agent 住进你写代码的地方」。** 但住进来之后各走各路:**Cursor 押「最强的原生编辑体验」,Copilot 押「缝进 GitHub 的整条 SDLC」,Cline 押「开源 + 你掌控一切」。** 三种赌注,对应三种不同的用户和信任模型。
