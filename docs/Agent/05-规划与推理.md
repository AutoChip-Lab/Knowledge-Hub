## 0. 先看全景:一个循环够不够用

前四篇我们把 agent 的五件套认全了:模型(想)、循环(转)、工具(做)、上下文/记忆(记),以及目标/终止(方向盘与刹车)。你现在应该有个印象:agent 之所以能干活,是靠**「想一步 → 做一步 → 看反馈 → 再想」这个循环**一圈圈磨出来的(第 ② 篇)。

那问题来了:**光靠这个循环「走一步看一步」,够不够应付所有任务?**

答案是——**看任务多难**。

```
简单任务:一个循环自然就走完了
   目标 ─▸ 想 ─▸ 做 ─▸ 看 ─▸ 想 ─▸ 做 ─▸ 完成
        (每一步都是"当下这步该干嘛",不用提前谋划)

复杂任务:走一步看一步容易迷路,得先谋后动 + 边做边反思
   目标 ─▸ ┌ 先拆解:整件事分成哪几大块? ┐
          │ 边做边反思:上一次为什么错了? │ ─▸ 完成
          └ 走偏了就改路线,别一条道走到黑 ┘
```

这一篇讲的就是这件事:**当任务复杂到「走一步看一步会迷路」时,agent 怎么「谋定而后动」,又怎么在走错时把自己拽回来。** 这就是 **规划(planning)** 与 **推理(reasoning)**。

它是「认识 Agent」这个板块的收官篇。读完你会明白:规划不是给 agent 加的「高级功能」,而是一种**用「想清楚」的成本换「少走弯路」收益的权衡**——用不用、用多重,全看任务配不配得上。

> 全篇继续用那个老场景回扣:**让 agent 去修好某开源仓库里的一个 bug——读 issue → 找到相关文件 → 改代码 → 跑测试 → 提 PR。** 这一篇我们专门看:这个 bug 修起来一旦不顺,agent 该怎么谋划、怎么纠错。

## 1. 「规划」在 agent 里到底指什么

一句话:

> **规划,就是让 agent 干两件事——把一个大目标拆成能一步步执行的小步(往前看),以及对已经走过的路做反思、从错误里学乖(往后看)。**

注意这个「往前看 + 往后看」的双向结构,是 agent 语境里「规划」最容易被漏掉的一半。很多人以为规划就是「开工前先列个提纲」,那只是**往前看**的那一半:

- **往前看 —— 任务分解(task decomposition)**:面对「修好这个 bug」,先想清楚这大概要经历「定位 → 改代码 → 跑测试 → 提 PR」几个阶段,而不是一上来就瞎读文件。
- **往后看 —— 反思与自我纠错(reflection / self-correction)**:测试挂了、改错了地方,agent 回头审视「我刚才哪一步的判断有问题」,把这份教训带进下一次尝试,而不是把同一个错误再犯一遍。

这个「Planning = 分解 + 反思」的划法,不是我们随口编的。Lilian Weng 在她 2023 年那篇被反复引用的《LLM Powered Autonomous Agents》里,把 LLM agent 拆成**大脑(LLM)+ 三大组件:规划(Planning)、记忆(Memory)、工具使用(Tool Use)**——其中「规划」这一块,她明确就是指**「子目标分解」加上「对过往行动的自我反思与自我批评」**这两半。

> 关键直觉:**规划的价值不在「想得多周全」,而在「走错时能回头」。** 一个只会往前列计划、不会往后反思的 agent,和一个背熟了菜谱却尝都不尝一口的厨子一样——菜咸了它也不知道。

## 2. 前提:什么时候「值得」规划

在讲怎么规划之前,先泼一盆冷水:**大多数任务根本不需要显式规划。**

回到第 ① 篇那把最重要的尺子——**workflow 还是 agent,看「谁在开车」**:步骤是人事先写死在代码里的,那是 workflow;步骤是模型每一步现想现决定的,那才是 agent。规划这件事,踩的是同一根弦,只是更细了一格:

**如果一件事的步骤,你事先就能枚举清楚,那就别给它上重型规划——直接把步骤写成固定流程(workflow)最省事、最可控、最好调试。**

举两个对照:

- 「把这段英文翻译成中文,再润色,再排成 Markdown」——三步,顺序固定,永远这三步。这种事**一个 workflow 就搞定**(第 ① 篇的 L2),让模型去「规划」纯属浪费 token,还平白引入不确定性。
- 「修好这个 bug」——你**根本不知道**要读哪几个文件、要试几次、第一次的诊断对不对、测试会不会牵出新问题。步骤事先无法枚举,方向盘只能交给模型——**这才是规划真正派上用场的地方**。

Anthropic 在《Building Effective Agents》(2024 年底)里给了一条同样朴素的建议:**从最简单的方案起步,只有当简单方案顶不住时,才往上加 agent 循环、加规划。** 这是对「一上来就套复杂框架」的明确反对。

> 一句话:**规划是给「事先没法排好步骤」的任务准备的。步骤能枚举的,写成流程就好——别拿大炮打蚊子。**

## 3. 谋定而后动:规划的几种玩法

这一节是本篇的核心。我们按「要不要规划 → 怎么往前拆 → 怎么往后纠 → 有哪些成套模式 → 代价在哪」的顺序,把规划这件事拆开看。

### 3.1 显式规划 vs 让循环自然涌现

先厘清一个最容易含糊的问题:**规划一定要「显式」地做吗?**

不一定。规划其实有两种存在形态,你得会分辨:

- **隐式规划(涌现式)**:你什么都不特意做,agent 就在那个 ReAct 循环里(第 ② 篇)一步步「想—做—看」。它每一步的「想」其实已经隐含了微型规划——「测试挂在 `parser.py` 第 40 行,那我下一步去看这个函数」。**对中小任务,这种走一步、看一步、自然涌现出来的路径,往往就够了。**
- **显式规划**:你专门让模型**先停下来,产出一份成文的计划**(「我打算:1. 定位 2. 改 3. 测 4. 提 PR」),再照着执行;或者在关键节点专门让它**反思一轮**。

什么时候该从「隐式」升级到「显式」?几条经验:

```
任务浅、路径短、每步反馈都很清楚  ─▸ 让循环自然涌现,别加戏
任务深、要跨很多步、容易中途迷路  ─▸ 显式列个计划,给循环一根主线
反复在同一个坑里栽、试了几次都不对 ─▸ 显式反思一轮,逼它总结教训
```

> 反直觉的点:**给弱任务硬加显式规划,常常帮倒忙。** 模型会一本正经列出一份漂亮但空洞的计划,然后被这份计划**框住**,反而不如它随机应变。规划是「按需调用」的重武器,不是默认开关。

### 3.2 任务分解:把「修 bug」拆成可执行的小步

**任务分解(task decomposition)** 是往前看那一半的主力:把一个大到无从下手的目标,切成几个「每一步都知道该干嘛」的小步。

还是那个 bug。「修好它」听着像一件事,拆开其实是一条链:

```
修好这个 bug
   │
   ├─▸ ① 定位:读 issue、复现、找到出问题的文件与函数
   ├─▸ ② 改代码:在定位到的地方动手
   ├─▸ ③ 跑测试:验证改动是否真的修好、有没有弄坏别的
   └─▸ ④ 提 PR:写清楚改了什么、为什么
```

拆解带来三个实打实的好处:

1. **把「无从下手」变成「知道下一步」**。面对整件事模型容易发懵,面对「先定位」它就有明确抓手了。
2. **每一小步都有独立的、可验证的产出**。「测试变绿」是个硬信号,比「感觉修好了」可靠得多——这正好接上第 ① 篇那句「反馈越真、越及时,agent 越强」。
3. **错了能局部回退**。如果测试挂在第 ③ 步,你知道问题多半出在第 ②(改错了)或第 ①(压根定位错了),不必把整件事推倒重来。

分解不必一次到位、也不必刻死。**好的分解是「够用就行、边走边细化」**:第 ① 步「定位」真做起来,可能又自然分出「grep 关键词 → 读三个候选文件 → 挑出真正的元凶」——这层细节没必要一开始就全想好,到了那步再涌现即可(呼应 3.1)。

> 一句话:**分解的目的不是画出完美路线图,而是把「一件想不清的大事」变成「一串能立刻动手的小事」。**

### 3.3 反思与自我纠错:用「说出来的自我批评」变聪明

这是往后看那一半,也是 agent 区别于「一次性问答」的灵魂之一。

设想 bug 场景里最常见的一幕:agent 改完代码,跑测试——**挂了**。一个不会反思的 agent,可能换个写法再改一遍,又挂;再换一个,还挂……在同一个坑里反复打转烧 token。

会反思的 agent 不一样。它会先停下来,用**大白话对自己做一次复盘**:「测试报的是 `KeyError`,说明我假设那个字段一定存在,但输入里可能没有——我上一次的诊断方向错了,根因不在格式化那步,而在取值那步。」然后**带着这条教训**去开始下一次尝试。

这套「用语言化的自我批评改进下一次尝试」的做法,有个经典名字:**Reflexion**(Shinn 等人,发表于 NeurIPS 2023,arXiv 2303.11366)。它的核心洞见很妙:

> **不用改模型的权重,只靠让模型把「我上次为什么错」写成一段自然语言、存进一个「教训本」(episodic buffer),下一次尝试就带着这段反思重来——模型就能一次比一次好。**

这为什么值得单拎出来讲?因为它把「让 AI 变强」这件事,从「重新训练(改权重,贵、慢)」搬到了「运行时改行为(写几句反思,便宜、快)」。在 Reflexion 当年的论文里,这套方法在 HumanEval 编程测试上拿到了 **91% 的 pass@1,而作为对照的 GPT-4 是 80%**。

这里必须诚实地加一个**重要限定**:

> ⚠️ 那个 **91% vs 80%** 是 **2023 年初论文里的历史数据**,用来说明「反思这套机制当年确实带来了显著提升」。它**不代表今天的能力水平**——如今的前沿模型,不加任何反思,分数也早已远超这两个数字。**请把它当成「机制有效」的历史佐证,而不是「现在能到多少」的当前战报。**

反思也不是免费午餐,它有两个硬前提:

- **得有多次尝试的预算**——反思的价值在「下一次」,一锤子买卖的任务用不上它。
- **得有可验证的结果**——编程(测试通不通)、解题(答案对不对)、玩游戏(赢没赢)这类「对错分明」的任务它最灵;对「这篇文案好不好」这种没有硬判据的开放任务,反思容易沦为自说自话(这个坑我们在第 5 节还会说)。

### 3.4 规划模式一览:CoT / ReAct / Plan-and-Execute / ToT

业界这几年攒下了几套成型的规划/推理「打法」。你不用全背,但得认识它们的名字和适用场景,将来看别人的 agent 设计才不会一头雾水。按「从轻到重」排:

- **思维链(Chain-of-Thought, CoT)**(Wei 等人,2022):最轻的一招——**让模型在给答案前,先把中间推理一步步「想出来、写出来」**。「让我们一步步来想」这句咒语就是它。它不调工具、不循环,纯粹靠「把思考显性化」提升推理质量,是后面一切复杂玩法的地基。
- **ReAct**(Reasoning + Acting,回顾第 ② 篇):在 CoT 的「想」之外,加上「做(调工具)」和「看(观察结果)」,让推理**踩在真实反馈上**交替推进。它是 agent 循环的经典范式,「边想边做边看」的那个循环就是它。
- **Plan-and-Execute(先规划、再执行)**:**先让一个强模型一次性产出完整计划,再由(可以更便宜的)执行者逐步照着做**。好处是「谋划」只用一次贵模型,比 ReAct 那种每步都重新推理省钱。这里要给个诚实的标注——

  > ⚠️ **Plan-and-Execute 没有单一的经典论文出处**,它是 LangChain 社区综合 BabyAGI、Plan-and-Solve(Wang 等,2023)等做法**归纳提炼出来的一种模式名**。所以谈它时点到「这是一种被广泛使用的模式」即可,别硬安一篇「奠基之作」。

- **思维树(Tree of Thoughts, ToT)**:把「一条线往下想」升级成「像下棋一样,同时展开多个候选思路、评估、择优,走不通就回溯」的**树状探索**。它最强,也最贵——探索多个分支意味着成倍的 token 和时间,只有当任务真的需要「试多条路再选一条」(如解数学难题、玩策略游戏)时才划算。

一张对比表帮你快速定位「什么时候用哪个、代价是多少」:

| 模式 | 一句话本质 | 什么时候用 | 主要代价 |
|---|---|---|---|
| **CoT**(Wei 2022) | 答题前先把推理写出来 | 单步就能答、但需要多步推理的题 | 几乎没有;只是输出更长 |
| **ReAct**(第②篇) | 想—做—看 交替循环 | 需要调工具、踩着真实反馈干活的任务 | 每步都要调一次模型,延迟高 |
| **Plan-and-Execute** | 先出完整计划,再逐步执行 | 步骤大致可预见、想省贵模型调用 | 计划会过时,现实一变就得重规划 |
| **Tree of Thoughts**(Yao 2023) | 树状展开多条思路、评估回溯 | 需要「多试几条路再择优」的难题 | 成倍的 token 与时间,最贵 |

> 一句话:**这四招是一条「从轻到重」的谱系,不是「越靠后越高级」。** 拿 ToT 去修一个改一行就好的 bug,是杀鸡用牛刀,还容易把自己绕晕。

### 3.5 规划的代价与边界

前面一路在讲规划的好,这一小节专门讲它的**账单**——因为不算这笔账,你就会不自觉地「过度规划」。

规划有三笔逃不掉的成本:

- **规划本身要花 token 和时间**。让模型「先停下来想清楚再动」,那一轮「想」是实打实的推理开销。任务越简单,这笔开销占比越亏——极端情况下,规划花的功夫比干活还多。
- **计划会过时**。这是 Plan-and-Execute 一类「先出完整计划」模式的阿喀琉斯之踵:计划是**基于开工前的认知**定的,可现实是边做边揭晓的。你计划里写「改 `parser.py`」,真读进去才发现根因在 `loader.py`——**计划立刻作废,不改就是照着错地图走**。
- **过度规划会让 agent 变僵**。列得太细、太死的计划,会把模型**锁死在既定路线上**,反而压制了它「随机应变」的那份灵活——而灵活恰恰是当初我们把方向盘交给它的全部理由。

所以规划有它清晰的**边界**:它擅长「给复杂长任务一根主线、在走错时提供复盘的支点」;它不擅长、也不应该「把每一步都提前钉死」。

> 一句话:**规划买的是「少走弯路」,付的是「想的成本 + 计划过时的风险 + 僵化的代价」。这笔买卖划不划算,完全取决于任务有多难、多长、多不可预测。**

## 4. 关键心态:规划不是越多越好

如果这一篇你只带走一句话,请带走这句:

> **规划不是「加得越多越好」的功能,而是一场交易——你用「想清楚」的前期成本,去换「少走弯路」的后期收益。这笔交易划不划算,要按任务的难度来配。**

把它掰成三条可操作的心态:

- **按难度配比,不搞一刀切。** 改一行的小 bug,让循环自然涌现就好(3.1);要重构一整个模块、跨几十步、中途极易迷路,才值得先显式列计划、并在关键节点安排反思。**给任务配规划,像给病配药——对症、按量,过量有毒。**
- **反思比列计划更金贵。** 往前列的计划开工五分钟就可能过时;而往后的反思,是从**已经发生的真实反馈**里榨取教训——信号更硬、更值钱。资源有限时,**优先保证 agent「走错了能回头」,而不是「开工前想得多周全」**。
- **让计划活着,而不是把它供起来。** 计划是**可以随时改的草稿**,不是签了字的合同。现实和计划一打架,该认输的是计划。一个好 agent 的标志,是**该改路线时真的会改**,而不是抱着一张过时的地图一条道走到黑。

## 5. 常被忽视的坑

规划这块的坑,几乎都长在「把规划当成越多越好的好东西」这个误解上。逐条拆:

1. **计划一旦定死就不肯改(计划僵化)。** 最常见、也最致命。agent(或写它的人)把开工前列的计划当成不可违抗的圣旨,现实早就变了还照着走,一路走到黑。**避法:把「重新规划」设成一等公民**——每当真实反馈和计划严重打架(测试反复挂、发现根因在别处),就该触发一次「要不要改计划」的判断,而不是闷头执行。
2. **反思变成空谈(自我批评沦为自我安慰)。** 让 agent 反思,它却写出一堆「我下次会更仔细」「我应该更全面地考虑」这类正确的废话——听着深刻,对下一步毫无指导。**避法:反思必须咬住可验证的信号**(具体是哪个测试、报了什么错、哪一步的假设被证伪),把它逼成「一条能改变下一步动作的具体教训」,而不是一句心灵鸡汤。这也正是 Reflexion 强调「要有可验证结果」的原因(3.3)。
3. **把规划当银弹,以为「加了规划就能搞定难任务」。** 规划不会凭空提升模型的智力天花板;它只是帮一个**本就有能力**的模型少走弯路。模型压根做不到的事,你让它规划一百遍也做不到——只会把「做不到」包装成一份看起来很专业的计划。**避法:先确认任务在模型能力圈内,再谈用规划优化路径;别指望规划替你补齐能力短板。**
4. **给简单任务硬上重型规划(过度工程)。** 三步固定的流水线,你非要让模型先「规划」一轮——白烧 token、平添延迟,还引入了本可避免的不确定性。**避法:回到第 2 节那把尺子——步骤能事先枚举的,写成 workflow;真需要 agent 自主时,也从最轻的 CoT/ReAct 起步,顶不住了再往 Plan-and-Execute、ToT 上加。**
5. **反思却没有「多次尝试」的预算。** 反思的全部价值都在「下一次」;可很多任务是一锤子买卖,或者你根本没给它重试的步数/预算。**避法:要用反思,就得在设计上真的留出多轮尝试的空间(和上限)**——否则那轮反思写完就没有下文,纯属白想。

## 6. 小结 & 收官:五件套靠什么粘起来

这一篇你需要带走的:

- **简单任务一个循环就够;复杂任务才需要「谋定而后动 + 边做边反思」。** 规划 = **往前看的任务分解** + **往后看的反思纠错**,两半缺一不可。
- **步骤能事先枚举的,别上重型规划**——写成 workflow 更省更稳(第 ① 篇那把尺子的延伸)。
- 规划有一条**从轻到重**的谱系:**CoT → ReAct → Plan-and-Execute → Tree of Thoughts**,按任务难度选,不是越靠后越高级。
- **反思(Reflexion)** 靠「说出来的自我批评」不改权重就变聪明,但要有**多次尝试的预算**和**可验证的结果**;那个 91% vs 80% 是**历史数据**,别当当前能力。
- 心态:规划是**用「想清楚」的成本换「少走弯路」的收益**,按难度配比;**让计划活着、优先保住反思**,别把它供成不可改的合同。

到这里,「认识 Agent」这个板块就走完了。回头看这五篇,我们其实是把一个 agent 彻底拆成了五件套:**模型(想,第①篇)、循环(转,第②篇)、工具(做,第③篇)、上下文与记忆(记,第④篇)、规划与推理(谋定而后动,本篇)。**

可你有没有发现一个悬而未决的问题:**这五件散装的零件,是靠什么被组装成一个能真跑起来的 agent 的?**

是谁决定循环转几圈才停、工具调用越权了拦不拦、上下文满了裁哪一段、什么时候该逼它反思一轮?——**不是模型自己,而是包在模型外面那一层「粘合代码」**。第 ① 篇结尾埋下的那个词,现在到了兑现的时候:

> **这层把五件套粘起来、并指挥它们怎么协作的代码,就是 agent harness。前五篇讲的是「零件」,下一个板块讲的是「把零件组装成机器的那套骨架」——而很多时候,一个 agent 好不好用,拼的恰恰不是零件,是这层 harness。**

> 👉 继续读 **第 ⑥ 篇:什么是 Harness**——从「认识零件」正式迈入「认识骨架」。(至于多个 agent 怎么分工协作,后面的 Harness 板块会专门讲。)
