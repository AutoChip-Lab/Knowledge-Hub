## 0. 先看全景:工具是模型伸向真实世界的「手」

上一篇我们把 agent 的循环拆开了:**想一步 → 做一步 → 看结果 → 再想**(见第 ② 篇)。这一篇我们盯住循环里的「做一步」——也就是 **Action**——到底是怎么发生的。

答案一句话:**靠工具。**

一个大模型,不管参数多大、多聪明,它自己能做的事只有一件:**根据你给的文字,生成下一段文字。** 它读不了你硬盘上的 `parser.py`,跑不了你的测试,发不出一个 HTTP 请求,更删不掉一个文件。它是一颗被关在玻璃缸里的大脑——能想,但够不着外面的世界。

**工具,就是从这颗大脑伸出去的手。** 读文件、跑 shell、搜网页、调 API,每一件「动手」的事,背后都是一次工具调用。

```
        ┌──────────────┐
        │   模型 (脑)   │   ← 只会生成文字,够不着真实世界
        └──────┬───────┘
               │ "我要调 read_file,参数 {path: parser.py}"
               ▼
        ┌──────────────┐
        │  工具 (手脚)  │   → 真的去读、去跑、去查
        └──────┬───────┘
               │ 把结果(文件内容/报错/测试输出)递回来
               ▼
        ┌──────────────┐
        │   真实环境    │
        └──────────────┘
```

> 关键直觉:**没有工具的模型,再聪明也只能「空谈」。** 它能给你讲一大套怎么修 bug 的道理,但它没看过你的代码、跑不了你的测试,对错全靠猜。**工具是把「会说」变成「能做」的那一步。**

这一篇讲清楚三件事:工具调用机理上怎么工作;什么样的工具算「好工具」;以及设计工具时那些看似小、实则决定 agent 好不好用的坑。全程回扣我们的老场景——**修一个开源仓库里的 bug**:读文件是工具调用,跑测试是工具调用,提 PR 也是工具调用。

## 1. 工具使用到底是什么

先给一句话定义:

> **工具使用(tool use / function calling)= 让模型不只是「吐文字」,而是能吐出一个「结构化的动作意图」——「我要调用哪个工具、传什么参数」——再由外面的代码真正去执行,把结果喂回给它。**

「工具使用」和「函数调用」这两个词,你可以先当同义词用:业界最早(OpenAI,2023 年 6 月 13 日)把这个能力叫 **function calling**;Anthropic 等把它叫 **tool use**。叫法不同,骨架一样。

直觉上,它把模型从「一张只能写字的嘴」升级成「一个会派活的调度员」。你给它一份「它能使唤的工具清单」,它在推理过程中判断:**「光靠我脑子里记的答不了这题,我得去查/去跑/去改」**,于是它不再直接编一个答案,而是发出一条指令:「请帮我运行 `run_tests`」。

对照修 bug 的场景:模型读懂 issue 后,心里想的是「我得先看看 `parser.py` 长什么样」。它没法自己看——于是它发出一个 `read_file` 的工具调用。这就是工具使用最朴素的样子:**把模型的「意图」翻译成一次真实世界里的「操作」。**

## 2. 前提:模型只会生成意图,执行永远在「外面」

这里有个新手极容易搞错、但一旦想通就豁然开朗的点,值得单独拎出来讲。

**模型自己从不「执行」任何东西。** 当模型「调用」一个工具时,它其实什么也没做——它只是**生成了一段特殊格式的文字**,内容是「我想调用名为 `run_tests` 的工具,参数是这些」。这段文字本身没有任何魔力,它跑不了测试,就像你在纸上写「运行测试」四个字,测试不会自己跑起来。

真正把测试跑起来的,是**模型之外的一段普通代码**——我们叫它**执行器(executor)**。它守在模型旁边,盯着模型的输出:一旦发现模型吐出了「我要调 `run_tests`」,它就真的去操作系统里把测试命令执行掉,拿到输出,再把输出**回填(feed back)**进对话,让模型接着往下想。

```
模型输出:  「调用 run_tests, 参数 {}」   ← 这只是一段文字(意图)
                     │
                     ▼
执行器:    真的去 shell 里跑 `pytest`     ← 真实动作发生在这里
                     │
                     ▼
执行器:    把测试输出塞回对话             ← observation 回填
                     │
                     ▼
模型:      「3 个测试挂了,我去看第一个」  ← 基于真实结果继续
```

> 记住这条分界线:**模型负责「决定做什么」,harness 里的执行器负责「真的去做」。** 模型的手其实是「借」来的——borrowed hands。这条线之所以重要,是因为**所有的安全护栏都建在这条线上**:既然动作是执行器落地的,那就可以在执行前拦一道(该不该让它删这个文件?),这正是权限与沙箱的立足点(细节见第 ⑨ 篇)。

这条分界线也解释了一个常见困惑:「模型会不会偷偷跑命令?」不会。它连碰真实世界的能力都没有,它能做的极限就是「说它想跑」。**给不给它手、给几只手、手能伸多远,完全由外面的 harness 决定。**

## 3. 核心内容:工具从机理到设计

### 3.1 function calling 怎么工作:一次完整的往返

把机理拆成四拍,你就彻底懂了。我们用修 bug 里最常见的两个工具——`read_file` 和 `run_tests`——当例子。

**第一拍:用 JSON Schema 描述工具,喂给模型。**
你不能只跟模型说「你有个读文件的工具」,它得知道这工具叫什么、干嘛的、要传哪些参数、参数什么类型。这份「说明书」用 **JSON Schema** 写:

```json
{
  "name": "read_file",
  "description": "读取仓库里一个文件的完整内容。用于查看源码、定位 bug。",
  "input_schema": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "相对仓库根目录的文件路径,如 src/parser.py"
      }
    },
    "required": ["path"]
  }
}
```

这份 schema 会和系统提示一起进上下文。模型「读」到它,就知道自己多了一只叫 `read_file` 的手,以及怎么使唤它。`description` 不是写给人看的注释——它是**写给模型的操作手册**,模型全靠它判断「什么时候该用、参数怎么填」。这个细节 3.2 会展开。

**第二拍:模型吐出一个「工具调用」。**
模型推理后,不再直接给最终答案,而是输出一个结构化的调用块,大意是:

```
{ "type": "tool_use", "name": "read_file", "input": { "path": "src/parser.py" } }
```

这时循环会停下来——按 Anthropic 官方文档的说法,模型返回一个 `stop_reason: "tool_use"` 的信号,等着外面把结果给它。

**第三拍:执行器执行,拿到 observation。**
harness 里的执行器接过这个调用,真的去读 `src/parser.py`,拿到文件内容(或者「文件不存在」的报错)。这个真实结果就叫**观察(observation)**。

**第四拍:observation 回填,循环继续。**
执行器把 observation 包成一个 `tool_result` 塞回对话,模型带着「文件里到底写了什么」再想下一步——也许是「问题在第 42 行,我要调 `edit_file` 改它」,也许是「还得先看看 `run_tests` 报什么错」。

```
① 定义(schema)  →  ② 模型: "调 X, 参数{...}"  →  ③ 执行器: 真的跑
        ▲                                                    │
        │                                                    ▼
        └────────  ④ observation 回填(tool_result)  ◀───────┘
                    循环回到第 ② 拍,直到不再需要工具
```

有一条协议纪律值得记住:**模型发出一个 `tool_use`,下一轮就必须收到一个配对的 `tool_result`,否则循环会断掉。** 这不是玄学,是工具往返的硬约束——发了指令没给回执,模型就悬在半空了。

> 一句话:function calling 的全部,就是这个「schema 定义 → 模型出意图 → 执行器落地 → 结果回填」的四拍往返。第 ② 篇讲的循环,每一次「Action」都是这么一拍一拍走完的。

### 3.2 一个「好工具」长什么样:工具是给模型用的界面,不是给人用的

这是本篇最反直觉、也最值钱的一节。

新手设计工具时,几乎都会犯同一个错:**把给人用的命令直接塞给模型。** 「人能用 `cat` 看文件,那就给模型一个 `cat` 工具好了。」听起来天经地义,实则是个大坑。

想想 `cat` 一个一万行的文件会发生什么:对人,你可以哗哗滚屏、一眼扫过、随时叫停;对模型,这一万行会**一次性灌满它的上下文窗口**(见第 ④ 篇),把真正重要的信息挤出去,还烧掉一大笔 token。**同一个命令,对人是趁手工具,对模型是灾难。**

Princeton 的 **SWE-agent**(NeurIPS 2024)把这件事点破,提出了一个影响深远的概念:**代理-计算机接口(ACI, agent-computer interface)**。核心洞见一句话:

> **大模型是一类「全新的用户」,它需要为自己量身定做的界面——就像人类需要 IDE、需要图形界面一样。** 工具不是给人设计的,是**给模型**设计的。

这个洞见有多硬?SWE-agent 论文做了个消融实验:**同一个 GPT-4,换掉底层界面(用为模型设计的 ACI 命令,而不是裸 Linux shell),在 300 个 issue 的子集上,问题解决率高出了 10.7 个百分点。** 注意——**模型没变,只换了界面,成绩就差一大截。** 界面设计,比这一步换不换模型更能决定成败。

那「为模型设计」具体意味着什么?SWE-agent 给了几条可落地的原则,每一条都对应一个它想防住的失败:

- **反馈要信息量足、又别太长。** `search_dir` 匹配到超过 100 个文件时,SWE-agent 直接**拒绝**并让你缩小查询,而不是把几百个路径糊你一脸——防的正是「一次搜索淹没上下文」。
- **粒度要合适。** 它的文件查看器**默认每轮只显示约 100 行**,配 `goto`/`scroll_up`/`scroll_down` 让模型一屏一屏地看——防的是 `cat` 万行文件的惨剧。
- **要防呆、能自纠。** 它的 `edit` 命令改完会跑一个 flake8 子集做静态检查;**如果这次改动引入了新的语法/未定义名错误,就自动回滚**,并把「改之前/改之后」的报错清楚地列给模型看。一次手滑不至于污染后面所有步骤。

> 一句话:**好工具的第一读者是模型,不是人。** 错误信息要让模型看得懂、粒度要刚好够它做下一步决策、危险操作要有防呆兜底。评判一个工具好不好,标准不是「人用着顺不顺手」,而是「模型用着会不会犯错、会不会被淹」。

### 3.3 常见工具类型:以 Claude Code 的真实工具集为例

工具五花八门,但按「它让模型多了什么能力」归类,主要就这么几种。我们用 **Claude Code**(Anthropic 官方的编码 agent)的真实内置工具当例子——它的公开工具参考表里(截至 2026 年中)列了 **42 个内置工具**(这是个版本快照,会随 CLI 频繁迭代而增减),恰好把每一类都覆盖了:

| 工具类型 | 让模型能… | Claude Code 里的真实工具 |
|---|---|---|
| 文件读写 | 看代码、改代码 | `Read`、`Edit`、`Write` |
| 代码/命令执行 | 跑测试、跑构建、用 git | `Bash`(Windows 上 `PowerShell`) |
| 代码检索 | 在大仓库里精准定位 | `Grep`(按内容搜)、`Glob`(按文件名搜) |
| 网页检索 | 查文档、查报错 | `WebSearch`、`WebFetch` |
| 生成子任务 | 把探索性工作丢给一个隔离的 subagent | `Agent`(见第 ⑩ 篇) |

把这张表映射回修 bug:模型用 `Grep` 在几万行代码里找到抛异常的那个函数,用 `Read` 看清上下文,用 `Edit` 打补丁,用 `Bash` 跑 `pytest` 验证——**一次修 bug,就是一连串不同类型工具调用的编织。**

这里藏着一个值得注意的设计选择:为什么单独给一个 `Grep`,而不是让模型用 `Bash` 自己敲 `grep`?**因为专用工具能把输出裁剪、格式统一、行为收敛**——这正是 3.2 的 ACI 思想在起作用。裸 `Bash` 什么都能干,但也什么都不「防呆」;专用工具窄,但对模型更友好。

`Bash` 这个「万能工具」本身也带着 ACI 式的护栏:据官方文档,它默认 **2 分钟超时**(可调到最多 10 分钟),输出**默认截断到 30,000 字符**(硬上限 15 万)——就是怕一条命令要么卡死循环,要么吐出天量输出撑爆上下文。**连最通用的工具,都得为「模型这个用户」上限流。**

> 一句话:工具类型无非「读、写、算、查、派」几大类;真正的功夫在于——每一类都为模型量体裁衣,而不是把人类命令原样搬过来。

### 3.4 MCP:把工具标准化,解决 N×M 的噩梦

到这里有个现实问题:工具是好东西,但**每个 agent 都要自己给每个外部系统写一遍工具接入代码**,这事不 scale。

设想有 N 个 agent(Claude Code、Cursor、你自研的业务 agent……),要接 M 个外部系统(GitHub、Slack、你的内部数据库、Jira……)。如果每一对都要单独写一套对接,你就得写 **N×M** 套胶水代码。加一个新 agent,得重新对接所有系统;加一个新系统,所有 agent 都得改。这是集成领域的经典噩梦。

**MCP(模型上下文协议,Model Context Protocol)** 就是来拆这个噩梦的。它是 **Anthropic 在 2024 年 11 月推出的一个开放标准**,思路和当年 USB 统一各种外设接口是一个套路:**定义一套通用协议,让「提供工具的一方」和「使用工具的一方」只需各自实现一次这套协议,就能互通。** 于是 N×M 塌缩成 **N+M**——每个 agent 实现一次协议,每个系统实现一次协议,任意两边即插即用。

它用的是 **client/server 模型**:
- **MCP server**:由「拥有工具/数据的一方」提供。比如一个 GitHub 的 MCP server,把「读 issue、提 PR」这些能力按 MCP 协议暴露出来。
- **MCP client**:内嵌在 agent/harness 里。Claude Code 就自带一个 MCP client,能连上任意符合协议的 server,把它暴露的工具「插」进自己的工具集。

```
                        ┌─────────────┐
        MCP client ────▶│ GitHub server│
       (在 agent 里)    ├─────────────┤
             ├─────────▶│ Slack server │
             │          ├─────────────┤
             └─────────▶│  DB   server │
        一个 client,连任意多个 server;
        server 写一次,任意 agent 都能用
```

有个和上下文相关的工程细节值得一提:据 Claude Code 官方文档,**MCP 工具的定义默认是「延迟加载」的——不会一股脑塞进上下文,而是按需通过 `ToolSearch` 拉进来。** 原因很实在:工具描述本身也占上下文,几十上百个 MCP 工具全展开会把窗口吃光(见第 ④ 篇)。

MCP 的更多协议细节和生态,这里保持克制、不深挖(标准仍在快速演进)。你只需带走一个判断:**当你发现自己在给多个 agent 重复写同一个系统的对接时,就该考虑把它做成一个 MCP server。** 工具在 harness 内部具体怎么被注册、分发,是另一个话题,留给第 ⑦ 篇。

> 一句话:function calling 解决「模型怎么调一个工具」;**MCP 解决「工具怎么被无数个 agent 复用」**——把 N×M 的集成噩梦,拍平成 N+M。

### 3.5 工具设计的坑:魔鬼都在细节里

工具能不能用,往往栽在几个不起眼的细节上。这里先点四个最常见的,展开的经验教训放到第 5 节。

- **错误信息设计。** 工具失败时回给模型什么,比成功时回什么更重要。一句 `Error: exit code 1` 等于没说,模型只能瞎猜;而「文件 `parser.py` 第 42 行:未定义变量 `x`」能让它精准下一步。**错误信息是模型自我纠错的唯一燃料**——回得含糊,agent 就在原地打转。
- **幂等性(idempotency)。** 模型是概率性的,可能重复发同一个调用。如果这个工具「多调一次就多干一次」(比如「转账」「发邮件」「追加写入」),重复就会造成真实损害。**能设计成「重复调用结果相同」的,尽量设计成幂等的**;做不到的(如转账),就得在工具外面加去重/确认。
- **返回过长。** 前面反复说过:一次返回把上下文撑爆,是工具设计头号杀手。截断、分页、摘要——总得选一个,不能让工具「有多少吐多少」。
- **权限。** 一个能跑 `Bash`、能 `Write` 文件的工具,威力和风险是一体两面。哪些工具能自动跑、哪些必须人点头,是工具设计绕不开的一环。这里只预告——**权限与沙箱是第 ⑨ 篇的主角**,本篇不展开。

> 一句话:工具的成败,一半在「它能干什么」,另一半在「它失败时说什么、被重复调用时怎么办、话太多时会不会噎着模型」。

## 4. 关键心态:工具设计常常比模型选择更能决定 agent 好不好用

这是本篇最想让你带走的一句话,也是整门课反复出现的主旋律。

新手评估 agent,第一反应总是问「它用的什么模型?换个更强的模型是不是就更好用?」模型当然重要——它是**智力天花板**(见第 ①、⑤ 篇)。但 SWE-agent 那个 **+10.7 个百分点**的消融实验冷冷地提醒我们:**模型没换,只把工具界面从裸 shell 换成为模型设计的 ACI,解决率就跳了一大截。**

换句话说,**在很多任务上,「工具设计得好不好」比「模型选得强不强」更能决定 agent 的实际表现。** 一个配了糟糕工具的强模型,会被含糊的报错、爆炸的输出、别扭的粒度反复绊倒;一个配了精心设计工具的中等模型,反而能稳稳把活干完。

> 一句话金句:**工具是模型的界面,而界面就是产品。** 你花在「怎么把工具做得让模型顺手」上的功夫,回报常常高于「再换一个更强的模型」。这也预告了整门课的核心命题——**agent 好不好用,拼的往往不是模型,而是包在模型外面的那层 harness**(第 ⑥ 篇起)。工具系统,正是 harness 里最能立竿见影的一块。

## 5. 常被忽视的坑(工具设计最容易翻车的几处)

1. **把「给人的命令」直接当「给模型的工具」。** 最经典也最致命的一坑。`cat` 一个大文件、`grep` 出几千行匹配、返回一大坨原始 JSON——对人无妨,对模型就是一次上下文灾难。**每个工具都要问一句:模型收到这个返回,还看得清重点、做得了下一步决策吗?** 记住 ACI:工具的第一读者是模型。
2. **错误信息含糊得等于没说。** 工具失败时甩一句 `exit code 1` 或一个空堆栈,模型就只能瞎猜、原地打转。**错误信息是 agent 自我纠错的唯一燃料**:要说清哪里错、错在哪一行、期望是什么。回得越具体,agent 越可能一次纠对。
3. **不管幂等性,让副作用工具能被安全重复。** 模型是概率机器,会重试、会手滑重发。若「发邮件/转账/追加写」被调两次就干两次,轻则脏数据,重则真实损失。**有副作用的工具,要么设计成幂等,要么在外层加去重与人工确认。**
4. **工具返回不设上限,一次撑爆上下文。** 这是烧 token、挤掉关键信息的头号元凶。Claude Code 连 `Bash` 都默认把输出截到 3 万字符、给 2 分钟超时,不是小气,是保命。**任何工具的返回都该有截断/分页/摘要策略**,别让它「有多少吐多少」。
5. **忽视「工具太多」本身也是负担。** 你可能以为工具越多 agent 越强,其实工具清单也占上下文、也会让模型在「该用哪个」上犹豫、选错。这正是 Claude Code 把 MCP 工具**默认延迟加载、按需检索**的原因。**工具不是越多越好,是「刚好够、且每个都好用」最好。**

---

## 6. 小结 & 下一步

这一篇你需要带走的:

- **工具是模型伸向真实世界的手**——没有工具,再聪明的模型也只能空谈。循环里的每一次「Action」,都是一次工具调用。
- **机理是一个四拍往返:** JSON Schema 定义工具 → 模型吐出「调 X、参数 {...}」→ 执行器真的去做 → observation 回填。模型只生成「意图」,**执行永远在模型之外的代码里**——这条分界线也是一切安全护栏的立足点。
- **好工具的第一读者是模型,不是人**(ACI 思想):错误信息可读、粒度合适、危险操作防呆。SWE-agent 的 +10.7 个百分点证明——**界面设计,常常比换模型更能决定成败。**
- **MCP 把工具标准化**:用 client/server 模型把 N×M 的集成噩梦,拍平成 N+M,让一个工具能被无数 agent 复用。
- 坑大多不在「工具能干什么」,而在「失败时说什么、被重复调用时怎么办、返回会不会噎着模型、工具是不是太多」。

工具解决了「模型怎么动手」。但你可能已经注意到:这一篇处处都在提「别撑爆上下文」「别把窗口吃光」。**模型每次「想」,都只能看见喂进它那一小段文字**——这段文字装不下的历史、装不下的知识,怎么办?这就是下一篇的主题。

> 👉 继续读 **第 ④ 篇:上下文与记忆**——agent 的「工作台」有多大、记性怎么维持、长任务里怎么不「失忆」。
