## 0. 先看全景:模型每次「想」,只能看见桌上这一叠纸

前三篇里,我们把 agent 拆成了「模型 + 循环 + 工具」,并且知道循环是靠**把工具返回的观察喂回给模型**转起来的(第 ②③ 篇)。现在有一个你一直没追问的问题:**这些「喂回去的历史」,到底喂到哪儿去了?模型是把它们「记住」了吗?**

答案会让很多人意外:**模型什么都没记住。** 它每一次「想」,都像一个失忆的天才——你把一叠纸推到它面前,它读完、给个回应,然后立刻忘光;下一次你得把**整叠纸(连同上一轮的回应)重新推过去**,它才知道之前发生了什么。这叠纸,就是**上下文(context)**;这张桌子有多大,就是**上下文窗口(context window)**。

于是麻烦来了:任务一长,纸越堆越多,桌子装不下了。**agent 能不能干「长活」,一大半不取决于模型多聪明,而取决于我们怎么在这张有限的桌子上,安排它该看见什么、忘掉什么、去哪儿翻旧账。** 这一整套手段,就是「记忆」。

```
        每一次模型调用 = 一次「失忆的天才」读一叠纸
   ┌──────────────────────────────────────────────┐
   │  系统提示 + 工具说明 + 历史对话 + 工具返回 + ……  │  ← 这一叠 = 上下文
   └──────────────────────────────────────────────┘
                        │  全部塞进去
                        ▼
                  ┌───────────┐
                  │   模型     │  读完→回应→立刻忘光
                  └───────────┘
                        │
       下一轮:把上面那叠 + 这一轮的回应,整个重新推一遍
```

> 全篇继续用那个跑腿的例子:**让 agent 修好某开源仓库里的一个 bug——读 issue → 找相关文件 → 改代码 → 跑测试 → 提 PR。** 这次我们盯住一件事:**一个上百万行的仓库,怎么塞进一张小小的桌子?**

## 1. 上下文窗口:agent 的「工作台」

先给直觉,再给术语。

> **上下文窗口,就是模型「一次能看见的全部文字」的上限——它是 agent 干活的工作台,台面有多大,决定了它一次能摊开多少东西。**

这台面上摆的不只是你的问题。一次调用里,上下文通常包含:**系统提示(system prompt,给它的角色和规则)+ 所有工具的说明书 + 到目前为止的全部对话历史 + 每个工具返回的结果**。模型「想」的时候,看到的就是这一整摊。

台面的计量单位不是「字」,而是 **token(词元)**——模型把文字切成的小块,一个 token 大致是几个字符或大半个英文单词(中文通常一个字一到两个 token)。所谓「200K 上下文窗口」,意思是**输入 + 输出加起来最多约 20 万个 token**。截至 2026 年中,前沿模型的窗口已经能到 **100 万 token**(据 Anthropic 官方文档,这一档位在其旗舰 Opus 与 Sonnet 5 上提供)——听起来大得吓人,几乎能装下一整本书。

但这里有第一个反直觉的点,请你现在就记牢:

> **上下文窗口是 agent 最硬的约束(binding constraint)。** 台面再大也是有限的,而一个真实任务产生的历史,几乎总能把任何台面撑爆。

为什么?回到修 bug 的场景:模型 `grep` 一次,可能返回几百行匹配;`Read` 一个文件,几千行进来了;跑一次测试,几百行报错糊上来;它试错三五个来回,历史就滚成了几万 token。**一个上百万行的仓库,你根本不可能把它整个摊上台面。** 桌子大只是让你晚一点撞墙,不是让你不撞墙。而且它还有代价——这是下一节的事。

> 一句话:**模型的智力有多高是「模型」的属性;它一次能看多少、记多久,是「上下文」的属性。这门课后半程讲的 harness,一大半功夫都花在经营这张桌子上。**

## 2. 前提:两次调用之间,它是「无状态」的

在讲各种记忆招式之前,必须先把一块地基夯死,否则后面全是空中楼阁:

> **LLM 的每一次调用之间,是完全无状态(stateless)的——你不主动把历史喂进去,它就什么都不记得。**

这不是模型「偷懒」,而是它的工作方式本就如此:模型是一个纯函数,**输入一叠文字,输出一段文字**,调用结束,内部不留任何痕迹。你以为的「它记得我们刚才聊过 X」,其实是**外面那层代码(harness)每一轮都偷偷把之前的全部对话重新塞了进去**,制造出「它有记忆」的错觉。

这一点解释了两件事,而且都直接关系到我们的场景:

- **为什么历史会「越滚越贵」。** 既然每轮都要把整段历史重新发一遍,那么对话越长,每一步要发的 token 就越多。到了第 30 步,光是「复述前 29 步」就可能占掉大半个窗口——这也是 agent 跑长任务时越跑越慢、越跑越烧钱的根源。
- **为什么「记忆」根本不是模型自带的功能,而是一项需要工程去搭的能力。** 模型只提供「一次看一叠纸」,「让它跨越几十步、几小时还记得住关键信息」这件事,是 harness 用检索、压缩、外部文件等手段**从外面拼出来的**。

> 关键直觉:**agent 的「记忆」是个假象——是外面的代码在每一轮精心决定「这次给它看哪叠纸」。** 谁来决定这叠纸怎么组、装什么、丢什么,才是本篇真正的主题。

## 3. 核心内容:一张桌子,五套记账法

模型只给你「一次看一叠纸」这个原语。要把它撑成一个能干长活的 agent,业界发展出了一整套围绕上下文的手法。下面五个子节,从「桌子本身」讲到「桌子之外的仓库」,层层往外扩。

### 3.1 上下文窗口:台面多大、token 怎么算、为什么「大窗口 ≠ 无限记忆」

前面说了台面是有限的。现在要戳破一个更顽固的幻觉:**「窗口够大就等于记性够好」。** 就算真给你 100 万 token 的桌子,把历史一股脑全铺上去,也是个坏主意,原因有三:

- **越长越贵。** 上下文是按 token 计费的,而且**每一轮都要把整段历史重发一遍**(见 3.2)。历史线性变长,累计花费近似平方级往上窜。Manus 团队公开过一组数字,很能说明 agent 的经济账:他们观察到 agent 任务的**输入与输出 token 比约为 100:1**——也就是说,agent 绝大部分开销花在「读那一大叠上下文」上,而不是「写回应」上。放到修 bug 场景里就是:真正「动笔改代码」的字数,远小于「反复重读 issue、代码、报错」的字数。上下文一胖,钱包立刻疼。
- **越长越慢。** token 越多,模型这一步「读完」花的时间越长,整个循环的节奏被拖慢。
- **注意力会被稀释。** 这是最反直觉、也最要命的一条:把关键信息埋在一大堆无关内容里,**模型反而更容易看漏**——尤其是埋在长上下文中段的信息(业界俗称「lost in the middle」,迷失于中段;这个现象最早由 Liu 等人 2023 年的同名研究系统地测出来)。桌子铺得越满,真正重要的那张纸越容易被淹没。Manus 之所以要在上下文末尾**反复重写一份 `todo.md` 清单**(见 3.5),正是为了对抗这种「读着读着就把目标读丢了」的毛病。

> 关键心态:**上下文窗口不是「越满越好」的仓库,而是「越精越好」的工作台。** 一个塞满噪声的大窗口,常常打不过一个只放了对味信息的小窗口。

所以真正的问题从来不是「窗口够不够大」,而是**「有限的台面上,此刻该摆哪几张纸」**。这就逼出了后面四节的全部手法。

### 3.2 短期记忆 vs 长期记忆:桌上的 vs 抽屉里的

有了「台面有限」这个前提,记忆就自然分成两类,用一个大白话的比喻:

- **短期记忆(short-term memory)= 此刻摊在工作台上的东西。** 也就是**已经放进上下文窗口的那部分历史**——最近的对话、刚读的文件、刚跑出的报错。它的专业说法是「上下文内(in-context)」。特点是**取用零成本**(模型直接就能看见),但**容量极其有限**,且**一压缩就可能丢**(见 3.4)。
- **长期记忆(long-term memory)= 存在抽屉、书架、档案室里的东西。** 也就是**存在上下文之外**的信息——外部数据库、向量库、磁盘上的文件。它的特点正好相反:**容量近乎无限,但要用之前必须先「取回来」**(检索、读文件),取回这个动作有成本、也可能取错。

这套划分并不是工程师拍脑袋想的,它对着**人类的记忆**照抄。Lilian Weng 在 2023 年那篇影响很大的《LLM Powered Autonomous Agents》里,把一个 agent 拆成三大件——**规划(Planning)、记忆(Memory)、工具使用(Tool Use)**,以 LLM 作大脑。而在「记忆」这一件的**内部**,她又按人类认知分了三层:

| 人类记忆 | agent 里对应什么 | 在我们修 bug 场景里 |
|---|---|---|
| 感觉记忆(转瞬即逝的原始感官) | 输入的向量表示(embedding) | 把 issue 文本、代码转成向量的那一瞬 |
| 短期 / 工作记忆 | 上下文窗口里的内容 | 手头正在读的这几个文件、刚才的报错 |
| 长期记忆 | 外部向量库,按需检索 | 整个仓库、历史 commit、以前踩过的坑 |

(顺带纠正一个常见的以讹传讹:Weng 的顶层拆分是 **Planning / Memory / Tool Use 三件**,不是四件——「Action / 行动」不是她列的独立顶层组件;上面这三层记忆,是「Memory」这一件**内部**的进一步细分,别把它和顶层三件混为一谈。)

> 一句话:**短期记忆快而小,长期记忆大而慢。agent 的记忆工程,本质是一门「什么时候把抽屉里的东西搬上桌、桌上又该收哪些进抽屉」的调度学问。** 3.3 到 3.5,就是这门调度学的三大招。

### 3.3 RAG:用不着的先收着,要用了再翻上来

第一招,专治「长期记忆怎么按需搬上桌」。

先说它解决的痛点。修那个 bug,你需要仓库里**相关的那几段代码**——但仓库有上百万行,既塞不进模型的权重(那是训练时才能干的事,而且成本高、更新难),也塞不进上下文窗口。怎么办?**别全搬,只搬相关的。** 这就是 **RAG(检索增强生成,retrieval-augmented generation)**:

> **RAG = 先把海量知识存进一个「按意思查」的库,用的时候只捞出跟当前问题最相关的几片,拼进上下文,再让模型作答。**

它的四步流水线是这样转的:

```
① 离线建库
   仓库文档 ──切块──▸ 每块转成 embedding(向量)──▸ 存进向量库
                            (一串数字,代表「这段话的意思」)

② 在线检索(每次要用时)
   当前问题 ──转成 embedding──▸ 在向量库里找「意思最接近」的几块
                                          │
                                          ▼
                    捞出 top-k 相关片段 ──拼进上下文──▸ 喂给模型作答
```

关键在第①步的 **embedding(嵌入向量)**:它把一段文字压成一串数字,让「意思相近」的文字在数字空间里也**离得近**。于是「查相关」就变成了「找最近邻」——你问「登录为什么报错」,库里那段讲 `auth.py` 的代码,哪怕一个字都没和你重合,也能因为「意思接近」被捞出来。

这就带出一个本篇绕不开的经典取舍——**RAG vs 长上下文**:

| | RAG(检索,只搬相关的) | 长上下文(全塞进窗口) |
|---|---|---|
| 能装的知识量 | 近乎无限(库多大都行) | 受窗口上限死死卡住 |
| 每次调用成本 | 低(只拼几片) | 高(整摊都在烧钱) |
| 知识更新 | 改库即可,不用动模型 | 每次都得把全量重塞 |
| 风险 | **检索错 / 捞进噪声**(见第 5 节) | 注意力稀释、贵、慢 |

没有谁全面碾压谁:窗口越做越大,一部分原本非 RAG 不可的场景,如今「直接全塞进去」反而更省心;但只要知识量远超窗口(比如整个代码仓库、整个公司文档),**RAG 依然是把「放不下的知识」喂给模型的主力手段**。

> 提醒一个界限(见第 ① 篇):**一次性的 RAG 是「检索一次 + 生成一次」的工作流,还不是 agent。** agent 的不同在于——它能**反复检索**、把检索和别的工具混着用、在循环里一步步逼近目标。修 bug 时它可能先检索「哪里处理登录」,读完发现线索指向别处,**再检索一次**,如此往复。RAG 是 agent 的一件趁手工具,不是 agent 本身。

### 3.4 上下文工程与压缩:把桌子当稀缺资源来管

第二招,专治「桌上的短期记忆总有一天会爆」。

就算有了 RAG 只搬相关的,循环转个几十步,历史照样会把窗口顶穿。这时不能任由它爆,得**主动经营这张桌子**——这项手艺,业界叫**上下文工程(context engineering)**:精心决定每一轮到底往窗口里放什么、放多少、以什么顺序放。它的一个核心动作,是**上下文压缩(context compaction)**:

> **压缩 = 当历史快撑爆窗口时,把陈旧、冗长的部分裁掉或总结成一小段精华,腾出台面给接下来的活。**

具体怎么裁?一般是「先便宜后昂贵」地来:**先丢**那些早就没用的原始工具输出(比如二十步前那次 `grep` 刷屏的几百行,现在完全用不上了)——这一步几乎无损、也几乎免费(丢掉的东西回头需要照着工具再取一次即可);实在还不够,**再总结**——让模型把一大段旧对话浓缩成几句话的摘要。总结这步是**有损且要花钱**的(它本身也得调一次模型,而且推理和对话细节一旦被压掉就取不回来了),所以通常留到最后才用。

为什么生产级(production)的 agent 都把上下文当**稀缺资源**来抠?因为前面 3.1 那三条账全在这儿汇合:**窗口有限、越长越贵、越满越蠢**。谁把这张桌子经营得干净,谁的 agent 就跑得又长又稳又省。Manus 团队甚至把「**KV-cache 命中率**」(一个衡量「上下文前缀有多稳定、能被缓存复用」的指标)称为他们**最重要的生产指标**——因为在他们的观察里,同样是 Claude Sonnet 的输入 token,**命中缓存与否,价格能差约 10 倍**($0.30 对 $3.00 每百万 token)。上下文管得好不好,直接写在账单上。

> 一句话:**上下文工程,是把「模型该看见什么」当成一等一的工程问题来认真对待。** 至于 harness 层面**具体怎么实现**这套压缩流水线(分几级、每级干什么、什么时候触发),是一门更细的手艺,**留到第 ⑧ 篇讲透**;这里你只要建立起「上下文是稀缺资源、要主动经营」这个直觉。

### 3.5 记忆文件与外部记忆:把关键信息落到纸上

第三招,专治「压缩会把重要的东西一起冲掉」和「换一轮会话就全忘了」。

短期记忆会被压缩冲走,长期的向量库又偏「检索式」、不适合放「我这次任务的进度和结论」。中间缺了一块:**一个稳定的、我自己能读能写的笔记本。** 于是有了**外部记忆文件**这一招:

> **把关键信息主动写到上下文之外的文件里,当作长期笔记;需要时再读回来。文件系统,成了 agent 一块「近乎无限、还不会被压缩误删」的记忆。**

这一招有两种典型用法,都能在真实产品里看到:

- **人写的长期须知——记忆文件。** 像 Claude Code 的 **`CLAUDE.md`**、以及跨工具通行的开放约定 `AGENTS.md` 这类文件,专门放「关于这个项目/这个人,agent 每次开工都该知道的事」——代码规范、常用命令、别碰哪些目录。据 Anthropic 官方文档,`CLAUDE.md` 会在**每次会话开始时读入**(嵌套子目录里的还会在 Claude 真正碰到那些文件时才懒加载)。它是**跨会话**的长期记忆:今天这轮结束、明天重开一轮,这份须知还在。(Claude Code 的完整记忆分层与它的 harness 设计,留到第 ⑫ 篇拆解。)

- **agent 自己写的工作记忆——文件即记忆。** Manus 把这条推到了极致,提出**「文件系统即记忆」**:任务中间产生的大块数据(网页正文、抓来的表格)不硬留在上下文里,而是**写到沙箱的文件系统**,上下文里只留一个「可还原」的指针(比如**只留 URL / 文件路径,丢掉正文**,回头要用照着路径再读回来)。这样既腾空了台面,又保证信息**一点没丢**。此外,它还在上下文末尾**反复重写一份 `todo.md`**,把「总目标和还没做完的步骤」一遍遍复述给自己听——用一份看得见的清单,对抗 3.1 说的「读着读着把目标读丢(lost in the middle)」。

回到我们的场景,这招的价值一目了然:修一个大 bug 往往要改好几个文件、跑好几轮测试。agent 完全可以在一份笔记里记下**「我已经改过 `auth.py` 和 `session.py`,`login_test` 已绿,还差 `logout` 分支」**——哪怕中途上下文被压缩、甚至会话中断重开,它读一眼这份笔记,就知道自己走到了哪、别再重复改一遍已经改好的文件。

> 关键直觉:**上下文是易失的短期记忆,文件是稳固的长期记忆。** 会被冲走的东西放桌上,要留住的东西落到纸上——这是 agent 干长活时最朴素、也最好用的一条纪律。

## 4. 关键心态:卡住长任务的,往往不是「不够聪明」,而是「记不住」

新手盯着模型的智商看,老手盯着上下文看。这是本篇最想让你换过来的一个视角:

> **一个 agent 做不完一件长任务,十有八九不是「模型太笨想不出来」,而是「它记不住」——上下文早就爆了、或者被压没了,它把自己前 20 步的成果忘得一干二净,在原地打转。**

想想那个失忆天才的比喻就懂了:再天才的人,如果每隔十分钟就失忆一次、还只准他看桌上那一小叠纸,他也做不成一件需要连续跟进几小时的复杂工作。**agent 干长活的真正瓶颈,常常在「记忆的带宽」,而不在「推理的深度」。** 这也正好呼应第 ① 篇那句反复出现的话——**模型强 ≠ agent 强**:同一个模型,配一套会经营上下文的 harness,和裸用它,干长活的能力差一大截。

所以当你的 agent 在长任务上翻车时,先别急着换更大的模型。先问三个上下文层面的问题:

- **它现在的桌上,还看得见目标吗?** 还是被一堆无关的工具输出淹没了(3.1)?
- **它有没有把该记的事落到笔记里?** 还是全指望易失的上下文、一压缩就丢(3.5)?
- **它检索回来的,是相关的信息,还是一堆噪声?**(3.3,下一节细说。)

> 一句话:**把上下文当成 agent 的「第一性约束」来设计,而不是事后才想起来的补丁。** 很多「模型不行」的锅,其实是「上下文没管好」在背。

## 5. 常被忽视的坑

上下文是最容易「看起来能跑、实际上悄悄坏掉」的地方——它不报错,只是让 agent 慢慢变傻。以下几处,新手几乎人人都栽过:

1. **上下文腐化(context rot)。** 循环转久了,窗口里会攒下大量陈旧、矛盾、失效的信息:早已被推翻的猜测、二十步前的过期报错、试错留下的死路。模型分不清哪些还作数,**被自己的旧垃圾带偏**。为什么致命:它不报错,只是让判断越来越飘。怎么避:主动清理和压缩(3.4),让过期信息尽早离场,别让它在桌上发酵。
2. **无脑塞满窗口。** 一看到「100 万 token 大窗口」就想着「全塞进去省事」。为什么致命:又贵、又慢,还会**稀释注意力**让模型看漏关键(3.1),常常打不过一个只放对味信息的小窗口。怎么避:把窗口当「越精越好」的工作台,而不是「越满越好」的仓库——**能不放的,就别放**。
3. **检索噪声(retrieval noise)。** RAG 捞回来的片段不总是对的——语义相近不等于真的相关,捞进一堆似是而非的内容,反而把模型**带向错误线索**。为什么致命:错误信息一旦进了上下文,模型往往当真,把假当真地往下推。怎么避:控制检索数量、给结果排序过滤、必要时让 agent 自己判断「这几片到底相不相关」再决定用不用。
4. **总结丢关键信息。** 压缩里的「总结」这一步是有损的——一句「已排查登录模块」可能把「其中 `token` 过期是根因」这条**唯一有用的线索**一起冲掉。为什么致命:被总结掉的信息**神不知鬼不觉地消失**,agent 后面完全意识不到自己丢了东西。怎么避:重要结论别只留在会被总结的对话里,**及时落到记忆文件**(3.5);总结策略上优先保留结论与决定,而非过程细节。
5. **把「有大窗口」当成「有记忆」。** 以为窗口够大就不用管记忆了。为什么致命:窗口只是「一次能看多少」,**跨会话、跨压缩地留住信息是另一回事**——重开一轮,桌上的东西全没了。怎么避:该跨会话留的,就写进 `CLAUDE.md` 这类文件,别指望窗口替你记。

## 6. 小结 & 下一步

这一篇你需要带走的:

- **模型是「无状态」的失忆天才:** 每次调用只看见喂进去的那叠纸(上下文),读完即忘;所谓「记忆」,是外面的 harness 每轮精心决定「给它看哪叠纸」拼出来的假象。
- **上下文窗口是 agent 最硬的约束:** 它是工作台,**有限、越长越贵越慢、越满越容易稀释注意力**;「大窗口 ≠ 无限记忆」。
- **记忆分两层:** 短期(桌上的 in-context 历史,快而小)+ 长期(桌外按需取回的,大而慢);记忆工程就是在两者间调度。
- **三大招:** RAG(只把相关知识检索上桌)、上下文压缩(裁剪/总结旧历史腾台面)、记忆文件(把关键信息落到 `CLAUDE.md`、`todo.md` 这类文件当长期笔记)。
- **心态:** 卡住长任务的常常是「记不住」,不是「不够聪明」——把上下文当第一性约束来设计。

但你可能已经注意到一个缺口:光把「该记的记住、该忘的忘掉」管好了,agent 就知道**先干哪一步、错了怎么回头、几个子目标怎么排序**吗?记忆解决的是「记得住」,还有一个「想得清」的问题没碰——那就是**规划(planning)**。
