## 0. 全景:承⑥,把「发动机」单独拆出来看

第 ⑥ 篇给了你一张 harness 的**七层栈**——从最外层的用户、接口,到里面的 agent loop、权限系统、工具、状态持久化,再到最底下的执行环境(shell / 文件系统 / 沙箱)。那一篇是「总纲」,让你看清 harness 一共有几个零件、各摆在哪一层。

这一篇,我们把栈里最靠近「引擎室」的那几层单独拎出来,拧开盖子看它转:

```
   七层栈(承⑥) ▸ 本篇聚焦真正驱动 agent 的那几层

   ┌─ 用户 ───────────────────────────────────┐
   ├─ 接口(CLI / SDK / IDE)──────────────────┤
   ├─ Agent loop ─────────────────┐            │  ◀── 本篇 3.1 主循环
   ├─ 权限系统 ──────────────────  │            │      (细节留⑨篇)
   ├─ 工具(注册表 + 分发)────────┤  本篇焦点  │  ◀── 本篇 3.2 系统提示 / 3.3 工具分发
   ├─ 状态与持久化 ───────────────┘            │  ◀── 本篇 3.4 会话状态
   ├─ 执行环境(shell / 沙箱)─────────────────┤
   └───────────────────────────────────────────┘
      + 每轮喂进循环的「系统提示」贯穿在 Agent loop 之上
```

具体说,本篇只讲四件事:**agent loop(主循环)**、**每轮喂进循环的系统提示(system prompt)**、**把模型意图落地执行的工具注册与分发(tool dispatch)**,以及给它们记账的**会话状态(session state)**。权限的细节留给第 ⑨ 篇,上下文压缩留给第 ⑧ 篇,子代理留给第 ⑩ 篇——这里只在它们和引擎「咬合」的地方点一下。

> 我们继续用第 ① 篇那个跑腿的例子:**让 agent 去修某开源仓库里的一个 bug——读 issue → 找相关文件 → 改代码 → 跑测试 → 提 PR。** 这一篇,你要能想象出「agent 每转一圈,harness 内部到底动了哪几个零件」。

## 1. 是什么:harness 的核心引擎,就三样东西

一句话:

> **Harness 的核心引擎 = 一个 `while` 循环 + 每轮喂进去的系统提示 + 把模型吐出的 `tool_use` 落地执行的分发器。** 循环负责「一圈圈转」,提示负责「告诉模型它是谁、有哪些手」,分发器负责「把模型说的『我要用这个工具』真的执行掉」。

第 ② 篇你已经认识了这个转圈的**概念**(ReAct:想一步、做一步、看一步)。本篇要回答的是一个更「工程」的问题:**这个循环在真实的 harness 代码里,到底长什么样?** 答案会让不少人意外——

**它简单得离谱。** 剥掉花哨的外壳,一个生产级 agent 的主循环,核心就是几十行:组装上下文、调一次模型、看它有没有要用工具、有就执行、把结果塞回去、回到开头。没有神秘的「智能调度器」,没有复杂的状态机。真正的工程量,全砸在循环**周边**——提示怎么拼、工具怎么注册过滤、结果怎么截断回填、状态怎么落盘——而不是循环本身。

> 反直觉钩子:**你以为 agent 的「聪明」藏在某个精巧的控制算法里,其实那个控制算法就是一个傻乎乎的 `while` 循环。** 聪明是从「简单循环 × 好提示 × 好工具分发」里**涌现**出来的,不是循环自己有多复杂。这一点想通了,你对 agent 的祛魅就完成了一大半。

## 3. 核心内容:拆开引擎室

### 3.1 agent loop 的真身:一个傻循环 + 精心工程

先看**结构**。业界最被反复引用的参考分解,是《Dive into Claude Code》(VILA-Lab,arXiv 2604.14228)对 Claude Code 主循环的刻画。**官方文档**把它描述成三个揉在一起的阶段——**gather context(组装上下文)→ take action(动手)→ verify results(看结果)**,循环往复直到做完;而**那篇论文的源码分析**(注意:不是官方文档)进一步把实现命名为 `query.ts` 里的 `queryLoop()` 异步生成器,刻画为一个「ReAct 风格的 `while` 循环、把过程当事件流式吐出来」。把两种说法合起来,循环体大致是这样:

```
        ┌─────────────────────────────────────────────────┐
        │                                                 │
        ▼                                                 │
  ①组装上下文  ──▸ ②调模型 ──▸ ③解析模型输出               │
  (系统提示+历史                    │                       │
   +工具schema)                    ├─ 只有文本,没工具调用 ─▸ ④做完了,退出循环
                                    │
                                    └─ 有 tool_use 块
                                            │
                                            ▼
                                   ⑤权限检查(deny-first,⑨篇)
                                            │ 通过
                                            ▼
                                   ⑥按名字派发到 handler ──▸ ⑦执行(shell/改文件/抓网页)
                                            │
                                            ▼
                                   ⑧结果截断/回填成 tool_result
                                            │
                                            └───────────────────┘(回①)

        另一个出口 ▸ 撞上安全上限(迭代数/token 上限/显式完成工具)→ 强制退出
```

对着修 bug 的例子走一遍:第 1 轮,①把系统提示 + issue 内容组装好 → ②模型说「我要 `Grep` 关键词」→ ⑤⑥⑦真的去搜 → ⑧把命中的文件路径塞回历史 → 回到①;第 2 轮,模型看到路径说「`Read` 这个文件」……如此十几二十圈,直到某一轮模型**不再要工具、只输出一段「已修好并跑绿测试」的文本**——那就是循环的**自然终点**(出口④)。

这里有两个**工程上死活要有、概念课不会强调**的细节:

- **两个出口,缺一不可。** 出口④是「模型觉得做完了」(它不再发 `tool_use`);但模型可能永远觉得没做完(死磕一个跑不过的测试),所以必须有另一个出口——**安全上限**:迭代次数封顶、token 预算封顶、或提供一个显式的「完成」工具让它主动收工。没有这道刹车,循环会无限烧钱(这是第 ① 篇提过的 AutoGPT 老毛病,也是本篇第 5 节的头号坑)。
- **结果必须「有界」。** 第 ⑦ 步执行工具时,输出可能是几十万字符的日志。直接塞回上下文会瞬间撑爆窗口,所以 handler 收集输出时要**截断 / 溢出落盘**。以 Claude Code 为例:Bash 工具默认把输出截到 30,000 字符,通过环境变量 `BASH_MAX_OUTPUT_LENGTH` 最高放到 150,000(截至 2026 年中的官方文档)。这就是「循环简单、周边精工」的第一个具体证据。

> 一句话:**主循环是发动机的曲轴,简单、可靠、转得稳最重要;真正的马力藏在曲轴周围那一圈精密配件里。**

### 3.2 系统提示的组装:按优先级现拼,而且最好「早拼」

模型是**无记忆**的:每一轮调用,它眼前只有你这次喂进去的那段文字。所以「它是谁、有哪些工具、什么能干什么不能干」这些身份信息,必须**每一轮都重新告诉它一遍**——这就是**系统提示(system prompt)**的活。

harness 不是把一大坨死文本塞进去,而是像拼乐高一样,把系统提示**按优先级、按条件**拼出来。典型的几块(合成自 OpenDev 的 arXiv 2603.05344 与 Claude Code 的实现):

```
  系统提示 = 拼装器(prompt composer)按优先级挑选并拼接 ▸

  ┌ 身份段    │ "你是一个软件工程 agent……"          (总在)
  ├ 工具指引  │ 每个工具怎么用、什么时候用            (随注册的工具变)
  ├ 权限策略  │ "改文件前要先……"、当前是 plan 模式吗  (随权限模式变,⑨篇)
  └ 上下文指令│ 项目约定、CLAUDE.md/AGENTS.md 层级    (随项目/会话变)
```

「按条件」很关键:比如当前处于「只读的计划模式」,权限策略段就会拼进一句「你现在不能写文件」;换了个只给读工具的子代理,工具指引段就只列读工具。**同一份拼装器,靠不同的输入,拼出不同的系统提示**——这个思想在 3.5 节会升华成一个漂亮的架构决策。

这里有个容易被忽略、但工程上很讲究的点:**什么时候拼?** OpenDev(arXiv 2603.05344)的做法是——**在构造期就把系统提示与工具 schema 建好(eager construction,急切构造)**,而不是等第一次调模型时才现拼。为什么?两个实打实的好处:

1. **消除首调延迟。** 提示和工具 schema 提前备好,第一次请求不用临时组装,响应更快。
2. **消除 MCP 发现竞态。** 外部工具(MCP,见第 ③、⑪ 篇)的发现是异步的;如果边跑边发现、边拼提示,就可能出现「模型已经在用某工具、而它的 schema 还没就位」的竞态。构造期一次性把该建的都建好,这类时序 bug 从根上没了。

> 一句话:**系统提示不是一段静态文案,是 harness 每一轮现场「拼」出来的产物;而「尽量早拼、拼一次」是个便宜又值钱的工程选择。**

### 3.3 工具注册与分发:一张注册表,模型点名、harness 执行

模型只会**生成文字**,它碰不到你的文件系统和 shell(这是安全的地基,第 ⑨ 篇细讲)。它能做的,只是在输出里吐一个结构化的 `tool_use` 块:「我要调 `Edit`,参数是这个文件、这段改动」。**把这句『我要用』变成真的执行,是 harness 的工具分发器(tool dispatch)干的活。** 整条链路:

```
  ┌ 工具注册表(registry)─────────────────────────────┐
  │  Read → handler_read()   Bash → handler_bash()     │
  │  Edit → handler_edit()   Grep → handler_grep()  …  │
  └──────────────┬─────────────────────────────────────┘
                 │ ①把每个工具的 JSON schema 暴露给模型
                 ▼
            模型看到「我有这些工具、各自参数长这样」
                 │ ②模型吐出 tool_use{ name:"Edit", input:{…} }
                 ▼
  ③harness 按 name 在注册表里查到 handler_edit
                 │ ④执行 handler(权限检查后)
                 ▼
  ⑤把返回值按 tool_result 回填进历史 ──▸ 回到循环①(见 3.1)
```

三个要点:

- **schema 是模型的「工具说明书」。** 注册表把每个工具的名字、描述、参数结构(JSON schema)一并暴露给模型——模型正是靠这份说明书,才知道自己「有哪些手、每只手怎么用」。工具设计得好不好(名字直不直白、参数清不清晰),直接决定模型会不会用错,这也是第 ③ 篇 ACI(代理-计算机接口)的主题。
- **分发就是「按名字查表 + 调函数」。** 没有魔法:模型说 `name:"Edit"`,harness 就在注册表里找到 `Edit` 对应的 handler,把 `input` 传进去执行。这一步之所以简单,恰恰是因为复杂度被推到了别处(权限、截断、错误处理)。
- **子代理拿到的是「过滤后的」schema。** 这是个漂亮且重要的模式:注册表不必把全部工具都给每个 agent。子代理(subagent)只会拿到它**该用的那几个工具**的 schema——这叫 `allowed_tools` 过滤。OpenDev(arXiv 2603.05344)把这一点用到了极致:让「计划者(Planner)」子代理的 schema 里**只有只读工具**,于是「写文件」这件事对它**从架构上就不可能**——因为模型根本没见过写工具的定义,压根无从调用。**约束从『运行时检查』前移到了『schema 层』**。这个思路的完整展开在第 ⑩ 篇。

> 一句话:**注册表是 harness 递给模型的一张「点菜单」,分发器是照单上菜的服务员;而『给谁看哪半张菜单』本身,就是一种最干净的安全设计。**

### 3.4 会话与状态:agent 的「只追加记账本」

循环转一圈,历史就长一截。这些历史存哪、怎么存,决定了 agent 能不能**中断后接着干、能不能回溯、能不能复盘**。

主流做法是把每一轮(用户消息、模型输出、工具结果)**只追加(append-only)**地写进一个转录(transcript)。Claude Code 的实现:会话写成 **JSONL 文件**,落在 `~/.claude/projects/` 下,顺序写入、**从不回改**(截至 2026 年中的官方文档;arXiv 2604.14228 强调其 append-only 特性)。OpenHands(All Hands AI 的开源 SWE agent 平台,前身 OpenDevin——注意它和上面 3.2 / 3.3 讲的 OpenDev 是两个不同项目)走得更远,把整个设计做成**事件溯源(event sourcing)**:每个动作和观察都是一条不可变的类型化事件,存进一个 append-only 的 **EventLog**,**所有状态都靠重放这条日志推导出来**(arXiv 2407.16741 / 2511.03690)。

只追加带来三个几乎「白送」的能力:

```
  只追加转录(append-only)──▸ 白送三件套

   resume(恢复)│ 进程挂了/关机了,读回日志、从断点接着跑
   fork(分叉)  │ 从某一步岔开,试另一条路,不动原线
   rewind(回溯)│ 退回到某一步之前的状态(Claude Code:每次编辑都有快照,连按两下 Esc 回退)
```

代价也诚实:**日志会无限增长**——这正是为什么必须有上下文压缩(第 ⑧ 篇)来给「喂给模型的那一段」瘦身;而且 append-only 意味着**不好对历史做复杂查询**(想「查所有失败的工具调用」得自己扫)。

> 一句话:**转录是 agent 的记账本——每一步都记、只记不改。正因为它诚实又完整,agent 才敢中断、敢回头、敢复盘。**

### 3.5 一个类 vs 一堆角色类:OpenDev 的反类爆炸决策

最后讲一个把前面几节串起来的**架构决策**,漂亮到值得单列。

很多人设计 agent 系统时,会本能地按「角色」建一堆类:`PlannerAgent`、`CoderAgent`、`ResearcherAgent`、`ReviewerAgent`……角色一多,类就爆炸,还容易陷进多重继承的泥潭(这个共享行为放哪个基类?那个覆盖冲突怎么办?)。

OpenDev(arXiv 2603.05344)的选择恰恰相反:**只有一个具体的 `MainAgent` 类,行为完全由构造参数决定**。想要一个「只读的计划者」?不新建类,而是用 `allowed_tools=只读工具 + 计划提示覆盖` 去**构造**同一个 `MainAgent`。想要一个「专注写代码的执行者」?换一组 `allowed_tools` 和提示再构造一次。

```
   传统:一堆角色类(类爆炸)          OpenDev:一个类 × 不同构造参数

   PlannerAgent  ┐                     ┌ MainAgent(allowed_tools=只读, prompt=计划)
   CoderAgent    ├ 各自维护、           │
   ResearcherAgt ├ 共享行为难放、  ▸    ├ MainAgent(allowed_tools=全量, prompt=执行)
   ReviewerAgent ┘ 继承打架            └ MainAgent(allowed_tools=读+web, prompt=调研)
                                          ▲ 同一个类,行为 = 构造参数的函数
```

看出来了吗?这正是 3.2(提示按条件拼)、3.3(schema 按 `allowed_tools` 过滤)两节的自然结论:**既然「一个 agent 是什么样」完全由『喂给它的提示』和『给它看的工具 schema』决定,那就没必要为每种角色单开一个类——把差异塞进构造参数就够了。** 于是「计划者不能写文件」不是某个 `PlannerAgent` 类里的一段检查代码,而是「构造时没给它写工具」这个事实(呼应 3.3 的 schema 层约束)。

无独有偶,另一个开源平台 **OpenHands** 在其 V1 SDK 里把「身份即数据」推到另一头:**agent 是无状态的不可变规格,所有可变状态集中在一个 `ConversationState` 里**(arXiv 2511.03690)——config 在构造时冻结,消灭了共享可变状态这类 bug。两条路殊途同归:agent 的身份归数据,可变状态归一处。

> 一句话:**「一个类 + 构造参数」胜过「一堆角色类」,是因为 agent 的身份本就是数据(提示 + 工具清单),不是代码。把身份当数据,类爆炸和继承地狱就一起消失了。**

## 4. 关键心态:聪明是涌现的,不是循环本身复杂

把这一篇的东西压成一句心法:

> **Agent 的「聪明」,几乎全从「简单循环 × 好的系统提示 × 干净的工具分发」里涌现,而不是来自循环本身有多精巧。** 曲轴越简单越好,马力靠的是周边配件的打磨。

这解释了两件常让新人困惑的事:

- **为什么 Claude Code 敢说自己的循环「刻意简单」?** 因为工程量本就不该压在循环上。那篇 arXiv 的社区源码估算给了个极端说法:**约 1.6% 的代码是 AI 决策逻辑,约 98.4% 是运维基础设施(harness)**——注意这是**社区对反编译源码的估算、不是官方指标**(arXiv 2604.14228 转述,方法未完全公开),别当精确数字用,但方向对得刺眼:大头永远在「循环周边」。
- **为什么同一个模型换个 harness,成绩天差地别?** 因为决定成败的,恰恰是这些「周边」——提示拼得好不好、工具 schema 干不干净、状态可不可靠。据 Claw-SWE-Bench(arXiv 2606.12344),在固定用 GLM 5.1 做后端时,一个**简陋适配器只拿到 19.1% Pass@1**(350 分之 67,约 69% 的补丁根本打不上),而一个**完整适配器拿到 73.4%**(350 分之 257,补丁失败率 <1.5%)——**模型一字没变,只换了 harness**。这正是第 ⑥ 篇「harness 常比模型更能决定成败」的硬证据。

一旦你把 agent 看成「一个傻循环 + 一堆精工周边」,你调 agent 的直觉就对了:**先别急着换更强的模型,先去看你的提示拼对没、工具给多了没、结果截断了没、状态存住了没。**

## 5. 常被忽视的坑(引擎室里最容易炸的几处)

1. **循环没有安全上限,无限打转烧钱。** 只靠「模型觉得做完了」当唯一出口是致命的——模型可能永远觉得没做完(死磕一个过不了的测试)。**必须有第二个出口**:迭代数封顶、token/预算封顶、或一个显式的「完成」工具。这是 AutoGPT 时代最惨痛的教训,也是 3.1 里那第二个出口存在的全部理由。别信「它应该会自己停」。

2. **系统提示每轮重复堆叠,悄悄吃光上下文。** 系统提示每一轮都要重发(模型无记忆),但如果你在提示里无脑追加「本轮新指令」「上一轮的提醒」,几十轮后光是系统提示就占掉半个窗口。**提示要按条件『拼』、不是按时间『堆』**(见 3.2):每轮从干净的模板重新拼,而不是在旧提示后面接龙。

3. **工具 schema 太多,撑爆上下文还把模型搞晕。** 每个工具的 schema 都占 token,还稀释模型注意力——给一个只需要读文件的子代理挂上 40 个工具,既浪费窗口又诱发误用。**按需过滤(`allowed_tools`)、按需发现(MCP 懒加载,见第 ⑪ 篇)**:只给它该看的那半张菜单(见 3.3)。Claude Code 的 MCP 工具默认延迟加载、用到才拉,就是这个道理。

4. **状态不持久,一崩全丢、无法恢复。** 如果转录只活在内存里,进程一挂、机器一重启,几十轮的进展灰飞烟灭,只能从头再来。**把每轮 append-only 落盘(JSONL / EventLog)**,resume/fork/rewind 才有可能(见 3.4)。别把「记账本」放在会断电的地方。

5. **把控制流硬编码成脆弱的状态机。** 一看到「计划模式 ↔ 执行模式」,就本能地写一个 `enter_plan()/exit_plan()` 的状态机——状态一多,转移条件就爆炸,边界一乱就出 bug。**能用『构造参数 + schema 过滤』表达的约束,就别写成运行时状态机**(见 3.5):让「计划者不能写」变成「它没有写工具」这个事实,而不是一段随时可能漏判的 `if`。约束前移到 schema 层,比运行时检查更难被绕过。

## 6. 小结 & 下一步

这一篇你要带走的:

- **Harness 的核心引擎 = 傻 `while` 循环 + 每轮现拼的系统提示 + 照单执行的工具分发器 + 只追加的会话状态。** 四个零件,咬合成 agent 的发动机。
- **主循环刻意简单**:组装上下文 → 调模型 → 解析 → 有 `tool_use` 就(权限检查后)派发并回填 → 回到开头,直到「无工具调用的终稿」或撞上安全上限。两个出口,缺一不可。
- **系统提示按优先级/条件现拼,且最好在构造期就建好**(eager construction),以消首调延迟与 MCP 发现竞态。
- **工具靠注册表暴露 schema、靠名字分发**;子代理只拿过滤后的 schema——**约束前移到 schema 层**是最干净的安全设计。
- **状态只追加(JSONL / EventLog)**,白送 resume/fork/rewind;代价是无限增长,得靠压缩来救。
- **一个 `MainAgent` 类 + 构造参数**(OpenDev)打败「一堆角色类」——因为 agent 的身份本就是数据(提示 + 工具清单),不是代码。
- 心态:**聪明是从简单循环里涌现的**;调不好 agent,先查周边(提示/工具/状态),再谈换模型。

那么,3.4 里埋的那个雷——**只追加的转录会无限增长,喂给模型的那一段迟早撑爆上下文窗口**——怎么办?这正是下一篇的主题。
