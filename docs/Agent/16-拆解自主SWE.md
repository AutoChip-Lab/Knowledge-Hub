## 0. 全景:从「你盯着它一步步干」到「派出去、它自己跑」

⑫⑬ 拆的 Claude Code 和 Codex,有一个共同的姿势:**你坐在旁边看着它干。** 它读一个文件、你看一眼;它想跑一条命令、弹窗问你;它改错了、你按 Esc 拦下来。人始终在环里(human-in-the-loop),一步一个脚印。

这一篇换一个流派——**自主 SWE agent(autonomous software-engineering agent)**:你把一个 GitHub issue 甩给它,说一句「把这个 bug 修了」,然后**关掉窗口去喝咖啡**。它自己去克隆仓库、翻代码、写方案、改文件、跑测试、看报错、再改、再跑……几十步甚至上百步,最后给你**开一个 pull request** 等你 review。整个过程可以完全没有人盯着。

```
   ⑫⑬ 陪跑型(human-in-the-loop)          ⑯ 长跑型(autonomous / long-horizon)

   你 ──▶ agent 走一步 ──▶ 你看/批             你 ──▶ 「修好 issue #123」
        ▲                │                          │
        └────── 反馈 ────┘                          ▼
   人在每一步的环里                          agent 自己跑几十~上百步
   随时可打断、逐步授权                              │
   目标:辅助你写代码                                ▼
                                            开一个 PR ──▶ 你只 review 结果
                                            人只在「进」和「出」两端
```

这不是「谁更好」的问题,是**两种形态**。陪跑型适合你在场、边想边改;长跑型适合「批量处理积压 issue」「让它去啃一个你不想碰的脏活」。这一篇拆三家长跑代表,而且它们三个**灵魂完全不同**:一个是学术界证明「界面比模型重要」的 **SWE-agent**,一个是开源社区的事件流引擎 **OpenHands**,一个是把整套东西托管到云上、按「算力单位」收费的商业产品 **Devin**。

> 读这一篇的正确姿势:**别记产品名,记它们各自赌了什么。** 同样是「让 agent 自主跑」,三家在「怎么让它跑得稳」这件事上,给出了三条截然不同的答案。

## 1. 是什么:长跑型 agent 的共同基因

先把三家的**共性**抽出来。一个「自主 SWE agent」,不管是谁做的,都长这样:

> **它是一个接下「整个软件工程任务」(而不是一次补全)、在沙箱里自己跑很久、以「解决 issue 并开 PR」为终点的 agent。** 关键词四个:**长程、高自主、跑沙箱、以 PR 收尾。**

逐个拆开这四个基因:

- **长程(long-horizon)**:它要连着走几十到上百步——读→改→测→再读→再改。这和「一问一答」是两个量级的难度,因为**错误会累积、上下文会漂移**(见 ④⑧)。长程能力,本质是「在几十步里不迷路」的能力。
- **高自主(high autonomy)**:目标是模糊的一句话(「修好这个 bug」),中间的规划、探索、试错**全由它自己决定**,不需要你逐步点头。这就把 ⑨ 讲的权限问题推到了极端:人不在场,谁来拦危险操作?
- **跑沙箱(sandbox)**:它要真的跑命令、真的改文件、真的执行测试——所以必须有一个隔离的执行环境(Docker 容器或云端 VM),既给它一个真环境,又不让它把你的机器搞坏。
- **以 PR 收尾(PR as output)**:终点不是「一段建议」,是一个**可以直接 review、可以直接 merge 的 diff**。这决定了它必须会用 git、会跑测试、会自证「我这个改动是对的」。

这四条是共性。但三家的**灵魂**——也就是「用什么招让它在这么长的自主跑动里不崩」——完全不同,这是本篇下半场的重头戏,先剧透:

| 家 | 一句话灵魂 |
|---|---|
| **SWE-agent** | 给模型一套**为模型设计的界面(ACI)**——不是给人用的 shell,是给 LLM 用的。 |
| **OpenHands** | 把每一步都记成**事件流(EventLog)**,动作**就是可执行代码(CodeAct)**。 |
| **Devin** | 每个任务**开一台云端 VM**,全托管、异步跑、按算力单位(ACU)收钱。 |

## 2. 一次任务在它体内怎么流动

三家实现不同,但「一个 issue 进去、一个 PR 出来」的**生命周期骨架是通用的**。先把这条通用流程看透,再点三家差异。设想你丢进去一句:

> 「issue #123:某个边界输入会让解析器崩溃,帮我修了。」

一个自主 SWE agent 内部大致这样走一大圈:

```
  issue #123
     │
     ▼
① 接任务      读 issue 文本 + 仓库地址,provision 一个沙箱(Docker / 云 VM),把 repo clone 进去
     │
     ▼
② 探索仓库    找相关文件:grep 报错信息 / 读目录结构 / 定位到 parser.py
     │        └─(污染上下文的脏活可外包给子代理去读,只回摘要 —— 见 ⑩)
     ▼
③ 规划        「先写一个能复现的测试 → 定位越界处 → 加边界判断 → 跑全套测试」
     │
     ▼
④ 改代码      打开 parser.py,定位到第 N 行,做一次最小编辑
     │
     ▼
⑤ 跑测试      在沙箱里执行 pytest ──▶ 读输出
     │
     ├──❌ 还挂 ──▶ 读报错 → 回到 ② 或 ④,修正假设,再来一圈
     │             (一圈圈 observe→think→act,自己纠错,可链几十步)
     ▼
   ✅ 全绿
     │
     ▼
⑥ 收尾提 PR   git diff 成 patch → 写 commit / PR 描述 → 开 pull request 等人 review
```

注意 ⑤ 那个**回环**——它才是「长跑型」的心脏。陪跑型 agent 每走一步等你确认;长跑型是**自己读测试报错、自己决定下一步**,这个 observe→think→act 的循环(② 篇的核心循环、⑦ 篇的循环工程)会自主地转几十上百圈,直到测试全绿或预算耗尽。

三家在这条骨架上的**分野**,恰好落在不同环节:

- **② 探索 / ④ 改代码这一层**,SWE-agent 的独门功夫最抢眼:它不给模型裸 shell,而是给一套「翻页看文件、有边界的搜索、改完自动 lint」的**定制命令**(见 3.1)。
- **整条流水的记账方式**,OpenHands 最有特色:每个动作和每个观察都被记成一条不可变**事件**,状态靠重放事件流还原(见 3.2)。
- **① 接任务 / ⑥ 收尾这两端**,Devin 做得最重:每个任务一台**全新云 VM**,还多了一个「先出方案给你看、你点头才动手」的 Ask 模式(见 3.3)。

> 一句话:**长跑型 agent 的本事,全在「⑤ 那个回环里不崩」。** 它怎么读文件、怎么记状态、怎么隔离环境、怎么在人不在场时不失控——就是下面三家各显神通的地方。

## 3. 拆开看三家

现在把三家分别搬上手术台。每家都对着上面那条生命周期,看它在哪一环下了重注。

### 3.1 SWE-agent(普林斯顿):灵魂是「为模型设计的界面」

**它是谁**:SWE-agent 出自**普林斯顿大学**(John Yang、Carlos Jimenez、Ofir Press 等),论文《SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering》发表在 **NeurIPS 2024**(arXiv 2405.15793)。它是最早证明「一个 LLM 能自主修真实 GitHub issue」的开源系统之一,也是学术界研究这个问题的**基线**。(截至 2026 年中,主仓库已进入**维护模式**,官方推荐用更简单的后继者 mini-swe-agent——这点后面讲。)

它的招牌不是某个模型,而是一个概念——**ACI(agent-computer interface,代理-计算机接口)**,回扣 ③ 篇讲工具时埋的伏笔。

**先给直觉**:你把一个 LLM 扔进裸 Linux shell,它会干什么蠢事?它会 `cat` 一个一万行的文件——瞬间把上下文窗口撑爆;它会 `grep` 一个太常见的词——刷屏几千行匹配;它会 `vim` 进入一个交互式编辑器——然后卡死,因为它根本不会退出 vim。**shell 是为人设计的,人有眼睛能滚屏、有肌肉记忆按 `:wq`。LLM 没有。**

ACI 的核心主张就一句话,也是 SWE-agent 最值得记住的思想:

> **LLM 是一类全新的「终端用户」,它需要为它专门设计的界面——就像人类需要 IDE 一样。** 别让模型去适配给人用的工具,而要为模型重造工具。

具体这套 ACI 长什么样(都是装到容器 `PATH` 上的 shell 命令/脚本):

- **带窗口的文件查看器**:`open` / `goto` / `scroll_up` / `scroll_down` / `create`。它**默认一屏只显示约 100 行**(官方原话「显示 100 行时效果最好」),`goto` 会把目标行放在窗口约 1/6 处(源码里 `offset_multiplier = 1/6`),滚动重叠行数可配置。为什么要「翻页」?就为了**不让模型一次性把大文件塞进上下文**——强制它增量地看。
- **有边界的搜索**:`search_dir` / `search_file` / `find_file`。妙处在 `search_dir` **当匹配文件超过 100 个时会拒绝**(源码 `if [ $num_files -gt 100 ]`),逼模型把查询写得更精确,而不是被几千条匹配淹死。
- **改完自动 lint 并回滚**:`edit <起>:<止> … end_of_edit`。每次编辑后跑一个 flake8 子集(`--select=F821,F822,F831,E111,E112,E113,E999,E902`,查未定义名、重复参数、缩进、语法等)。关键设计:**只有「新引入」的错误才触发自动回滚**——改动前就存在的老错误会被行号平移、不算在模型头上。这样一次手滑的坏编辑不会污染后面所有步骤。
- **提交哨兵**:`submit` 命令吐出一个特殊标记 `<<SWE_AGENT_SUBMISSION>>`,循环检测到它就结束,`git diff` 直接变成最终 patch。

底层的执行是一个叫 **SWE-ReX** 的独立沙箱包,支持 Local / Docker / Podman / AWS Fargate / Modal 等多种后端,agent 代码在哪个后端都一样——这让它能在云上**大规模并行**跑评测。成本控制上,它不数「步数」而是设**每实例花费上限**(`per_instance_cost_limit` 默认 $3.00),因为不同模型走一步花的钱天差地别,**用钱当统一货币**才能横向比较。

**为什么 ACI 重要?一个数字**:论文里做了消融——同一个 GPT-4,配上 ACI 比裸 Linux shell 基线**在 300 个 issue 子集(SWE-bench Lite)上多解出 10.7 个百分点**。而最初的头条成绩是 **GPT-4 Turbo 在完整 SWE-bench 测试集上 12.5% pass@1**(准确说 12.47%,即 2294 题里对 286 题),而在此之前非交互式 RAG 方案的最好成绩只有 **3.8%**。**注意:换个测法、换个数据集(Lite / Verified),分子分母都不一样——这正是第 5 节要讲的「SWE-bench 数字横向不可比」。**

> 反直觉的点:**让 agent 变强,常常不是换个更强的模型,而是给同一个模型换一套更好的界面。** SWE-agent 用一个消融实验把这句话钉死了——「界面即产品」。这也是为什么 ③ 篇要专门讲「工具是为模型设计的接口」。

**后继者 mini-swe-agent**:有意思的是,普林斯顿(与斯坦福)那批做 SWE-bench / SWE-agent 的人后来把这套哲学**反着做了一遍**。mini-swe-agent 只有约 100 行 Python,**没有任何定制工具、只用裸 bash**、每个动作用无状态的 `subprocess.run` 执行,却在现代强模型上拿到 **>74% 的 SWE-bench Verified**。寓意很深:**当底座模型足够强,精心设计的脚手架收益递减**——2024 年非 ACI 不可的活,2026 年一百行 bash 就够了。这条「由繁入简」的弧线,本身就是一堂关于「模型 vs 脚手架」的课。

### 3.2 OpenHands(All Hands AI,原 OpenDevin):事件流 + 动作即代码

**它是谁**:OpenHands 是一个 **MIT 许可**的开源 agent 平台,由 **All Hands AI** 公司维护。它的前身叫 **OpenDevin**,是 2024 年初社区为对标 Cognition 那个闭源的 Devin 而攒起来的,同年底改名 OpenHands(平台论文发在 **ICLR 2025**,arXiv 2407.16741)。截至 2026 年中,GitHub 上约 **7.9 万 star**,既能自托管,也有 OpenHands Cloud 托管版;通过 LiteLLM 支持 100+ 家模型,**不绑定任何一家**。(仓库当前定位已从「Devin 克隆」演进为「面向编码 agent 的自托管开发者控制中心」。)

它有两个招牌机制,一个管**记忆**,一个管**动作**。

**招牌一:事件溯源(event sourcing)的 EventLog。** OpenHands 把 agent 干的每一件事——每个动作(action)、每个观察(observation)——都记成一条带类型的**不可变事件**,追加进一个 **EventLog**;整个会话的状态**不是存在某个变量里,而是靠重放这串事件还原出来的**。

先给直觉:普通 agent 的「记忆」是一团活的可变状态,进程一崩就没了。OpenHands 反过来——**它只有一本永远只追加、从不修改的流水账**。这本账带来三个直接好处:

- **随时暂停 / 恢复**:任何时刻停下,之后重放事件就能精确回到原状态。
- **确定性重放(deterministic replay)**:同一串事件重放出同一个结果,便于调试「它当时到底怎么想的」。
- **天然审计**:每一步都留痕,谁在什么时候干了什么一目了然。

代价是这本账会**无限增长**——所以需要一个 **Condenser(`LLMSummarizingCondenser`)**,在日志太长时把旧事件总结压缩成一条摘要事件(据其 SDK 论文,能把 API 成本压到约一半、且宣称「不损失表现」——但这句「不掉点」是厂商自报、未指明测的是哪个基准,当参考看)。这正是 ⑧ 篇「长任务会撑爆上下文」的一个具体工业答案。(这条设计和 ⑫ 篇 Claude Code 的「只追加 JSONL 会话」殊途同归——都拿「最笨的存储」换「最强的可恢复性」。)

**招牌二:CodeAct——动作即可执行代码。** 大多数 agent 的动作空间是一堆固定的 JSON 工具:调 `read_file`、调 `run_command`……OpenHands 的核心 **CodeActAgent** 反其道:它的动作**本身就是一段可执行代码**——一段 Python(`IPythonRunCellAction`)或一条 bash(`CmdRunAction`),丢进沙箱直接跑。

为什么这样更强?因为**代码天生会组合**:模型可以在一个动作里写循环、写条件、把几步串起来、甚至临时定义一个函数当工具用,还能读到报错后**自我调试**。CodeAct 那篇论文(arXiv 2402.01030)报告:相比固定 JSON 工具的基线,在 17 个 LLM 上成功率最高提升约 **20%**。这就是 ⑬⑭ 反复出现的「代码是最通用的动作空间」这一思潮的源头之一。

其余组件对着生命周期扫一遍:

- **Runtime / 沙箱**:一个 Docker 运行时,里面装了 bash、Jupyter/IPython、一个浏览器和 VSCode,由容器内的一个「动作执行服务器(Action Execution Server)」统一调度。V1 架构里**沙箱是可选的**(本地优先,需要时再上远程)——这是从 V0「强制 Docker、导致本地和生产环境不一致」踩坑后改的。
- **Skills / 微代理(microagents)**:把「某个仓库的专属知识」写成 Markdown 丢进 `.openhands/microagents/`,`repo.md` 总是加载,带关键词触发的文件在消息里出现触发词时才激活。**把领域知识外置**,而不是硬编进 prompt——和 ⑫ 篇的 CLAUDE.md 是同一类思想。(现在主要叫 Skills,旧路径为兼容保留。)
- **多代理委派(delegation)**:`CodeActAgent` 可以发一个 `AgentDelegateAction` 把子任务甩给专门的 `BrowsingAgent`(浏览专家),后者干完回一个 `AgentDelegateObservation`。这是 ⑩ 篇「通才 orchestrator + 专才 subagent」最干净的一个活样板。
- **安全闸门**:`SecurityAnalyzer` 给每个动作打风险分(低/中/高/未知),`ConfirmationPolicy` 决定要不要在执行前暂停等人确认——低风险自主、破坏性操作才叫人,正好回应 ⑨ 篇。

**成绩(据其 SDK 论文,截至 2026 年中)**:配 Claude Sonnet 4.5,在 SWE-bench Verified 上 **72.8%**(同论文里 GAIA 67.9%)。(注意:它仓库 README 上还挂着一个「77.6」的徽章,但没标模型、日期、方法,和论文里的 72.8% 也对不上,**当营销数字看,别当可引用成绩**——这是核查笔记特意点出的。)

### 3.3 Devin(Cognition):每任务一台云 VM,全托管、按算力收费

**它是谁**:Devin 是 **Cognition** 公司做的**云端托管、自主 AI 软件工程师**,2024 年 3 月 12 日发布时被高调宣传为「第一个 AI 软件工程师」。它和前两家最大的不同:**闭源、商业、全托管**。你不部署它、也看不到源码,你在网页/Slack/Jira 里派任务,它在**自己的云上**跑完把 PR 给你。截至 2026 年中,Cognition 在 2026 年 5 月的一轮约 10 亿美元融资把估值推到 **250 亿美元(pre-money)/ 260 亿美元(post-money)**(据 TechCrunch,较半年前的 102 亿美元翻了一倍多),是这个赛道商业化最激进的一家。

**这里必须把「营销宣称」和「已验证能力」严格分开——这是拆 Devin 时最重要的纪律。**

先说**架构里已确认的事实**:

- **每会话一台云端 VM**:每个任务 provision 一台**全新、隔离、用完即弃**的沙箱 VM,里面自带 shell(bash)、基于 VS Code 的编辑器、和一个多标签浏览器,把 repo clone 进去。好处是天然并行、可复现、会话间不串数据;代价是有启动延迟、且**没法在你本地设备上测**(比如测手机 app)。
- **Ask-then-Agent 两段式**:先是 **Ask 模式**(只读:扫仓库、出一个带编号的方案,还标 **Confidence Scores** 绿/黄/红置信度),你 review/改完再授权进 **Agent 模式**(全自主执行)。这一步是专门对付「模糊 prompt」这个头号失败模式——**先花小钱探路、你点头了才花大钱干活**。
- **ACU 计费**:Devin 按 **Agent Compute Unit(ACU,代理算力单位)** 收费——一个把「VM 时间 + 推理 + 网络」打包的归一化单位,约等于 15 分钟活跃工作,Core/Pro 计划下约 **$2.25/ACU**。好处是每个任务成本可预测;代价是**比看原始 token 数更不透明**。
- **单 agent 架构**(不是多 agent 群):Cognition 有一篇立场鲜明的博客《Don't Build Multi-Agents》(Walden Yan),主张「多个独立 agent 协调会引入脆弱性」,所以他们把力气全押在**为一个连贯 agent 做上下文工程**上,只用「只读的辅助子代理」(行为像工具调用)。**特别注意:把 Devin 描述成「规划器+编码器+批评者+浏览器」几个专门子模型的说法是被驳斥的架构臆测**,别写。2026 年他们又加了一层「Managed Devins」——一个协调 Devin 调度多个子 Devin(各自一台 VM)并行干活再汇总,但这是**叠在单 agent 之上的编排层**,不是把核心改成了多 agent。
- **配套产品**:DeepWiki(把公开 GitHub 仓库变成可交互文档,也是 Devin 上手代码库用的同一套索引技术)、Devin Review(免费 PR 分析)、Dana(数据分析变体)等。

再说**必须打问号的宣称**——这些数字要么是厂商自报未经独立审计,要么是历史旧值:

- **SWE-bench 争议**:Devin 发布时报的是「解决 **13.86%** 的 SWE-bench issue」(2024 年 3 月,即 570 题里对 79 题,超过当时 Claude 2 的 4.80% 有辅助 / 1.96% 无辅助的旧 SOTA)。**这是一个历史数字,只能当 2024 年发布成绩讲**——前沿模型早已远远超过它,而 Cognition **并未公布** Devin 2.x 对阵 Claude/GPT/Gemini 的最新头对头 SWE-bench 成绩。谁要拿 13.86% 说 Devin「今天很弱」或「今天很强」,都是错的:它就是个 2024 年的旧快照。
- **演示争议**:2024 年那个「Devin 在 Upwork 上接活赚钱」的爆款演示,被 Internet of Bugs 的视频逐帧扒皮(《Debunking Devin》),指出演示里 Devin「修的 bug」有一部分是它自己制造出来又修掉的。**这是一堂关于「精心剪辑的 demo ≠ 真实能力」的经典课。**
- **一堆 vendor 自报指标**:「2025 年 67% 的 PR 被合并(前一年 34%)」「快 4 倍、省 2 倍」「Devin 2.0 每 ACU 多完成 83% 的初级任务」——**全部是 Cognition 自己报的,没有独立审计**。当「厂商声称」读,别当「已验证事实」引用。

> 拆 Devin 的中立姿态:**它在「产品化云端自主 agent」这件事上是真做出了完整形态**(VM 隔离、Ask 探路、ACU 计费、企业集成),这些工程选择实打实、值得学;**但凡是带具体百分比的能力宣称,都要标清「厂商自报」并降低确定性。** 诚实地讲,才对得起读者。

## 4. 设计决策对比:三条「让 agent 自主」的路

退一步看,三家其实是在回答同一个问题——「**人不在场,怎么让一个 agent 连续跑几十步还不崩?**」——但下注的位置完全不同。

- **SWE-agent 押「界面」:ACI——为模型重造工具。** 赌注是:agent 崩,常常崩在「工具是给人用的、模型用不好」。所以它不碰模型、不碰云,只把 shell 换成一套 LLM 友好的窗口化命令。**收益**:同模型下实打实涨 10.7 个点,而且这套思想可迁移(连 CTF 安全题都提升了 >3 倍)。**代价**:窄——它基本只围绕 shell 编辑做文章,不上浏览器、不做多模态。
- **OpenHands 押「记账 + 动作」:事件流让它可暂停可重放,CodeAct 让动作能自我组合。** 赌注是:自主 agent 的关键是**状态可控**(崩了能恢复、能审计)和**动作够通用**(一段代码顶十个工具)。**收益**:开源、可自托管、模型无关、能委派、能审计,是研究和企业都能拿来改的底座。**代价**:事件流无限增长要靠 Condenser 兜底(压缩即有损),灵活也意味着组装成本更高。
- **Devin 押「托管」:全云 VM + 异步 + ACU 计费。** 赌注是:普通用户不想管沙箱、不想管 orchestration,只想「派任务、收 PR」。所以它把一切藏进云端,你只碰「进」和「出」两端。**收益**:体验最像「雇一个远程工程师」,企业集成、并行、计费都产品化了。**代价**:闭源、不透明(模型、机制都不公开)、按 ACU 花钱、没法本地测,而且**能力宣称难以独立核验**。

一句话把三条路对齐:**SWE-agent 改造 agent 看世界的「界面」,OpenHands 改造 agent 记事和行动的「机制」,Devin 改造你和 agent 之间的「交付形态」。** 三者不冲突——你完全能想象一个产品同时借鉴 ACI 式界面、事件流记忆、和云 VM 托管。

> 一个贯穿的反直觉:**这三家没有一家是靠「训一个更强的模型」赢的。** SWE-agent 靠界面、OpenHands 靠架构、Devin 靠产品化——全是 ⑥ 篇那句「harness 比模型更重要」的不同侧面。模型是发动机,但能不能连续自主跑几十步不散架,拼的是发动机外面那套东西。

## 5. 局限与坑

这个流派最容易被神化,也最容易踩坑。诚实列几条(机器会渲染成⚠️警示卡):

1. **SWE-bench 分数有水分,且横向不可比。** 不同系统在 SWE-bench(及其 Verified/Lite 变体)上的成绩,受**提示词、思考预算、重试次数、评分器可靠性**影响极大,厂商各报各的口径。**同一个 72% 和 88%,可能根本不是同一种测法。** 引用任何 SWE-bench 数字务必带「什么模型、哪个变体、什么时候、谁测的」;别拿两家的裸数字直接排名。这也是为什么本篇给的所有分数都标了「截至 2026 中」和来源。
2. **Devin 的演示 ≠ Devin 的实测。** 2024 年那个 Upwork 演示被逐帧证伪(见 3.3)。**营销视频里的丝滑,和你真派它去修一个陌生仓库的坑,是两回事。** 凡带具体百分比的能力宣称(PR 合并率、快几倍、每 ACU 多干多少),默认按「厂商自报、未经独立审计」对待。
3. **长程漂移 + 烧钱是长跑型的原罪。** 连跑几十上百步,错误会累积、上下文会被压缩「吃掉」早先的决定(见 ④⑧),它可能**忘了自己两小时前定的方案**、在错误方向上越走越远。而每一步都要调模型、占 VM——**跑得越久烧得越多**。所以 SWE-agent 用「每实例花费上限」、Devin 用「ACU 预算」硬性刹车:不是它们抠,是**长跑本质上必须给自主性设一个钱包上限**,否则它能替你烧穿账单。
4. **人不在场,权限的锅全在你。** 陪跑型你能逐步批;长跑型你一句话丢出去就走了。这意味着**沙箱逃逸、危险命令、把密钥打到日志里**这些事,没有一个实时的人来兜。所以三家都很重沙箱隔离(Devin 一次性 VM、OpenHands 的 SecurityAnalyzer、SWE-agent 的容器)——但隔离**降低**风险不**消除**风险。给它的仓库权限、云凭证越大,爆炸半径越大。
5. **数据外泄与提示注入,在自主流派下被放大。** 它会自己上网、自己读仓库里的任意文件(见 ⑨ 讲的 prompt injection)。一个藏在 issue 里或依赖包 README 里的恶意指令,可能诱导一个**无人盯着**的 agent 去 exfiltrate 代码或密钥。自主度越高,这类攻击面越危险——因为没有人在环里当最后一道人肉防线。
6. **别把「开 PR」等于「做对了」。** 它给你一个绿灯 PR,只说明「它跑过的那些测试过了」。**测试覆盖不到的地方、它自己没想到要测的边界,照样可能是错的。** 长跑型 agent 的产出**必须 review**——它把你从「写代码」解放到「审代码」,但没把你从「负责」里解放出来。

## 6. 一句话记住它们 + 对比表

> **自主 SWE agent = 接一句模糊需求、在沙箱里自己长跑几十上百步、最后开一个 PR 的长程 agent。SWE-agent 赌「为模型设计的界面」,OpenHands 赌「事件流 + 动作即代码」,Devin 赌「全托管云 VM + 按算力收费」——三条让 agent 自主的路,没有一条是靠更强的模型赢的。**

三家横向对齐(数字均为截至 2026 年中的快照,SWE-bench 会持续漂移):

| 维度 | SWE-agent(普林斯顿) | OpenHands(All Hands AI) | Devin(Cognition) |
|---|---|---|---|
| **开放性** | 开源(研究导向,已维护模式) | 开源(MIT,可自托管+云) | 闭源商业 SaaS |
| **形态** | CLI + 沙箱包(SWE-ReX) | 自托管平台 + Cloud + SDK | 全托管云端异步 agent |
| **招牌机制** | ACI(为模型设计的命令界面) | EventLog 事件流 + CodeAct(动作即代码) | 每任务一台云 VM + Ask/Agent 两段式 |
| **执行环境** | Docker/Fargate/Modal 等多后端 | Docker 运行时(bash/Jupyter/浏览器/VSCode),V1 沙箱可选 | 每会话一台隔离云 VM(用完即弃) |
| **上下文管理** | HistoryProcessors(丢旧观察/保留最近 N) | Condenser(LLM 总结压缩) | 上下文工程 + Knowledge/Playbooks |
| **多代理** | 无(单 agent 单循环) | 委派 `AgentDelegateAction → BrowsingAgent` | 单 agent + Managed Devins 编排层 |
| **计费** | 每实例花费上限($3.00 默认) | 取决于所选模型(自付 token) | ACU(≈15 分钟/单位,约 $2.25) |
| **自主度** | 高(可全自主跑评测) | 高(可配确认策略) | 最高(异步、人只在进出两端) |
| **招牌成绩** | GPT-4T 12.5% pass@1(界面消融 +10.7 pts);mini-swe-agent >74% | 72.8% SWE-bench Verified(Sonnet 4.5) | 13.86%(2024 发布历史值;新值未公布) |

**怎么选**:想**研究/教学「界面如何影响 agent」**、想要一个能 hack 的最小基线,看 SWE-agent(或它 100 行的后继 mini-swe-agent);想**自托管、要开源可控、要审计和可重放、模型自由**,看 OpenHands;想**开箱即用、把任务外包给云、不想管基础设施**、且能接受闭源与 vendor 数字,看 Devin。

对照 ⑫⑬,这一篇补上了「自主流派」这块拼图:⑫ Claude Code 是**本地、可随时打断**的陪跑型,⑬ Codex Cloud 已经有一只脚踏进云端异步,而这一篇的三家把「派出去、它自己跑」推到了极致。

下一篇,我们从「专攻编码」抬头看更大的图景——**通用 agent 与 agent 框架**(LangGraph、AutoGPT 这一类),看当 agent 不再只为「修 issue」而生、要去做各种任务时,harness 又长成什么样。
