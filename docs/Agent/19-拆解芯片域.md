## 0. 全景:当 agent 真的走进硅片工厂

前面十八篇,我们拆的都是**软件世界**里的 agent——它们读代码、改文件、跑测试。这一篇,我们把镜头转向一个硬核得多的战场:**芯片设计**。这里的「代码」是 Verilog/SystemVerilog 这类硬件描述语言(HDL,hardware description language),「跑一下」意味着调动一整套沉重的 EDA(electronic design automation,电子设计自动化)工具链,而「错一个」的代价可能是几百万美元的流片(tape-out)打水漂。

先给整张地图一个坐标系。有一篇综述——《The Dawn of Agentic EDA:A Survey of Autonomous Digital Chip Design》(arXiv 2512.23189,提交于 2025 年 12 月底)——把「AI 参与芯片设计」分成一条 **L0–L4 自主性阶梯**:

```
L0/L1  工具辅助的人类设计     人在开车,工具只是扳手(手工 & 算法脚本)
   │
L2     AI 辅助副驾(预测)     模型帮你补全/预测,但不自己动手(Copilot 式)
   │
L3     Agentic EDA(编排)     agent 自己规划、调工具、看反馈、迭代 ▸ 本篇主角在这里
   │
L4     自主设计(自动驾驶)     端到端无人参与——综述说:至今没人真做到过
```

这张阶梯之所以重要,是因为它逼你分清一个**全篇最关键、也最容易被营销话术搞糊涂的区分**:

> 💡 **「agent」和「微调模型」是两种东西。** 微调模型(如 RTLCoder、ChipNeMo)是**单次生成器**:喂进去一段需求,吐出一段 Verilog,一锤子买卖。agent 则是**推理时的工具循环**:生成 → 编译/仿真 → 看报错 → 改 → 再来一遍。前者靠改权重变强,后者靠「会用工具、会看反馈」变强。本篇讲的是后者。这个区分你在①②篇已经见过雏形,这里它是分辨真假「芯片 agent」的照妖镜。

这是**板块四(案例拆解)的收官篇**,也是通往㉛篇的桥:最后的落地案例会把这些真实拼图串成一个高约束领域里的端到端 agent 蓝图。所以这一篇,你不妨当成一次「侦察」——看看这条路上,前人已经踩出了哪些真实的脚印。

> 读这一篇的正确姿势:**别被厂商的「Level-5」「全自主虚拟工程师」晃花眼。** 每看到一个系统,先问三件事:它是 agent 还是微调模型?它管芯片流程里的哪一环?它的招牌机制到底解决了什么问题?带着这三问往下走。

## 1. 芯片设计域 agent 是什么:两大战场,和一个「特别难」

一句话:

> **芯片设计域 agent,就是把「推理时工具循环」这套 agent 范式,搬进 EDA 流程里——让 LLM 自己去写 HDL、调 EDA 工具、看仿真/编译反馈、迭代改进,替代原本纯靠工程师手工完成的某个环节。**

这个领域眼下有**两大主战场**:

- **RTL 生成(RTL generation)**:让 agent 从自然语言需求或规格,写出**寄存器传输级(RTL,register-transfer level)**的 Verilog 代码——相当于芯片的「源代码」。代表:AutoChip、ChipMATE。
- **功能验证(functional verification)**:芯片写出来还得**证明它是对的**——写测试平台(testbench)、跑仿真、算覆盖率、抓 bug。这活儿吃掉了 IC 开发**约 70% 的时间**(UCAgent 论文引用的数字)。代表:本篇的深拆主角 **UCAgent**。

那么,**为什么这个领域比「让 agent 写 Python」难得多?** 三个根本约束:

1. **HDL 训练数据极度稀缺。** LLM 的能力和它见过的数据量强相关。而硬件描述语言在互联网上少得可怜——这是理解本篇一大半设计决策的钥匙,§3.1 会给出震撼的具体数字。
2. **正确性判据严苛。** 软件「跑起来不崩」往往就算过关了;芯片要求**功能覆盖率(functional coverage)**——你得证明「所有该测的情况都测到了」,还得抓出边界条件里的隐蔽 bug。判对错不能靠 LLM「感觉一下」,得靠确定性的检查器(checker)。
3. **EDA 工具链又重又专。** 编译器、仿真器、综合工具、布局布线……每一个都是庞然大物,接口古老、许可证昂贵。agent 想「调工具看反馈」,先得把这套重家伙接进循环里。

> 💡 一句话:**芯片域 agent 的所有精巧设计,几乎都是在回答同一个问题——「训练数据这么少、判对错这么严、工具这么重,我怎么还能让 agent 靠谱地干活?」** 记住这个问题,下面每个系统的招牌机制你都会秒懂。

## 2. 深拆:UCAgent 的一次功能验证怎么流动

这一节是全篇的灵魂。我们用一个具体任务——**验证一个整数除法器(IntegerDivider)**——把 UCAgent 的血液走一遍。

UCAgent 是中科院计算所(ICT/CAS,包云岗团队)牵头开源的**端到端块级功能验证 agent**,属于 **XS-MLVP / UnityChip(「万众一芯开放验证」)** 开放生态(§3.1 细拆)。它的核心动作是:驱动一条**可配置的 31 阶段验证工作流**,而这条工作流**完全用 Python 实现**——这是它最反直觉的赌注,先记住,§3.1 揭晓原因。

先看它的生命周期(对照②篇的 agent 单步、⑤篇的规划)。注意:下面的 ①–⑦ 是**七个宏观阶段**,真实工作流把它们进一步细分成 31 个小阶段:

```
工程师给三样东西:RTL(.v/.sv) + 规格文档 + (可选) YAML 工作流配置
   │
   ▼
① Picker 转换      Picker 把 Verilog/SystemVerilog 编译成 PyDUT
                  └─ PyDUT = 一个 Python 包(背后是编译好的 .so),
                     暴露出和 DUT 端口/信号一一对应的 Python API
   │
   ▼
② 需求分析阶段     agent 读规格 → 产出带 FG/FC/CK 标注的「功能分解」
                  └─ ✅ 确定性 checker 校验标注完整性,过了 StageManager 才放行
   │
   ▼
③ 基建阶段        agent 用 Toffee 的 Bundle/Agent 抽象,给 PyDUT 写 Python 驱动
                  └─ ✅ checker 确认代码能跑通(无 import/运行时错误)
   │
   ▼
④ 覆盖模型阶段     agent 建 Toffee 功能覆盖模型(groups/bins),挂上 VCLM 标签
                  └─ ✅ checker 校验「规格 ↔ 覆盖模型」标签对得上
   │
   ▼
⑤ 测试用例阶段     agent 填模板 → 可执行 Python 测试 → 跑 PyDUT →
                  拿覆盖率当工具反馈 → 迭代(每阶段有重试预算)直到打满 bin 或预算耗尽
   │
   ▼
⑥ bug 分析阶段     出现意外失败 → agent 拿到 trace → 做面向 bug 的分析 →
                  记录疑似 DUT 缺陷(可回溯到具体的 FG/FC/CK 点)
   │
   ▼
⑦ 收尾            最终状态 + 覆盖率指标 + 报告序列化(代码覆盖率%、功能 bin 数、发现的 bug)
```

有三个细节值得当场记住:

- **每一阶段都有确定性 checker 把关,而不是让 LLM 当裁判。** 这是 UCAgent 对抗「单次 prompt 方法」最大失败模式(错误逐步累积)的核心武器。checker 不过,`StageManager` 就不让你往下走。
- **状态在每次阶段切换时都序列化。** 于是它能**从任一阶段续跑**:工程师随时可以介入,改一个产物,再让 agent 从**那个阶段**接着跑,而不是从头重来。这正是①⑯篇讲的「human-in-the-loop」的教科书实现。
- **反馈是真实的、可执行的。** 覆盖率不是 LLM 猜的,是真跑了 PyDUT 拿回来的数字——这和 AutoChip「编译器反馈」的思路一脉相承(§3.2)。

> 💡 一句话:**UCAgent 把「验证一个模块」这个模糊大任务,切成 31 个带确定性关卡的小阶段,每关都用真实工具反馈驱动 LLM 迭代——它赌的是「细分 + 硬关卡」能遏制住长程 agent 最怕的错误累积。**

## 3. 组件与全景

这一节我们先把 UCAgent 彻底拆开(§3.1),再拉开视野,把整个开放芯片-agent 全景一家家过一遍(§3.2–§3.4)。每一家都给一个像样的小拆解:招牌机制 + 设计取舍。

### 3.1 UCAgent 拆解:四层架构 + Python-over-SystemVerilog 的赌注

UCAgent 的架构是**四层 + 一个横切机制**(据其论文,arXiv 2603.25768,提交 2026 年 3 月,CC BY 4.0;来自 ICT/CAS、UCAS、北京开源芯片研究院 BOSC):

```
┌─────────────────────────────────────────────────────────┐
│ 工作流管理层   VerifyStage + StageManager                │
│               阶段定义在用户可配的 YAML;每阶段引 Markdown  │
│               模板;每次切换序列化状态;过 checker 才前进    │
├─────────────────────────────────────────────────────────┤
│ Agent 中间件层  基于 LangChain,统一多 LLM 接口 →         │
│               中央 VerifyAgent,由 ReAct(推理→行动)循环驱动 │
├─────────────────────────────────────────────────────────┤
│ MCP 工具层     经 Model Context Protocol 暴露能力,两类:  │
│               (a) 工作流控制(查阶段/触发 checker/前进回退) │
│               (b) 验证环境(编译/跑测试/取覆盖率)          │
├─────────────────────────────────────────────────────────┤
│ Python 验证环境层  Picker(RTL→PyDUT) + Toffee(UVM 式抽象) │
└─────────────────────────────────────────────────────────┘
    横切:VCLM —— FG / FC / CK 三级标签,贯穿全流程做可追溯性
```

逐层看关键点:

- **工作流管理层**用 `VerifyStage` + `StageManager` 两个类实现。妙处在于**工作流即数据**:阶段写在 YAML、模板写在 Markdown(带 `{PROJECT}` 之类变量替换),领域专家改流程**不用碰框架代码**。这个 31 阶段的粒度,是从一个更粗的工作流出发、根据 checker 失败分析**经验性迭代**出来的——细分能定位错误,但也增加编排开销,所以阶段数**可配置**。(论文只在正文描述了需求分析 / 基建 / 覆盖模型 / 测试与 bug 分析这四个宏观阶段,31 个小阶段的完整命名指向论文的图,未在正文逐一列出。)
- **Agent 中间件层**基于 **LangChain**,把多家 LLM 的接口统一起来,汇聚成一个中央 `VerifyAgent`,由 **ReAct(推理→行动)** 循环驱动——和⑫篇 Claude Code 那个「gather→act→verify」循环是同一家族。
- **MCP 工具层**是最能体现「可组合」的一层(对照③篇 MCP)。它既可以由 UCAgent 自带的 LLM 客户端驱动,**也可以把控制权交给一个外部 Code Agent**——OpenHands、GitHub Copilot、Claude Code、Gemini-CLI、Qwen-Code 都行——经 MCP 来驱动 UCAgent。这就形成了一个漂亮的**「agent 既是工具、工具也是 agent」**的反转(§4 会点透)。
- **VCLM(验证一致性标注机制,Verification Consistency Labeling Mechanism)**是横切全篇的灵魂机制:三级标签 **FG(Function Group,功能组)/ FC(Function Checkpoint,功能检查点)/ CK(Check Point,检查点)**注入到 prompt 里,由各阶段专用 checker 校验,保证「规格 → 覆盖模型 → 测试用例」始终**可追溯、语义不漂移**;双向抽取还能标记出 LLM 的幻觉。

现在揭晓那个反直觉的核心赌注——**Python-over-SystemVerilog**。传统验证工程师手写 UVM/SystemVerilog testbench;UCAgent 让 LLM **一行 SystemVerilog 都不写**,全部用 Python(经 Toffee)完成。为什么?因为:

> 💡 **训练数据的悬殊,就是答案。** 据 The Stack v2 数据集:**Python 有 178.44 GB,而 SystemVerilog 只有 0.84 GB**(Verilog 另计,为 9.348 GB)。LLM 写 Python 的能力,和写 SystemVerilog 的能力,根本不在一个量级。UCAgent 的整套 Picker/PyDUT/Toffee 栈,本质就是一句话:**别让 LLM 在它最弱的地方硬拼,把输出模态换到它最强的地方去。**

代价当然存在:你得维护 Picker/PyDUT/Toffee 这一整套栈,而工业 EDA 流程期待的是 SystemVerilog 输出——这道鸿沟是 UCAgent 落地时绕不开的(§5)。

**评测数据点(据论文)**:UCAgent 在六个 DUT 上验证——从 268 行的 UART-16550、到 1,024 行的 IntegerDivider、再到 155,408 行的 PageTableWalker。招牌成绩是**最高约 98.5% 代码覆盖率、最高 100% 功能覆盖率**;就我们这节的例子 **IntegerDivider,三个模型的代码覆盖率都超过 97%,功能覆盖率稳定打到 100%**。它还在 **LaneFAdd** 里真抓出过此前未知的边界条件缺陷,并回溯到具体的 FG/FC/CK 点——这不是刷分,是真在找 bug。

**状态与生态(截至 2026 中快照)**:UCAgent 仓库 `XS-MLVP/UCAgent`,Apache-2.0,约 78.6% Python,183 stars / 41 forks,最新发布 v26.06.24(2026 年 6 月 24 日)。它和 Picker(`XS-MLVP/picker`,支持 Verilator / Synopsys VCS / GSIM / UniVista Simulator 等仿真器)、Toffee(`XS-MLVP/toffee`,融入 UVM 方法学要素的 Python 验证框架)同属一个开放生态。

> ⚠️ 诚实说明:UCAgent 是否已在 **UnityChipForXiangShan / 香山处理器**的验证中**投入生产**,目前**没有可查证的源文档确认**。二者是同一 XS-MLVP 组织下、共享「Python-first」理念的姊妹仓库,但「实际生产集成」这一步未被证实,别当既成事实传播。

### 3.2 RTL 生成 agent:从「最小可行」到「多 agent 交叉验证」

如果说 UCAgent 管「验证」,这一节的三家管的是芯片的另一半——**把需求变成 RTL 代码**。

**AutoChip——最小可行的 EDA agent。**(Thakur 等,arXiv 2311.04887)它朴素到堪称「教学标本」:**一个 LLM + 一个 Icarus Verilog(iverilog)编译器 + 一个反馈循环**。LLM 写 Verilog → iverilog 编译+仿真 → 把报错喂回去 → 改。招牌数据点极有说服力:**把编译器工具反馈加进上下文,比不加时 Verilog 准确率高出 24.20%**。

> 💡 AutoChip 的价值在于它是**「grounding(接地)最纯粹的一课」**:同一个模型,只是让它看得见编译器报错,准确率就跳一大截。这印证了②篇反复强调的——**agent 变强,常常不靠更聪明的模型,靠让它「看得见真实反馈」。** ㉛篇我们自己的 AutoChip 落地,起点正是这个循环。

**ChatEDA——从 RTL 到 GDSII 的全流程编排 agent。**(Wu、He 等,CUHK + 上海 AI Lab,arXiv 2308.10204)它管的不是写一个模块,而是**编排整条流程**:自然语言指令 → 任务规划 → 生成 Tcl/shell 脚本 → 派发给 EDA 工具 → 一路做到 RTL-to-GDSII(把 RTL 变成可流片的版图)。驱动它的是一个**微调模型 AutoMage**(Llama2-70B + QLoRA)。

> ⚠️ 准确性红线:ChatEDA 的招牌成绩是**在 ChatEDA-Bench(50 个任务)上,AutoMage 拿到 88% 的「Grade A」,而 GPT-4 只有 58%**。(网上流传的「98.3% 成功率」在原论文里**根本不存在**,是讹传,别引用。)

**ChipMATE——无金标 oracle 的多 agent 交叉验证。**(arXiv 2605.12857,2026 年 5 月)它专治 RTL 生成的一个老大难:**你怎么知道生成的 RTL 是对的?** 传统要靠人写的「金标准 testbench(golden oracle)」。ChipMATE 不要这个——它让**一个 Verilog agent** 和**一个 Python 参考模型 agent 互相验证**(自称「首个自训练的多 agent RTL 框架」),用**回溯式推理(backtrack inference)**防止错误传播,再用**两阶段 RL**(个体饱和 → 团队训练)打磨。成绩:VerilogEval V2 上 **75.0% / 80.1% pass@1(4B / 9B 模型)**,论文称超过了 DeepSeek V4(1600B)。

> 💡 ChipMATE 揭示了一个深刻的工程逻辑(对照⑩篇多代理):**当你没有「标准答案」时,让两个视角不同的 agent 互相挑错,能逼近正确性。** 这不是炫技,是在模仿真实芯片团队里「设计和验证两拨人互相较劲」的做法——**工程约束催生了多 agent 架构。**

顺带划清一条界:**RTLCoder、ChipNeMo 这类是「微调模型」,不是 agent。** RTLCoder 是 7B 开源 Verilog 模型(可 4-bit 量化到笔记本上跑),ChipNeMo 是 NVIDIA 在 24B token 上对 LLaMA2-70B 做领域自适应预训练(DAPT,domain-adaptive pretraining)的模型——它们强,但都是**单次生成**,没有推理时工具循环。别把它们混进「agent」这一栏(这正是§0 那把照妖镜的用处)。

### 3.3 物理与系统层:参数调优 agent 与 HW+SW 协同

前两节都在「前端」(写代码、验证)。芯片流程还有**后端物理设计**和**更宏大的系统层**。

**ORFS-agent——在物理设计里做参数调优。**(Ghose、Kahng 等,arXiv 2506.08332)它是一个 **model-agnostic、免训练**的 LLM agent,活在 **OpenROAD-flow-scripts(ORFS)**这个开源物理设计流程里,专门调布局布线的参数,还能把贝叶斯优化(Bayesian optimization)库当**子工具**调用。成绩:相比 OR-AutoTuner,在线长/时钟周期/协同优化上分别提升**最多 1.0% / 1.3% / 2.7%**,而且**少用 40% 的迭代**(6 个 benchmark,ASAP7 + SKY130HD 工艺)。它的招牌取舍是**「不改权重、不挑模型」**——纯靠工具循环 + 会调子工具,就把重活干了。

**QiMeng(启蒙)——三层的硬件+软件协同设计。**(Zhang、Wen 等,中科院,arXiv 2506.05007)这是野心最大的一个:**三层架构**——底层是 **LPCM(Large Processor Chip Model,大处理器芯片模型)**这个领域基础模型;中层是**硬件设计 agent + 软件设计 agent**;顶层是应用。它演示了 RISC-V CPU 前端、HDL 生成、编译器设计、OS 调优——**软硬件一起协同设计**,这是别家都没碰的维度。(具体量化指标目前未从可查证来源充分核实,当方向性理解。)

### 3.4 商业化:两家巨头的新闻稿(带 vendor 免责)

上面全是学术/开源。产业界呢?两家 EDA 巨头下场了——但**下面这些都是厂商新闻稿口径,不是同行评审,请戴着怀疑的眼镜读**。

**Cadence ChipStack AI「Super Agent」。** Cadence 于 2025 年 11 月收购 ChipStack,在 COMPUTEX 2026(新闻稿 2026-05-31)宣布**「Level-5 自主」**。它用 NVIDIA Nemotron 构建、跑在 NVIDIA OpenShell 沙箱里,用 Xcelium + Jasper Formal 跑上百个动态仿真,**宣称** RTL 验证快 40 倍以上、把一个 5 周的循环压到不到 1 天。

## 4. 与众不同的设计决策:四个可复用模式

把上面这些系统拉到一起看,会浮现出几个**跨系统复用的设计模式**——这些才是㉛篇能直接搬走的东西。

- **输出模态匹配 LLM 能力,而非匹配行业惯例(Python-over-SV)。** UCAgent 最狠的一招:行业惯例是 SystemVerilog,但 LLM 的数据在 Python 那边(178.44 GB vs 0.84 GB),于是它果断把输出模态换成 Python。**通用启示:当 agent 在某种语言/格式上表现差,先问「有没有一种它更擅长的等价表达」,而不是硬逼它。**
- **分阶段 + 确定性 checker = 错误遏制。** UCAgent 的 31 阶段各带一个**确定性** checker(不是 LLM 裁判),把长程任务切成带硬关卡的小段,阻断错误累积——这是单次 prompt 方法的头号死因。**这是本篇最值得抄进㉛的可靠性模式。**
- **agent-as-tool / tool-as-agent(经 MCP 反转)。** UCAgent 既**使用**工具,自己又**是**一个别的 Code Agent 能驱动的工具(经 MCP)。这是③篇 MCP + ⑩篇多代理的完美合流:**能力经标准协议暴露,agent 就变得可组合、可嵌套。**
- **无金标 oracle 的交叉验证。** ChipMATE 用「Verilog agent × Python 参考模型 agent」互验,绕开了「必须有人写金标 testbench」的瓶颈。**当没有标准答案时,让异构视角互相挑错,是逼近正确性的通用手段。**

> 💡 一句话把这四条串起来:**芯片域 agent 的靠谱,来自「扬长(换到 LLM 擅长的模态)+ 遏错(确定性关卡切段)+ 组合(MCP 可嵌套)+ 互验(多视角交叉)」——没有一条是靠「模型再大一点」。** 这和⑫篇 Claude Code「极简循环、极重 harness」的信条,底层是同一句话。

## 5. 局限与坑:别信「全自主」的营销

诚实地把这条路上的雷区列清楚(这一节尤其要冷静——芯片是最容易被 hype 淹没的领域之一):

1. **「全自主流片」目前不存在——这是硬事实。** §0 那篇综述明确写道:**至今没有任何 LLM 驱动的 agent,完成过工业级芯片的、完全无人参与的完整流片(tape-out)。** 谁的新闻稿说「全自主造芯片」,先默认打个问号。这是全篇最需要你记住的一条。
2. **规模一大,「全自动」就顶不住,得靠 human-in-the-loop。** UCAgent 的诚实数据点:面对 **155,408 行代码的 PageTableWalker**,**默认的全自动工作流跑不下来**;换成**人在环**的策略,才在**约 8 小时**内达到预期验证效果。这不是「失败」,但它是当下 LLM-agent **规模上限**的活样本——大模块仍需人来拆解、介入。
3. **厂商的「Level-5」≠ 综述的 L4。** Cadence 的「Level-5」是它自定的刻度;学术综述的天花板是 L4,且伴随「无人流片尚未实现」的明确声明。**把营销刻度当学术结论,是这个领域最典型的认知陷阱。**
4. **benchmark 有水分,且不能直接横比。** VerilogEval V1(156 题)已被普遍认为**接近饱和**;V2(209 题,含工业级任务)才更有区分度。不同系统常在**不同的 DUT / 不同 benchmark** 上报数(UCAgent 用它自己那六个 DUT,ChatEDA 用 ChatEDA-Bench),**跨框架的覆盖率/pass@1 对比只能当方向性参考,不是苹果对苹果。** 别拿 A 的 88% 直接去比 B 的 80%。
5. **HDL 数据稀缺是根本约束,绕不开只能迂回。** Python-over-SV 是**迂回**,不是**根治**:它把问题换了个战场,代价是要维护一整套转换栈、且输出不是工业期待的 SystemVerilog。这个约束会长期存在——它决定了「芯片 agent」不可能简单照搬「软件 agent」的打法。
6. **vendor 宣称与同行评审必须分开记账。** ChipStack、Fuse 的所有「×倍」数字都是新闻稿;ChipAgents 的 Renoir 模型、RLVR(Reinforcement Learning from Verification Results)机制也只有厂商自家博客背书、无独立验证。引用时务必标明来源级别——**这不是吹毛求疵,是这个领域的基本诚信。**

## 6. 一句话记住它 + 对比表

> **芯片设计域 agent = 把「推理时工具循环」搬进 EDA 流程:让 LLM 自己写 HDL / 调工具 / 看真实反馈 / 迭代。UCAgent 是其中最完整的开放代表——用「Python-over-SystemVerilog + 31 阶段确定性关卡 + MCP 可组合」三招,把功能验证做成了一个能靠谱跑、能从任一阶段人工续跑的 agent。而整个领域的天花板,仍卡在综述那句冷静的判断上:全自主流片,尚未有人做到。**

把本篇五个代表系统拉齐(招牌机制 + 属性对比):

| 系统 | 目标环节 | agent 还是模型 | 开放性 | 招牌机制 |
|---|---|---|---|---|
| **UCAgent** | 块级功能验证 | agent(ReAct + 31 阶段工作流) | 开源(Apache-2.0,ICT/CAS) | Python-over-SV + 确定性 checker + VCLM(FG/FC/CK)+ MCP 可组合 |
| **AutoChip** | RTL 生成(模块) | agent(单 LLM + 编译器循环) | 开源(学术) | Icarus Verilog 反馈接地(+24.20% 准确) |
| **ChatEDA** | RTL→GDSII 全流程编排 | agent(由微调 AutoMage 驱动) | 开源仓库 + 微调模型 | NL→规划→Tcl→EDA 工具;ChatEDA-Bench 88% Grade A |
| **ChipMATE** | RTL 生成(正确性) | agent(多 agent + RL) | 学术(论文) | 无金标 oracle 交叉验证 + 回溯推理 + 两阶段 RL(V2 75.0%/80.1% pass@1) |
| **ORFS-agent** | 物理设计参数调优 | agent(免训练,model-agnostic) | 开源(OpenROAD 生态) | 函数调用 + 贝叶斯优化子工具(少 40% 迭代) |

(注:上表「pass@1 / 覆盖率」等数字均出自各自论文、benchmark 各异,不可直接横比——见 §5 第 4 条;商业系统 ChipStack/Fuse 未入表,因其成绩仅为厂商宣称、无同行评审。)
