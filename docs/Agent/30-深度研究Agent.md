## 0. 全景:把「搜索」换成「研究」

你打开一个普通搜索框,敲进一个问题,得到十条蓝色链接——**筛选、交叉验证、下结论,全是你的活。** 这就是过去三十年搜索引擎的合同:它给你原料,你做研究员。

深度研究 agent(deep research agent)把这份合同**反过来签**。你把一个开放问题丢给它——「帮我把国产长上下文推理模型这两年的竞争格局摸清楚,带引用」——然后它自己去干那个研究员的活:先把问题拆成一串子问题,同时开几十路去搜、去读、去抓网页,一边读一边给每个来源做摘要、记下出处,最后把几十个来源**综合成一份带引用的长报告**交给你。你从「搜索的操作员」变成了「报告的验收员」。

这一类系统,是当今 agent **最成熟、落地最扎实的形态之一**——因为它踩中了一个甜蜜点:任务足够复杂(单次 LLM 调用做不了),但又足够「只读」(主要是搜网页、读文档、写字,不用改你的生产代码库),风险可控、价值肉眼可见。它把前面几篇讲的所有零件——规划、并行子代理、上下文压缩、工具接口——拧成了一个特别清晰的范式:

```
   普通搜索 / 单次问答                    深度研究 agent(planner → 扇出 → 综合)

   你 ──▶ 一个 query                       你 ──▶ 一个开放问题
        │                                        │
        ▼                                        ▼
   10 条链接 ──▶ 你自己读、筛、            ① Planner 拆成 N 个子问题
              交叉验证、写结论                    │
   (研究员是「你」)                        ┌────┼────┬────┐  ② 并行扇出(fan-out)
                                          ▼    ▼    ▼    ▼
                                        搜+读 搜+读 搜+读 搜+读   每路各自检索→逐源摘要→记来源
                                          └────┴──┬─┴────┘
                                                  ▼
                                          ③ 聚合去重 / 综合(reduce)
                                                  │
                                                  ▼
                                          一份带引用的长报告 ──▶ 你只验收
                                          (研究员是「agent」)
```

这一篇拆这个范式最有代表性的四家。深拆两个**开源、能把控制回路看到底**的:字节跳动的 **DeerFlow** 和个人开源项目 **GPT-Researcher**;再用一个「群像」子节横扫两个**定位鲜明的另类**:斯坦福的学术系统 **STORM / Co-STORM**,和商业闭源的 **Perplexity**。

> 读这一篇的正确姿势:**别记谁的报告更漂亮,记那条贯穿所有人的骨架——planner → 并行检索 → 综合成带引用报告。** 四家的差异,全是在这条骨架的不同环节上下的不同注。

## 1. 这一类到底是什么、边界在哪

先给一句话定义:

> **深度研究 agent 是一个接下「一个开放研究问题」、自己规划子问题、并行去多个来源检索与阅读、逐源摘要后综合出一份带引用长报告的 agent。** 关键词四个:**开放问题、自主规划、并行多源、带引用综合。**

逐个拆这四条边界,顺便把它和容易混的东西划清:

- **开放问题(而非可检索的事实)**:你问「东京时间几点」,那是查询,一个 API 就够了。你问「过去两年 agentic 浏览器的安全争议有哪些、各家怎么回应」——这没有单一答案,得**读很多篇、比对、归纳**。深度研究 agent 是为后者生的。
- **自主规划(而非人给步骤)**:你不告诉它「先搜 A 再搜 B」;它自己把大问题拆成一组「合起来能形成客观结论」的子问题。这就是前面讲规划那几篇的直接落地。
- **并行多源(而非串行单跳)**:它同时开好几路去搜、去抓。GPT-Researcher 的文档口径是**单次研究平均聚合 20+ 个网页来源**;Perplexity 的 Deep Research 更号称能读到**上百个源文档**。并行,是「快」的来源。
- **带引用综合(而非返回原始结果)**:终点不是一堆链接,是一篇**每条断言都能溯源**的报告。这一条,是它区别于「把搜索结果贴给 LLM 总结一下」的分水岭——它逐源记出处,让你能核查。

为什么值得单独一章?因为它是 **RAG(检索增强)这条老思路,被「agent 化」之后长出的最完整形态**。传统 RAG 是「检索一次 → 塞进上下文 → 生成一次」,一锤子买卖;深度研究 agent 把它变成**一个会自己迭代的回路**:搜了、读了、发现还不够,就改写研究计划、再搜一轮——直到覆盖足够才收笔。Perplexity 官方把这个内核描述得很干净:*"iteratively searches, reads documents, and reasons about what to do next, refining its research plan"*(迭代地搜、读、推理下一步该干什么,不断精炼研究计划)。

> 一句话:**深度研究 agent = 把 RAG 从「检索一次」升级成「自己决定要检索几次」的 agent。** 会迭代、会拆解、会记引用——这三样让它从「搜索工具」跨进了「agent」。

还有个边界要划清:这一类和前面拆的**自主 SWE agent**(修 issue、开 PR 那一篇)是**两个物种**。SWE agent 的动作是「写文件、跑测试、改代码」,风险在于它真会动你的仓库;深度研究 agent 的动作绝大多数是「搜、读、写字」,是**只读的**——GPT-Researcher 和 STORM 的文档里都明确点出:因为不执行任意代码,**沙箱不是它们的核心关注点**。这个「只读」属性,正是它敢于高度自主、也最早被大规模采用的原因。

## 2. 一次任务在这类系统里怎么流动

挑两个代表走一遍生命周期。先走**最能看清骨架**的 GPT-Researcher 标准模式,再看 DeerFlow 2.0 把这套东西「电池全含」化之后的样子。

设想你丢进去一句:

> 「帮我研究:agentic 浏览器的间接提示注入(prompt injection)风险,各家现状如何?」

**代表一:GPT-Researcher 标准模式(一次 `.conduct_research()` → `.write_report()`)**

```
  一个开放问题
     │
     ▼
① 建域 agent    根据问题定 prompt / 报告类型 / 检索器 / 来源范围
     │
     ▼
② 规划子问题    Planner 让 LLM 拆出一组子问题:
     │          「什么是间接注入」「Comet 被曝了什么」「Brave 说了什么」「各家如何缓解」……
     │          (力求合起来覆盖主题、客观)
     ▼
③ 并行检索      每个子问题 → 触发一个 crawler agent,用检索器(默认 Tavily)搜网抓 top 资源
     │          ┌──────┬──────┬──────┐   ← 这一步是 async/await 并行的,多路同时跑
     │        子问题1 子问题2 子问题3 …     这是「快」的关键
     ▼          └──────┴──┬───┴──────┘
④ 逐源摘要+记源  对每个抓到的资源,让 LLM 按「与该子问题的相关性」压缩成摘要,同时存下 URL/出处
     │
     ▼
⑤ 聚合去重      Publisher 过滤、去重、把所有摘要 reduce 成一个统一 context
     │          (这一步就是 map-reduce 式的上下文压缩,避免撑爆窗口)
     ▼
⑥ 成文          聚合后的 context 喂给 LLM 生成 5–6 页报告,附引用 → 导出 PDF / Docx / Markdown
```

注意 ③④⑤ 这三步:**一个 planner 扇出 N 个并行的「搜索→摘要」分支,再 reduce 成一份带引用的报告。** 这就是整个流派的心脏。而且这里藏着它对付「上下文窗口塞不下几十个网页」的核心手法——**不是把原始网页硬塞进窗口,而是「逐源先摘要、再聚合」**(map-reduce)。据其文档,标准模式跑完这一大圈约 **3 分钟、约 $0.1**(用 GPT-4o-mini 做便宜摘要 + GPT-4o 成文的混合配置)。

如果这一圈还不够深,GPT-Researcher 有个 **Deep Research 递归模式**:在上面的扇出之上再加**递归深度**,模拟人「顺着线索往下挖」——每层生成多个 query 探索不同侧面(breadth,默认 4),对每个分支再**递归下钻**(depth,默认 2 层),用一个「Smart Context Management」跨所有分支聚合发现。代价是更贵更慢——约 **5 分钟、约 $0.4**,推荐上推理型模型(如 `o3-mini`)。

**代表二:DeerFlow 2.0——同一条骨架,被做成「一台真电脑 + 主从子代理」**

DeerFlow 2.0 的循环长得不太一样,但拆开看,骨架还是那条:

```
  用户请求(Web UI 或某个 IM 渠道 → 落到一个会话线程)
     │
     ▼
① 主智能体推理    lead agent 用 LLM 判断:直接答?还是要拆解 / 动工具 / 派子代理?
     │
     ▼
② 动态派子代理    任务复杂时,主体动态 spawn 若干子智能体,每个带:
     │            独立上下文 + 限定工具集 + 各自的终止条件(彼此互不可见)
     │            ┌────────┬────────┬────────┐   能并行则并行
     │          子代理A   子代理B   子代理C          每个在沙盒容器里「真执行」
     ▼            └────────┴───┬────┴────────┘     (bash / 读写文件 / 跑 Python / web search)
③ 结果合成        子代理完成后回传结构化结果,主体聚合
     │
     ▼
④ 上下文压缩(关键)  完成的子任务做摘要、中间产物落盘到文件系统、压缩不再相关的内容
     │
     ▼
⑤ 长期记忆持久化   把关于用户/项目的持久信息写进 memory,跨 session 复用(去重)
     │
     ▼
   报告 / 幻灯片 / 网页 / 图像 / 视频……
```

对比两家,你会发现:**GPT-Researcher 把「planner → 扇出 → 综合」固化成了一条明确的流水线;DeerFlow 2.0 把它交给一个 lead agent 动态决定「什么时候拆、拆几个」。** 前者简单可预测,后者灵活能干更杂的活。这个「静态流水线 vs 动态主从」的分野,正是下一节深拆的重点。

> 一句话:**这一类 agent 的本事,全在「扇出多少路、每路读多深、怎么把几十个来源压回一个窗口、怎么保住引用」。** 怎么在这四件事上取舍,就是下面四家各显神通的地方。

## 3. 拆开看

现在把四家分别搬上手术台。前两个深拆——盯着它的**控制回路与核心组件**一步步看;最后一个子节做「群像横扫」,把定位鲜明的另外两家点透。

### 3.1 DeerFlow(字节跳动):从「深度研究框架」长成「超级 agent 运行时」

**它是谁**:DeerFlow(**D**eep **E**xploration and **E**fficient **R**esearch **Flow**)是字节跳动做的**开源系统**,许可 **MIT**,仓库 `bytedance/deer-flow`。它的身世有个关键转折:**1.x 是一个「深度研究框架」**(Deep Research);而 **2.0 是彻底重写(ground-up rewrite),与 1.x 不共享任何代码**,发布于 **2026 年 2 月 28 日**,官方把它重新定位成 *"an open-source long-horizon SuperAgent harness that researches, codes, and creates"*——一个能跑分钟级到小时级长任务的「超级 agent 运行时/骨架」。截至 2026 年中,仓库活跃(约 **7.6 万 star**),1.x 仍在 `1.x` 分支维护,主线已转向 2.0。

拆 DeerFlow,最好的办法是**把 1.x 和 2.0 并排看**——它俩恰好演示了本篇范式的两种极端:一个是**显式流水线**,一个是**动态主从**。

**先看 1.x,它是「planner → 检索 → 综合」这条范式的教科书式实现。** 它是一个 LangGraph 状态图,节点清清楚楚:

- **Coordinator(协调器)**:入口,管生命周期。
- **Planner(规划器)**:把研究问题拆成结构化计划,**并决定何时该收笔出报告**。
- **Research Team**:两个干活的——**Researcher**(搜索/爬取/走 MCP)+ **Coder**(Python REPL,做数据处理)。计划里的各步在这里并行执行。
- **Reporter(汇总器)**:把研究结果汇成报告(Markdown,可再转 PPT / 播客音频 / 可编辑块)。
- 外加一个 **Human Feedback 节点**:计划出来后**人可以审**——回 `[ACCEPTED]` 接受,或 `[EDIT PLAN] ...` 修改;也可设 `auto_accepted_plan: true` 自动放行。

看出来了吗?1.x 就是把 §2 那条「planner → 扇出 → 综合」骨架,画成了一张能在 LangGraph Studio 里可视化调试的**静态图**——节点固定、流向清晰。这是深度研究 agent 最经典、最好懂的长相。

**再看 2.0,它把「显式图」换成了「动态主从 + 一台真电脑」。** 三个核心变化:

- **Lead agent + Sub-agents(主从子代理)**:不再是固定的 Researcher/Coder 节点,而是主智能体**按需动态拉起**子智能体。每个子代理有**自己独立的上下文、工具集、终止条件**,且**看不到主体或兄弟子代理的上下文**——这是刻意的**上下文隔离**,防止主体上下文被子任务的原始素材撑爆。子代理干完只回传**结构化结果**,不把原始一堆网页塞回主体。(这套「通才主体 + 隔离子代理只回摘要」的思路,和前面讲子代理那篇是同一个母题。)
- **Sandbox——「给 agent 一台真电脑」**:这是 2.0 最鲜明的一句主张——*"The agent does not suggest a bash command. It runs it."*(agent 不是建议一条 bash 命令,而是真的执行它。)每个任务拿到一个隔离容器,里面有完整文件系统、bash 终端、能读写和执行文件。沙盒三档:**Local / Docker / Kubernetes pod**。⚠️ 有个安全事实要记牢:`LocalSandboxProvider` 默认禁用 host bash,官方**明说本地模式不是安全边界**,生产要上 Docker/K8s 隔离。
- **激进的上下文管理**:官方原话——*"summarizing completed sub-tasks, offloading intermediate results to the filesystem, compressing what's no longer immediately relevant"*(把完成的子任务摘要掉、把中间产物落盘到文件系统、压缩不再相关的内容)。**「把中间产物落盘、需要时再读回」这一手很关键**——它把文件系统当成上下文窗口之外的「外部记忆」,和 §2 里 GPT-Researcher「逐源摘要」是同一类思想的不同实现。

再点几个 2.0 有特色的部件:

- **Skills = Markdown + 渐进式加载**:一个技能就是一个 Markdown 文件,定义工作流/最佳实践/引用,**用到才加载**(progressive loading),`/skill-name` 单轮激活。内置技能覆盖网络研究、报告、幻灯片、网页、图像、视频。(把「领域知识外置成 Markdown、按需加载」——和讲 Claude Code 那篇的 skills 理念同源;官方甚至提供一个 `claude-to-deerflow` skill,让你从 Claude Code 终端直接驱动一个 DeerFlow 实例。)
- **Message Gateway(消息网关)**:这是同类里少见的一等公民能力——多 IM 渠道接入(Telegram / Slack / 飞书 / 企业微信 / 钉钉 / 微信),**自启动、无需公网 IP**。面向「把研究 agent 直接挂进团队 IM」的落地场景。
- **`/goal` 目标驱动的自动续跑**:这是 2.0 的 HITL(human-in-the-loop)护栏。你可以给线程设一个完成条件,之后每轮跑完 DeerFlow 拿对话对照目标做评估,只有没满足才注入「隐藏续跑」继续干——**安全上限默认 8 次隐藏续跑**,外加「无进展熔断」:连续 **2 次**评估都是「没进展」就停。这是给「长时程自动化」上的双保险刹车。

**为什么 DeerFlow 值得单独深拆?** 因为它是这一流派里少数**明确自称「harness 而非框架」**的:*"DeerFlow 2.0 is no longer a framework you wire together. It's a super agent harness — batteries included, fully extensible."*(不再是你自己接线的框架,而是电池全含、完全可扩展的超级 agent 运行时。)它模型无关(任何 OpenAI 兼容 API 都行),坐在 LLM 之上做编排/工具/沙盒/记忆。

> 反直觉的点:**DeerFlow 从 1.x 到 2.0,走的不是「加更多研究功能」,而是「把研究降级成一个普通任务」。** 1.x 是专门为「深度研究」画的流水线;2.0 里研究只是那台「真电脑 + 主从子代理」能干的众多活之一。这说明一件事——**当底座够强,你不必为「研究」造一个专用管线,一个足够通用的 agent 循环顺手就把研究做了。** 这正是深度研究从「专用系统」向「通用 agent 的一种用法」演化的缩影。

### 3.2 GPT-Researcher:把「扇出—综合」范式做到最透的开源基线

**它是谁**:GPT-Researcher 是一个**开源的自主深度研究 agent**,作者 **Assaf Elovic**(2023 年 5 月建仓,个人发起;他同时是检索 API 公司 Tavily 的联合创始人,Tavily 是本项目默认检索后端,但项目本身独立)。许可 **Apache-2.0**(⚠️ 一个溯源坑:官网首页某处写「License: MIT」,但 MIT 实际是**独立的 `gptr-mcp` 仓库**的许可,主项目以仓库为准 = Apache-2.0,官网把两者混写了)。截至 2026 年中活跃演进,约 **2.8 万 star**,最新 v3.5.x,近期新增了 MCP server 集成等能力。

如果说 DeerFlow 展示了「研究被通用化」的一端,GPT-Researcher 就是**把「planner 扇出 → 综合」这条范式本身做到最透、最好读**的另一端。它明确借鉴两个经典思路:**Plan-and-Solve**(先规划再解)和 **RAG**。它要治的是单次 LLM 做研究的三个病:**幻觉、上下文塞不下、结果不可复现**。

它有**两套架构**,对应两档复杂度——这个「双档」设计本身就是一个好范例。

**A. 标准架构(默认,`GPTResearcher` 类)——干净的三层 agent:**

- **Planner Agent(规划器)**:把用户 query 拆成一组「合起来能形成客观结论」的研究子问题。
- **Execution / Crawler Agents(执行/爬取器)**:**每个子问题触发一个爬取 agent**,抓在线资源,逐源摘要并**记来源出处**。
- **Publisher Agent(发布器)**:过滤、去重、聚合所有摘要,生成最终报告。

这三层,就是 §2 那张图的最小骨架实现。它的控制回路(§2 已走过)清晰到可以当教学模板:**建域 agent → 规划子问题 → 并行检索(async/await)→ 逐源摘要+记源 → 聚合去重 → 成文**。

**B. 多智能体架构(`multi_agents/`,基于 LangGraph)——一个「研究编辑部」:**

有意思的是,这套架构**明确受 STORM 论文启发**(官方文档里点名 arXiv 2402.14207)——这正好把本篇要横扫的 STORM 串了进来。它仿一个编辑部,角色分工:

- **Chief Editor(主编)**:总协调,用 LangGraph 驱动整个团队。
- **Editor(编辑)**:规划报告大纲与结构。
- **Researcher**:就是 GPT-Researcher 本体,做深度自主研究。
- **Reviewer(审稿)→ Revisor(修订)**:按标准校验、按反馈修正。
- **Writer(撰稿)→ Publisher(发布)**:汇编、导出 PDF/Docx/Markdown。
- **Human**(可选):`include_human_feedback` 打开可插入人工反馈环。

关键设计:**对大纲里的每个主题,并行跑「Researcher → Reviewer → Revisor」的迭代精修**,再由 Writer 汇编。这是「多角色协作写长报告」的另一条路——和 §2 那条单纯扇出的流水线相比,它多了「审稿—修订」的质量回路。

再点几个 harness 级的设计,都是本篇范式的典型手法:

- **上下文管理靠 map-reduce 压缩**:核心手法就是「逐源摘要 → 聚合」,把 20+ 来源压成能进窗口的浓缩 context,而不是硬塞原始网页。混合模式下同时管 web + 本地文档两路。
- **provider / retriever 双不锁定**:LLM 可插拔(默认 OpenAI,可换 Claude / Gemini / Mistral / Ollama / DeepSeek 等,或任意 OpenAI 兼容端点),检索器也可插拔(默认 Tavily,可换 Google / Bing / DuckDuckGo / Serper / SearX / Arxiv / Exa,甚至逗号串联多个)。**这是开源深度研究 agent 的典型姿态:不把你绑死在任何一家。**
- **成本优化的模型混用**:标准模式用便宜的 GPT-4o-mini 做逐源摘要、用 GPT-4o 成文;只有 Deep Research 才上贵的推理模型。**「便宜模型干量大的摘要、贵模型干最后成文」——这是一个所有并行检索系统都该抄的省钱模式。**
- **本地/私有数据 + 报告类型**:支持 PDF/CSV/Excel/Word 等本地文档(`report_source="local"/"hybrid"`),报告类型可选标准 / 自定义(如「APA 格式含引用」)/ 指定 URL 等。

**它和商业深度研究产品的分界线在哪?** 相比 Perplexity Deep Research / OpenAI Deep Research 这类**闭源产品**,GPT-Researcher 的卖点是**可自托管、可定制、可审计引用、不锁厂商**——本体免费开源,你只付底层 LLM/检索的账单;代价是**得自己配 key、自己运维**。

> 一个必须诚实标注的坑:官网宣称在卡内基梅隆的 **DeepResearchGym** 基准上「排名第一、领先 Perplexity / OpenAI」。**但原论文(arXiv:2505.19253)并未评出任何「综合第一」**——成绩是按维度分散的(比如 GPT-Researcher 在引用精度/召回上强,Perplexity 在 Key Point Recall 上领先)。「#1」是官网的营销概括,**不是基准结论**。拆一个开源项目也要把「官网自夸」和「论文事实」分开——这正是本站的准确性纪律。

**一句话收束 GPT-Researcher**:它是深度研究范式最好读的开源标本——一个 planner 扇出 N 个并行分支、每路先摘要后聚合、reduce 成带引用报告。想搞懂这个流派,先读它。

### 3.3 群像横扫:STORM / Co-STORM 与 Perplexity

前两家把「开源、可拆解」的两端讲透了。剩下两家定位鲜明,值得各点两笔——一个是**学术界把「怎么问对问题」当核心难点来攻**的知识策展系统,一个是**商业闭源、把深度研究做成极速产品**的答案引擎。

**STORM / Co-STORM(斯坦福 OVAL,开源 · MIT):把「提问」当一等公民的知识策展系统。**

STORM(**S**ynthesis of **T**opic **O**utlines through **R**etrieval and **M**ulti-perspective Question Asking)出自斯坦福 OVAL 实验室(PI 是 Monica Lam),两篇会议主论文压阵(**NAACL 2024** + **EMNLP 2024**)。它的定位不是「深度研究直答工具」,而是**知识策展(knowledge curation)**:给它一个主题,它生成一篇带引用的、**维基百科式**的长文。

它最独特的赌注,是把重心押在**「问对问题」这个前期调研(pre-writing)难点**上,而不是「下笔写作」。具体做法很巧:先去**检索与该主题相似的现有维基条目、抽取它们的目录结构,归纳出 N 个「视角/persona」**;再让扮演不同 persona 的「维基写手」LLM,向一个 `TopicExpert` 提问——Expert 把问题**改写成搜索 query、联网检索、给出基于检索片段的 grounded 回答并附来源**;写手据此追问下一问。所有对话轮次和引用落进一个外部化的 `StormInformationTable`。这套「知识策展(刚说的多视角提问)→ 大纲生成 → 逐节带引用写作 → 润色」是它固定的四阶段管线。论文报告:相较 outline-driven RAG 基线,STORM 生成文章的「组织性」绝对提升约 **25%**、「广度」约 **10%**。

**Co-STORM** 是它的人机协作升级版,主攻一个绝妙的问题——**「unknown unknowns」(你根本不知道自己该问什么)**。它让多个 LM agent(领域专家 `CoStormExpert` + 主持人 `Moderator` 等)就主题**相互提问、回答**,由一个中心化的**回合调度器 `DiscourseManager`** 按 turn policy 决定下一个谁发言;人类可以**只旁观、也可以插话把话题引向自己关心的方向**。整个对话的知识沉淀进一棵**动态思维导图(`KnowledgeBase` 层级树)**,新信息靠「LLM 判断 + 向量相似度」挂到合适节点,节点过载了自动拆子节。最后 `generate_report()` 把思维导图的节点名当章节标题,组织成结构化报告。

**它和其他三家最不一样的地方**:别人是「你提问、agent 给答案」;Co-STORM 是「**你围观 agent 之间的对话,去撞见自己想不到的盲区**」。思维导图是它给你和 agent 共享的认知支架。取舍也清楚:**专而不通**(只做知识合成/写作,不写代码、不做通用任务);输出偏「维基风格」。状态上,截至 2026 年中**开源、活跃度中等**——打 tag 的 GitHub release 停在 v1.1.0(2025-01),但 PyPI 上有更新的补丁版(v1.1.1,2025-09),约每半年一个补丁,热度更多来自「经典参考实现」的地位。约 **3 万 star**,官方称 7 万+ 人试过在线 demo。

**Perplexity(Perplexity AI,闭源商业):把深度研究做成「快而免费」的答案引擎。**

Perplexity 是这几家里**唯一的闭源商业产品**,核心定位是「answer engine(带引用的实时搜索问答)」,再向上叠 agentic 研究和 agentic 浏览器。它横跨多层——**既是模型厂**(自研 `Sonar`,据官方说明基于 Meta Llama 3.3 70B 再训练,但未开源权重),**又是模型路由器/编排层**。

它的 **Deep Research** 内核,就是本篇范式的商业化极致版,官方描述完全对得上骨架:*"iteratively searches, reads documents, and reasons about what to do next, refining its research plan"*。一次请求的生命周期是**意图展开成研究计划+子查询 → 数十路并行检索 → 读上百个源、抽取证据+重排 → 推理并动态改写研究计划(回到检索,迭代)→ 合成 + 每条断言带引用 → 交付**。它的独特赌注有两个:

- **走「快而免费」路线**:官方称 Deep Research 大多数任务在 **2–4 分钟**内完成(对手常要 5–30 分钟)。速度背后是硬件——它和 **Cerebras** 合作,用 WSE-3 晶圆级引擎跑 Sonar,官方称约 **1200 tokens/s** 解码。这是「牺牲最强通用能力、换搜索场景下够好且极快」的取舍。
- **2026 的 Deep Research 2.0 / "Computer" 走「多模型编排」**:引入两个具名构件——**「Search as Code」**(模型写代码来「组装搜索本身」,从而并行跑数千个检索步)和**多模型子任务路由**(把研究拆成子任务,按专长派给 20+ 前沿模型,如法律模型审合同、数据模型核表格;据报道核心 reasoning engine 用 Claude Opus 4.6)。这已经从「一个 agent 做研究」演化成「一个编排层调度一群模型做研究」。

Perplexity 还有个 **Comet 浏览器**(agentic browser),是「让 agent 在真实网页上替你做事」的执行面。这里必须挂一个⚠️:据 Brave 安全团队公开(2025-08 起),Comet 存在**间接提示注入(indirect prompt injection)**漏洞——「Summarize this webpage」会把不可信页面内容直喂 LLM 而不区分「指令」与「内容」,页面里隐藏的 HTML 甚至截图里近乎隐形的文字都可能被当命令执行,以用户已认证的权限跨域行动。**这是 agentic 浏览器 harness 的结构性风险,也是本篇第 5 节要展开的头号坑。**(采用度方面:核心产品闭源、无官方 star 数;用户/估值等数字均来自第三方统计与媒体报道、无官方财报,一律当估算看。)

> 把两个另类对齐一句:**STORM 赌「问对问题」(让 agent 替你发现盲区),Perplexity 赌「答得又快又带引用」(把研究做成秒级可用的产品)。** 一个是学术界对「研究质量」的深挖,一个是工业界对「研究速度与产品化」的极致——两头都没离开「planner → 并行检索 → 带引用综合」这条骨架。

## 4. 它们与众不同的设计决策与取舍

退一步看,四家其实在回答同一个问题——「**怎么让一个 agent 把一个开放问题研究得又全、又快、又可信?**」——但下注的位置各不相同。挑几个真正分野的维度对齐看:

- **静态流水线 vs 动态主从**。GPT-Researcher 标准模式和 DeerFlow 1.x、STORM 都是**显式的固定管线**(planner → 检索 → 综合,节点画得清清楚楚)——好懂、可预测、好调试。DeerFlow 2.0 则押**动态主从**:lead agent 自己决定何时拆、拆几个子代理。**取舍**:流水线简单可控但天花板低;动态主从灵活能干杂活,但行为更难预测、组装成本更高。

- **专用系统 vs 通用 agent 的一种用法**。STORM 把「研究/写维基长文」做成一个**专用管线**——专而精。DeerFlow 2.0 反过来,把研究**降级成通用 agent 顺手能干的一件事**。这条演化弧线很值得玩味:**当底座模型够强,专用研究管线的收益在递减**——一个足够通用的 agent 循环加几个检索工具,就能把研究做得不错。这和讲自主 SWE 那篇里「模型变强、精心脚手架收益递减」是同一个道理的另一面。

- **开源可审计 vs 闭源快产品**。DeerFlow / GPT-Researcher / STORM 全开源——你能自托管、能审计引用、能换模型、不被锁厂。Perplexity 闭源商业——开箱即用、极速、产品化到位,但模型和编排都不公开、能力宣称难独立核验。**取舍**:开源换来的是掌控与透明,代价是自己配 key、自己运维;闭源换来的是丝滑体验,代价是黑箱和 vendor 数字。

- **上下文压缩的四种实现,同一个母题**。全都要解决「几十上百个来源塞不进窗口」:GPT-Researcher 用**逐源摘要 → 聚合(map-reduce)**;DeerFlow 2.0 用**摘要 + 把中间产物落盘到文件系统 + 压缩无关内容**;STORM 用**外部化的 `StormInformationTable` / `KnowledgeBase` 思维导图**,写作时按 section 精准取用;Perplexity 用**迭代中动态维护的「研究计划 + 已读证据集」**。四种手法,同一个母题——**别把原始网页硬塞窗口,先蒸馏、再综合。** 这是深度研究 agent 的立身之本。

一句话把四条路对齐:**GPT-Researcher 把范式做到最透最好读,DeerFlow 把范式通用化成一台真电脑,STORM 把范式深挖到「问对问题」,Perplexity 把范式产品化到秒级带引用。** 四者不冲突——你完全能想象一个产品同时借鉴 GPT-Researcher 的 map-reduce、DeerFlow 的落盘外部记忆、STORM 的多视角提问、和 Perplexity 的多模型路由。

> 一个贯穿的反直觉:**这四家没有一家是靠「训一个更强的模型」赢的。** GPT-Researcher 靠清晰的编排、DeerFlow 靠 harness 化、STORM 靠「提问方法论」、Perplexity 靠推理硬件+产品化。模型是发动机,但能不能把一个开放问题研究到位,拼的是发动机外面那套「怎么拆、怎么搜、怎么压、怎么引用」的东西——这正是本站反复讲的「harness 比模型更重要」。

## 5. 局限与坑

这个流派看着最「无害」(只读、不改代码),但恰恰因为让人放松警惕,坑更隐蔽。诚实列几条(机器会渲染成⚠️警示卡):

1. **「带引用」不等于「引用是对的」——最致命的错觉。** 报告里每句话后面挂个 `[1][2]`,会给你一种「这是核查过的」的强烈安全感。但引用只保证「这句话来自那个源」,**不保证那个源本身可靠、不保证摘要没扭曲原意、更不保证 agent 没把不相关的源硬凑上去**。深度研究 agent 从**低质量来源、营销软文、甚至彼此抄来抄去的内容农场**里综合出来的报告,照样能挂满引用、读着很权威。怎么避:**把带引用报告当「研究的起点」而非「结论」——至少抽查几条关键断言,点开引用看原文对不对得上。**

2. **间接提示注入,在深度研究流派下被放大成结构性风险。** 这一类 agent 的动作核心就是「自己上网、读任意页面」——而它读的内容**未必区分「这是数据」还是「这是指令」**。一段藏在网页 HTML、PDF、甚至截图里近乎隐形的恶意文字,可能诱导一个**无人盯着**的 agent 去执行页面里的命令。Brave 安全团队 2025 年就公开了 Perplexity Comet 的这类漏洞——「总结这个网页」能被网页内容劫持,以用户已认证的权限跨域行动。**自主度和联网面越大,这个攻击面越危险。** 怎么避:对能替你操作(而非只读)的 agentic 浏览器保持警惕,别让它带着你的登录态去读不可信页面;只读研究相对安全,但结论仍要防「被投毒的来源」。

3. **信息覆盖度的幻觉:它只研究了「搜得到的」那部分。** 一份 5 页带引用的报告会让你以为「这个主题被覆盖全了」,但 agent 的世界边界**就是它的检索器能搜到的东西**——付费墙后的、没被索引的、非英文的、太新还没被收录的,它统统看不见。CMU 的 DeepResearchGym 基准里明确指出:**即使表现最好的系统,在信息覆盖度上仍逊于写作质量**——也就是说,报告「读起来很全」和「实际上很全」是两回事。怎么避:对结论的**完整性**保持怀疑,尤其涉及小众/最新/非英文主题时;必要时换检索器、加本地私有文档。

4. **厂商的 benchmark 数字,别照单全收。** 深度研究是营销重灾区:GPT-Researcher 官网的「DeepResearchGym 排名第一」经核查**与原论文不符**(论文根本没评综合第一);Perplexity 的一串 HLE / BrowseComp 提升数字**多来自报道与厂商自述、无独立审计**。同一个基准,换个测法、换个数据集变体、换个重试次数,分子分母就全变了。怎么避:引用任何深度研究成绩,务必带「什么模型、哪个基准的哪个变体、什么时候、谁测的」;**把「官网自夸」和「论文/独立评测」严格分开**——这是拆任何一家(哪怕开源项目)都要守的纪律。

5. **并行扇出得越宽,烧钱越快、越可能「宽而不深」。** 「并行开几十路检索」是这个流派提速的法宝,但每一路都要调 LLM 做摘要——**扇出越宽、下钻越深,token 账单涨得越猛**(GPT-Researcher 标准模式 ~$0.1,Deep Research 递归模式就跳到 ~$0.4,量级差在广度×深度)。更隐蔽的是:一味求广会**摊薄深度**——读了 50 个源每个只瞄一眼,不如读 10 个源每个读透。怎么避:按预算和主题选档(简单主题用标准模式,复杂主题才上递归/多智能体);把 breadth/depth 当旋钮调,别默认「越宽越好」。

6. **溯源要溯到「代码/论文」,别停在「官网首页」。** 一个看似无害的坑:GPT-Researcher 主项目是 **Apache-2.0**,但官网首页写着「License: MIT」——因为它把**独立的 `gptr-mcp` 仓库**的 MIT 许可和主项目混写了。许可证、benchmark、版本号这些「硬事实」,官网口径经常滞后或混淆。怎么避:关于开源项目的硬事实,**以仓库(LICENSE 文件、GitHub API、release tag)为准**,官网营销页只当线索。

## 6. 一句话记住它们 + 对比表

> **深度研究 agent = 接一个开放问题、自己拆成子问题、并行去多源检索与阅读、逐源摘要后综合成一份带引用长报告的 agent。它是 RAG 被「agent 化」后最成熟的形态——GPT-Researcher 把范式做到最透,DeerFlow 把范式通用化成一台真电脑,STORM 把范式深挖到「问对问题」,Perplexity 把范式产品化到秒级带引用。骨架只有一条:planner → 并行检索 → 带引用综合;四家的差异,全是在这条骨架不同环节下的不同注,没有一家是靠更强的模型赢的。**

四家横向对齐(数字均为截至 2026 年中的快照,会持续漂移):

| 维度 | DeerFlow(字节跳动) | GPT-Researcher(Assaf Elovic) | STORM / Co-STORM(斯坦福 OVAL) | Perplexity(Perplexity AI) |
|---|---|---|---|---|
| **定位** | 长时程「超级 agent 运行时/harness」,研究只是其一 | 开源自主深度研究 agent,可嵌可跑 | LLM 知识策展系统(写维基式长文) | 带引用答案引擎 + agentic 研究/浏览器 |
| **内核** | 1.x 静态图(Planner/Researcher/Coder/Reporter);2.0 动态主从 + 真电脑沙盒 | planner 扇出 N 路「搜索→摘要」→ map-reduce 综合 | 多视角提问 + 模拟对话 grounding → 四阶段管线;Co-STORM 多 agent 回合制 + 思维导图 | 迭代 search-read-reason 回路;2.0 多模型子任务路由 + Search as Code |

**怎么选**:想**读懂这个流派、要一个能 hack 的最小基线**,从 GPT-Researcher 入手(它的三层结构就是范式本身);想**自托管一个能长跑、能挂进团队 IM、还能顺手写代码/造网页的通用运行时**,看 DeerFlow(2.0);想**研究学术级的「知识策展 / 怎么问对问题」或撞见自己的盲区**,看 STORM / Co-STORM;想**开箱即用、要又快又带引用的答案**、且能接受闭源和厂商自报数字,用 Perplexity。

对照前面几篇,这一篇补上了「深度研究」这块拼图:讲自主 SWE 那篇的 agent 会**动你的代码**(写文件、跑测试、开 PR),风险在「改」;而这一篇的四家绝大多数是**只读的研究员**(搜、读、综合),风险在「信」——它给你的每份带引用报告都很有说服力,但**「带引用」不等于「可信」,你依然是那个最后负责的人**。这也是为什么它成了 agent 最早、最稳的落地形态之一:价值肉眼可见,而爆炸半径,比会改代码的 agent 小得多。

> 👉 深度研究把「planner → 并行 → 综合」这条范式跑通了;接下来的融会篇,我们把这些拆过的形态收拢,看它们如何在真实工程里拼成一个能用的系统。
