## 0. 全景:有了「怎么设计」,还得会「怎么判断好不好」

上一篇(㉙)给了你一套**设计方法**:任务倒推、笨循环、上下文当预算、权限分层……当你照着搭出(或挑出)一个 harness,一个问题立刻砸过来:**你凭什么说它「好」?换个模型、换套 harness,是真变强了,还是这次运气好?**

这就是评测(evaluation)要回答的。它听着像收尾的杂活,其实是**整个工程闭环里最容易骗到自己的一环**——因为 agent 评测几乎每一步都埋着陷阱。

```
㉙ 怎么设计            ㉚ 怎么判断好不好(你在这)         ㉛ 落到芯片
┌──────────┐         ┌───────────────────────────┐    ┌──────────┐
│ 决策框架  │         │ 评什么 · 用什么基准 · 怎么防坑 │    │ 覆盖率    │
│ 一串取舍  │  ═════▶  │ 一个数字几乎说明不了任何事    │ ═▶ │ 功能正确  │
└──────────┘         └───────────────────────────┘    │ 能否流片  │
  「怎么搭」               「搭得好不好,怎么才知道」          └──────────┘
```

这一篇不教你「刷高某个榜」——恰恰相反,它教你**看穿榜单**:当有人甩给你一句「我们 SWE-bench 87%」,你要能立刻反问出一串问题,把这个孤零零的数字还原成它真正的含义。

> 读这一篇的正确姿势:**把它当成一副「反诈眼镜」。** agent 评测领域充斥着好看但误导的数字;戴上这副眼镜,你才不会为一个不完整的分数买单——无论那分数是别人给你的,还是你给自己的。

## 1. 为什么评测 agent 特别难

先建立一个反直觉的认知:**评测一个 agent,比评测一段传统软件难一个量级。** 原因藏在它的本性里(①篇——它是个「会犯错、靠反馈、需护栏的概率循环」)。传统软件是**确定的**:同样输入永远同样输出,单元测试红了就是错、绿了就是对。而 agent 违反了这里几乎每一条前提:

- **它是概率的,不是确定的。** 同一任务跑两次,路径可能完全不同、结果一次过一次挂。你测的不是一个函数,是一个**分布**——单跑一次拿到的,只是分布里的一个采样点。
- **它是多步的,不是一锤子的。** 一次「修好 bug」要读文件、改代码、跑测试、再改……动辄几十步;中途任一步走岔都可能连锁崩,而「最终对了」也可能是绕了大段冤枉路、烧十倍成本蒙对的。**只看终点,看不见它一路上干了什么。**
- **它和环境强耦合。** 能力一大半来自真实反馈(测试报错、编译失败)。换个 Docker 镜像、换个仓库版本、网络慢一点,表现就能变样——它不是在真空里做题,是在一个**活的、会变的环境**里摸爬。
- **「对」常常没有唯一标准。** 补丁 A、B 都能让测试变绿,哪个更好?覆盖率到 95% 算过还是要 98%?很多 agent 任务的「正确」是**开放式**的,没有能机器判定的标准答案。

四条叠起来,就明白 agent 评测为什么这么滑:**你面对的是一个概率的、多步的、和环境纠缠的开放式系统,却总想用一个像单元测试那样干脆的数字去概括它。** 这个「想」,就是一切陷阱的源头。

> **传统软件评测是「测一个函数」,agent 评测是「测一个会随机、会漂移、走很多步、还没标准答案的活物」。** 别用测函数的直觉去测 agent。

## 2. 前提:先想清「评什么」

在挑基准、跑分之前,先问一个更根本的问题:**你到底想知道它哪方面好?** 因为「好」不是一个维度。新手最常见的错误,就是把「评测 agent」默认等同于「测任务成功率」——好像只有一个数字似的。真实的 agent 好坏,至少铺在**五个维度**上,而且它们经常互相拉扯:

| 维度 | 它回答什么 | 典型指标 | 为什么常被忽视 |
|---|---|---|---|
| **任务成功率** | 能不能把事干成 | Pass@1、resolve rate | 最直观,常被当成唯一维度 |
| **成本** | 干成花了多少 | token、美元、ACU(见 §3.4) | 榜单几乎从不并列,一算就现形 |
| **可靠性 / 方差** | 稳不稳——每次都行还是碰运气 | pass^k、多次跑的方差 | 单次跑根本测不出来 |
| **安全 / 轨迹合规** | 中途有没有越权、绕路、干坏事 | 轨迹级审计(⑨) | 只看最终输出时完全隐形 |
| **轨迹质量** | 干净利落还是瞎折腾蒙对的 | 步数、绕路次数、人评 | 需看过程,不只看结果 |

这五个维度经常**打架**:一个 agent 可能成功率高但巨贵;另一个便宜但方差大(时好时坏);第三个又快又准却会在中途偷偷 `rm` 掉不该动的东西(⑨)。**没有哪个 agent 在五维上全赢**——所以「评什么」永远得先服从「你的真实任务最在乎哪几个」。举个勾㉛的例子:给芯片写验证平台的 agent,你最在乎的多半是**功能覆盖率**(该测的有没有测到)和**可靠性**(能不能稳定复现);而要在生产 CI 里无人值守跑的,**成本和安全**的权重会陡然上升。

而且「在乎哪几维」还随 **agent 形态**变:编码 agent 的重心是任务成功率 + 成本;**深度研究 agent**(如 DeerFlow、GPT-Researcher、STORM、Perplexity)几乎没有单一「通过率」可言,你在乎的是**报告质量 + 引用可核查性 + 覆盖度**;而**应用生成器 / Vibe Coding**(Replit、Bolt.new、v0、Lovable)连机器判定的正确答案都没有,更多看**一次成活率、部署成不成功、产品留不留得住**这类产品指标。换个形态,五维的权重就得重排。

> **「评测 agent」不是「测一个数」,是「在你真正在乎的几个维度上各测一笔」。** 顺序反了,你会拿着一堆精确但不相干的数字。

## 3. 核心内容

五个子节:主流基准(3.1)→ 本篇最硬的一条铁律(3.2)→ 评测的陷阱(3.3)→ 超越通过率的维度(3.4)→ 怎么务实地评一套东西(3.5)。

### 3.1 主流基准速览:各自测什么、怎么判定

「基准(benchmark)」就是一套**固定题目 + 一套判定对错的机器规则**,让不同 agent 在同一张卷子上比。了解它们不是为了背分数(分数天天变),而是看懂**各自在测什么、用什么方式判定**——这决定了一个分数到底能说明什么。

**通用编码 / 软件工程:**

- **SWE-bench / SWE-bench Verified** —— 最有影响力的编码 agent 基准。题目是**真实 GitHub 仓库里的真实 issue**,判定极硬核:**把配套测试跑一遍,补丁能让原本失败的测试变绿(且不弄挂原本绿的)就算过**。用测试当裁判,好处是客观可机器判定;代价是「测试没覆盖到的正确」它看不见。**Verified** 是经人工筛掉含糊/坏题的高质量子集,当下引用最多。⚠️ 第⑯篇讲的 **SWE-agent** 正是为了跑通 SWE-bench 而生——它的 **ACI(代理-计算机接口)** 把 GPT-4 从非交互 RAG 的 3.8% 拉到 12.5%(截至 2026 中榜首已推到 87–88% 量级,但**分数飞快过时,引用必须带 as-of 日期**)。
- **Terminal-Bench** —— 测 agent 在**真实终端环境**里完成任务(不止编码,还有环境配置、命令行操作)。它对 harness 特别敏感:⑥篇提过,光换 harness 就能在这类榜上从三十名外窜进前五。
- **GAIA** —— 测**通用助理**能力:需多步推理 + 工具使用(搜索、读文件、计算)才能答对的真实问题。题「对人很简单、对 AI 很难」,专戳多步工具链的短板。这类需要多步检索、跨源汇总的基准,也最贴近**深度研究 agent**(DeerFlow、GPT-Researcher 一类)要考的能力。

> 💡 **换个 agent 形态,就得换一整套基准。** SWE-bench 这类「跑测试判绿」的卷子只适合编码 agent;**GUI / 计算机使用 / 浏览器 agent**(browser-use、Anthropic Computer Use、OpenAI Operator)得用**模拟真实网页 / 桌面任务**的环境类基准——网页操作看 **WebArena**、桌面操作看 **OSWorld** 这类,判定的是「在一个活的环境里有没有把事办成」,而非「补丁能不能编译」。不确定该用哪套时,记住原则就够:**这类 agent 需要一套面向真实环境的基准,而不是拿编码榜硬套。**

**领域基准(以芯片为例,勾㉛):** 通用榜衡量不了领域能力,于是有专门的领域基准(第⑲篇拆过这一域):

- **VerilogEval** —— 测 LLM **生成 Verilog RTL** 的能力,判定「生成代码功能是否正确」。
- **ChatEDA-Bench 一类** —— 测 agent 驱动 **EDA 流程 / 工具链**的能力。
- **以覆盖率为准的验证评测** —— 最贴近⑲篇那类**功能验证 agent**(如 UCAgent)的评法:判据不是「代码对不对」,而是**它写的验证平台达到了多少代码 / 功能覆盖率**。比如 UCAgent 在 IntegerDivider 上「三个模型都超 97% 代码覆盖率、稳定 100% 功能覆盖率」——这里的「过」是用**覆盖率数字**判定的,不是一个 GitHub 测试。

把判定方式并起来看:

| 基准 | 测什么 | 判定方式 |
|---|---|---|
| SWE-bench Verified | 修真实 GitHub issue | 配套测试是否变绿(且不弄挂) |
| Terminal-Bench | 终端里完成任务 | 任务级成功判据 |
| GAIA | 通用多步助理 | 标准答案比对 |
| VerilogEval | 生成 Verilog | 生成代码功能是否正确 |
| 覆盖率验证评测 | 写验证平台 | 达到的代码 / 功能覆盖率 |

> **看一个分数前,先看它背后的「判定方式」——跑测试、比标准答案,还是量覆盖率。** 判定方式决定了这个数字「能证明什么、又对什么盲」。

### 3.2 本篇最硬的一条:不披露 harness,就别比 agent

如果这一篇你只带走一句话,就是这句——它推翻了绝大多数人对 agent 排行榜的默认理解。

默认的直觉是:**「SWE-bench 榜上分数高 = 这个模型强。」** 错。一个 agent 的分数,是 **模型 × harness** 一起跑出来的(⑥篇「harness 常比模型更重要」那条主线)。harness 的影响有多大?看两篇学术工作(截至 2026 中):

- **Claw-SWE-Bench**(arXiv 2606.12344,350 个 issue / 8 语言 / 43 仓库):**同一个模型 Qwen 3.6-flash,只换 harness**,五套 harness 的 Pass@1 从 **38.6%(GenericAgent)一路铺到 66.0%(OpenClaw)——差 27.4 个百分点**,已经**赶上换一整档模型**的量级。更极端:同模型 GLM 5.1,「裸适配器」只拿 19.1%(约 69% 补丁根本没打上),换「完整适配器」直接到 73.4%(打补丁失败率 <1.5%)。**换的不是模型,是模型外面那层壳。**
- **《Stop Comparing LLM Agents Without Disclosing the Harness》**(arXiv 2605.23950,Zhang 等):把现象提炼成一条命题——**Binding Constraint Thesis(约束绑定论)**:对能力相近的前沿模型、在长程任务上,**harness 带来的方差可以超过模型带来的方差**,严重到能把两个模型的排名**颠倒过来**。结论一针见血:**不披露 harness 的 agent 排行榜是「不完整、可能误导」的。**

```
              同一个模型,只换 harness
   Pass@1
   66% ┤                    ██  OpenClaw
       │                ██
       │            ██
       │        ██
   39% ┤    ██  GenericAgent
       └────┴─────────────┴──── 27.4 个百分点,全来自「壳」
         裸适配器 19% ─────────▶ 完整适配器 73%(GLM 5.1)
```

这意味着:当你看到「模型 A 在 SWE-bench 上 82%、模型 B 84%」,若**没告诉你各自套的什么 harness**,这个比较**什么都证明不了**——很可能 A 只是配了套烂 harness,换个好壳就反超 B 了。这正是⑥篇「harness > model」在评测上的直接后果。

> **不披露 harness 的 agent 分数,和不披露路况的「百公里油耗」一样——不是假,是没法用。** 比模型就固定并公开 harness;比 harness 就固定并公开模型。混着比,排名可能是反的。

### 3.3 评测的陷阱:好看的数字怎么骗人

披露 harness 只防住了一个坑。agent 评测里还有一整排陷阱,每个都能让「亮眼数字」变误导:

- **数据污染(contamination)。** 基准题目(某个 GitHub issue 和它的答案)**混进了模型的训练集**——模型不是「解出来」是**背出来**的,分数虚高,一换新题就现原形。老基准尤其危险:越出名越可能被爬进训练数据。
- **过拟合基准(刷榜 ≠ 真强)。** 团队为名次专门针对这套题调 harness、调提示、调 few-shot;分数真涨了,涨的却是「应付这套卷子」而非通用能力。**榜单一高就以为无敌,换个真实任务就露馅。**
- **樱桃挑选(cherry-picking)。** 跑十次挑最好那次报,或五个基准只报自己赢的那两个。每个数字都「真」,合起来是精心裁剪的假象。
- **pass@k 和 pass@1 被混为一谈。** `pass@1` 是「一次就过」的概率;`pass@k` 是「给 k 次机会至少过一次」——k 越大越好看。**「pass@10 九成」和「pass@1 九成」天差地别**:前者可能意味着单次只有 40%、靠多试蒙的。看到 pass@k,先问 **k 是几**。
- **单次跑 vs 多次方差。** agent 是概率的(§1),单跑一次拿到的数字可能只是分布里运气好的那一端。**用一次跑下结论,等于用一次掷骰子判断骰子是几点。**
- **把成本排除在外。** 几乎所有榜单只比「过没过」,绝口不提「花了多少」。一个 pass@1 高 2 个点却贵 10 倍的 agent,榜上「赢」了、账单上是灾难。**成本被系统性地从比较里抹掉**,这是最普遍也最隐蔽的失真。

> **一个 agent 分数至少要过七道检疫——没污染、没过拟合、没樱桃挑选、pass@k 说清、跑够多次、算进成本、披露 harness。** 少一道,可信度就打一次折。

### 3.4 超越「通过率」的维度:成本、可靠性、轨迹、谁来评判

通过率只是五维里的一维(§2)。真正把 agent 评明白,得看另外几个更难量、却往往更要命的维度:

**① 成本(cost)。** 别只问「过没过」,要问「过它花了多少」。度量有几种:**token 数**(最底层)、**美元**(最直接)、以及产品自定义的抽象单位——比如商用 agent Devin 用 **ACU(Agent Compute Unit)** 把「虚拟机时间 + 模型推理 + 网络」归一成一个可计费的算力单位。⑯篇讲的 **SWE-agent** 更干脆,把**成本当第一控制旋钮**:`per_instance_cost_limit` 默认 **$3.00**,一超预算就自动交卷——因为不同模型的「步数」根本不可比,只有花掉的钱是可通约的货币(⑯)。**评测里带上成本,排名常常会重排。**

**② 可靠性(reliability):方差与 pass^k。** 「平均能过」不等于「靠得住」。一个 pass@1 是 70% 的 agent,可能意味着**每次跑都有三成概率翻车**——放进无人值守的生产流水线不可接受。这里有个和 pass@k 长得像、意思却相反的指标:**pass^k = 连续 k 次全都过的概率**。pass@k(至少过一次)奖励「碰运气」,pass^k(k 次全过)考验「稳」——对生产系统,**pass^k 往往比 pass@1 更能说明它能不能真用**。

**③ 轨迹级评估(trajectory-level)。** §1 说过,只看最终输出,看不见 agent 一路上干了什么——而那里藏着最危险的信息:**它有没有在中途越权、绕路、干不该干的事?** 第⑨篇讲过一个反直觉事实:一个 agent 完全可能**给出正确的最终答案,却在轨迹中途违反权限**(偷读不该读的文件、试图执行危险命令)。这正是「审计 agent harness 安全」那类工作(HarnessAudit 一类,⑨)的核心洞察:**违规发生在轨迹中途,而不是终点输出**——只看输出的评测对它完全盲。要评安全,就得**审整条轨迹**。

**④ 人评 vs LLM 当裁判(LLM-as-judge)。** 当「对」没有机器判据(§1 的开放式任务),你得靠人打分,或者让另一个 LLM 来打分(LLM-as-judge):

| 评判方式 | 优点 | 代价 / 风险 |
|---|---|---|
| **人评** | 最贴近真实质量判断 | 贵、慢、难规模化、评委间不一致 |
| **LLM-as-judge** | 便宜、快、可规模化 | 有系统偏见(偏爱长答案、同源风格),**必须先校准** |

LLM-as-judge 是省钱的诱惑,但不是免费午餐:**用它可以,但必须先拿一批人评样本校准过**——否则你是在用一把有偏的尺子量东西,还以为自己客观。这一维在**深度研究 agent**(GPT-Researcher、STORM 一类)上尤其吃重:一份研究报告没有机器判定的「标准答案」,你评的是**内容覆盖度 + 每条论断的引用能不能核查**,而引用可核查性(去核对它引的源真存在、真支持那句话)恰恰是 LLM-as-judge 最容易睁眼说瞎话、最需要人评兜底的地方。

> **通过率告诉你「能不能成」,成本告诉你「值不值」、方差告诉你「稳不稳」、轨迹告诉你「干不干净」、裁判的选择决定「你信的这把尺准不准」。** 缺了后几维,「通过率高」是一句危险的半真话。

### 3.5 怎么务实地评一套 harness / agent

当你真要评一套东西时,记住四条务实法则,每一条都直接堵住前面某个坑:

1. **别混着比:固定一边、变另一边。** 想知道 harness 好不好?**固定模型只换 harness**(像 Claw-SWE-Bench 那样)。想知道模型强不强?**固定 harness 只换模型。** 一次只动一个变量——控制变量法的常识,却是 agent 评测里最常被违反的一条(§3.2)。
2. **可复现:公开 harness 与全部配置。** 别人复现不了的分数等于没有分数。**把 harness 版本、模型快照、提示、工具集、环境镜像、随机种子全记下来、能公开就公开。** 不可复现的评测,连你自己下次都对不上。评**多智能体 / 工程框架**(CrewAI、AutoGen、LangGraph、DSPy、Mastra 一类)时,这一条尤其吃紧——你评的往往不是单个答案对不对,而是**编排本身稳不稳、可不可复现**;有些框架干脆把评测内建进流程(如 DSPy 的优化本身就靠一个 metric 驱动、Mastra 等内置 evals),等于逼你先把「怎么算好」这把尺子定下来。
3. **看分布,不看单点:多跑几次。** agent 是概率的(§1)。**同一配置至少跑若干次,报分布(均值 + 方差 / pass^k),不是某一次的幸运值。**
4. **贴近你的真实任务,别只盯公开榜。** 公开基准测的是「一般性能力」,未必等于「在你的代码库 / 芯片 / 流程上的能力」。**最有说服力的评测,是拿一批你自己的真实任务当私有基准**——它没被污染(§3.3)、又最贴近你要它干的活。公开榜看趋势,私有集做决策。

```
   ① 固定一边、变另一边  ─▶ 堵「混着比」
   ② 全配置可复现       ─▶ 堵「不可复现」
   ③ 多跑看分布         ─▶ 堵「单次跑」
   ④ 用你自己的私有题    ─▶ 堵「污染 + 榜≠你」
```

---

## 4. 关键心态:一个孤零零的数字,几乎什么都说明不了

把前面浓缩成一个心态:**「SWE-bench 87%」这样一个光秃秃的数字,几乎什么都证明不了。** 因为一个 agent 分数被至少**五个隐藏变量**同时决定。听到任何 agent 分数,都该像条件反射一样追问这五件事:

```
   「SWE-bench 87%」—— 慢着,先问清这五件事:
   ┌─────────────────────────────────────────────────────┐
   │ 什么模型?     ── 哪个模型、哪个版本快照?              │
   │ × 什么 harness? ── 套的什么壳?(§3.2:换壳能差 27 个点)│
   │ × 什么任务集?   ── 全集还是 Verified 子集?有没有污染?  │
   │ × 跑几次?       ── 单次还是多次?方差多大?pass@ 还是 ^? │
   │ × 花多少钱?     ── 每题成本?贵 10 倍换来的 2 个点值吗?  │
   └─────────────────────────────────────────────────────┘
      这五个空没填满之前,那个「87%」是一句没法用的话
```

五个空里有一个没填,那个漂亮数字就是**悬空**的。别人给你悬空数字,你有权不信;你给自己报悬空数字,那是自欺。

> **判断一个 agent,不看它报了多高的分,看它敢不敢把「什么模型 × 什么 harness × 什么任务集 × 跑几次 × 花多少钱」五格全填满。** 敢填满、且填得难看的,往往比只敢报一个漂亮数字的更可信。

## 5. 常被忽视的坑(评测里最容易栽的几处)

1. **唯 SWE-bench 论。** 把一个基准当 agent 好坏的唯一裁判。它只测「修 GitHub issue」、只用测试判定(§3.1),测不到你的领域(如芯片覆盖率)、也测不到成本和安全——更别说换了形态它根本不适用:浏览器 / 计算机使用 agent 该看 WebArena、OSWorld 这类真实环境基准,深度研究 agent 该看报告质量与引用可核查性,应用生成器该看一次成活率与部署留存(§2、§3.1)。**避:按真实任务和 agent 形态选维度和基准,公开榜看趋势、私有集做决策。**
2. **不算成本。** 只比「过没过」、不比「花多少」;贵 10 倍只高 2 个点的 agent 榜上赢、账单上输(§3.4)。**避:把 token/$/ACU 并进每次比较。**
3. **单次跑就下结论。** 拿一次掷骰子判断骰子。agent 是概率的,单点数字掩盖巨大方差(§1)。**避:同配置多跑几次,报分布(均值 + 方差 / pass^k)。**
4. **基准污染没查。** 题目早被爬进训练集,模型是背答案不是解题,分数虚高(§3.3)。**避:优先用新题 / 私有题,对老基准的高分保持怀疑。**
5. **拿不同 harness 的分横比模型。** 最硬的一条(§3.2):同模型换 harness 能差 27.4 个点、甚至颠倒排名(Binding Constraint Thesis,⑥)。**避:一次只动一个变量,且披露 harness。**
6. **只看通过率,不看轨迹安全。** agent 可能答对却在中途越权、绕路、干坏事——只看输出完全看不见(⑨)。**避:审整条轨迹,危险动作在 schema 层就杜绝。**
7. **用 LLM-as-judge 却不校准。** 图它便宜快,却忘了它有系统偏见(偏爱长答案、同源风格),分是有偏尺子量的(§3.4)。**避:先拿人评样本校准再规模化。**
8. **把 pass@k 当 pass@1 读。** 「pass@10 九成」听着无敌,单次可能只有四成靠多试蒙的(§3.3)。**避:看到 pass@k 先问 k 是几,生产系统更该看 pass^k。**

## 6. 小结 & 下一步

这一篇你需要带走的:

- **agent 评测特别难**:它是**概率的、多步的、和环境强耦合的、还常没标准答案**——别用「测函数」的直觉去测它(§1)。
- **先想清「评什么」**:成功率只是五维之一,还有**成本、可靠性/方差、安全、轨迹质量**,它们经常互相打架(§2)。
- **最硬的铁律**:**不披露 harness,就别比 agent**——同模型换 harness 能差 27.4 个点、甚至颠倒排名(§3.2、⑥)。
- **一整排陷阱** + **务实四法则**:防污染/过拟合/樱桃挑选/pass@k 混淆/单次跑/漏成本;一次只变一个变量、全配置可复现、多跑看分布、贴近真实任务(§3.3、§3.5)。
- 核心心态:**一个孤零零的数字几乎什么都说明不了**——必须问清「什么模型 × 什么 harness × 什么任务集 × 跑几次 × 花多少钱」(§4)。

方法(㉙)和评测(㉚)都有了,最后一步是看一个**真实、高约束领域里的落地案例**——它会给评测提出几道通用榜答不了的难题:验证**覆盖率**够不够、功能**正确性**能不能保证、以及那个终极判据——**系统产出的东西到底能不能进入真实工程闭环?** 这就是最后一篇要谈的。
