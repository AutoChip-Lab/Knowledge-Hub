## 0. 全景:前面的 agent 都在「读写文件」,这一篇的 agent 在「点鼠标」

到目前为止,我们拆过的绝大多数 agent,动作空间其实很干净:**读文件、写文件、跑命令、调 API**。它们活在一个由文本和结构化数据构成的世界里——一个 `read_file` 拿到的是字符串,一个 `run_command` 回来的是 stdout。哪怕是拆 Manus 那篇里的云端虚机 agent,底层的手脚也大多是「命令行 + 文件系统」这套。

这一篇换一个物种:**会用电脑的 agent(computer-using agent)**——让模型**像人一样看屏幕、移动光标、点按钮、敲键盘**。它面对的不是 API,而是一块屏幕上花花绿绿的像素;它要解决的是一类特别顽固的问题:**那些没有 API、只有图形界面(GUI)的软件**——遗留桌面程序、复杂 JS 网站、订票订餐的下单流程、跨系统抄数据的报销表单。

```
   前面各篇(文件/命令 agent)              这一篇(会用电脑的 agent)

   任务 ──▶ read_file("parser.py")         任务 ──▶ screenshot()  ← 看一张屏幕截图
        ──▶ run_command("pytest")               ──▶ 「点坐标 (512,300)」
        ──▶ 拿到 stdout(文本)                  ──▶ 「type 'cat photo'」
                                                 ──▶ screenshot() ← 再看一眼
   世界 = 文本 + 结构化数据                   世界 = 一块屏幕的像素(或一棵 DOM 树)
   动作 = 函数调用                           动作 = 鼠标 / 键盘事件
```

放进整张地图看:这一类是**「感知—动作」离屏幕最近的一层**。它逼着我们回答一个前面各篇都绕开了的问题——**当动作不再是干净的函数调用,而是「点这个坐标」这种又脆又慢的物理操作时,harness 该怎么设计?** 本篇深拆两家路线截然相反的代表:开源、DOM 优先的 **browser-use**,和纯视觉、能力烧进模型里的 **Anthropic Computer Use**;再横扫 **OpenAI Operator / ChatGPT Agent** 这条商业托管线。

> 读这一篇记住一条主线就够了:**同样是「让 agent 会用电脑」,有两条根本不同的路——一条给它看「截图 + 坐标」(视觉派),一条给它看「DOM 里的可点元素清单」(结构派)。这条分野决定了它快不快、贵不贵、脆不脆、能不能碰非浏览器的软件。**

## 1. 这一类到底是什么、边界在哪、为什么值得单独一章

先给一句话定义:

> **「会用电脑的 agent」是这样一类 agent:它不通过 API,而是通过操作图形界面(看屏幕、动鼠标键盘)来完成任务;它的动作空间是「人在电脑前会做的事」——点击、输入、滚动、拖拽、翻页。**

它的边界,靠和两个近亲划清:

- **和「函数调用式 agent」的界限**:前面各篇的 agent,面对一个网站会去调它的 REST API;这一类 agent 面对同一个网站,是**打开浏览器、真的去点那个「提交」按钮**。为什么要这么笨?因为**大量真实系统根本没有 API**——一个 1998 年的内部 ERP、一个只在浏览器里跑的 SaaS、一个需要登录态和验证码的下单页。有 API 时这一类是杀鸡用牛刀;没 API 时它是**唯一**能自动化的办法。
- **和「爬虫 / RPA 脚本」的界限**:传统 RPA(机器人流程自动化)是**录制死脚本**——「点屏幕第 300 个像素」,页面一改版就全崩。这一类 agent 是**每一步都重新看、重新想、重新决定下一步**,布局变了它能自己适应。这就是 agent 和脚本的根本差别(见讲 agent 与 workflow 界限的那篇)。

为什么值得单独一章?因为它把前面所有篇讲的机制,都推到了一个**更极端、更物理**的版本:

- **感知**变了:不再是「把文件内容塞进上下文」,而是「把一块屏幕(或一棵 DOM 树)压缩成模型看得懂的东西」——这里冒出了本篇的核心分野(§3)。
- **动作**变得**又慢又脆**:一次点击可能点偏几个像素、可能点到一个还没加载完的按钮。前面 agent 的动作大多是确定性的(写文件几乎不会「写偏」),这一类的动作**天生带误差**。
- **安全**被放到了一等位置:一个会自己点「确认支付」「接受条款」的 agent,一旦被网页里藏的恶意指令带偏(见讲权限与提示注入的那篇),后果是真金白银。所以本篇每一家都在「人在环」和「沙箱」上下了重注。

## 2. 一次任务在这类系统里怎么流动

不管视觉派还是结构派,这类系统的生命周期骨架是**同一个循环**——只是「看世界」那一步的实现不同。先看骨架,设想你丢进去一句自然语言任务:

> 「帮我在这个网站上查到最便宜的那趟航班,填好乘客信息,停在支付页等我。」

```
   任务(自然语言)
      │
      ▼
 ① 感知      看当前屏幕:视觉派 → 截一张图;结构派 → 抓 DOM,列出带编号的可点元素
      │
      ▼
 ② 思考      模型读「当前状态 + 任务 + 历史」,决定下一步动作
      │        (计划藏在模型每一步的推理里,通常没有显式的 planner/DAG)
      ▼
 ③ 出动作    产出结构化动作:视觉派 →「click 坐标(512,300)」;结构派 →「click element 42」
      │
      ▼
 ④ 执行      harness 在真实浏览器/桌面上落地这个动作(CDP 事件 / Xvfb 上的鼠标事件)
      │        ├─ 危险动作(支付/发邮件)?→ 停下,等人确认(HITL 闸门)
      │        └─ 执行完 → 回到 ① 重新截图/抓 DOM
      ▼
 ⑤ 回灌      把新状态(新截图 / 新 DOM)追加进历史,连同全部上下文再喂给模型
      │
      ├──❌ 没到目标 ──▶ 回到 ②,模型看新状态、纠正、再走一步
      │                  (observe→think→act 自主转几十步)
      ▼
   ✅ 到达目标(或触及步数/预算上限)──▶ 停在支付页,交还给你
```

这个循环的骨架,和讲核心循环那篇的 ReAct(想一步—做一步—看结果)是**同一个东西**,只是「看」变成了「看屏幕」。三个关键点,决定了这类系统跟前面各篇的不同气质:

- **① 感知是全篇的分水岭。** 视觉派看**截图(像素)**,结构派看**DOM(带索引的元素清单)**。这一步选什么,后面的快慢、成本、脆度全被它锁死。
- **② 思考里通常没有显式规划器。** 三家(browser-use / Computer Use / Operator)都**不写一个外部的 plan 图**——多步计划和「点错了自我纠正」的能力,都藏在模型每一步的推理里(见讲规划与推理那篇)。harness 主要是个「渲染器 + 执行器 + 安全闸门」。
- **④ 执行里那个 HITL 闸门是刚需,不是可选项。** 因为动作有真实后果,这类系统几乎都在「支付、发邮件、接受条款」这种 **consequential action(有后果的动作)**前强制暂停等人。这是前面写文件类 agent 不太需要、这一类却绕不开的东西。

循环骨架不新鲜,新鲜的是「①怎么看屏幕」和「④动作会造成真实后果」这两头——下面拆的三家,分野恰好就落在这两头。

## 3. 拆开看

现在把代表们分别搬上手术台。深拆两家(browser-use、Anthropic Computer Use),最后一节做「群像横扫」覆盖 OpenAI 这条线。

### 3.1 browser-use(Browser Use):灵魂是「把网页变成带编号的元素清单」

**它是谁**:browser-use 是一个 **MIT 许可**的开源 Python agent 框架,让 LLM 操控真实浏览器完成网页任务。出品方是同名公司 **Browser Use**(旧金山),创始人 **Magnus Müller** 与 **Gregor Žunič**,起源于 2024 年 ETH Zurich 的一个学生项目,后进 YC(2025 冬季批次)。截至 2026 年中,它**高度活跃、未停更、未改名**:经 GitHub API 核验(2026-07),约 **10.3 万 star**、1.1 万 fork,最新 release 已到 0.13.x(2026-07),每日级提交;已完成 1700 万美元种子轮(2025-03,Felicis 领投)。另有一个闭源商业云平台 **Browser Use Cloud**。

它的招牌不是某个模型,而是一个鲜明的立场——**别让 agent 靠「看图」操作网页,让它靠「读结构」**。创始人 Müller 说得很直白:很多 agent 依赖视觉系统,「东西就崩了」,他们的做法是**把网页转成 agent 能理解的东西**;官方也把纯视觉自动化批评为「慢、贵、不可靠」。

**先给直觉**:你把一张网页截图丢给模型,让它「点登录按钮」,它得先在一堆像素里认出哪块是按钮、算出它的坐标、然后祈祷点得准。这又慢(每步一张大图)、又贵(图像 token 是大户)、又脆(点偏了就废)。browser-use 反过来——它不给模型看图,而是**先把这一页的 DOM 扒下来,挑出所有「可点/可填」的元素,给每个编上一个整数号码**,然后告诉模型:

```
   [ 12] <button> 登录
   [ 13] <input>  邮箱
   [ 14] <input>  密码
   [ 42] <a>      忘记密码?
   ...
```

模型只需要说「**click element 42**」,harness 就知道该点哪个真实节点——**不用坐标、不用 XPath、不用视觉**。这就是 browser-use 最值得记住的思想:

> **网页对人是「一张要用眼睛看的图」,对 agent 应该是「一份带编号的可交互元素清单」。** 把感知从「像素」降维到「结构化文本」,快、便宜、可靠,而且任何会 function-calling 的 LLM 都能用——不必是视觉模型。

**这套「感知降维」具体怎么做的**(点名真实构件):

- **DOM 处理引擎**是核心。`DomService` 通过 **CDP(Chrome DevTools Protocol,Chrome 开发者工具协议)** 抓当前页的语义化 DOM 树;`ClickableElementDetector` 用一套**四档启发式**挑出「可交互元素」:(a) 原生标签(`button/input/select/a` 等);(b) 挂了 JS 点击监听器的元素(经 CDP 探测 `click/mousedown` 等);(c) 带 ARIA 角色/属性的(`role=button`、`aria-expanded` 等);(d) 表单控件的后代。挑出来后,`DOMSelectorMap` 给每个可见可交互节点分配一个**稳定的顺序整数**(整数 ID → 真实节点)。
- **视觉是补充,不是主角**:参数 `use_vision` 默认是 `"auto"`——**只在需要时才附一张截图**(比如纯 canvas 布局、DOM 说不清的情况),而不是每步都给。**DOM 文本是主输入,像素是备胎。** 这跟 §3.2 的 Computer Use 正好相反。
- **一步能出多个动作**:`max_actions_per_step` 默认 **4**——模型可以在一步里说「填第 13 个框、填第 14 个框、勾第 20 个框、点第 12 个按钮」,一批执行以省往返;但**页面一变化就停下来重新观察**(因为编号可能失效了)。

**它的控制回路**(对着 §2 骨架落地):用户调 `Agent(task="...", llm=...).run()`,进入 step 循环——`DomService` 抓 DOM → `ClickableElementDetector` 挑元素 → `DOMSelectorMap` 编号 → `MessageManager` 把 [系统提示 + 任务 + 序列化 DOM 清单 +(可选)截图 + 历史] 拼成 prompt → 调 LLM 要**结构化输出**(一段思考 `AgentBrain` + 一批动作)→ `Controller`(别名 `Tools`)把动作解析、经 CDP 执行 → 结果写进 `AgentHistoryList` → 回环。容错也很实:单步失败重试上限默认 3 次,LLM 侧限流/5xx 重试后切 `fallback_llm`,还有 `step_timeout`(120s)等看门狗。

**两个有意思的工程决定**:一是 **2025-08 从 Playwright 迁到裸 CDP**——早期靠 Playwright(浏览器自动化库)当适配层,后来官方发文《Closer to the Metal》**扔掉 Playwright、直接说浏览器母语 CDP**,提取/截图/动作全提速,顺带引入**事件总线(自研库 bubus)+ watchdog 架构**(各种 watchdog 订阅 CDP 事件、扩展功能而不动核心)。二是**工具用装饰器 + 按参数名注入**:自定义动作用 `@tools.action(...)` 注册,函数签名里声明 `browser_session: BrowserSession` 就自动被注进去;动作返回一个 `ActionResult`(带 `extracted_content`、`long_term_memory`、`is_done`、`error` 等字段)。其中 `long_term_memory` 值得一提——它让动作把「要长期记住的要点」写进长期记忆,区别于一次性的抽取结果,是对抗长任务上下文漂移的小设计(呼应讲上下文压缩那篇)。

**它的边界要诚实说**:核心库主打**单 agent 循环**;并行/多任务、隐身指纹、代理轮换、验证码求解这些主要由**商业的 Browser Use Cloud** 承担。框架内是否有正式的「多 agent 交接」原语、是否有内置的「暂停等人确认」一等 HITL 机制,**底稿未找到确切的一手来源**,这里就不下结论。性能上,官方 SOTA 技术报告称在 WebVoyager 基准上 **89.1% 成功率(586 个任务)**——但 2026 年已有竞品(如 Magnitude 宣称 93.9%)反超,且各家任务集不同**不可直接横比**,所以别拿这个数字当「它最强」的证据。

> 反直觉的点:**让浏览器 agent 变强,browser-use 的答案不是「上更强的视觉模型」,而是「干脆别看图」。** 把网页从像素降维成带编号的清单,它用一个 model-agnostic(模型无关)的框架,让任何会函数调用的普通 LLM 都能可靠地操作网页——据报道,拆 Manus 那篇里的通用 agent 早期的浏览器能力就被曝基于 browser-use(此为外部报道口径)。

### 3.2 Anthropic Computer Use:灵魂是「纯像素视觉,能力烧进模型」

**它是谁**:Computer Use **不是一个独立框架**,而是 **Claude 开发者平台(Messages API)上的一个内置工具类型**——`type` 叫 `computer_20251124`(截至 2026 年中的版本)。它让 Claude 通过「**截图看屏幕 → 输出鼠标/键盘动作 → 你在自己的沙箱里执行 → 再截图回传**」的循环,自主操作任意桌面 GUI。出品方是 **Anthropic**;工具能力本身闭源、随 Claude 模型提供、按 token 计费;官方开源了一个 MIT 许可的参考实现 `computer-use-demo`(在 `anthropic-quickstarts` 仓库里)。

状态:2024-10 首发(随 Claude 3.5 Sonnet,是「第一个提供 computer use 公开 beta 的前沿模型」);截至 2026 年中**仍标注 beta、未 GA**,但迭代很快——工具版本已三代演进(`computer_20241022` → `computer_20250124` 加了 scroll/hold_key/wait → `computer_20251124` 加了 `zoom`),并已成为 Anthropic 多条 agent 产品线(如浏览器 agent「Claude for Chrome」)的底层能力。

它和 browser-use 站在**光谱的两端**:browser-use 是「DOM 优先、模型可插拔的框架」,Computer Use 是「**纯像素视觉、能力烧进 Claude 权重、harness 极薄的能力原语**」。

**先给直觉**:browser-use 的思路是「网页有 DOM,我把结构扒出来给模型」。但**很多东西没有 DOM**——一个桌面版 Photoshop、一个远程桌面、一个 canvas 游戏、一个 1998 年的 Win32 程序。这些东西你扒不出「可点元素清单」,唯一能看的就是**屏幕上的像素**。Computer Use 的赌注就是:

> **人能点的,Claude 就能点。** 不依赖 DOM、不依赖无障碍树,就看屏幕像素、输出通用的鼠标键盘动作——所以它能泛化到**任何** GUI,而不只是浏览器。代价是:点击靠坐标,天生更容易点偏、更慢、更贵(每步一张截图)。

**它由三块拼成**(这是理解「它到底是什么」的关键):

**A. 工具契约(内置在模型里,Anthropic 侧)。** 这三个内置工具是**「schema-less」**的——你不用提供输入 schema,因为 *"schema 内置在 Claude 模型里、不可修改"*:`computer_20251124`(图形操作)、`bash_20250124`(跑 shell)、`text_editor_20250728`(改文件)。当你的请求带上这些工具时,API 会**自动注入一段 computer-use 专用的系统前缀**(告诉模型「你有一个沙箱计算环境……」),再拼到你自己的 system 提示上。还有一个安全亮点:**内置的提示注入分类器**——截图会自动过一层分类,命中疑似注入时会**自动转成让 agent 先向用户求确认**。

**B. Harness / Agent Loop(参考实现 `loop.py`,你侧)。** 这一层薄到几乎只是「执行动作 + 回传截图」。核心是 `sampling_loop`:取工具组 → 拼系统提示 → 反复调 API → 跑工具 → 回填结果。里面两个上下文小机制值得记(呼应讲上下文管理那篇):`_maybe_filter_to_n_most_recent_images` **只保留最近 N 张截图**(截图是 token 大户,更早的成块删掉);`_inject_prompt_caching` 给最近 3 个用户轮次和系统提示打 `ephemeral` 缓存标记省钱。(注:开 prompt caching 时那个删图会被关掉——缓存命中读起来更便宜,留着反而好。)它**没有长期记忆、没有向量库**——记忆就是「消息历史 + 最近几张截图」。

**C. 沙箱 / 执行器(参考实现是个 Docker 容器)。** 里面用 `Xvfb`(X 虚拟帧缓冲)渲染一个看不见的桌面,`Mutter` 当窗口管理器,预装了 Firefox、LibreOffice 等,还开了 VNC/noVNC 端口让你能看它在干什么。官方明确说这个 demo **有意最小化**:agent loop 和被控桌面**跑在同一个容器里(弱隔离)**,是 demo 取舍,生产环境要另做隔离。

**它的控制回路**(用官方例子「把一张猫的图片存到桌面」走一遍):你的 harness 调 `POST /v1/messages`,带上 `anthropic-beta: computer-use-2025-11-24` 头和那组工具 → **Claude 决定「先看一眼」,返回一个 `{"action":"screenshot"}` 的 tool_use** → 你的 harness 在 Xvfb 上截图、转 base64 → 封成 `tool_result` 追加为一条 user 消息、连同全部历史再发给 API → Claude 看到截图、决定下一步(`{"action":"left_click","coordinate":[500,300]}` 打开浏览器 → `type` 搜索 → 保存)→ **执行、回传、再看** 无人介入地反复,直到 Claude 返回**不含任何 tool_use 的纯文本**(任务完成),或触及 `max_iterations`(参考实现示例默认 10)这个防死循环的硬保险。

**这个回路里有三个「讲课要点」级别的细节**:

1. **Claude 永远不直接连沙箱。** 官方原话:*"你的应用必须显式地运行 computer use 工具;Claude 不能直接运行它。"* 模型只**产出动作意图**,落地永远是你的代码干——这是一道天然的安全边界。
2. **它是「视觉闭环」,不是状态机。** 没有显式规划器/DAG,计划就藏在模型每一步推理里。官方甚至建议你在 prompt 里让它 *"每一步之后都截个图、仔细评估是否达到了预期结果"*——因为模型有个毛病:**会假设动作成功了而不去检查**。这句话本身就是一堂关于「视觉 agent 为什么脆」的课。
3. **坐标要自己缩放。** Claude 按它「看到的图像分辨率」返回坐标,如果你的真实屏幕比模型的图像上限大,你必须**自己把截图缩小、再把返回的坐标反向放大**回真实屏幕。官方推荐工作分辨率是 **XGA 1024×768**(准确率最佳);还有个新动作 `zoom`——传一个区域让模型看局部的全分辨率,专门用来看小字、文件名、行号。

**能力势头**(以 OSWorld 这个衡量 GUI 操作的核心基准看,逐代跃升很清晰):首发的 Claude 3.5 Sonnet 只有 screenshot-only **14.9%**(当时次优系统才 7.8%),加了步骤后 **22.0%**;此后逐代更新一路涨到 Opus 4.8 在 OSWorld-Verified 上约 **87.1%**。**但要打个星号**:14.9% / 22.0% 是官方首发公告数字,后面那串逐代分数来自第三方汇编(Vellum 等)、**未逐一在官方 system card 复核**,当趋势看、别当可引用的精确成绩(引用基准数字的坑,见讲怎么评测 agent 那篇)。

> Computer Use 最值得记住的一句:**它把「会用电脑」这件事的智能,烧进了模型权重,而不是写进 harness 代码。** 所以 harness 薄到你几乎只需实现「执行动作 + 回传截图」——代价是**强绑 Claude、不可换模型**(非 Claude 模型调不了这个 beta 工具),换来的是省掉了外部的视觉定位/OCR 模型。这跟 model-agnostic 的 browser-use,是两种世界观。

### 3.3 群像横扫:OpenAI Operator / ChatGPT Agent 这条商业托管线

深拆了「DOM 派」和「视觉派」两端,再横扫一个**把视觉派产品化、托管到云上**的代表——它把「会用电脑的 agent」做成了给普通用户的成品。

**OpenAI Operator / ChatGPT Agent(OpenAI)**:定位是「一个能像人一样看屏幕截图、发鼠标键盘动作来操作浏览器与桌面的 agent 产品」,底层是 OpenAI 自研的 **CUA 模型(Computer-Using Agent,视觉 + 强化学习训出来的 GUI 操作能力)**。它是**视觉派**——和 Computer Use 同一路线(截图 + 坐标动作),而非 browser-use 的 DOM 路线。

关于它的**状态**,有个必须讲清的点:**独立的 Operator 产品已经停用了**——2025-07 并入 **ChatGPT Agent**,独立站点在 2025-08 前后关闭。但**能力只是改名没消失**:CUA 模型现在活在 ChatGPT Agent 的「agent 模式 / 虚拟浏览器」里,以及开发者能调的 **computer use 工具(Responses API)** 中。模型谱系也一直在升级(`computer-use-preview` 基于 GPT-4o → 2025-05 升到基于 o3 的版本 → 2026-03 官方示例仓库默认改到 `gpt-5.4`)。

**它的独特点**,值得单独点出来的有三处:

- **能力烧进模型 + 执行沙箱解耦(开发侧)。** 和 Computer Use 一样,规划/纠错在模型权重里,harness 极薄;但 API 版**不代管执行环境**——模型发动作(`computer_call`,含 `click/type/scroll/drag` 等),真正落地的沙箱**由你提供**(本地 Playwright / Docker / Browserbase / Scrapybara 都行)。官方还特意说:已经有 Playwright/Selenium/VNC 自动化栈的开发者**不必重建**,把现有栈包成一个工具暴露给模型即可。
- **消费侧是全托管 + 三档 HITL。** ChatGPT Agent 里 OpenAI 帮你**托管云端浏览器/虚拟计算机**,你不碰沙箱;遇到登录/支付会**暂停、把控制权交还给你(takeover 接管模式),此时不截图以保护密码**;提交订单、发邮件这种**有后果的动作前需要你批准**,敏感站点还有 **watch mode** 要求你在场监督。这套「三档人在环」是它作为**面向大众的成品**必须做的安全护栏。
- **从「单点产品」收敛成「平台默认能力」。** 2025-07 把 Operator(操作)+ Deep Research(检索分析)+ 浏览/终端整合进**一个 agent 面**(ChatGPT Agent),这是它最标志性的一步——不是多 agent 网,而是把多种工具(视觉浏览器 / 文本浏览器 / terminal / Gmail 等 connector)塞给**一个** agent 自己按需选用。

**要诚实标注的短板**:截图-动作回路**慢且脆**,复杂结账(验证码、复杂 JS、会话管理)成功率偏低——据外部报道,旗舰用例在真实站点上成功率曾 <50%(**未经 OpenAI 官方确认**,当参考),这也被视为独立 Operator 被并回 ChatGPT 的原因之一。开发者示例仓库 `openai/openai-cua-sample-app`(MIT)约 1700+ star,但**这是示例仓库,star 数不代表产品装机量**;Operator/ChatGPT Agent 的真实活跃用户数,底稿**未找到公开数据**。

## 4. 它们与众不同的设计决策与取舍

退一步看,这三家其实在回答同一个问题——「**怎么让一个 agent 可靠地操作图形界面?**」——但下注的位置完全不同。核心张力落在三个维度上。

**取舍一:感知——给模型看「DOM」还是「像素」?** 这是全篇最大的分野。

- **browser-use 押 DOM**:赌注是「网页有结构,别浪费视觉」。**收益**:快、便宜、可靠,而且 model-agnostic——任何会函数调用的 LLM 都能用。**代价**:窄——只吃**浏览器**(DOM 才有意义),桌面软件、canvas、远程桌面它碰不了;重排版/纯视觉布局信息也弱,得靠 `use_vision="auto"` 打补丁。
- **Computer Use / Operator 押像素**:赌注是「人能点的它都能点」。**收益**:泛化到**任意 GUI**,不限浏览器。**代价**:慢(每步一张大截图)、贵(图像 token)、脆(坐标点击容易偏),而且强绑各自的模型(Computer Use 绑 Claude,CUA 绑 OpenAI 谱系)。

**取舍二:智能在「模型」还是在「harness」?**

- **browser-use 把不少工程逻辑放在 harness**:DOM 提取、元素检测、编号映射、事件总线、重试与看门狗——这些都是**代码**,模型只管「看清单、选编号」。所以它**模型可插拔**。
- **Computer Use / CUA 把智能烧进模型权重**:规划、纠错、决定点哪个坐标,全在权重里,harness 薄到只剩「执行 + 回传」。所以它**上手门槛低**(几乎不用写 agent 逻辑),但**对行为的可控性也低**(想改行为只能靠 prompt 和工具设计,没有一个 planner 代码给你改),而且**换不了模型**。

**取舍三:交付形态——你自己搭,还是全托管?**

- **browser-use / Computer Use / CUA 开发侧**:你拿到库或工具契约,**自己组装 harness、自己提供沙箱**。灵活、可控、可审计——代价是安全和运维的锅落到你头上。
- **ChatGPT Agent 消费侧**:全托管,你只管「派任务、看结果」,沙箱和护栏 OpenAI 包了。体验最顺——代价是闭源黑盒、按订阅花钱、把安全信任交给厂商。

> 一个贯穿全篇的反直觉:**这三家没有一家是靠「模型能读懂网页」这么朴素的能力赢的。** browser-use 靠的是「**别让模型读网页**,改读一份降维后的清单」;Computer Use 靠的是「把**看屏幕点鼠标**这件事训进权重」;Operator 靠的是「把这套能力**托管产品化**」。同样一句「让 agent 会用电脑」,拆开是三条完全不同的工程哲学——再次印证了讲 harness 那篇那句「harness 与产品形态,常常比模型本身更决定体验」。

## 5. 局限与坑

这一类 agent 最容易被 demo 带着高估,也最容易在真实场景里踩坑。诚实列几条(机器会渲染成 ⚠️ 警示卡):

1. **别信任何一家的裸基准数字——它们横向不可比。** WebVoyager 89.1%、OSWorld 87.1% 这些数字,受**任务集、提示词、思考预算、重试次数、评分器**影响极大,各家各测各的。**同一个「87%」和「89%」,很可能根本不是同一种测法、甚至不是同一个任务集。** 引用时务必带「什么模型、哪个基准、什么时候、谁测的」,尤其是那些来自第三方汇编、未经官方 system card 复核的逐代分数(见讲怎么评测 agent 那篇)。本篇给的每个数字都标了来源和「截至 2026 年中」,就是这个原因。

2. **截图-坐标回路「慢且脆」是视觉派的原罪。** 每一步都要截一张大图、模型算一个坐标、然后点下去——图像 token 贵、往返慢、坐标还可能点偏一个未加载完的按钮。据外部报道,视觉派在真实复杂结账流程(验证码、复杂 JS)上成功率曾 <50%(未经官方确认)。**怎么避**:能用 DOM 就别用视觉(browser-use 的选择);用视觉时,官方那句「每步截图后自查是否成功」不是废话——**模型天生会假设动作成功而不检查**,不强制自查它就会在错误状态上越走越远。

3. **提示注入在「会用电脑的 agent」上被放大成一等威胁。** 这类 agent 会自己上网、自己读网页里的任意文字。一段**藏在页面里、评论里、甚至一张图片里的恶意指令**,可能诱导一个无人盯着的 agent 去点「授权」「转账」或把你的数据发出去(见讲权限与提示注入那篇)。这比文件类 agent 危险得多,因为它的动作有**真实世界后果**。**怎么避**:这正是 Computer Use 内置提示注入分类器、Operator 做 takeover + watch mode 的原因——**consequential action 前必须有人确认**,别把这道闸门关掉图省事。

4. **「会自己点鼠标」意味着爆炸半径是整台电脑,不只是一个仓库。** 文件类 agent 最多把一个 repo 改乱;会用电脑的 agent 能点「删除账户」、能在你登录态下操作银行页、能接受一份法律条款。**怎么避**:三家都重沙箱不是偶然——用**专用的低权限 VM/容器**、别给敏感凭据、把上网限制到域名 allowlist。但记住:**隔离降低风险,不消除风险**;而且要特别警惕参考实现的「弱隔离」(Computer Use demo 明说 agent loop 和被控桌面同容器,是 demo 取舍,别直接上生产)。

5. **别把 demo 的丝滑当成真实能力。** 这条对整个 agent 赛道都成立,对「会用电脑」这条线尤其致命——因为一个流畅的屏幕录像视觉冲击力极强,极易让人高估。真实站点会改版、会弹验证码、会有登录态和风控。**怎么避**:凡是带具体百分比的能力宣称(尤其成功率、比人快几倍),默认按「厂商自报、未经独立审计」对待;真要上生产,拿**你自己的真实站点**跑一批,而不是信官方 demo。

6. **DOM 派有它自己的盲区:没有 DOM 的东西它全瞎。** browser-use 这类结构派又快又稳,但一遇到 canvas 画布、`<iframe>` 里的复杂内容、纯图片布局、或者干脆不是浏览器的桌面软件,它的「可点元素清单」就空了或不准。**怎么避**:清楚它的适用边界——**它是浏览器专家,不是通用 GUI agent**;需要碰非浏览器软件时,得换视觉派(Computer Use 一路),或用它 `use_vision` 的视觉补丁兜底,但别指望补丁能顶替主视觉派。

## 6. 一句话记住它们 + 对比表

> **会用电脑的 agent = 不靠 API、靠操作图形界面(看屏幕、动鼠标键盘)来干活的 agent。核心分野只有一条:给模型看「DOM 里的带编号元素清单」(结构派,browser-use)还是「屏幕截图 + 坐标」(视觉派,Computer Use / Operator)。结构派快、便宜、可靠但只吃浏览器;视觉派泛化到任意 GUI 但慢、贵、脆。三家没有一家靠「模型更能读网页」赢——靠的是感知降维、能力烧进权重、和托管产品化这三种不同哲学。**

三家横向对齐(数字均为截至 2026 年中的快照,基准会持续漂移):

| 维度 | browser-use(Browser Use) | Anthropic Computer Use | OpenAI Operator / ChatGPT Agent |
|---|---|---|---|
| **定位** | 开源浏览器 agent 框架 | Claude 平台上的内置工具能力(原语) | 商业托管的通用 GUI agent 产品 |
| **感知内核** | **DOM 优先**:带整数索引的可交互元素清单(视觉为补充) | **纯像素视觉**:截图 + 坐标,不依赖 DOM/无障碍树 | **纯像素视觉**:CUA 模型看截图、发坐标动作 |
| **智能在哪** | 部分在 harness(DOM 提取/检测/编号是代码),模型可插拔 | 烧进 Claude 权重,harness 极薄,强绑 Claude | 烧进 CUA 模型权重(GPT-4o→o3→gpt-5.4),harness 薄 |
| **开源闭源** | 开源(MIT);另有闭源 Browser Use Cloud | 工具能力闭源(随 Claude);参考实现 `computer-use-demo` MIT | 消费产品闭源订阅制;开发者示例 `openai-cua-sample-app` MIT |
| **执行环境** | 自带浏览器控制层(裸 CDP + bubus 事件总线 + watchdog) | 参考实现 Docker(Xvfb/Mutter/VNC),生产需自建隔离 | 开发侧你自备沙箱(Playwright/Docker/Browserbase…);消费侧 OpenAI 全托管云 VM |
| **人在环 / 安全** | `sensitive_data` 脱敏、域名过滤、文件白名单;内置一等 HITL 未确认 | 内置提示注入分类器 + 自动求确认;官方强建议高后果动作人工确认 | 三档:API 级安全确认、takeover 接管(登录/支付停截图)、watch mode 监督 |
| **最适合谁** | 要**开源可控 + DOM 可靠性 + 模型自由**、且任务在浏览器里的团队 | 已在 Claude 生态、要操作**无 API 的任意桌面/网站**、能接受 beta 与像素误差 | 想**开箱即用**、把 GUI 任务外包给云、不想管沙箱、接受闭源与厂商数字的用户 |
| **2026 中状态** | 高度活跃(约 10.3 万 star,日级提交,已融资 $17M) | 活跃、仍 beta 未 GA;工具三代演进,已成 Claude for Chrome 底层 | 独立 Operator 已停用,能力并入 ChatGPT Agent;模型持续升级 |
| **招牌数字(带星号)** | WebVoyager 89.1%(官方,586 任务;已被竞品反超,不可横比) | OSWorld 首发 14.9%→22.0%(官方);逐代至 ~87.1%(第三方汇编,未官方复核) | 真实结账成功率曾 <50%(外部报道,未经官方确认);活跃用户未公开 |

**怎么选**:任务**在浏览器里**、要**开源可控、模型自由、又快又稳**,看 browser-use(接受它碰不了非浏览器软件);要操作**没有 API 的桌面软件 / 任意 GUI**、又已经在 Claude 生态里,看 Anthropic Computer Use(接受纯视觉的慢与脆、且换不了模型);想**开箱即用、把 GUI 任务丢给云、连沙箱都不想管**,看 ChatGPT Agent(接受闭源与厂商自报数字)。

对照前面各篇:拆 Manus 那篇里的通用 agent,据外部报道其早期的浏览器能力就搭在 browser-use 这类框架上——这一篇等于把「agent 怎么会用浏览器」这块拼图补齐了。而这一篇最该带走的,不是三个产品名,是那条分野:**你要 agent 会用电脑,第一个、也是最要命的决定,是让它看「结构」还是看「像素」。**
