这门课要回答一个更基础的问题:**「agent」到底是什么?它凭什么能自己把一件复杂的事做完,而不是像 ChatGPT 那样你问一句它答一句?**


这一篇是入口。读完你会有一张清晰的心智图:**agent = 模型 + 循环 + 工具**,以及它和聊天机器人、和「工作流」到底差在哪。后面每一章,都是把这张图里的某个部件放大来讲。

> 你脑子里一直想着这个画面,概念就不会飘。

## 1. Agent 到底是什么

一句话:

> **Agent 是一个让大模型"自己决定下一步做什么、动手去做、再根据结果决定接下来怎么做"的系统——它围着一个目标转圈,直到把事做完。**

拆开这句话里的三个关键词,就是 agent 的三大件:

- **模型(大脑)**:负责「想」——理解目标、判断现在该干嘛、下一步调用什么。
- **工具(手脚)**:负责「做」——读文件、跑命令、查网页、调 API。没有工具的模型只能「说」,有了工具才能「动」。
- **循环(心跳)**:负责「转起来」——做一步 → 看结果 → 再想 → 再做,一圈一圈,直到达成目标或放弃。

对照上面那个修 bug 的例子:模型读懂 issue、判断「先去看看 `parser.py`」;工具真的把 `parser.py` 的内容读出来、把改动写回去、把测试跑起来;循环让它「读完文件发现不对 → 再去读另一个文件 → 改 → 测试失败 → 再改」,反复逼近,直到测试变绿。

**这就是 agent 和「一次性问答」最根本的区别。** 你把同一个 bug 甩给普通的聊天框,它顶多给你一段它「猜」出来的补丁——它没读过你的代码、跑不了你的测试,对错全靠运气。而 agent 会**真的去环境里摸爬滚打**,用一次次真实反馈把答案磨出来。

> 关键直觉:**普通 LLM 是「一锤子买卖」——输入进,输出出,结束。Agent 是「一个装了工具、会看反馈、会自己接着干的循环」。** 从「回答问题」到「完成任务」,中间隔的就是这个循环。

## 2. 从 LLM 到 Agent:自主性的阶梯

「Agent」这个词被用得很滥,从「能查天气的聊天机器人」到「Devin 这种号称能独立干活的 AI 工程师」都自称 agent。要不被营销词绕晕,最好的办法是把它摆到一条**自主性阶梯**上看——问一个核心问题:**「下一步做什么」这个决定,是人预先写死的,还是模型当场自己拿的?**

```
自主性  低 ────────────────────────────────────────────────→ 高

L0 光秃秃的 LLM      输入→输出,一问一答。不能查、不能动手。            "帮我写段正则"
L1 LLM + 工具        能调工具补齐能力,但仍是你问一次它答一次。         "查下今天汇率"(它调 API)
L2 工作流 workflow    人把步骤编排成固定流水线,LLM 在每一格里填空。      "翻译→润色→排版"三步固定
L3 Agent            模型自己决定步骤、自己循环、自己看反馈纠错。         "修好这个 bug"(它自己摸)
L4 多 Agent / 长程    多个 agent 分工协作,或单 agent 跑几小时的长任务。   "把这个功能从头做到上线"
```

这条阶梯上,**L2 和 L3 之间那道线最关键**,也最容易含糊。Anthropic 在《Building Effective Agents》(2024 年底)里把这道线讲得很干脆:

- **工作流(workflow)**:大模型和工具被**预先写好的代码路径**编排起来。步骤是人定的,LLM 只在每一步里发挥。流程可预测、可控,但不灵活。
- **Agent**:大模型**自己动态决定**流程和工具用法,自己掌控怎么完成任务。灵活、能应对没预料到的情况,但也更难控、更贵、更容易跑偏。

一句话记住这道线:

> **看「谁在开车」。步骤是人写死在代码里的 → 那是 workflow;步骤是模型每一步现想现决定的 → 那才是 agent。**

这不是说 L3 一定比 L2「高级、更该用」。**恰恰相反**:工程实践里,能用固定工作流解决的,就别上全自主 agent——工作流更便宜、更可预测、更好调试。**只有当任务的步骤事先无法枚举**(比如「修 bug」——你根本不知道要读哪几个文件、要试几次),才值得把方向盘交给模型,忍受它的不确定性去换那份灵活。这个取舍我们在第 5 篇和第 20 篇会反复谈。

## 3. 一个 Agent 由哪些部分构成(先鸟瞰,后面逐章放大)

把一个能跑起来的 agent 彻底拆开,你会看到**五个必备零件**。这一节只做「鸟瞰式」介绍——让你先认全这几块、知道它们各管什么;每一块背后的机理,后面都有专章展开(标注了「见第 X 篇」)。

### 3.1 模型(大脑)——负责想

agent 的推理内核,通常是一个大语言模型(LLM)。它干三件事:**理解**当前目标和处境、**决策**下一步该调哪个工具/给什么参数、**综合**工具返回的结果继续往下推。

模型强不强,决定了 agent 的「智力天花板」。但请记住一句后面会反复出现的话:**模型强 ≠ agent 强**。同一个模型,配一套好工具、好上下文管理,和裸用它,干活能力差一大截——这正是「harness」存在的意义(第 6 篇)。

### 3.2 循环(心跳)——负责转起来

让 agent 之所以是 agent 的那个「转圈」机制:**想一步 → 做一步 → 观察结果 → 再想**,直到收工。最经典的表述是 **ReAct**(Reasoning + Acting,Yao 等人 2022):让模型交替产出「推理(想)」和「行动(调工具)」,把工具返回的「观察」再喂回去,循环推进。这是整个 agent 领域的地基,第 2 篇专门讲透。

### 3.3 工具(手脚)——负责做

模型本身只会生成文字。**工具是它伸向真实世界的手**:读写文件、执行 shell、搜索网页、查数据库、调外部 API。模型通过「函数调用(function calling)」说出「我要用哪个工具、参数是什么」,由外面的代码真正去执行,再把结果递回来。工具设计得好不好,极大地影响 agent 好不好用——第 3 篇细讲,还会介绍把工具标准化的 **MCP**。

### 3.4 上下文与记忆(工作台 + 笔记本)——负责记住

模型每次「想」,都只能看见喂给它的那一段文字,也就是**上下文窗口(context window)**。任务一长,历史就会超出窗口装不下。于是就有了各种**记忆**手段:把长历史压缩总结(context compaction)、把知识存进外部库按需检索(RAG)、把关键信息落到文件里当长期笔记。这决定了 agent 能不能干「长活」,第 4 篇专讲。

### 3.5 目标与终止(方向盘 + 刹车)——负责知道何时收手

agent 得知道**要干成什么**(目标)、以及**什么时候算干完了 / 该放弃了**(终止条件)。听起来简单,却是最容易被忽视、也最容易出事的地方——终止条件没设好,agent 会陷在死循环里反复烧钱(这正是早期 AutoGPT 的著名毛病,第 17 篇会讲)。

```
        ┌──────────────── 目标 / 终止条件 ────────────────┐
        │                                                 │
   ┌────▼─────┐   决策    ┌─────────┐   执行   ┌──────────┐
   │  模型/大脑 │ ───────→ │  工具调用 │ ──────→ │ 真实环境  │
   └────▲─────┘           └─────────┘          └────┬─────┘
        │                                            │
        └──────────── 观察(把结果喂回)───────────────┘
                  ↑ 上下文/记忆:把这一圈圈的历史攒起来
```

> 把这五件套连起来的,是一层**粘合代码**——它决定循环怎么转、工具怎么派发、上下文怎么裁剪、越权了要不要拦。这层代码就是 **agent harness**,是这门课的第二大板块(第 6 篇起)。**很多时候,一个 agent 好不好用,拼的不是模型,而是这层 harness。**

## 4. 关键心态:agent 不是魔法,是「装了工具、会看反馈的概率机器」

新手对 agent 最常见的误解,是把它想象成一个「无所不能、给个目标就搞定」的黑魔法。要用好它、也要设计好它,你得先建立三个更贴近真相的心态:

- **它是概率性的,不是确定性的。** 同一个任务跑两次,路径可能不一样,偶尔会犯低级错。别指望它像传统程序那样每次都走同一条路、给同一个结果。**设计时要为「它会出错」做准备**,而不是假设它不会。
- **它靠反馈闭环续命。** agent 之所以能比「一次问答」强,全靠**每一步都能看到真实结果**(测试跑挂了、文件不存在、编译报错)并据此纠偏。**反馈信号越真、越及时,agent 越强。** 一个跑不了测试、看不到报错的 agent,和闭着眼睛猜没区别。
- **它需要护栏。** 一个能执行 shell、能改文件、能调 API 的循环,威力和风险是一体两面——它可能帮你修好 bug,也可能 `rm -rf` 掉不该删的东西,或被网页里的一段恶意文字骗着去干坏事(提示注入)。所以「让它动手」和「不让它闯祸」必须同时设计,这是第 9 篇的主题。

> 一句话:**agent 的强大来自「循环 + 工具 + 反馈」,它的脆弱也来自这三样。** 理解它是一个「会犯错、需要反馈、需要护栏的概率循环」,你就既不会低估它,也不会盲目信它。

## 5. 常被忽视的坑(新手最容易栽的几处)

1. **把 agent 当成「更强的聊天机器人」。** 两者的本质差别是**有没有那个「动手—看反馈—再动手」的循环**。只会答话、不能在环境里试错的,再能说也不是 agent。搞混这一点,你会既高估聊天机器人、又低估真 agent。
2. **以为「agent 强全靠模型强」。** 模型是天花板,但地板和梯子是 harness 搭的:工具设计、上下文管理、循环控制、错误恢复。业界一个反直觉但被反复验证的事实——**同一个模型,配不同 harness,干活能力能差出一大截**。这也是本课要专门用一半篇幅讲 harness 的原因。
3. **一上来就追求「全自主」。** 自主性越高≠越好。全自主 agent 更难控、更贵、更容易跑偏。**真正好用的产品(Claude Code、Cursor 等)大多是「半自主 + 人在环」**:关键动作要你点头,方向错了你能随时拽回来。先会用半自主,再谈全自主。
4. **不设终止条件,让它无限打转。** 没有明确的「做完了」或「该停了」,agent 会在死胡同里反复尝试、反复烧 token。**目标要可判定、要有步数/时间/预算上限、要有失败退出**。这是 AutoGPT 时代最惨痛的教训之一。
5. **期待确定性、可复现。** 它是概率机器。别把它塞进一个要求「每次输出逐字节相同」的流程里;要在它外面包上校验、重试、人审这些确定性护栏,而不是奢望它自己变确定。

## 6. 小结 & 下一步

这一篇你需要带走的:

- **Agent = 模型(想)+ 工具(做)+ 循环(转起来),围着一个目标反复「做—看反馈—再做」,直到完成。**
- 用**自主性阶梯**给形形色色的「agent」定位;区分 workflow 和 agent 的那把尺子是:**下一步是人写死的,还是模型现场决定的。**
- 一个 agent 的**五个零件**:模型、循环、工具、上下文/记忆、目标/终止——后面逐章放大。
- 心态:它是**会犯错、靠反馈、需护栏的概率循环**,不是魔法。
- 把这些零件粘起来、并决定它们怎么协作的那层代码,叫 **harness**——常常比模型更能决定成败。

下一篇,我们把五件套里最核心的那个——**循环**——拆开,看 ReAct「想一步、做一步、看一步」到底是怎么转起来的,以及一个 agent 从收到任务到交活,内部走过了哪些步。
