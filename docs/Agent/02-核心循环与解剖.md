## 0. 全景:五件套里,这章只拆「循环」这一件

第①篇我们把 agent 拆成了**五件套**:模型(想)、循环(转起来)、工具(做)、上下文/记忆(记住)、目标/终止(何时收手)。这一篇,我们只放大其中一件——**循环(agent loop)**。

为什么先拆它?因为循环是那个让「五件套」真正**活过来**的东西。工具再全、模型再强,如果没有一个「做一步 → 看结果 → 再想 → 再做」的圈把它们串起来,你手里就只是一个「会调一次函数的聊天机器人」,而不是 agent。第①篇那句话——**从「回答问题」到「完成任务」,中间隔的就是这个循环**——这一篇就是来兑现它的。

```
第①篇给了你这张鸟瞰图:

   模型 · 循环 · 工具 · 上下文 · 目标/终止
          │
          └──▶ 本篇把「循环」这一件拆到底:
               单步里发生了什么 → 前提是什么 → ReAct → 怎么攒成多步 → 何时停 → 有哪些变体
```

其余四件在别处细讲:工具的内部机制见第③篇,上下文怎么裁剪、怎么压缩见第④篇,规划(先想好整盘棋)的细节见第⑤篇。这一篇聚焦一个问题:**这个圈,到底是怎么一圈圈转起来的?**

> 全篇继续用第①篇那个跑腿的例子回扣:**让 agent 去修好某开源仓库里的一个 bug——读 issue → 找相关文件 → 改代码 → 跑测试 → 提 PR。** 这一次,我们盯着它「转圈」的每一下看。

## 1. 一次「想-做-看」到底发生了什么

在讲「循环」之前,先把**一圈**看清楚。很多人对 agent 的想象是模糊的一团「AI 在自己干活」,但拆到底,它其实是一个**极其机械、可以逐字节复述**的过程。我们把这一圈叫做**单步(single turn)**:模型被调用一次、吐出一次结果、外面的代码处理一次。

先说一个反直觉的点:**模型自己什么都「做」不了。** 它不能读文件、不能跑测试、甚至不能「记住」上一步——它是一个纯函数,你喂给它一段文字,它吐回一段文字,仅此而已。所有「动手」的活,都是它**外面**那层代码(我们后面会叫它 harness)干的。所以一圈里,真正的主角有两个:**模型**和**外面的代码**,它们轮流上场。

**① 模型收到什么(输入)。** 每次调用模型,外面的代码都把三样东西拼成一段文字喂进去:

- **系统提示(system prompt)**:告诉模型「你是谁、你的任务是什么、有哪些规矩」。修 bug 场景里,大意是「你是一个修复代码的 agent,目标是让失败的测试通过,别乱删无关文件」。
- **工具清单(tool schemas)**:一份「你能调用哪些工具」的说明书,每个工具用 **JSON Schema** 描述名字、干什么、要哪些参数。比如 `read_file(path)`、`run_tests()`、`edit_file(path, patch)`。(工具怎么定义、为什么用 JSON Schema,见第③篇。)
- **历史(history)**:到目前为止发生过的一切——用户最初的目标、模型之前的每次发言、每次工具调用、每次工具返回的结果。**这就是 agent 的「短期记忆」**,它全靠这段历史来知道「我刚才干到哪了」。

**② 模型吐出什么(输出)。** 模型看完这一大段,产出的东西只有两种可能:

- **一段文字**——它想说话:解释思路、或者宣布「任务完成了,这是结论」。
- **一个工具调用(tool call / `tool_use`)**——它想动手:结构化地说出「我要调 `read_file`,参数 `path="parser.py"`」。

**③ 外面的代码怎么执行。** 如果模型只是说了段话,那这一圈可能就结束了(它认为活干完了)。但如果它吐出的是一个工具调用,控制权就**交回给外面的代码**:代码把 `read_file("parser.py")` 真正跑起来,去磁盘上把文件读出来。

**④ 观察怎么回填。** 工具跑完的结果——文件内容、测试输出、报错栈——被打包成一个**观察(observation)**,追加回那段历史的末尾。然后,外面的代码带着**变长了的历史**,再一次调用模型。新的一圈开始了。

```
        ┌─────────────────────────── 一圈(single turn)───────────────────────────┐
        │                                                                         │
  系统提示 + 工具清单 + 历史                                                          │
        │                                                                         │
        ▼                                                                         │
   ┌─────────┐   吐出 tool_use   ┌──────────────┐   真正执行   ┌──────────┐        │
   │  模型   │ ───────────────▶ │  外面的代码    │ ──────────▶ │ 真实环境  │        │
   │ (纯文字)│                  │  (harness)    │             │ 文件/测试 │        │
   └─────────┘                  └──────────────┘             └────┬─────┘        │
        ▲                                                          │             │
        │              observation(把结果打包追加进历史)             │             │
        └──────────────────────────────────────────────────────────┘─────────────┘
                        历史又长了一截 → 带着它再调一次模型
```

> 一句话:**一圈 = 模型看一眼全部历史,决定「说话还是动手」;动手就由外面的代码去执行、把结果塞回历史,再来一圈。** 控制权在「模型」和「代码」之间来回传,像打乒乓球。

## 2. 前提:要有循环,先要有「可回填的反馈」

在往下讲「怎么循环」之前,必须先钉死一个前提,否则整座大厦是空的:**没有 observation,就没有 agent。**

回到第 1 节那张图——循环之所以能转,是因为每一圈末尾都有一个**观察(observation)** 被塞回历史,让下一圈的模型「看见」上一步的真实后果。抽掉这一步,会发生什么?模型改完 `parser.py`,永远不知道测试到底过没过;它调完 `read_file`,永远看不到文件里到底写了啥。它就退化成第①篇讲的那个「闭着眼睛猜」的聊天机器人——只不过多喊了几嗓子「我要读文件」而已,喊完什么反馈都没有。

这就是**反馈闭环(feedback loop)**:动作 → 真实结果 → 回填给模型 → 影响下一个动作。它是 agent 全部「智能」的来源。第①篇那句「反馈信号越真、越及时,agent 越强」,机理就在这里——模型本身不会变聪明,是**一次次真实的观察在帮它纠偏**。

有一个经典的、比 LLM 老得多的说法能帮你记住这件事。1995 年 Russell & Norvig 的经典教材《人工智能:一种现代方法》里给 agent 下的定义是:**「任何能通过传感器感知环境、通过执行器作用于环境的东西。」** 感知(perceive)+ 行动(act)。LLM agent 只是把「传感器」换成了工具返回的 observation,把「执行器」换成了工具调用——**内核是同一个:一个能看见自己动作后果、并据此调整的闭环。**

> 关键前提:**agent 的动作必须落到一个能「回话」的环境里。** 能跑测试、能看报错、能读到文件真实内容——环境每次都给它一个诚实的回音。**一个动作没有回音的 agent,不是 agent,是在自言自语。**

这也解释了一个工程直觉:**给 agent 配环境,比给它配算力更重要。** 你想让它修 bug,首先得保证它真能把测试跑起来、真能拿到红/绿结果。如果测试环境是坏的、报错被吞掉了,再强的模型也只能瞎猜。(observation 该怎么喂、太长太脏怎么办,是第 5 节的坑,也牵扯第④篇的上下文管理。)

## 3. 循环的核心机理

前面两节铺好了地基:一圈是什么(第 1 节)、圈能转起来的前提是什么(第 2 节)。这一节把**「圈怎么转、什么时候停」** 彻底拆开。我们从这个领域的地基范式 ReAct 讲起,再钻进单步内幕,然后看单步怎么攒成多步,以及怎么收尾。

### 3.1 ReAct:想一步、做一步、看一步

先给你一个可以「五分钟看懂、当天就能实现」的经典范式。

2022 年 10 月,Yao 等人发表了 **ReAct**(**Rea**soning + **Act**ing 的缩写,arXiv 2210.03629,后中 ICLR 2023)。在它之前,人们让模型「想」(思维链,Chain-of-Thought——一步步把推理写出来)和让模型「做」(调工具)是两件分开的事。ReAct 的洞见简单得让人拍腿:**为什么不让它俩交替进行?** 想一点、就据此做一点;做完看到结果、再想下一点。

它的记法就三个词,循环往复:

```
   Thought(想)  ──▶  Action(做)  ──▶  Observation(看)
       ▲                                     │
       └─────────────────────────────────────┘
              把看到的塞回去,接着想下一步
```

套到修 bug 上,一段真实的 ReAct 轨迹长这样:

```
Thought:      issue 说排序结果偶尔乱掉,先看看 sort 相关的实现在哪。
Action:       search_code("def sort")
Observation:  找到 parser.py 第 88 行 def sort_items(...)
Thought:      读一下这个函数,怀疑是比较逻辑写反了。
Action:       read_file("parser.py", 80, 100)
Observation:  (函数内容)……果然,88 行用了 < 但注释说要降序。
Thought:      改成 >,然后跑测试验证。
Action:       edit_file("parser.py", ...)  →  run_tests()
Observation:  17 passed, 0 failed. ✅
Thought:      测试全绿,任务完成,去提 PR。
```

看清楚这里的关键:**每一个 Thought 都不是凭空想的,而是踩着上一个 Observation 想的。** 它「怀疑比较逻辑写反了」是因为看到了函数内容;它「确认可以提 PR」是因为看到了 `17 passed`。这就是第 2 节那个反馈闭环在跑。

ReAct 论文里量化过这个「踩着现实想」的价值:在 ALFWorld、WebShop 这类需要多步交互的任务上,它比当时的模仿学习 / 强化学习基线分别高出约 **34%** 和 **10%**(绝对成功率),而且只用了 **1–2 个示例**。为什么这么有效?因为「先想再做、做完看着结果再想」能**把幻觉摁住**——模型不会一口气编出一长串脱离现实的计划,每一步都被真实观察拉回地面。

> 记住这个三拍子:**Thought → Action → Observation,循环。** 它是整个 agent 领域的心跳节奏。你后面看到的所有花哨循环,拆到底几乎都是这三拍的变奏。

代价也要诚实说:ReAct 式的「一步一想一调」意味着**每一步都要调一次模型**——API 调用多、延迟高、花钱多。想省这个钱,就有了 3.5 会提的「先规划后执行」等变体,那是拿灵活性换效率的另一种取舍。

### 3.2 单步内幕:模型如何「决定」调哪个工具

3.1 给了直觉,现在钻进一个 turn 内部,看**「模型决定调 `run_tests`」这件事在工程上到底是怎么发生的**——因为这里藏着 agent 最核心的控制权交接。

你可能以为模型「调用」了工具。**其实模型一个工具都调不了。** 它做的全部事情,是**吐出一段结构化的文字**,说「我想调用名为 `run_tests` 的工具,参数是空」。这段结构化输出,在 Anthropic 的 API 里叫 `tool_use` 块,在 OpenAI 那边(2023 年 6 月 13 日推出函数调用,同年 11 月统一为 `tools` 参数)叫 tool call。名字不同,机理一样:**模型只负责「点菜」,真正「下厨」的是外面的代码。**

那模型怎么知道有哪些菜、怎么点?靠的就是第 1 节说的**工具清单**——每个工具的 JSON Schema 在系统层面告诉它「有这么个函数、叫这名、吃这些参数」。模型在生成时,会像选词一样**选出一个工具名、填好参数**,输出成结构化的块。(为什么用 Schema、Schema 怎么影响模型选得准不准,是第③篇的核心。)

控制权的交接,有一个非常具体、可以照着写代码的信号。以 Anthropic 的工具协议为例(据其官方文档,截至 2026 年中):

- 模型这一轮返回时,会带一个**停止原因(`stop_reason`)**。如果它是 `"tool_use"`,意思是「我不说了,我要动手,轮到你(外面的代码)执行了」。
- 这时外面的代码**必须**执行对应工具,并在下一轮把结果作为**匹配的 `tool_result` 块**塞回去。**如果你不回填 `tool_result`,循环就断了**——模型下一轮会一脸懵,因为它点了菜却没等到上菜。

```
   模型这轮输出          harness 读到的信号            harness 该做什么
   ─────────────        ──────────────────         ─────────────────────────
   纯文字,没有工具调用    stop_reason = "end_turn"    多半是收工了 → 结束循环
   带 tool_use 块        stop_reason = "tool_use"    执行工具 → 回填 tool_result → 再调模型
```

顺带一个第③篇会展开、但这里得先埋点的区别:工具在**谁那儿执行**是有讲究的。有些是**客户端工具(client tools)**,比如跑 shell、编辑文件,由你自己的代码执行,完全可定制;有些是**服务端工具(server tools)**,比如某些厂商内置的联网搜索、代码执行,由模型提供方在自己的服务器上跑完、结果直接内联返回。对循环来说,前者是「模型 ↔ 你的代码」打乒乓,后者有一部分在「模型提供方那侧」就闭环了。

> 一句话:**模型不调工具,它只是「说」它想调哪个;`stop_reason: "tool_use"` 是它把方向盘递给你的那一下,你执行完、把 `tool_result` 递回去,才算把方向盘还给它。** 这一递一还,就是 agent 循环的最小传动机构。

### 3.3 从单步到多步:状态怎么攒起来,循环何时继续

现在把镜头拉远。**一圈**看清楚了,**多圈**其实就是**同一圈的重复**——但有一个决定性的细节:**每一圈的产物,都被追加进那段历史里,带进下一圈。**

修 bug 的完整历史,随着循环推进,是这样滚雪球长大的:

```
起点:  [系统提示] + [工具清单] + [用户目标:修好这个排序 bug]

第1圈后追加:  Thought「先搜 sort」+ Action(search_code) + Observation(找到 parser.py:88)
第2圈后追加:  Thought「读这个函数」+ Action(read_file) + Observation(函数内容)
第3圈后追加:  Thought「改比较符」+ Action(edit_file→run_tests) + Observation(17 passed)
第4圈:        模型看到「全绿」→ 只输出一段文字「完成」→ 不再有 tool_use
```

看出来了吗?**agent 的「记忆」不是什么神秘装置,就是这段越滚越长的历史文本。** 模型本身没有记忆(第 1 节说过它是纯函数),是外面的代码每一圈都把新的 thought/action/observation **拼接**上去,再整段喂回模型,制造出「它记得前因后果」的效果。这也是为什么第①篇说上下文就是 agent 的「工作台」。

于是「循环何时继续」有了一个朴素到近乎机械的答案:

- **模型这一圈吐出了 `tool_use`** → 说明它还想动手、活没干完 → **执行、回填、继续下一圈。**
- **模型这一圈只吐了文字、没有 `tool_use`** → 说明它认为不需要再动手了 → **循环停下**(它多半是在交活了)。

就这么简单:**「还有没有下一个动作要做」,决定了「圈要不要接着转」。** 主动权在模型手里,但**闸门在外面代码手里**——下一节讲的正是这道闸门。

这里也埋下第④篇的伏笔:历史滚雪球是有代价的,它会**越滚越大,直到撑爆上下文窗口**。修一个小 bug 可能十几圈就完了,但让 agent 跑一个几小时的长任务,历史会长到装不下。到那时就需要「压缩、摘要、把关键信息落盘」等手段——那是第④篇「上下文与记忆」的主场,这里只需要你记住:**多步的本质是历史的累积,而历史是有物理上限的。**

### 3.4 终止与失败处理:什么时候算「做完」,出错了怎么办

循环最危险的地方,不是它不转,而是它**停不下来**。第①篇提过 AutoGPT 时代最惨痛的教训:没设好终止条件,agent 在死胡同里反复烧钱。所以「怎么停」和「怎么转」同等重要。

一个健壮的循环,通常有**三类**终止方式,缺一不可:

| 终止方式 | 触发时机 | 修 bug 场景里的样子 |
|---|---|---|
| **正常完成** | 模型不再发起工具调用,给出最终答复 | 测试全绿,模型输出「已修复,PR 已提」 |
| **上限熔断** | 到达步数 / 时间 / token / 预算的硬上限 | 转了 40 圈还没绿,强制停下交人 |
| **失败退出** | 遇到无法恢复的错误或明确的「放弃」信号 | 测试环境根本起不来,无路可走 |

**为什么「正常完成」这一条不能单独指望?** 因为模型是概率性的(第①篇),它有时会**过早宣布成功**(测试其实没全过,它却说「搞定了」),有时会**永远不肯认输**(一个改法不行,换个几乎一样的改法再试,无限打转)。所以你必须在模型的「自我判断」外面,套上**上限熔断**这道不讲道理的硬闸——不管模型怎么想,转够 N 圈就停。这道闸不是给模型用的,是给「模型犯轴」时兜底的。

**失败处理**则是循环的「韧性」所在。工具调用天然会出错:文件不存在、命令超时、测试挂了、补丁打不上。好的循环不会一遇错就崩,而是把**错误本身当成一个 observation 回填给模型**——「你要读的文件不存在」也是一条有用的反馈。模型看到它,往往能自己纠正:换个路径、换个工具、换种改法。这里的常见策略有几种:

```
   工具报错  ──▶  把报错当 observation 回填  ──▶  模型看情况:
                                                  ├─ 重试(retry):同样的动作再来一次(适合超时等偶发错)
                                                  ├─ 换路(reroute):换个工具/换条思路(适合此路不通)
                                                  └─ 上报(escalate):实在不行,交还给人
```

**注意这里的分寸。** 无脑重试同一个失败动作,是死循环的一大来源(第 5 节的坑);而「把报错喂回去让模型自己想办法」才是 agent 相对于传统程序真正强的地方——传统程序遇到没预料的错误就抛异常停摆,agent 却能**读懂报错、临场改道**。但这份灵活也要靠上限熔断兜底,否则「改道」会变成「原地打转」。

> 一句话:**会转的循环只是及格,会停、会认错、会兜底的循环才是能上生产的。** 终止条件要「可判定 + 有硬上限 + 有失败出口」;错误不是循环的敌人,回填给模型的错误反而是最有价值的反馈之一——只要别让它无脑重试。

### 3.5 循环的变体速览:不止 ReAct 一种转法

ReAct 的「想一步、做一步、看一步」是最基础的转法,但不是唯一的。这里只**点到**两个最常被提起的变体,建立你的「地图感」;它们的机理细节留给第⑤篇(规划推理)。

**变体一:先规划,后执行(Plan-and-Execute)。** ReAct 是「走一步看一步」,每步都劳烦大模型想一次,贵且慢。另一种思路是:**先让一个强模型一次性想好整个计划**(第 1 步读 issue、第 2 步定位文件、第 3 步改、第 4 步测),再让一个便宜的执行器**照着计划一条条做**。好处是省下大量昂贵的「大模型思考」调用;代价是**计划一旦和现实脱节**(比如第 2 步发现文件根本不在预想的位置),就得停下来重新规划。(这个模式没有单一的经典源头论文,是社区综合 BabyAGI、Plan-and-Solve 等思路、主要经 LangChain 推广而流行起来的说法,细节与取舍见第⑤篇。)

**变体二:带反思的循环(Reflexion 式)。** 前面讲的循环是「一次性把活干完」。但有些任务允许**多试几次**:第一次做完,让一个「批评者」评判好坏,把「哪里没做好、下次注意什么」**用自然语言写成一段反思**存起来,下一次尝试时把这段反思带上,争取做得更好。这就是 Reflexion(Shinn 等,arXiv 2303.11366,NeurIPS 2023)的核心——**不改模型权重,只靠语言反馈自我改进**。它的前提是任务得能「多试 + 可验证结果」(比如写代码能跑测试)。(细节同样见第⑤篇。)

```
   ReAct:          想→做→看→想→做→看……(一次通关,走一步看一步)
   Plan-Execute:   [大模型规划一整盘] → 便宜执行器逐条做 → (脱节了才回头重规划)
   Reflexion:      做完一整轮 → 批评+写反思 → 带着反思重来一轮 → 又一轮……
```

你现在只需要记住:**这些都是同一个「反馈闭环」的不同调度方式**——在「每步都想 vs 一次想好」「一次通关 vs 反复重来」之间做取舍。选哪种,取决于任务是短是长、模型调用贵不贵、结果能不能被验证。第⑤篇会把这几种规划/反思策略讲透。

## 4. 关键心态:循环的强大,全在「每步都能拿真实反馈纠偏」

如果这一篇你只带走一句话,是这句:**循环之所以让 agent 强,不是因为它「转得多」,而是因为它每转一圈都能拿到一次真实反馈、据此纠偏。**

对比着看就清楚了。你让一个没有循环的模型「修 bug」,它只能凭训练里见过的模式**一口气猜**出一个补丁——对错全靠运气。而有循环的 agent,是**把一个大而难的猜测,拆成了几十个小而可验证的动作**:每个动作都立刻撞上现实(文件里到底写了啥、测试到底过没过),现实立刻给它一个诚实的回音,它据此修正下一步。**几十次「小猜测 + 即时校验」串起来,逼近正确答案的概率,远高于「一次大猜测」。**

这也解释了第①篇那个反直觉结论的一半——**为什么同一个模型换个 harness 能力差一大截**:harness 决定了反馈闭环的**质量**。测试报错有没有干净地回填?文件读出来是不是被截断了?错误是被吞掉了还是被喂回给了模型?这些不改模型一个参数,却直接决定了模型每一步「睁着眼」还是「闭着眼」做决定。

由此可以推出几条可操作的判断法则:

- **反馈越真越好。** 能给真实测试结果,就别给模型一个「大概能过吧」的模糊描述。真环境 > 模拟环境 > 无环境。
- **反馈越及时越好。** 别攒一大堆动作再统一看结果——那样错误会层层累积、难以归因。ReAct 那种「做一步立刻看一步」之所以强,就强在及时。
- **反馈要能被模型看懂。** 一屏乱码的堆栈、一个静默的失败,对模型等于没反馈。让 observation 清晰、聚焦,和让 agent 变聪明是同一件事(具体怎么做,牵涉第③篇工具设计和第④篇上下文管理)。

> 一句话:**agent 不是靠「一次想得多准」赢的,是靠「转很多圈、每圈都被现实纠正一点」赢的。** 你的活儿,不是让模型更聪明(那是模型厂商的事),而是让它每一圈都能睁着眼睛做决定——**把反馈闭环修好,就是把 agent 修好。**

## 5. 常被忽视的坑(循环最容易栽的几处)

1. **死循环:无脑重复同一个失败动作。** 补丁打不上,模型换个几乎一样的补丁再打,还是不上,再来……在死胡同里反复烧 token。**为什么致命:** 它不会自己意识到「我在原地转」,而每一圈都在花钱。**怎么避:** 上限熔断(第 3.4)是底线;更进一步,检测「连续 N 步的动作/观察高度重复」就强制换路或上报,别只靠模型自觉。

2. **把「想」当成「做」。** 模型在 Thought 里写「我现在去跑测试」,然后……就没有然后了——它只是**说**要跑,并没有真的吐出 `run_tests` 的 `tool_use`。历史里留下一句空话,循环却以为动作发生了。**为什么致命:** 你会看到 agent「口头很勤快」,实际啥也没干,结果全是幻觉。**怎么避:** 分清「文字」和「工具调用」是两种不同的输出(第 3.2);只认真正的 `tool_use` 块为「动作」,Thought 里的豪言壮语不算数。

3. **observation 太脏或太长。** 把一整个 50 万字的日志文件、或一屏 ANSI 颜色码的报错,原封不动塞回历史。**为什么致命:** 一是瞬间撑爆上下文窗口(第 3.3 说的滚雪球),二是把真正有用的那行报错淹没在噪声里,模型反而看不见重点、纠不了偏(直接违背第 4 节的心态)。**怎么避:** observation 要**裁剪、聚焦**——只回填有信息量的部分(报错的关键几行、测试的 pass/fail 汇总)。怎么裁,是第③篇工具设计和第④篇上下文管理的活儿。

4. **没有终止条件,让它无限打转。** 只写了「修好这个 bug」,却没定义「多少圈还没绿就停」「多长时间算超时」。**为什么致命:** 遇到它搞不定的 bug,它会一直试到你账单爆炸——AutoGPT 时代的经典翻车。**怎么避:** 每个循环都必须有**硬上限**(步数/时间/预算)和**失败出口**,不管模型怎么想都能刹住(第 3.4)。

5. **把模型的「我做完了」全盘当真。** 模型说「测试都过了,已修复」,你就信了——但它可能只跑了部分测试、或压根看错了输出。**为什么致命:** 概率机器会**过早宣布成功**,你拿到一个「自称修好」实则没修好的 PR。**怎么避:** 别让「完成」只由模型自己说了算;在循环外面套一道**确定性校验**(比如独立地再跑一遍全量测试、检查 exit code),用代码而不是用信任来判定「做完」。

6. **误以为 agent 在「自己执行」工具。** 新手常把 agent 想成一个能直接读写磁盘的黑箱,于是忘了工具是**你的代码在执行**,该加的权限、超时、错误处理一个没加。**为什么致命:** 一个能跑 shell 的循环,权限没兜住,可能 `rm -rf` 掉不该删的东西。**怎么避:** 牢记第 3.2——模型只「点菜」,执行永远是外面代码的责任,所以护栏也得加在那层代码上(权限与沙箱是第⑨篇的主题)。

## 6. 小结 & 下一步

这一篇你需要带走的:

- **一圈(single turn)** = 模型看全部历史 → 决定「说话还是动手」→ 动手就由外面的代码执行、把 **observation** 回填 → 再来一圈。模型只「点菜」,不「下厨」。
- **前提:没有 observation 就没有 agent。** 循环的全部力量来自「动作 → 真实结果 → 回填」这个反馈闭环;动作没有回音的 agent,是在自言自语。
- **ReAct(Yao 等 2022)** 是心跳节奏:**Thought → Action → Observation,循环**。每个想法都踩着上一个观察,把幻觉摁在地面上。
- **多步 = 历史的累积。** agent 的「记忆」就是那段越滚越长、每圈追加的历史文本;吐出 `tool_use` 就继续,只吐文字就停。
- **会停比会转更难。** 终止要「可判定 + 硬上限 + 失败出口」;错误回填给模型是宝贵反馈,但别让它无脑重试。
- **心态:agent 靠「转很多圈、每圈被现实纠正一点」取胜。把反馈闭环修好,就是把 agent 修好。**

我们已经知道循环靠「动作 → 观察」转起来。但那个让模型能真正「动手」、也让 observation 有质量的东西,到底长什么样、怎么设计才好用?下一篇,我们拆开 agent 的**手脚**——工具(tool),看看一个好工具和一个坑爹工具的差别,以及把工具标准化的 **MCP** 是怎么回事。
