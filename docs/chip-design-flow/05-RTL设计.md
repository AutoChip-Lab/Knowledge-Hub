# 芯片设计完整流程 · 第 ⑤ 步:RTL 设计

> 「芯片设计完整流程」学习系列第五篇。
> 全流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产
```

**RTL 设计夹在「微架构(④)」和「逻辑综合(⑦)」之间**:微架构画出了每个块的流水线图、状态机、缓冲深度、接口时序——但那还是**图和文档**。RTL 这一步,就是用硬件描述语言(Verilog/SystemVerilog/VHDL/Chisel)把这份蓝图**写成可仿真、可综合的代码**。这是整条流程里**第一次产出"真正能跑、能被工具变成电路"的东西**,也是后面功能验证(⑥)和综合(⑦)的直接输入。

> 一句话定位:**微架构是图纸,RTL 是按图纸写出的、机器能读懂并造成电路的"源代码"。**

> **🔧 关于贯穿例:从这一篇起,把"LLM 推理卡"落到真实公开案例 TPU v1**。前四篇(①–④)用的是一个**前瞻性的、假想的数据中心 LLM 推理卡**(HBM、FP8、PCIe Gen5)来讲"产品 / 规格 / 架构怎么定";从 ⑤ 起,为了让 RTL / 验证 / 后端 / 制造这些**实现环节**有真实可考的数字,我们把它**落到 Google 公开的 TPU v1**(Jouppi et al., ISCA 2017):28nm、700MHz、256×256 **INT8** 脉动阵列、24 MiB Unified Buffer、**DDR3**、≤331mm²。**注意两者口径不同**——TPU v1 是 2015 年的推理芯片,用 **INT8 + DDR3 + PCIe Gen3**,而前几章那张"前瞻 LLM 卡"用的是 **FP8 + HBM + PCIe Gen5**;它们是"同一类加速器、不同代际 / 不同取舍"的两个例子,**不是同一颗芯片**。后面看到精度 / 内存 / 接口代际对不上时,就是这个原因。

## 1. RTL 设计到底是什么

一句话:**用硬件描述语言(HDL),按"寄存器之间的数据如何在每个时钟沿流动"来描述硬件,产出一份既能仿真验证、又能被综合成门级电路的代码。**

**RTL = Register Transfer Level(寄存器传输级)。** 这个名字就说清了它的抽象层次:你不直接画门和连线(那是综合器的活),而是描述"哪些**寄存器**,在每个时钟沿,从哪些**组合逻辑**接收什么值"。比这更高是行为级(算法),比这更低是门级(与或非门)——RTL 正好卡在"人能写、工具能综合"的甜点上。

### 一个必须先建立的认知:RTL 不是软件代码

这是新手最大的坑。你写的 `a <= b + c;` **不是"先算 b+c 再赋给 a"这个执行步骤**,而是"综合出一个加法器,它的输出在每个时钟沿被锁进寄存器 a"——它描述的是**一块每时每刻都同时存在、并行工作的硬件**。

- 软件:语句**顺序**执行,一行做完做下一行。
- RTL:所有 `always` 块、所有赋值,**每个时钟沿同时发生**(并行)。

> 灵魂句:**你写 RTL,不是在写"做事的步骤",而是在写"每个时钟沿同时发生的硬件"。** 用写软件的脑子写 RTL,几乎必然出错。

### 和微架构(④)的边界
- 微架构:流水线几级、FSM 有哪些状态、FIFO 多深、接口什么协议——**蓝图,不可仿真**。
- RTL:把这些用 HDL **写成代码**,综合器能把它变成网表,仿真器能逐周期跑它——**可仿真、可综合**。

*在 TPU 上*:④ 说"PE 内部是 8-bit 乘 + 32-bit 累加、一拍一次";⑤ 就要把它写成 `always @(posedge clk) acc <= acc + a*b;` 这样的可综合代码,再把 256×256 个这样的 PE 用 `generate` 拼成阵列。

## 2. 谁来做 + 需要什么输入

**谁来做**:**RTL 设计工程师 / 逻辑设计工程师**(常常就是定微架构的那批人继续往下做),按块分工。**验证、综合、物理设计(PD)、DFT、低功耗团队继续并行介入**——RTL 的每个写法都直接决定他们好不好做。

**需要什么输入**(主要来自 ④ 的交付物):
- **微架构规格**:流水线图、FSM 状态转移图、接口时序图、缓冲深度、bank/端口方案、时序预算。
- **编码规范(coding guidelines)**:团队/公司的 HDL 风格约定(命名、复位风格、禁用构造)。
- **目标工艺库与工具**:综合用哪家库、仿真/lint/综合工具链,决定可综合子集与某些写法。
- **接口与寄存器定义**:对外协议、CSR 映射(常用工具从规格自动生成寄存器 RTL)。

> 关键心态:RTL 工程师要**一边写代码,一边在脑子里"看见"它综合出的电路**。写不出电路图像的人,会写出仿真过、但综合出 latch、跑不到频率、或验证爆炸的代码。

## 3. RTL 设计的核心内容(尽量穷尽地列)

按"写 RTL 要掌握/决定哪些事"组织,每项给:**要写什么 + 怎么写(经验法则/铁律) + 在 TPU 上长什么样**。

### 3.1 HDL 选择与可综合子集
- **要决定**:用哪种语言——**Verilog/SystemVerilog**(业界主流)、VHDL(欧洲/航天多)、**Chisel/SpinalHDL**(生成式,RISC-V 生态)、或 HLS(C→RTL)。
- **怎么写**:无论哪种,只能用**可综合子集**——综合器认识的那部分语法。`#延迟`、`initial`、`$display`、不可综合的循环只能用在 testbench,不能进设计。SystemVerilog 的 `always_ff/always_comb/always_latch` 能把"意图"写明确,让工具替你查错(强烈推荐)。
- *在 TPU 上*:阵列规整、要参数化复用 → 用支持 `generate` 的 Verilog/SV,或用 Chisel 这种生成式语言一键拉出 256×256。

### 3.2 组合逻辑 vs 时序逻辑(阻塞/非阻塞铁律)
- **要写**:两类电路——**组合逻辑**(输出只看当前输入,`always @(*)` / `always_comb`)和**时序逻辑**(带寄存器,`always @(posedge clk)` / `always_ff`)。
- **铁律(背下来)**:**时序逻辑用非阻塞赋值 `<=`,组合逻辑用阻塞赋值 `=`。** 反了就会出"仿真和综合不一致(simulation-synthesis mismatch)"或竞争冒险——这是 RTL 头号经典 bug。
- *在 TPU 上*:PE 的乘法是组合(`always_comb prod = a*b;`),累加寄存器是时序(`always_ff @(posedge clk) acc <= acc + prod;`)。

> 灵魂句:**`<=` 给寄存器,`=` 给组合;记混这一条,后面用几周 debug 偿还。**

### 3.3 寄存器、时钟与复位
- **要写**:触发器(flip-flop)、统一的时钟纪律、**复位策略**(同步 vs 异步复位、复位如何安全释放)。
- **怎么写**:全设计**复位风格一致**;复位要**覆盖所有需要确定初值的寄存器**(漏了 → 仿真出 X、硅上行为不定、bring-up 起不来);数据通路寄存器常可不复位以省面积,但控制寄存器必须复位。
- *在 TPU 上*:控制 FSM、计数器必须复位到确定态;阵列里成千上万的累加器可选择不复位(靠流程清零),省掉巨大的复位网络。

### 3.4 数据通路的 RTL(位宽/运算符/有符号)
- **要写**:用 `+ - * & | << >>` 等运算符描述运算,综合器会映射成加法器/乘法器/移位器;以及 mux、寄存器、拼接。
- **怎么写**:**位宽要精确算并写明**(位宽不匹配是 lint/bug 重灾区);**有符号/无符号**要声明清楚(`signed`),否则比较、移位、扩展会错;乘累加注意**累加位宽要够防溢出**。
- *在 TPU 上*:`logic signed [7:0] a,b; logic signed [31:0] acc;` —— INT8 乘进 INT32 累加,位宽必须算准防溢出。

### 3.5 状态机(FSM)的 RTL 编码
- **要写**:把 ④ 的状态转移图写成 RTL——状态寄存器 + 次态逻辑 + 输出逻辑。
- **怎么写**:推荐**两段式/三段式**写法(时序块只更新状态寄存器、组合块算次态与输出),清晰且不易出 latch;状态编码(二进制/one-hot)可由综合器选或手工定;**次态组合逻辑必须有 default**,否则推断出 latch。
- *在 TPU 上*:控制分块矩阵乘的 `Idle→LoadW→Stream→Accumulate→WriteBack` 就是一个这样的 FSM。

### 3.6 存储器的 RTL(推断 vs 例化)
- **要写**:片上存储有两种落法——**推断(infer)**:用特定 always 写法让综合器认出这是 RAM/regfile;**例化(instantiate)**:直接调用内存编译器(memory compiler)生成的 SRAM 宏(macro)。
- **怎么写**:大块 SRAM 几乎都用**例化宏**(密度/时序/功耗远好于推断);小寄存器堆可推断。注意读写端口、时序(读延迟几拍)、bank 划分要和微架构对齐。
- *在 TPU 上*:24 MiB Unified Buffer 用例化的多 bank SRAM 宏;小的控制寄存器堆可推断成触发器。

### 3.7 流水线、握手与 FIFO 的 RTL
- **要写**:流水线寄存器(逐级打拍)、**valid/ready 握手**、FIFO、背压(backpressure)。
- **怎么写**:握手要**两边都正确响应**(valid 不能等 ready、ready 不能组合依赖 valid 形成环);FIFO 的满/空、读写指针、深度要和微架构一致;背压要能**端到端传播**不死锁。
- *在 TPU 上*:Weight FIFO、DMA→Unified Buffer→阵列 之间的握手与流水寄存器,保证阵列不空转又不溢出。

### 3.8 跨时钟域(CDC)的 RTL
- **要写**:跨时钟域信号的**同步器**——单比特用**双触发器同步器**,多比特用**异步 FIFO 或握手**。
- **怎么写**:**绝不让裸信号直接跨域**(会亚稳态);多比特绝不能各 bit 单独同步(会采到不一致的中间值);CDC 要专门用 **CDC lint 工具**查。
- *在 TPU 上*:核心阵列域 ↔ PCIe 域 ↔ DDR3 域 之间的控制/状态信号都要正确同步——这是最容易"仿真过、流片后随机崩"的地方(见 §7)。

### 3.9 低功耗 RTL(门控与功耗意图)
- **要写**:**时钟门控(clock gating)**(整组寄存器不更新时停掉它们的时钟)、**操作数隔离**(不用的运算单元别让输入翻转)、与 **UPF/CPF 功耗意图**配合的结构。
- **怎么写**:**别手写门控时钟逻辑**(易出毛刺、坏 DFT/时序)——用工具自动插入门控单元(ICG),RTL 里写成"使能更新"的标准模式让工具识别。
- *在 TPU 上*:decode/小负载时空闲的 PE 列做时钟门控,"不干活就不耗动态功耗"——RTL 写成带 enable 的更新,综合时换成门控单元。

### 3.10 可测试性友好的 RTL(DFT)
- **要写**:让结构**可扫描(scan-friendly)**、为 SRAM 留 MBIST、避免破坏可测性的写法。
- **怎么写**:**别用裸的门控时钟、组合环、锁存器(latch)、内部生成的异步复位**——它们都会破坏扫描和时序;时钟/复位走规范结构;让所有触发器都能被串进扫描链。
- *在 TPU 上*:成千上万 PE 的触发器要能串进 scan 链、Unified Buffer 要能 MBIST —— RTL 阶段就得按 DFT 友好规则写。

### 3.11 参数化与模块复用
- **要写**:用 **`parameter`/`generate`/模块例化** 把设计写成**可配置、可复用**的;同一份 RTL 通过参数裁出不同规模。
- **怎么写**:把"会变的量"(位宽、深度、阵列边长、通道数)做成参数;重复结构用 `generate` 循环;这既减重复代码,也为**产品家族**(裁/扩衍生型号)留好旋钮。
- *在 TPU 上*:一个参数化的 PE 模块 + `generate` 双重循环 → 拉出整块 N×N 阵列;改 N 就能裁出不同算力的版本。

### 3.12 编码规范、Lint 与面向综合的写法
- **要写**:遵守**编码规范(coding standard)**,过 **Lint**,并按"综合得好、跑得到频率"的方式写。
- **怎么写**:**Lint 必须干净**(它专抓 latch、不完整敏感列表/case、位宽不匹配、多驱动、X 风险、CDC 问题);避免超长组合路径(会跑不到频率);写法要让综合器容易优化。**"仿真通过"只是起点,"Lint 干净 + 可综合 + 时序意图清晰"才算合格。**
- *在 TPU 上*:阵列规整、关键路径短(一个 MAC),只要 lint 干净、不写出长组合链,就能轻松收敛到 700 MHz。

## 4. 贯穿始终、必须反复权衡的东西

- **可读性 vs 对综合结果的掌控**:写得太行为级,综合结果难预测;写得太结构化,啰嗦难维护。要在能掌控电路的前提下尽量清晰。
- **抽象 vs 精确**:RTL 既要表达意图,又要对位宽、时序、复位等"硬"细节零含糊。
- **三处一致**:同一份 RTL 要在**仿真、综合、形式验证**里行为一致——尤其 X 语义、阻塞/非阻塞,差一点就 sim≠synth。
- **写得快 vs 写得对**:latch、X、CDC 是隐形杀手,省下的时间后面加倍偿还。
- **复用 vs 专用**:参数化复用省力,但过度通用会牺牲面积/时序。

> 灵魂句:**综合器照单全收——你写出 latch,它就给你 latch;你写出长路径,它就给你跑不到频率的电路。RTL 的每一行都会变成硅。**

---

## 5. RTL 是"怎么做"出来的(方法与工具)

- **仿真器**:VCS / Questa / Xcelium(商业)、**Verilator**(开源、快)——边写边跑 testbench 看波形。
- **Lint**:SpyGlass / Verilator lint —— 静态查 latch、X、位宽、CDC 等,**写完先过 lint**。
- **CDC 工具**:专门查跨时钟域同步是否正确(普通 lint/仿真抓不全)。
- **波形查看**:Verdi / GTKWave —— 逐周期看信号,RTL 调试主力。
- **断言(SVA)/ 形式**:写 SystemVerilog Assertion 表达意图,配形式工具早抓 bug。
- **试综合**:把关键块快速综合,看时序/面积是否如预期(承接 ④ 的时序预算)。
- **版本控制 + 代码评审**:RTL 是源代码,走 Git + review,和软件一样。

---

## 6. RTL 阶段的交付物(交给下游什么)

- **可综合 RTL 代码**:整套设计的 HDL,是 **⑦ 综合的直接输入**。
- **testbench / 验证环境骨架**:交给 **⑥ 功能验证**继续搭(也常由验证团队主导)。
- **Lint / CDC 清洁报告**:证明代码合规、可综合。
- **寄存器(CSR)文档与生成代码**:对软件可见的接口。
- **与微架构 spec 的对应关系**:每块 RTL 实现了哪页蓝图,便于追溯。

> RTL 是分水岭:它一旦冻结,验证(⑥)对着它测、综合(⑦)把它变电路——**RTL 里的 bug,要么验证抓住,要么变成硅里的缺陷。**

---

## 7. 容易被忽视、却致命的坑

1. **阻塞/非阻塞用反**:时序逻辑用了 `=`、组合用了 `<=` → **仿真-综合不一致**或竞争——RTL 头号经典 bug。
2. **意外推断出 latch**:组合 `always` 漏了 `else`/`default`,综合器给你塞个锁存器 → 时序/功耗/可测性全坏,还极难发现。
3. **复位不全**:有的寄存器没复位 → 仿真出 X 或假"正常",**芯片回来 bring-up 起不来**。
4. **X 态传播**:复位/初值没管好,X 在仿真里要么乐观掩盖 bug、要么悲观淹没真值;硅上是不确定。
5. **CDC 漏同步**:裸信号跨时钟域 → 亚稳态,**仿真过、流片后在客户现场随机崩**。
6. **不完整敏感列表(老 Verilog)**:`always @(a)` 漏了 `b` → 仿真假象,综合却按全敏感来,sim≠synth。用 `always_comb` 根治。
7. **手写门控时钟 / 组合环**:综合、时序、DFT 噩梦——门控交给工具,环坚决不写。
8. **只追"仿真通过"**:不过 lint/不管可综合性 → 到综合(⑦)才暴雷,返工最贵。
9. **位宽不匹配 / 有符号搞错**:静默截断或符号扩展错,结果对一半——lint 能抓,别忽略。

---

## 8. 小结 + 下一步

**RTL 设计 = 用 HDL 把微架构蓝图写成"每个时钟沿同时发生的硬件"——既能仿真验证、又能被综合成电路的可综合代码。** 它的灵魂是 **"RTL 不是软件,是硬件;你写的每一行都会变成硅"**;最该背下的铁律是 **"`<=` 给时序、`=` 给组合"**;最该敬畏的隐形杀手是 **latch、X、CDC**;最该养成的纪律是 **"仿真通过只是起点,Lint 干净 + 可综合 + CDC 安全才算合格"**。

下一步(第 ⑥ 步)是 **功能验证**:搭测试平台、用海量激励和覆盖率,**证明这份 RTL 的行为真的符合规格**——它通常是整条流程里工作量最大的一环,因为"造对芯片"的前提是"先证明设计是对的"。
