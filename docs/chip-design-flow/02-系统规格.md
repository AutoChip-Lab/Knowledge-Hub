# 芯片设计完整流程 · 第 ② 步:系统规格(System / Chip Specification)

> 「芯片设计完整流程」学习系列第二篇。
> 全流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产
```

**你现在在第 ② 步**——把 ① 的商业"目标"(Target Spec)翻成可执行、可验证的技术契约,再交给 ③ 翻成硬件骨架。系统规格夹在**"想要什么(商业)"和"怎么搭(工程结构)"之间**。上一步产出的 Target Spec 还是商业语言("要做一款省电的 LLM 推理卡,300W 内、Q4'27 上市");这一步把它**翻译成精确、无歧义、可测量的工程规格**:到底支持哪些精度、哪代接口、寄存器怎么定义、工作温区多少、每条指标怎么验收……

一句关键认知:**系统规格是整颗芯片的"合同"和"单一事实来源"。** 架构必须满足它,验证逐条对照它测,它定义了什么叫"做对了"。**规格里的歧义和漏洞,会原封不动变成硅片里的 bug。**

---

## 1. 这一步到底是什么

一句话:**把商业目标翻译成一份完整、无歧义、可验证的技术规格——精确说清"芯片必须做到什么、做到什么程度、在什么条件下、怎么算达标"。**

三层关系要分清:
- **PRD / Target Spec** = 市场/产品视角的"要什么、为什么"(商业语言)。
- **系统规格** = 工程视角的"芯片必须做到什么,精确且可测"(**WHAT**,但还**不谈内部怎么实现**)。
- **架构** = "用什么结构去满足规格"(**HOW**)。

核心铁律:**规格说 WHAT,不说 HOW。**
- ✅ 规格:"batch=32 时,单 token 解码延迟 < 10ms;支持 FP16/FP8;PCIe Gen5 x16。"
- ❌ 规格里不该写:"用一个 256×256 的脉动阵列"——那是架构的选择,写进规格就**过度约束**,堵死了更优解。

它同时是三件事的依据:**架构要满足它 · 验证要测它 · 它定义"done"**。

> 贯穿例子继续用那款**数据中心 LLM 推理加速卡**。

---

## 2. 谁来写

- **系统架构师 / 系统工程师**主笔(常有一个明确的 "spec owner")。
- **产品经理**把关:规格忠实反映 PRD。
- **设计、验证、软件、关键客户**都要参与评审——尤其**验证必须早期介入**,否则会混进"测不了"的需求。

---

## 3. 核心内容(尽量穷尽地列)

一份完整的芯片规格通常是**一套文档**(顶层规格 + 各接口规格 + 寄存器规格 + 子系统规格),内容覆盖:

### 3.1 功能规格(Functional spec)
- 完整、精确定义的**功能清单**;每个功能的行为、参数、边界。
- **工作模式 / 状态**:正常、低功耗、测试、调试、安全模式等,以及之间的转换。
- 必须支持的用例 / 场景(LLM 卡:prefill / decode、不同 batch、不同序列长度、不同精度)。

### 3.2 接口规格(Interface spec)
- 每个对外接口:**协议 + 版本**(如 PCIe Gen5 x16、DDR5-6400 / HBM3、UCIe……)。
- 带宽 / 延迟指标;协议合规性要求(要过哪些一致性测试)。
- 引脚/球级定义(逐步细化):pinout、信号定义、电平。
- 常单独成**接口控制文档(ICD)**,因为接口是和外部世界的硬约定。

### 3.3 性能规格(Performance spec)
- 量化的吞吐 / 延迟 / 频率目标,**并写明在什么条件下**(负载、batch、精度)。
- 性能 corner(典型 / 最差);QoS 保证。
- 对 LLM 卡:吞吐(tokens/s)、TTFT、TPOT、有效算力利用率,且要分场景给数。

### 3.4 功耗与热规格(Power & thermal spec)
- 各模式功耗(active / idle / sleep)、TDP、功耗状态与转换。
- 热限制(结温 Tj)、能效目标(perf/W)。
- 供电要求:电压轨、电流、容差。

### 3.5 物理 / 封装规格(Physical / package spec)
- 封装类型、尺寸、球数 / 引脚数、ball map。
- die 面积预算(初步)、**已确定的工艺节点**。
- 工作条件:电压范围、温区(商规 / 工规 / 车规)。

### 3.6 电气规格(Electrical spec)
- 供电电压与容差、各电源轨。
- IO 标准、信号完整性要求。
- 时钟:输入时钟、频率、抖动(jitter)要求。

### 3.7 软件 / 编程接口规格(Programming spec)
- **寄存器映射 / CSR 定义**(程序员模型)、**内存映射(memory map)**。
- 启动 / 初始化序列;编程模型、驱动需求;ISA(若有)。
- (这部分和架构有重叠——规格层面给"对软件可见的契约",内部实现归架构。)

### 3.8 可靠性 / 安全 / 质量规格(RAS / Safety / Quality)
- 可靠性目标(FIT 率、寿命)、ECC / 错误处理要求。
- 功能安全(汽车 ASIL 等级)、安全需求(secure boot、加解密、隔离)。
- 质量 / 测试 / 缺陷目标。

### 3.9 可测性 / 调试规格(Test / Debug spec)
- DFT 要求(scan、BIST 及**覆盖率目标**)。
- 调试 / 可观测接口(JTAG、trace);量产测试要求。

### 3.10 环境与合规规格(Environmental & Compliance)
- 工作 / 存储温湿度、ESD、latch-up、EMI/EMC。
- 认证(安规、法规)、各协议的合规要求。

### 3.11 验收标准(Acceptance criteria)——最容易被漏、却最关键
- **每一条需求都要写清"怎么验、怎么测、过/不过的判据"。**
- 这正是**验证团队和芯片回来后 bring-up 的对照标准**。一条无法测量的需求,等于没有需求。

---

## 4. 一份"好规格"的标准

- **完整(Complete)**:没有未定义的空白——空白会被工程师用假设填上,假设 = 后期 bug。
- **无歧义(Unambiguous)**:只有一种解读。
- **可验证(Verifiable / Testable)**:每条都能被测量。"应该够快"不是需求。
- **一致(Consistent)**:内部不自相矛盾。
- **可追溯(Traceable)**:向上链回 PRD,向下链到设计与测试。
- **可行(Feasible)**:工程已确认做得到(必须有可行性反馈在场)。
- **抽象层级正确**:说 WHAT 不说 HOW,别过度约束架构。

---

## 5. 怎么管理这份规格(工程实务)

- **需求管理**:每条需求**唯一编号、可追溯、可测试**;维护**追溯矩阵**(需求 → 设计 → 测试),保证"建了 + 测了"每一条。
- **层次化**:顶层芯片规格 → 子系统规格 → 块级规格(后者就接到微架构)。
- **版本与变更控制**:规格要冻结;冻结后的改动走正式变更流程(类似 ECO),因为越晚改越贵。
- **与架构协同迭代**:规格和架构不是先后,而是**来回对**——架构反馈可行性,规格据此调整现实性,直到都收敛。

---

## 6. 关键心态与张力

- **规格 = 单一事实来源**:验证字面对照它测,所以**规格 bug 直接变硅片 bug**。值得花时间抠到精确。
- **精确 vs 不过度约束**:既要精确到能当合同,又别细到替架构做决定(留出实现空间)。这条线就是 "WHAT not HOW"。
- **冻结时机**:适时冻结规格;冻太晚 → 架构/RTL 反复返工;冻太早 → 可能锁死错误假设。
- **验证共签**:规格必须有验证团队签字"这些我都能测",否则会混进无法验收的条款。

---

## 7. 交付物

- **芯片 / 系统规格文档**(通常一套:顶层 + 接口规格 + 寄存器规格 + 子系统规格)。
- **需求追溯矩阵**。
- **接口控制文档(ICD)**。
- **规格冻结 / 签核闸门(spec sign-off)**——过了才放心进架构/设计的全力投入。

---

## 8. 容易被忽视、却致命的点

1. **歧义 / 不完整**:工程师各自用假设补,集成时才爆雷。
2. **规格写了 HOW**:过度约束架构,堵死更优解。
3. **不可验证的需求**("反应要快")——验证没法测,等于没写。
4. **没有追溯**:说不清是否每条都建了、测了。
5. **规格漂移 / 无变更控制**:多个版本各执一词。
6. **验证没共签**:无法验收的条款溜进去。
7. **冻结太晚**:下游一直在动的地基上施工,反复返工。
8. **漏掉"非功能"需求**:功耗、温区、可靠性、可测性、合规——这些不写清,后期签核/认证才发现做不到。

---

## 9. 小结 + 下一步

**系统规格 = 把商业目标翻译成一份完整、无歧义、可验证的技术契约,精确说清芯片必须做到什么、什么条件下、怎么验收;它是单一事实来源,架构满足它、验证对照它测。** 灵魂是 **"说 WHAT 不说 HOW" + 每条都可测量、可追溯**。

下一步(第 ③ 步)就是已经写好的 **架构设计**(`03-架构设计.md`):把这份"芯片必须做到什么"的规格,翻译成"由哪些块组成、怎么连、能否在功耗/面积/成本内达标"的硬件骨架。至此 ①→②→③ 这条"需求 → 规格 → 架构"的上游链就接上了。
