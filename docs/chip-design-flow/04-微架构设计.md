# 芯片设计完整流程 · 第 ④ 步:微架构设计

> 「芯片设计完整流程」学习系列第四篇。
> 全流程:① 产品/市场需求 → ② 系统规格 → ③ 架构 → ④ 微架构 → ⑤ RTL → ⑥ 验证
> → ⑦ 综合 → ⑧ 物理设计 → ⑨ 签核 → ⑩ 流片 → ⑪ 制造 → ⑫ 封测 → ⑬ bring-up/量产。

---

## 0. 先看全景:芯片从想法到出片,大致这几步

```
① 产品/市场需求 → ② 系统规格(Spec) → ③ 架构设计 → ④ 微架构设计 → ⑤ RTL 设计
   → ⑥ 功能验证 → ⑦ 逻辑综合 → ⑧ 物理设计(布局规划/布局/时钟树/布线)
   → ⑨ 签核(STA/功耗/EM-IR/物理验证/等价性) → ⑩ 流片(tape-out)
   → ⑪ 制造(Fab) → ⑫ 封装测试 → ⑬ 芯片回来后 bring-up/量产
```

**微架构设计夹在「架构(③)」和「RTL(⑤)」之间**:架构把芯片切成了一堆块、给每块定了 PPA 配额和对外接口;微架构则**钻进每一个块内部**,精确到 cycle 地决定"它到底怎么实现"——流水线分几级、状态机怎么转、缓冲多深、关键路径能不能跑到目标频率。它仍然**不写可综合的 RTL**,但产出的是一份**精确到让 RTL 工程师可以照着敲代码**的实现蓝图。

> 一句话定位:**架构开支票(承诺 PPA),微架构兑现金(把承诺做成真实的时序与结构)。** 架构里超的预算可以回炉重定;到了微架构还兑不出来,RTL 基本救不回来。

---

## 1. 微架构设计到底是什么

一句话:**把架构定下的每一个块,从"是什么"细化到"内部每个时钟周期发生什么"——确定流水线、状态机、数据通路、缓冲、时序,使这个块在目标频率下真正达到架构给它的性能/功耗/面积配额。**

### 架构 vs 微架构:这一篇必须先钉死的边界

这是最常被混淆的一对概念。一句话区分:**架构是"对软件可见的契约",微架构是"对软件透明的实现"。**

| | **架构(Architecture)** | **微架构(Microarchitecture)** |
|---|---|---|
| 回答 | 这个块**是什么、对外什么行为** | 这个块**内部怎么实现** |
| 对软件 | 可见(换实现也不用改软件) | 透明(随便换,软件无感) |
| 例子 | ISA、寄存器/内存映射、块级接口、算法选择 | 流水线级数、FSM、FIFO 深度、关键路径、bank 划分 |
| 谁定的 | ③ 架构阶段 | ④ 微架构阶段(本篇) |
| 一对多 | 一个架构 | 可以有**无数种**微架构实现 |

**最经典的例子**:同一个 ISA(比如 ARMv9),可以有性能/功耗天差地别的多种微架构实现——有的是浅流水顺序执行(省电的小核),有的是深流水乱序超标量(高性能大核),软件却跑同一套指令、完全无感。**ISA 是契约,微架构是兑现方式。**

> 这也解释了为什么 Intel/AMD/Apple 用同一个 x86/ARM ISA 却能做出完全不同的芯片——**他们拼的是微架构,不是架构。** 微架构是真正"分胜负"的地方。

实务里 ③ 和 ④ 的边界是连续的:架构阶段往往已经下探到"块级微架构"(比如定了用脉动阵列、大概多大),微架构阶段则把这些块**逐个做实**。

*放到 LLM 推理卡上*:③ 说"用一个 256×256 的脉动阵列、FP8、weight-stationary 数据流、配 X TB/s HBM、Y MB 片上 SRAM"(② 章正是拿"256×256 阵列"当过"规格不该写 HOW"的反例——到了架构/微架构这层,才是把这个具体数字定下来的地方);④ 则要回答——**每个 PE 内部 MAC 怎么流水(几级)?权重怎么一行行压进阵列?这块 SRAM 分几个 bank、几个读口才能每周期喂满阵列?HBM-DMA → SRAM → 阵列 之间的 FIFO 各多深才不让阵列空转?控制整个 GEMM 分块循环的状态机长什么样?** 这些就是微架构。

## 2. 谁来做 + 需要什么输入

**谁来做**:**微架构师 / 资深设计工程师**(常常就是接下来要带队写 RTL 的那批人),按块分工(每个块有 owner)。架构师交接并答疑;**验证、DFT、综合/物理设计(PD)、低功耗团队都要早期介入**——因为微架构的每个选择都直接决定它们后面好不好做。

**需要什么输入**(主要来自 ③ 的交付物):
- **架构规格**:顶层框图、**每个块的 PPA 配额**(多少面积/功耗、要跑多高频率)、块的对外接口与协议、memory map。
- **目标频率 / 时序约束**:这个块要在多少 MHz/GHz 下工作。
- **工艺库特征**:典型门延迟、SRAM 时序/面积参数——用来估关键路径和面积(虽然还没正式综合)。
- **性能关键块的负载特征 / trace**:对吞吐敏感的块,需要真实访问模式来定缓冲深度、bank 数、流水策略。

> 关键心态:微架构师必须**同时戴三顶帽子**——性能(达不达得到吞吐)、时序(跑不跑得到频率)、可实现性(验证/综合/PD 扛不扛得住)。只盯性能,会做出验不完、综合后跑不到频率的"漂亮怪物"。

## 3. 微架构设计的核心内容(尽量穷尽地列)

按"钻进一个块内部要定哪些东西"组织,每项给:**要决策什么 + 怎么入手(经验法则) + 在 LLM 卡上长什么样**。

### 3.1 流水线设计(pipelining)——微架构的头号主题
- **要决策**:流水线**级数(深度)**、每一级放什么、怎么处理**流水线冒险(hazards)**。
- **怎么入手**:级数由"目标频率"反推——把组合逻辑切成若干级,使**最长一级的延迟 < 时钟周期**。深流水→高频但延迟大、冒险多、功耗大;浅流水→反之。**没有最优深度,只有甜点(sweet spot)。** 冒险三类:结构冒险(资源抢用)、数据冒险(后指令要前指令还没出的结果)、控制冒险(分支);对策是**前递/旁路(forwarding/bypass)、暂停(stall/interlock)、预测**。
- **关键纪律**:**各级要配平**——最慢那一级决定整条流水线的频率,一级太重就成了全块的瓶颈。
- *LLM 卡*:MAC 单元的乘法→累加做成流水;脉动阵列本身就是一个二维流水结构,数据像波一样穿过 PE。

> 灵魂句:**流水线是用"延迟换吞吐"——多切一级,单次更慢,但每周期都能吐一个结果。**

### 3.2 数据通路 vs 控制通路(datapath / control 拆分)
- **要决策**:把块拆成**数据通路**(做实际运算:ALU、乘加器、寄存器、mux、各种位宽)和**控制通路**(产生控制信号的状态机)。
- **怎么入手**:这是写 RTL 前的经典分法——先画数据通路(数据怎么流过哪些运算单元),再设计控制器去"指挥"它。RTL 基本就照这个结构落代码。
- *LLM 卡*:数据通路 = 阵列 + 累加器 + 量化单元;控制通路 = 驱动分块循环、决定何时载权重/何时算/何时写回的状态机。

### 3.3 有限状态机(FSM)设计
- **要决策**:控制逻辑用 FSM 描述——**Moore vs Mealy**、状态数、状态编码(二进制 / one-hot / 格雷码)、转移条件。
- **怎么入手**:凡是"按步骤序列做事"的控制器(协议引擎、DMA 控制、启动序列、调度)都是 FSM。状态多、追频率→倾向 one-hot(快但费触发器);状态少→二进制(省面积)。**画状态转移图是这一步的标准动作。**
- *LLM 卡*:控制 GEMM tiling 的"载入权重→流入激活→累加→写回→下一块"就是一个核心 FSM。

### 3.4 时序与频率预算(timing budget)
- **要决策**:估**关键路径(critical path)**、判断能否达到目标频率、给每段逻辑分配**逻辑级数(logic depth)预算**。
- **怎么入手**:虽然还没正式综合,但要对门延迟有量级概念——估"这条路径串了多少级逻辑",超了就用**插流水级、retiming、并行化、预计算**来缩短。**这一步常用"试综合(trial synthesis)"快速验证关键路径能不能到频率**(见 §5)。
- *LLM 卡*:累加链(长加法树)往往是关键路径——用 Wallace tree、进位保存、或多打一拍来压。

### 3.5 缓冲与流控(buffering & flow control)——"FIFO 到底多深"
- **要决策**:块间/块内 **FIFO/buffer 的深度**,以及**流控机制**。
- **怎么入手**:深度**不是拍脑袋,是算出来的**——用 **Little's Law、带宽×延迟积(BDP)、最大突发长度** 估:太浅→频繁背压、丢吞吐;太深→白白浪费面积/功耗。流控常用 **valid/ready 握手** 或 **credit-based(信用)**,并要保证**背压能端到端传播**而不死锁。
- *LLM 卡*:HBM-DMA → SRAM → 阵列 之间各级 FIFO,要深到能"盖住 HBM 访问延迟",让阵列在等数据时不空转(计算与搬运重叠)。

> 灵魂句:**FIFO 深度是算出来的,不是拍出来的;拍出来的深度,要么死锁,要么丢吞吐。**

### 3.6 存储器微架构(SRAM / 寄存器堆 内部)
- **要决策**:用**寄存器堆(register file)/ SRAM / 触发器阵列**哪种;**bank 划分、端口数(单口/双口/多口)、读写带宽、访问延迟**;若是 cache,还有 **tag/data array、组相联实现、替换逻辑、MSHR(未命中状态保持寄存器)、写缓冲**。
- **怎么入手**:小容量多端口→寄存器堆/flop;大容量→SRAM(但 SRAM 端口贵)。**带宽不够就切 bank**——多 bank 并行读写,把"每周期要喂的数据量"摊到多个 bank 上。端口数、bank 数由"峰值访问并发"反推。
- *LLM 卡*:③ 定了片上 SRAM 总量;④ 要决定它**分几个 bank、几个读口**,才能每个周期同时喂满阵列的一整行/列,否则阵列利用率上不去。

### 3.7 并行与资源权衡(replication vs sharing)
- **要决策**:某个运算**复制多份(replicate)/ 循环展开(unroll)** 换吞吐,还是**时分复用(time-multiplex / 资源共享)** 换面积。
- **怎么入手**:这是**块级 PPA 的总旋钮**——吞吐不够就复制/展开(费面积功耗);面积紧张、吞吐有余就共享一份硬件分时用(省面积、降吞吐、控制变复杂)。
- *LLM 卡*:阵列规模(PE 数)就是"复制"到极致;而像 softmax 这种低频算子,可能一份单元分时处理多行,省面积。

> 灵魂句:**并行换面积、复用换时间——块级 PPA 几乎都在这一根旋钮上。**

### 3.8 算术与数值实现(arithmetic)
- **要决策**:加法器/乘法器的**结构**(行波进位 / 超前进位 / Wallace tree……)、**定点 vs 浮点**、位宽、舍入、饱和;流水化的**乘累加(MAC)**;量化/反量化。
- **怎么入手**:按"速度/面积/功耗"选结构(要快→Wallace tree + 超前进位,省面积→行波);精度按 ③ 定的数值格式落地。
- *LLM 卡*:FP8/INT 的 MAC 单元设计,以及累加用更高精度(防溢出)再量化回低精度——是能效与精度的关键折中。

### 3.9 块接口与握手协议(interfaces)
- **要决策**:块对外用什么协议(标准 **AXI/自定义 valid-ready**)、是否做**对延迟不敏感(latency-insensitive)**设计、接口的时序契约。
- **怎么入手**:用规范的握手接口,块才能**独立验证、独立复用、独立改内部实现**而不影响邻居。这是"模块化"的工程基础。
- *LLM 卡*:阵列、DMA、SRAM 控制器之间用统一握手,任一块内部微架构改了,只要接口时序契约不变,别的块不用动。

### 3.10 跨时钟域与复位(CDC / reset,块级落地)
- **要决策**:块内/块间的 **CDC 同步器、异步 FIFO、握手同步**;块级**复位策略**(同步/异步复位、复位如何安全释放)。
- **怎么入手**:③ 划了时钟域,④ 要在每个跨域信号上**真正放对同步器**(单比特用双触发器同步,多比特用异步 FIFO 或握手),漏一个就是亚稳态隐患。
- *LLM 卡*:阵列高频域 ↔ HBM PHY 域 的边界必须正确 CDC,否则会出"仿真过、流片后随机崩"的鬼故事(见 §7)。

### 3.11 低功耗微架构(power hooks)
- **要决策**:**时钟门控的粒度**、操作数隔离(operand isolation,不用的运算单元不让输入翻转)、bank 级/块级电源门控、空闲关断。
- **怎么入手**:③ 定了功耗域和预算,④ 要把**门控真正插进结构里**——能关就关、不算就别让它翻转。
- *LLM 卡*:decode 阶段阵列经常等数据,要能**细粒度门控**让空闲 PE 不耗动态功耗;这是 perf/W 的实打实来源。

### 3.12 可测性与可调试钩子(DFT / debug)
- **要决策**:预留 **scan 友好结构、MBIST 接口(测 SRAM)、性能计数器、关键内部信号的可观测点**。
- **怎么入手**:DFT 不是后面才加——微架构期就要让结构"可扫描、可自测、可观测",否则后面插不进去(见 ③ 的同款提醒)。
- *LLM 卡*:大块 SRAM 留 MBIST;阵列利用率、stall 计数等做成性能计数器,供 bring-up 和调优观测。

### 3.13 微架构级性能建模与验证规划
- **要决策**:建一个**块级/cycle 级模型**确认"这套微架构真能达到 ③ 承诺的吞吐/延迟",并同步定义**这个块怎么验**(覆盖点、激励、checker)。
- **怎么入手**:在写 RTL 前用模型把性能跑出来——**别等 RTL 写完才发现微架构本身就达不到指标**。同时让验证团队基于微架构定测试计划。
- *LLM 卡*:用 cycle 模型确认"阵列利用率 ≥ 目标、decode 延迟达标",不达标就回到 §3.1/3.5/3.6 调流水、FIFO、bank。

## 4. 贯穿始终、必须反复权衡的东西

- **延迟 vs 吞吐**:深流水/大缓冲提吞吐但加延迟;这块是延迟敏感还是吞吐敏感,决定它整体的"形状"。
- **面积 vs 速度**:并行/展开/快算术单元提速但涨面积。
- **功耗 vs 性能**:高频、宽数据通路都吃功耗;门控/降并行省电但降性能。
- **流水线深度的甜点**:太浅频率上不去,太深冒险多、功耗大、收益递减。
- **复杂度 vs 可验证性**:乱序、复杂调度这类微架构性能高,但**验证复杂度爆炸**——很多 bug 就藏在这里(见 §7)。
- **兑现 vs 回炉**:微架构若兑不出架构预算,只有两条路——**回 ③ 改架构**,或认栽。**RTL 阶段几乎无力回天。**

> 这些旋钮同时在转;微架构师的功力,就是在"达标"的前提下,把面积/功耗/复杂度压到最低。

## 5. 微架构是"怎么做"出来的(方法与工具)

- **框图 / 流水线图 / 状态转移图 / 时序图**:微架构的主要表达方式,画清楚了 RTL 才好写。
- **块级 cycle-accurate 模型(C++/SystemC)**:验证性能、定缓冲深度与流水策略。
- **解析估算(电子表格)**:估关键路径逻辑级数、面积、功耗。
- **试综合(trial synthesis)**:把关键块/关键路径快速过一遍逻辑综合,看**能不能到目标频率、面积多大**——这是微架构特有、极其重要的"提前验时序"手段,避免到 ⑦ 才发现跑不到频率。
- **高层次综合(HLS)**:用 C/C++ 描述快速探索不同微架构(尤其数据通路密集的加速器),加速设计空间探索。
- **参考已有 IP / 上一代设计**:成熟微架构是最好的起点(和 ③ 一样,别从白纸开始)。

## 6. 微架构阶段的交付物(交给下游什么)

- **微架构规格文档(每个块一份)**,通常包含:
  - 块级框图、**流水线图**、**FSM 状态转移图**、**接口时序图**;
  - 关键寄存器/内部寄存器定义、**缓冲深度**、**bank/端口**方案;
  - **时序预算**(关键路径、逻辑级数、目标频率)、**面积/功耗估算**;
  - 数据通路位宽、数值格式、算术单元选型。
- **块级验证计划**(覆盖点、激励策略)。
- **性能模型与结论**(证明这套微架构达标)。

> 这份蓝图就是 **⑤ RTL 设计的直接输入**——RTL 工程师照着它把行为翻译成可综合的 Verilog/VHDL。

## 7. 容易被忽视、却致命的坑

1. **CDC 没规划好 → 亚稳态**:跨时钟域漏放/放错同步器,是**会过仿真、过流片、却在客户现场随机崩**的那种鬼 bug——微架构期最该认真对待的隐患。
2. **流水线冒险漏 case**:forwarding/stall 少考虑一种数据相关,功能就错;而且常常只在特定指令序列下暴露。
3. **FIFO 深度拍脑袋**:太浅丢吞吐,太深费面积;更糟的是流控没端到端考虑,直接**死锁(deadlock)**。
4. **复位策略不一致**:复位域/释放顺序没理清,芯片回来 **bring-up 起不来**,定位极痛苦。
5. **低估关键路径**:微架构以为能到频率,综合后(⑦)发现差一截,被迫回炉加流水级——**而改流水级 = 改时序行为 = 验证全部重来**,代价巨大。
6. **微架构过度复杂**:为一点性能上乱序/复杂调度,换来**验证验不完、面积功耗失控**——复杂度要配得上收益。
7. **没给 DFT/debug 留钩子**:后期插不进 scan/MBIST/观测点,掉良率、难调试。
8. **只建功能模型、不建性能模型**:等 RTL 写完才发现微架构本身达不到吞吐——这时候改,几个月没了。

---

## 8. 小结 + 下一步

**微架构设计 = 钻进架构定下的每一个块,把它的内部实现精确到 cycle 级——流水线、状态机、数据通路、缓冲、时序、bank,使这个块在目标频率下真正兑现架构给的 PPA 配额,并产出 RTL 可以照着写的实现蓝图。** 它的灵魂是 **"架构开支票、微架构兑现金"**;最常被混淆的边界是 **"架构是对软件可见的契约,微架构是对软件透明的实现——同一个 ISA 可以有无数种微架构"**;对新手最实用的纪律是 **"FIFO/流水深度是算出来的、关键路径要靠试综合提前验、性能模型要先于 RTL 跑出来"**。

下一步(第 ⑤ 步)是 **RTL 设计**:用 Verilog/VHDL 把这份微架构蓝图翻译成**可综合的寄存器传输级代码**——微架构图上的每一级流水、每一个状态、每一条数据通路,都将变成真正的硬件描述。
