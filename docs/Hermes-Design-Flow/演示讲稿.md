# Hermes Agent 演示讲稿

本文件只保存讲解顺序和现场提示；完整技术内容已经显示在 `index.html`。网页顶部的 **Hermes 源码基线** 是本次讲解对应的 commit，若标记 `dirty`，说明构建时还有未提交的 tracked 源码改动。

## 开场（1 分钟）

> Hermes 是个人 AI Agent 运行时。CLI、TUI、消息平台、Desktop、ACP 和 HTTP API 中的 LLM 驱动路径共享 `AIAgent`；工具、状态和第三方能力在核心外扩展。

立即补两个例外，避免把架构讲成绝对规则：

- Cron `no_agent` job 只运行脚本，不创建 Agent。
- `codex_app_server` 由 `AIAgent` 接入，但使用 Codex 原生 runtime 和工具循环。

## 架构主线（4 分钟）

按首页交互图从上到下讲：

1. **入口与交付**：处理平台事件、授权、UI 和投递。
2. **AIAgent 窄腰**：处理提示词、provider、工具、预算、中断和压缩。
3. **执行边缘**：直接工具、`execute_code`、`delegate_task`、MCP 和插件。
4. **状态与知识**：SessionDB 保存历史，memory 保存耐久事实，skills 保存流程。

讲到缓存时只强调一个不变量：系统提示构建后在会话内冻结。`stable/context/volatile` 是构建来源分段，不是每轮刷新；真正的 per-turn memory prefetch 和 plugin context 在 API 请求时附加。

## 执行闭环（3 分钟）

在第 ④ 章时指出三个可见事件：

- 用户输入先进入会话存储，再开始模型调用。
- 模型返回 tool calls 后，结果写回消息并继续循环；多组工具调用不是多次独立聊天。
- 消息、用量和退出状态增量落库；trajectory、memory sync 和 skill review 是条件性路径。

Fallback 是缓存稳定规则的显式例外：切换 provider 后，系统提示中的 Model/Provider 身份行也会同步更新。

## 状态与压缩（3 分钟）

用一句话区分三层：

> Memory 放少量长期事实，SessionDB 放完整历史，Skill 放可重复方法。

当前默认是原位压缩：旧消息 soft-archive 为 `compacted=1`，同一个 session id 写入新的 active view；恢复只读 active rows，`session_search` 仍能搜索 compacted rows。`parent_session_id` 轮换是保留的旧路径，不是默认行为。

阈值口径：配置基值是 50%，但小于 512K context 的模型实际至少 75%，Gateway 还有 85%/消息数安全网。不要只说“默认 50%”。

## 扩展与交付（4 分钟）

- **Plugin**：代码能力；通过公开 `PluginContext` / registry 注册。
- **Skill**：按需加载的流程知识。
- **Memory provider**：可选的外部长期状态增强。
- **Cron**：schedule + fresh session/script + delivery；默认跳过 memory，连续性用文件或 `context_from`。
- **Dashboard Chat**：浏览器里嵌入真正的 TUI PTY。
- **Desktop**：独立 React chat，通过 WebSocket/JSON-RPC 连接 headless `hermes serve`，不嵌入 Dashboard 或 TUI。

Curator 当前默认做确定性的闲置维护；LLM consolidation 默认关闭，不要把“自动合并和改进技能”说成每次都会发生。

## Hermes 的不足：需要补做的系统设计（5–6 分钟）

这一节只讨论平台缺口和待设计的系统能力，不把 Hermes 包装成已经完成的工业系统。

### 1. Durable workflow，而不是会话级委派

当前 `delegate_task` 的后台任务是进程内能力，Cron 也主要围绕文件 job 运转；Kanban 有 claim/heartbeat，但不是完整 workflow 语义。我们需要设计：

- 有版本的任务状态机、依赖关系、条件分支和人工 gate；
- 跨进程/跨主机的 durable queue、lease、heartbeat、reclaim 和恢复；
- 明确的 at-most-once / at-least-once / exactly-once 边界；
- 每一步的幂等键、重试策略、补偿动作和可重放记录。

演示时强调：**后台返回消息 ≠ 工作已经可靠完成**。

### 2. Artifact 与 provenance 模型

Hermes 的 `SessionDB` 主要保存对话，trajectory 也是可选的；任务 metadata 和 attachment 不能单独证明一个结果如何产生。我们需要设计不可变的 artifact manifest，记录：

- 输入/输出 artifact 的内容 digest 和父子关系；
- 源码、配置、工具、模型、容器、环境和参数版本；
- 命令、日志、退出状态、指标、审批和生成时间；
- artifact 的 schema、生命周期、保留、归档和删除策略。

摘要和自然语言 handoff 只能用于阅读，不能作为正式事实来源。

### 3. 可重复执行与严格配置快照

Hermes 支持 provider fallback、动态配置和多种执行后端，这对交互体验有帮助，但会削弱严格复现。我们需要设计：

- 一次运行的完整输入、配置和环境快照；
- 模型/provider、工具版本、随机性和外部依赖的 pinning；
- “允许 fallback”与“禁止 fallback”的明确策略；
- 从 manifest 重新执行并比较结果的 replay 机制。

### 4. 多维资源、租约与长任务控制

当前 Agent 预算主要是 API 调用/迭代次数，不能表达长任务的 CPU、内存、磁盘、并发、外部配额或专用资源。我们需要设计统一的资源账本、预留、配额、优先级、超时、抢占和跨节点释放语义，并把资源消耗写入运行记录。

### 5. Fail-closed 安全与变更控制

Hermes 的审批主要保护工具调用和危险命令；Local/SSH、同进程插件和可配置容器并不自动构成强隔离。我们需要设计：

- 身份、租户、项目、workspace、凭证和网络的最小权限边界；
- 插件/工具的版本、签名、加载和撤销策略；
- 关键变更的不可绕过 policy、多人审批、签名和 promotion 状态；
- 审计记录不可被 Agent、插件或 worker 自己删除或覆盖。

### 6. 类型化任务契约与独立验证

当前很多子 Agent 通过 summary 和自由格式 metadata 交接。我们需要为每类任务定义输入、输出、错误、证据和验收 schema，并在系统层验证：

- 输出是否满足 schema；
- 引用的文件、运行和 artifact 是否真实存在；
- 声称完成的检查是否确实执行过；
- 失败、部分成功和未知状态是否被区分。

Agent 的自报结果只能是候选事实，不能自动升级为系统状态。

### 7. 可观测性、评测与故障复盘

可选 trajectory 和日志不足以支撑长期质量管理。我们需要设计统一 event、trace、metric 和 audit 模型，能够重建一次运行的完整因果链，并建立固定的回归集、失败分类、成本统计、质量指标和人工复核闭环。

### 8. 状态库的规模与一致性边界

SQLite + FTS5 很适合单机 session search，但不等于多实例控制面，也不应承载大型二进制结果。我们需要明确元数据、事件、会话、缓存和大文件的存储边界，以及备份、迁移、一致性、灾备和跨版本兼容策略。

收束句：

> Hermes 已经把“Agent 如何思考、调用工具、保存会话和扩展能力”做成了可研究的实现；上面这些系统能力，才是从交互式 Agent 走向工业级运行平台时必须另外完成的设计。

## Live Demo（8–10 分钟）

### 1. 先用网页代码锚点讲实现

打开任意章节，先指向章节顶部的 **CODE ANCHORS · 可指着讲** 卡片。每张卡片都链接到本次构建记录的 Hermes commit，并定位到具体文件和行号。推荐顺序：

1. 第 ③ 章：工具自动发现与按会话权限 dispatch；
2. 第 ④ 章：预算、中断与工具回环；
3. 第 ⑥ 章：`leaf / orchestrator` 与工具继承；
4. 第 ⑧/⑨ 章：SessionDB、压缩和缓存；
5. 第 ⑪ 章：PluginContext、hook、技能和记忆。

讲解时不要只读文件名：说明每个锚点解决了什么问题、哪些状态在代码中被显式约束，以及它为什么值得借鉴。

### 2. 读取真实仓库

```bash
hermes --tui
```

```text
读取当前仓库的 AGENTS.md、run_agent.py、agent/conversation_loop.py 和 toolsets.py。
用一张表说明 Agent 核心、工具运行时和扩展面的责任，并给出文件证据。
```

讲解：工具结果进入同一 Agent turn，回答必须能够回指真实文件。

### 3. 后台并行审查

```text
并行审查当前改动：一个子代理检查正确性，一个检查测试缺口。
把仓库绝对路径、禁止修改文件和需要引用行号写进每个任务。
你自己同时检查 git diff；收到结果后只汇总可验证的问题。
```

讲解：顶层 `delegate_task` 立即返回，子代理在后台执行，完成结果随后以新消息回流。父 Agent 不应轮询等待，而应继续本地检查。子代理自报的文件或外部副作用仍要回读验证。

### 4. 恢复同一 TUI 会话

退出后执行：

```bash
hermes --tui -c
```

```text
基于刚才的审查，列出已验证事实和仍需运行的检查。不要重新猜测仓库状态。
```

讲解：状态来自 profile-scoped `state.db`，不是前端内存。

### 5. 提炼技能草案

```text
把刚才可重复的“并行审查 + 父代理复核”整理成技能草案。
先检查已有技能；只有没有重复项时才创建，并包含验证与常见误区。
```

讲解：Skill 是程序性知识。是否立即写入受 `skills.write_approval` 控制；当前会话的冻结提示也不会因此自动刷新。

## 常见问答

**为什么不把所有能力做成工具？**

每个核心工具 schema 都随每次 API 调用发送。优先扩展现有代码、CLI + skill、门控工具、插件或 MCP。

**审批能否保证安全？**

不能。审批是启发式保护；不可信输入需要权限、挂载、网络和凭证都受限的 sandbox、容器或 VM。普通同用户进程不是安全边界。

**Hermes 是否会自动学习每次对话？**

不会。会话自动持久化；memory、skills、Curator 和 trajectory 都有各自的显式或条件性写入路径。

## 收尾

> Hermes 的核心价值不是工具数量，而是把模型—工具闭环、会话状态、缓存约束和多种交付面放进一套可扩展运行时，同时保留清晰边界。
